# 設計決策 D02：效用 AI 而非行為樹

**日期**: 2026-06-01  
**狀態**: 已採納

---

## 決策

`action_decision` 任務使用**效用 AI（Utility AI）**評分選擇動作，而非行為樹（Behavior Tree）或 GOAP（Goal-Oriented Action Planning）。

---

## 選項對比

| 方案 | 原理 | 優點 | 缺點 |
|---|---|---|---|
| **效用 AI（採用）** | 每個動作有評分函數，選最高分 | 資料驅動、易擴充、自然處理多因子權衡 | 評分函數需人工設計；可能出現「邊界切換」現象 |
| 行為樹 | 條件節點樹，依優先級執行 | 直觀、易調試、行為可預測 | 修仙動作空間大（43 個），樹會非常深；新動作要重排樹結構 |
| GOAP | 根據目標倒推行動序列 | 理論上最接近 LLM 的規劃能力 | 實作複雜；每個動作需定義 preconditions + effects；修仙世界的 state space 太大 |
| 規則 if/else | 直接硬編碼規則 | 最簡單 | 不可擴展；新動作 = 新 if 分支；無法處理多因子權衡 |

---

## 理由

選效用 AI 的核心原因：

1. **問題形狀天然適合**：LLM 做的事是「從 N 個選項中選最合理的一個」，這正是效用 AI 的強項。行為樹更適合「有固定優先序的行為」（如格鬥動作鏈）。

2. **資料驅動**：`BASE_WEIGHT`、`PERSONA_EFFECT`、`GOAL_ACTION_ALIGNMENT` 都是 YAML/Python dict，可以在不改代碼的情況下調整 NPC 行為傾向。

3. **43 個動作管理**：行為樹的節點數量會爆炸；GOAP 的 state/action 建模工作量更大。效用函數可以用分類繼承（先分 `combat/cultivation/economy`，再細分）。

4. **Softmax 採樣保留多樣性**：直接 argmax 會讓所有 NPC 做完全相同的決策；Softmax + temperature 讓行為有可控的隨機性，還原 LLM 的「合理但不確定」特性。

---

## 注意事項

**邊界切換（Discontinuity）**：當兩個動作效用分數非常接近時，一點點狀態變化就可能讓決策翻轉。  
緩解：Softmax 採樣而非 argmax，讓接近分數的動作都有機率被選中。

**冷啟動**：第一批 `PERSONA_EFFECT` 權重完全靠設計者直覺，需要跑若干月模擬後觀察行為再調整。建議 Phase 1 完成後做一次「100 月行為觀察」，記錄每類角色的動作分布是否合理。
