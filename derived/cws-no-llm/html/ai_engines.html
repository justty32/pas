7 個引擎模組的設計原理、關鍵邏輯與實作細節

路由入口

dispatcher.py — 統一路由入口

src/local_ai/dispatcher.py
dispatch(task_name, infos) 查詢 handler 字典， 命中則呼叫對應引擎函數並返回 JSON dict； 未命中或引擎返回 None 則返回 None，由呼叫端 fallback 到 LLM。 所有引擎模組都在此集中 import，上層代碼只需知道 dispatcher。

路由表（Phase 0→4 累積）：

HANDLERS = {
  "action_decision":      gen_action_decision,
  "long_term_objective":  gen_long_term_objective,
  "relation_delta":       gen_relation_delta,
  "relation_resolver":    _stub_relation_resolver,
  "story_teller":         gen_story,
  "interaction_feedback": gen_interaction_feedback,
  "backstory":            gen_backstory,
  "nickname":             gen_nickname,
  "sect_decider":         gen_sect_decision,
  "sect_thinker":         gen_sect_thinking,
  "random_minor_event":   gen_minor_event,
  # None = fallback to LLM:
  "history_influence":    None,
  "single_choice":        _stub_single_choice,
  "custom_content_generation": None,
}

dispatch 邏輯：

def dispatch(task_name, infos):
    handler = HANDLERS.get(task_name)
    if handler is None:
        return None  # fallback
    try:
        return handler(infos)
    except Exception:
        return None  # 防守性 fallback
Phase 0 時所有 handler 都是 stub lambda。 Phase 1→4 逐步用真正引擎替換 stub。 dispatcher 本身不改，只換 HANDLERS 表中的函數。
Phase 1 — Utility AI

decision.py — Softmax 效用 AI（action_decision）

src/local_ai/decision.py
效用評分 Softmax 採樣 43+ 動作
對每個可用動作（來自 general_action_infos）計算效用分數， 以 Softmax + temperature 採樣選出動作，而非直接 argmax。 這還原了 LLM 的「合理但不確定」特性，避免所有 NPC 做完全相同決策。

效用函數結構：

utility(action) =
    base_weight(action_type)
  × persona_modifier(avatar.persona)
  × goal_modifier(avatar.long_term_goal)
  × realm_modifier(avatar.realm)
  × resource_modifier(avatar.stats)

# 硬性覆蓋：
# HP < 30%  → self_heal/rest 強制最高
# 仇敵在同區域 → flee/attack 上升
# 宗門任務中  → sect_mission × 2.0

輸出格式：

{avatar_name: {
    "action_name_params_pairs": [
        ["Respire", {}],
        # 或 ["Retreat", {"days": 30}]
        # 或 ["Plant", {"location_id": ...}]
    ],
    "avatar_thinking": "境界壓力高，選擇修煉",
    "short_term_objective": "提升靈力",
    "current_emotion": "專注",
}}
動作名稱必須 PascalCase，且必須從 general_action_infos 的可用動作中選取（Respire > Meditate > Retreat > Rest 降級順序）。
Phase 1 — 關係公式引擎

relations.py — 互動公式 + 性格矩陣（relation_delta）

src/local_ai/relations.py
公式引擎 性格相容矩陣 境界差修正
依互動類型決定 friendliness 變化的 base 範圍（-6 ~ +6）， 再疊加性格相容度、境界差、近期事件三個修正項。 最終裁剪到 config.yml 的 social.relation.llm_delta.min/max。

互動類型基礎範圍：

BASE_INTERACTION_DELTA = {
  "conversation":        (-2, +4),
  "gift":                (+2, +5),
  "spar":                (-3, +2),
  "gathering":           (-2, +4),
  "random_minor_event":  (-1, +2),
  ...
}

# 計算流程：
# 1. 從 base 範圍隨機一個值
# 2. +性格相容度 modifier
# 3. +境界差 modifier
# 4. +近期事件 modifier
# 5. 裁剪到 config 允許範圍

性格相容矩陣（部分）：

PERSONA_COMPATIBILITY = {
  ("正義", "邪惡"):   -2,
  ("謙遜", "傲慢"):   -1,
  ("俠義", "怯懦"):   +1,
  ("好學", "好學"):   +2,
  ("貪財", "俠義"):   -1,
  ...
}

# 返回格式：
{"delta_a_to_b": int,
 "delta_b_to_a": int}
Phase 1 — 目標模板生成器

goals.py — 情境感知模板池（long_term_objective）

src/local_ai/goals.py
情境分支 7 個模板池 Phase 4 Bug 修復
從 avatar_info 提取線索（has_sect / has_enemy / at_war / magic_stone_low）， 依優先級疊加模板池，最終隨機選一句。

7 個模板池：

_WAR_GOALS       → at_war == True
_REVENGE_GOALS   → has_enemy == True
_SECT_GOALS      → has_sect == True
_CULTIVATION_GOALS → has_sect == True（疊加）
_WEALTH_GOALS    → magic_stone_low == True
_WANDERER_GOALS  → not has_sect
_CULTIVATION_GOALS → 永遠疊加（最後補底）
_GENERIC_GOALS   → 所有條件都不中時

Phase 4 Bug 修復：magic_stone 解析

# 原版（崩潰）：
magic_stone = int(flat.get("magic stone", 0))
# 原因：欄位值為 "50靈石" 格式，無法 int()

# 修復後：
_ms_raw = str(flat.get("magic stone", 0) or 0)
_ms_match = re.match(r"\d+", _ms_raw)
magic_stone = int(_ms_match.group()) if _ms_match else 0
此 bug 自 Phase 1 起潛伏，直到 Phase 4 的 100 月基準測試才觸發， 因為 make_avatars() 初始化的角色此欄位格式不同。

Caves of Qud 風格 動作分類回應 五段身世模板

epithets.py — 外號生成（nickname）

src/local_ai/epithets.py
前綴+主題+後綴 境界感知 5% 稀有
外號由三部分拼接：境界感知前綴池 + 主題詞庫 + 後綴池。 5% 機率從稀有特殊外號池中選取（如「天妒英才」「劍癡」），不遵循拼接規則。 10 次連續生成驗證皆唯一（seed=42 測試）。 
# 一般外號：前綴 + 主題 + 後綴
# 例：「煉氣」（境界前綴）+「劍」（主題）+「客」（後綴）= 「煉氣劍客」

# 5% 稀有：直接從特殊外號池選取
# 例：「天劍尊者」「修羅魔頭」「半步仙人」

# 返回格式：
{"nickname": "煉氣劍客",
 "reason": "境界精進，以劍稱名",
 "thinking": ""}
Phase 3 — 宗門 AI

sect_ai.py — 五維度決策樹 + 年度思考模板

src/local_ai/sect_ai.py
決策樹 五個決策維度 狀態感知模板
sect_decider：依宗門狀態（成員數、資源、戰況、外交關係） 分別評估五個維度（外交 / 招募 / 驅逐 / 獎勵 / 扶持）， 各維度獨立輸出 ID 列表。這與原版 LLM 返回格式一致（六個 ID 列表 + thinking）。

決策樹（sect_decider）邏輯：

# 外交維度
if 有宗門 AND 戰力佔優 AND 關係極差:
    declare_war_target_ids = [最弱敵宗門]
if 正在戰爭 AND 實力劣勢:
    seek_peace_target_ids = [攻擊我的宗門]

# 招募維度
if 成員數 < 閾值:
    recruit_avatar_ids = [散修列表前 N]

# 驅逐維度
if 有叛徒/負關係成員:
    expel_avatar_ids = [...]

# 獎勵/扶持維度（依貢獻排序）

sect_thinker 模板（境界/財力/戰況感知）：

# 戰況感知：
"宗門正值戰時，上下戮力同心，
以實力奠定局面。"

# 人才匱乏：
"門內可造之材寥寥，廣納賢才
乃當務之急。"

# 資源充足：
"靈石充盈，正是擴張版圖的良機。"

# 返回：{"sect_thinking": str}  # 30-100 字
7 項單元測試全通過：招募/驅逐/求和邏輯正確， thinker 字數在 30–100 字範圍內。
Phase 4 — 隨機小事件詞庫

minor_events.py — 12 event_key 模板（random_minor_event）

src/local_ai/minor_events.py
12 個 event_key SOLO / PAIR 偵測 槽位填充
12 個 event_key 各有 3–4 句不同措辭的模板。 自動偵測 SOLO（avatar_info）vs PAIR（avatar_a_name）類型， 填入 {name} / {a} / {b} / {location} 槽位。 13 項單元斷言（6 SOLO + 6 PAIR + 1 fallback）全通過（seed=42）。

SOLO 事件（6 種）：

inner_mood_shift:
  "{name}打坐間忽然思緒萬千，沉寂
   片刻後重回定境。"

inner_obsession / daily_practice /
environment_response / sect_errand /
comic_incident

PAIR 事件（6 種）：

passing_interaction:
  "{a}與{b}在{location}擦肩而過，
   彼此頷首示意，未多言語。"

asymmetric_attention / subtle_goodwill /
social_friction / resource_competition /
small_mutual_help
