一、核心策略：Shim Layer（共 4 行改動）

在 src/utils/llm/client.py::call_llm_with_task_name() 加入本地 AI 前置查詢。 上層 20 個模擬 Phase 代碼完全不動。dispatcher 返回 None 時自動降級到 LLM。 

async def call_llm_with_task_name(task_name, template_path, infos, ...):
    from src.local_ai.dispatcher import dispatch   # +1
    local_result = dispatch(task_name, infos)       # +2
    if local_result is not None:                    # +3
        return local_result                         # +4

    # 原版 LLM 邏輯（降級）
    mode = get_task_mode(task_name)
    return await call_llm_with_template(...)

為什麼選 Shim Layer？

替代方案對比：

方案改動面結論
A. Shim（採用）1 個函數 4 行最小侵入
B. 替換各 Phase~10 個 Phase 檔案改動面廣
C. Mock HTTP 服務外部進程啟動複雜

詳見 D01_shim_layer.md

二、local_ai 模組架構

dispatcher.py

唯一路由入口。dispatch(task_name, infos) 查詢 handler 表，返回 JSON dict 或 None（fallback）。

decision.py

action_decision — Softmax 效用 AI。對每個可用動作評分，按溫度採樣，輸出 action_name_params_pairs。

relations.py

relation_delta — 互動類型 base 範圍 + 性格相容矩陣 + 境界差修正。輸出 delta_a_to_b / delta_b_to_a。

goals.py

long_term_objective — 7 個模板池（宗門/戰爭/靈石/散修/修煉）。依 avatar_info 線索選池，隨機抽句。

narrative.py

story_teller / interaction_feedback / backstory — 動作分類回應詞庫、五段身世模板，Caves of Qud 風格槽位填充。

epithets.py

nickname — 前綴 + 主題 + 後綴詞庫，境界感知前綴池，5% 稀有特殊外號，10 次生成皆唯一。

sect_ai.py

sect_decider / sect_thinker — 五維度決策樹（外交/招募/驅逐/獎勵/扶持），30–100 字思考模板。

minor_events.py

random_minor_event — 12 個 event_key（solo 6 + pair 6），各 3–4 句模板，SOLO/PAIR 自動偵測，location / name 槽位填充。

__init__.py

空白入口。所有引用透過 dispatcher.dispatch() 統一進入，不直接 import 各引擎。

三、不走 dispatcher 的特殊 Patch

sect_random_event.py — _generate_reason_fragment()

此 task 直接呼叫 call_llm_with_template()，繞過 call_llm_with_task_name()，dispatcher 無法攔截。

解法：直接 patch _generate_reason_fragment()，在函數開頭加本地 shim 判斷，返回固定 reason fragment 跳過 LLM 呼叫。

autonomous_custom_content_service.py — should_trigger()

custom_content_generation 由此 service 控制觸發。本地 AI 模式下無法生成高品質自訂內容。

解法：直接 patch should_trigger() 在本地模式下返回 False，完全停用此功能（不影響其他遊戲邏輯）。

五、關鍵設計決策

D01 — Shim Layer vs 替換各 Phase

結論：Shim Layer（採用）
只改 client.py 一個函數，影響面最小；保留 LLM fallback； dispatcher 是純函數，可獨立單元測試不需啟動伺服器。

風險：返回格式必須與消費端完全一致，任何 key 不符導致靜默錯誤。 緩解：Phase 0 用 debug log 記錄每個 task 的實際消費端格式，再逐一核對。

D02 — Utility AI vs 行為樹 vs GOAP

結論：效用 AI + Softmax 採樣（採用）
問題形狀天然適合：從 N 個選項中選最合理的一個。 行為樹適合固定優先序行為（格鬥），GOAP 的 state space 建模工作量太大。

Softmax 的作用：避免所有 NPC 做完全相同決策（argmax 問題）， temperature 控制隨機性，還原 LLM 的「合理但不確定」特性。
