# TODO：消耗端與產出端脫鉤（負成長料盡仍照產正成長）

> 記於 2026-06-10。待回家後實際修改。源碼核對於 2026-06-10。

## 問題摘要

在目前實作中，「用黃金產出食物」這類情境並沒有被綁成一條配方。
`ProductivitySnapshot.dailyRates`（`Source/ProductivitySnapshot.cs:10`，`Dictionary<ThingDef, float>`）
把每種資源各自記成一筆**獨立的每日有號淨流**：

- 正 rate（食物）→ 走 `base.Produce()` 投遞回主基地
- 負 rate（黃金）→ 每週期從哨站 `containedItems` 緩衝扣

兩者之間**沒有任何耦合判斷**。

## 缺口位置

`Source/Outpost_Sampled.cs:36-54` 的 `Produce()`，執行順序：

1. 先扣負成長（黃金）：`TakeItems(kv.Key, want)` —「取出最多 `want` 個，不足取到 0」
   （`Outpost_Sampled.cs:43-51`），取出即 `Destroy()`。
2. 再產正成長（食物）：`base.Produce()` **無條件執行**（`Outpost_Sampled.cs:53`），
   完全不檢查第 1 步是否真的扣到足量黃金。

## 後果

- 庫存有黃金 → 扣黃金、產食物。正常。
- 庫存黃金歸零 → `TakeItems` 回傳剩餘（可能 0 個），`base.Produce()` **照樣產食物**。
- 等於黃金耗盡後食物變「免費」無限產出 —— 非設計本意。

## 修法選項（待選）

1. **全有全無**：扣料前先檢查庫存是否 ≥ `want`，不足則本週期跳過對應的正成長產出。
   - 難點：負成長品項與正成長品項在 dailyRates 裡彼此獨立，沒有「哪個料對應哪個產物」的關聯資訊。
     若要全有全無，需先決定「料盡時擋掉哪些產出」的規則（全擋？還是只擋特定關聯？）。
2. **比例縮放**：按「實際扣到量 / 應扣量」等比例縮減該週期所有正成長產出。
   - 需在扣料時記錄達成比例 r = 實扣 / 應扣，再以 r 縮放 `base.Produce()` 的產量
     （可能要 override ResultOptions 動態調整，或自行投遞而不呼叫 base）。
3. **維持現狀**：若黃金只是象徵性維護費、扣完當補貼，現行行為可接受 → 不改。
