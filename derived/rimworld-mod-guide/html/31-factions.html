派系與外交關係

FactionDef 定義「世界上有哪些勢力」，FactionManager 管理它們的好感、戰爭與互動。本章聚焦如何新增派系、操控外交，以及派系如何驅動突襲與貿易。

Chapter 31 · Factions & Diplomacy
§1 概念分層：Def（模板）vs 執行期實例

派系系統與其他 RimWorld 子系統一樣，遵循「Def 是模板、執行期物件是實例」的二分。釐清這條界線是寫派系 mod 的第一課，否則你會把「該寫進 XML 的常數」與「該在 C# 操控的狀態」搞混。

FactionDef（模板）

由 XML 定義，全域唯一、唯讀。描述「世界上『可能』存在這樣一種勢力」：科技等級、組隊規則、起始好感範圍、圖示、配色頻譜。型別為 RimWorld.FactionDef，由 DefDatabase<FactionDef> 持有。

Faction（執行期實例）

世界生成時依 FactionDef 實體化出來的具體勢力，型別 RimWorld.Faction。它有名字、領袖、配色、以及對其他每個 Faction 的關係。存於存檔，由 Find.FactionManager 管理。一個 FactionDef 可被實體化成 0、1 或多個 Faction。

關鍵概念是：玩家殖民地本身也是一個 Faction。透過 Faction.OfPlayer 取得（其 FactionDef 為 PlayerColony，isPlayer=true）。外交頁上每一條「你 vs 某派系」的好感，其實就是玩家 Faction 與對方 Faction 之間的一條 FactionRelation。

派系的三種「壽命」分類

類型判定欄位 / API說明
永久派系 def.requiredCountAtGameStart > 0 世界生成時必定生成至少 N 個。例如海盜聯盟（Pirate）、機械蟲（Insect，隱藏）。即使所有據點被毀，Faction 物件仍存在於 FactionManager。
可被消滅派系 無 required 數量、非 permanentEnemy 失去所有 Settlement／成員後被標記 defeated，最終由 FactionManager 內部回收（Remove 為 private，外部不可直接呼叫）。常見於可被玩家征服的小型部落。
隱藏派系（hidden） def.hidden == true 不顯示在外交頁，玩家無法與之談判。用於機械蟲群、瘋狂/無主動物（AnimalsRoamer 等）、墜落機械。它們仍是合法的攻擊來源 Faction。
為何動物需要 Faction？RimWorld 的傷害／敵我判定建立在 Faction 關係上。瘋狂動物被歸進一個隱藏的「無主動物」派系並與玩家設為敵對，戰鬥系統才知道牠們會主動攻擊。這也是 hidden faction 存在的主要理由——提供敵我語意，但不汙染外交介面。

常用入口 API 速覽

API回傳用途

§2 FactionDef 關鍵欄位與完整範例

定義位於 RimWorld.FactionDef（繼承 Verse.Def）。欄位很多，下表挑出與 mod 製作最相關者。建議用 dnSpy 打開 FactionDef 對照 XML 名稱與型別。

欄位型別說明
isPlayerbool是否為玩家派系。自訂派系一律 false（玩家派系只有內建 PlayerColony / PlayerTribe）。
permanentEnemybool永遠敵對、不可談和。設 true 時起始好感強制為 -100、naturalEnemy 被蓋過，外交頁不顯示談判按鈕。
permanentEnemyToEveryoneExceptPlayerbool對所有人敵對，但仍可與玩家建立關係（用於「人人喊打但你能拉攏」的勢力）。
naturalEnemybool傾向與玩家敵對的「自然敵人」（海盜系常設）。設 true 時起始好感為 -80（敵對開局），影響某些事件選取。
categoryTagstring分類標籤（如 Pirate、Outlander、Tribe），決定某些事件/任務的派系篩選。
pawnGroupMakersList<PawnGroupMaker>核心欄位：定義此派系如何組隊（突襲、貿易商、訪客）。詳見下方。
basicMemberKindPawnKindDef派系基本成員的 PawnKind 模板（銜接第 11 章）。
apparelStuffFilterThingFilter限制此派系成員可穿戴的材質（如部落只用皮革/木材）。
techLevelTechLevel科技等級（Neolithic / Medieval / Industrial / Spacer 等），影響裝備、貿易品、AI。
naturalEnemy / permanentEnemybool起始好感不是 XML 欄位，而由這兩個旗標推導：permanentEnemy → -100、naturalEnemy → -80、皆無 → 0（中立）。見 Faction.TryMakeInitialRelationsWith。
canSiege / canStageAttacksbool是否能發動圍攻 / 能否作為主動攻擊來源。隱藏派系常 false。
settlementGenerationWeightfloat世界生成時撒據點的權重。0 代表不在世界地圖上有據點。
requiredCountAtGameStartint新世界至少生成幾個此派系的 Faction 實例。
maxCountAtGameStartint上限。required 與 max 之間取隨機。
factionIconPathstring外交頁與世界地圖圖示（Textures 路徑，銜接第 8 章）。
colorSpectrumList<Color>派系配色頻譜；實例化時從此漸層挑一色，避免兩派系撞色。

PawnGroupMaker：突襲與來訪組成的引擎

RimWorld.PawnGroupMaker 是「給定一筆點數預算（points），產出一組 Pawn」的規則。每個 kindDef 帶權重 selectionWeight 與成本（取自 PawnKindDef.combatPower）。突襲系統會反覆抽 kindDef、直到耗盡 points 預算。每個 PawnGroupMaker 的 kindDef（型別 PawnGroupKindDef）區分這套規則用於哪種場合：

Combat

突襲、圍攻的戰鬥隊伍。被 IncidentWorker_RaidEnemy 取用。

Trader

貿易商隊組成（商人＋護衛＋馱獸）。被商隊到訪事件取用。

Peaceful / Settlement

友好訪客、據點駐守人口。決定玩家攻打據點時遇到誰。

完整範例：自訂中立貿易派系

下面定義一個工業級、預設中立、會派貿易商隊、可被拉攏為盟友的「自由商會」。

<!-- Defs/FactionDefs/Factions_FreeGuild.xml -->
<Defs>
  <FactionDef ParentName="FactionBase">
    <defName>MyMod_FreeGuild</defName>
    <label>自由商會</label>
    <description>一群以貿易維生的中立商人聯盟，沒有固定意識形態，只認利潤。</description>
    <categoryTag>Outlander</categoryTag>
    <pawnsPlural>商會成員</pawnsPlural>
    <listOrderPriority>40</listOrderPriority>

    <!-- 起始中立、可被拉攏：沒有 startingGoodwill 欄位，靠下列旗標 -->
    <!-- naturalEnemy=false + permanentEnemy=false ⇒ 起始好感 0（中立） -->
    <mustStartOneEnemy>false</mustStartOneEnemy>
    <naturalEnemy>false</naturalEnemy>
    <permanentEnemy>false</permanentEnemy>

    <techLevel>Industrial</techLevel>
    <backstoryFilters>
      <li>
        <categories>
          <li>Offworld</li>
        </categories>
      </li>
    </backstoryFilters>

    <!-- 世界地圖會撒據點 -->
    <requiredCountAtGameStart>1</requiredCountAtGameStart>
    <maxCountAtGameStart>2</maxCountAtGameStart>
    <settlementGenerationWeight>1.0</settlementGenerationWeight>
    <canSiege>false</canSiege>
    <canStageAttacks>true</canStageAttacks>
    <allowedArrivalTemperatureRange>-40~45</allowedArrivalTemperatureRange>

    <!-- 圖示與配色 -->
    <factionIconPath>World/WorldObjects/Expanding/MyMod_FreeGuild</factionIconPath>
    <colorSpectrum>
      <li>(0.85, 0.70, 0.20)</li>
      <li>(0.90, 0.80, 0.35)</li>
    </colorSpectrum>

    <!-- 限制成員裝備材質：工業派系允許金屬與布料 -->
    <apparelStuffFilter>
      <categories>
        <li>Leathers</li>
        <li>Fabric</li>
      </categories>
    </apparelStuffFilter>

    <!-- 組隊規則 -->
    <pawnGroupMakers>
      <!-- 貿易商隊 -->
      <li>
        <kindDef>Trader</kindDef>
        <traders>
          <Town_Trader>10</Town_Trader>
        </traders>
        <carriers>
          <Muffalo>10</Muffalo>
          <Dromedary>10</Dromedary>
        </carriers>
        <guards>
          <Town_Guard>10</Town_Guard>
        </guards>
      </li>
      <!-- 戰鬥隊（被拉到敵對時的突襲組成） -->
      <li>
        <kindDef>Combat</kindDef>
        <options>
          <Town_Guard>10</Town_Guard>
          <Mercenary_Gunner>6</Mercenary_Gunner>
          <Mercenary_Sniper>3</Mercenary_Sniper>
        </options>
      </li>
      <!-- 友好訪客 -->
      <li>
        <kindDef>Peaceful</kindDef>
        <options>
          <Town_Pawn>10</Town_Pawn>
          <Town_Guard>5</Town_Guard>
        </options>
      </li>
    </pawnGroupMakers>
  </FactionDef>
</Defs>
善用 ParentName="FactionBase"。核心 FactionBase 抽象 def 已填好一堆預設（領袖頭銜、relativeName 等）。繼承它能省掉大量樣板，只覆寫你關心的欄位。可在 Core/Defs/FactionDefs/Factions_Misc.xml 找到 FactionBase 與各內建派系作模板。
§3 好感度與關係：FactionRelation 與門檻機制

兩個 Faction 之間的關係由一個 RimWorld.FactionRelation 物件描述，掛在每個 Faction 內部的關係表上。它有兩個核心欄位：

  • baseGoodwill（int，範圍 -100~100）：好感數值。
  • kind（FactionRelationKind 列舉）：Hostile / Neutral / Ally。它是由 goodwill 推導出來的「離散狀態」。

好感 → 關係種類的門檻

goodwill 區間FactionRelationKind意義
<= -75Hostile敵對，會互相突襲。
-74 ~ +74Neutral中立，可貿易、可談判。
>= +75Ally同盟，可請求軍援、共同作戰。
門檻常數位於 GoodwillSituation / Faction 內。實作上由 FactionRelation.CheckKindThresholds 在 goodwill 變動後重算 kind，跨過 ±75 會觸發狀態切換並（依旗標）發出通知信。請務必透過官方 API 改 goodwill，別直接寫 baseGoodwill，否則 kind 不會自動更新、外交頁與戰鬥判定會錯亂。

讀取關係

using RimWorld;
using Verse;

Faction player = Faction.OfPlayer;
Faction guild  = Find.FactionManager
    .FirstFactionOfDef(FactionDefOf.MyMod_FreeGuild);

// 取得關係物件
FactionRelation rel = player.RelationWith(guild);
Log.Message($"goodwill={rel.baseGoodwill}, kind={rel.kind}");

// 便捷屬性：直接問「玩家對某派系」的關係種類
if (guild.HostileTo(player)) { /* 敵對 */ }
FactionRelationKind k = guild.PlayerRelationKind;   // Hostile/Neutral/Ally
int gw = guild.PlayerGoodwill;                       // -100~100

改變好感：永遠用 TryAffectGoodwillWith

Faction.TryAffectGoodwillWith(other, goodwillChange, canSendMessage, canSendHostilityLetter, reason, lookTarget) 是唯一正確的入口。它會：套用增量、夾在 -100~100、重算 kind、處理跨門檻的關係切換、（可選）發信通知、寫入歷史記錄供外交頁顯示「為何變動」。

// 賄賂讓商會更喜歡你（+15），並在外交頁標註原因
bool changed = guild.TryAffectGoodwillWith(
    Faction.OfPlayer,
    goodwillChange: 15,
    canSendMessage: true,
    canSendHostilityLetter: true,
    reason: HistoryEventDefOf.UsedHarmfulAbility,  // 例；正向事件用對應 def
    lookTarget: null);

// 跨過 +75 時，changed 內部已自動切成 Ally 並發通知信
reason 參數在 1.5 是 HistoryEventDef。舊版（1.3 以前）是 string，網路上不少教學過時。1.4+ 之後外交歷史走 RimWorld.HistoryEventDef 系統（Find.HistoryEventsManager），goodwill 變動會記成可追溯的歷史事件，顯示在派系資訊面板的「最近事件」清單。傳 null 也合法，只是不留原因。

設定為「絕對狀態」

使用 goodwill 的派系（自訂派系皆是）沒有 TrySetRelationKind 這個方法——關係種類永遠由 goodwill 推導，所以「硬切換」其實是把 goodwill 一次拉過門檻。例如要立即翻臉，可用 GoodwillToMakeHostile 算出所需負增量再交給 TryAffectGoodwillWith：

// 直接設為敵對：算出把 goodwill 拉到 -75 以下所需的增量
Faction player = Faction.OfPlayer;
int delta = player.GoodwillToMakeHostile(guild);   // 例：-75 - 現有goodwill
player.TryAffectGoodwillWith(
    guild, delta,
    canSendMessage: true,
    canSendHostilityLetter: true,
    reason: null);

// 註：SetRelationDirect 只適用「不使用 goodwill」的派系，
// 對 goodwill 派系呼叫它只會 Log.Error 並被忽略。
方法適用情境
TryAffectGoodwillWith(±delta)漸進式調整（任務獎懲、賄賂、誤傷）。最常用，也是「硬切換」的實作途徑（搭配 GoodwillToMakeHostile 等）。
SetRelationDirect(other, kind)僅限「不使用 goodwill」的派系直接設定 kind；對 goodwill 派系會被拒絕（Log.Error）。
SetRelation(FactionRelation)底層；初始化／特殊情形才用，直接塞入一個關係物件。
§4 派系生成與世界：撒據點與舊存檔後補

世界生成階段，RimWorld.FactionGenerator 依各 FactionDef 的 requiredCountAtGameStart / maxCountAtGameStart 實體化出 Faction（呼叫 FactionGenerator.NewGeneratedFaction，入口為 FactionGenerator.GenerateFactionsIntoWorldLayer），再由 FactionManager.Add 註冊。接著世界地圖生成器依 settlementGenerationWeight 為這些派系撒 RimWorld.Planet.Settlement（一種 WorldObject，銜接第 13 章）。

生成流程

  1. FactionGenerator.GenerateFactionsIntoWorldLayer(layer, factions)：遍歷所有非隱藏、required>0 的 FactionDef，產生對應數量的 Faction 實例。
  2. 對每個新 Faction，呼叫 TryMakeInitialRelationsWith，依 permanentEnemy/naturalEnemy 推導與玩家及彼此的起始關係。
  3. 世界生成把據點撒在地圖上（依 settlementGenerationWeight 與 tile 適居性）。
  4. 隱藏派系（機械蟲、無主動物）由 FactionGenerator 以固定邏輯補上，不吃 settlementGenerationWeight。
致命陷阱：加派系後讀舊存檔會缺實例。GenerateFactionsIntoWorldLayer 只在「新世界」跑一次。玩家把你的 mod 加進既有存檔時，世界早就生成完畢，你的 FactionDef 不會有任何 Faction 實例——任何 FirstFactionOfDef 都回傳 null，依賴它的事件全部失效。

解法：載入既有存檔時後補生成

標準作法是掛在 FactionManager 載入後的時機，檢查每個「應存在」的 FactionDef 是否有實例，缺了就補。可用 Harmony postfix（銜接第 3 章）勾 Game.FinalizeInit 或 FactionManager.ExposeData 之後：

using HarmonyLib;
using RimWorld;
using RimWorld.Planet;
using Verse;

[HarmonyPatch(typeof(Game), nameof(Game.FinalizeInit))]
public static class Patch_EnsureMyFactions
{
    public static void Postfix()
    {
        EnsureFactionExists(FactionDefOf.MyMod_FreeGuild);
    }

    private static void EnsureFactionExists(FactionDef def)
    {
        // 已有實例就跳過
        if (Find.FactionManager.FirstFactionOfDef(def) != null)
            return;

        // 後補生成一個 Faction
        Faction faction = FactionGenerator.NewGeneratedFaction(
            new FactionGeneratorParms(def));
        faction.ideos?.SetPrimary(faction.ideos.PrimaryIdeo); // 若啟用 Ideology
        Find.FactionManager.Add(faction);

        // 為它撒一個世界據點，讓它「真的存在於星球上」
        TryGenerateSettlementFor(faction);
        Log.Message($"[MyMod] 後補生成派系 {def.defName}");
    }

    private static void TryGenerateSettlementFor(Faction faction)
    {
        if (!TileFinder.TryFindNewSiteTile(out PlanetTile tile))
            return;

        Settlement s = (Settlement)WorldObjectMaker
            .MakeWorldObject(WorldObjectDefOf.Settlement);
        s.SetFaction(faction);
        s.Tile = tile;
        s.Name = SettlementNameGenerator
            .GenerateSettlementName(s);
        Find.WorldObjects.Add(s);
    }
}
更輕量的替代：不要硬撒據點。如果你的派系只需要「能發突襲/能來貿易」而不要求世界地圖上有圖示，可以只 FactionManager.Add 一個無據點的 Faction（settlementGenerationWeight 設 0）。少了 TileFinder 的失敗風險，相容性更好。是否需要實體據點，取決於玩法是否允許玩家攻打它。
§5 派系驅動的玩法：突襲、貿易、外交事件

派系不是裝飾品，它是 RimWorld 大量事件的「來源資料」。三條主要管線都從 Faction 取資料：

突襲

IncidentWorker_RaidEnemy 從敵對 faction 的 Combat pawnGroupMaker 依 points 組隊，再交給 Lord（第 30 章）執行 AI。

貿易

商隊到訪事件依 TraderKindDef 與 Trader pawnGroupMaker 產生會走進殖民地的商人。

外交事件

和平談判、賄賂、共同敵人結盟等，最終都呼叫 TryAffectGoodwillWith 改關係。

突襲是怎麼選 faction 的

當敘事引擎（第 12 章）決定要放一場 ThreatBig 突襲，IncidentWorker_RaidEnemy.TryExecuteWorker 大致流程：

  1. 呼叫 parms.faction；若沒指定，用 Find.FactionManager.RandomRaidableEnemyFaction() 挑一個敵對且可突襲的派系。
  2. 用 parms.points（威脅點數，隨財富縮放）呼叫 PawnGroupMakerUtility.GeneratePawns，從該 faction 的 Combat groupMaker 組隊。
  3. 用 RaidStrategyDef（如 ImmediateAttack、Siege）與 PawnsArrivalModeDef（如 EdgeWalkIn、EdgeDrop）決定戰術與進場方式。
  4. 把生成的 pawns 交給對應的 Lord + LordJob（如 LordJob_AssaultColony）接管群體 AI。
所以「自訂突襲組成」其實是在編 pawnGroupMaker。你不需要重寫 IncidentWorker。只要在 FactionDef 的 Combat groupMaker 裡放對 PawnKindDef 與權重，這個派系突襲時自然就用那批兵。要更複雜的編隊（固定帶一個 boss）才需要自訂 PawnGroupKindWorker 或 patch。

讓自訂派系派出貿易商隊

商隊到訪走 IncidentDefOf.TraderCaravanArrival（category FactionArrival）。它會挑一個有 Trader groupMaker 且非敵對的派系，並選一個 TraderKindDef 決定賣什麼。把 TraderKindDef 綁到你的派系：

<!-- Defs/TraderKindDefs/TraderKinds_FreeGuild.xml -->
<Defs>
  <TraderKindDef>
    <defName>Caravan_FreeGuild_General</defName>
    <label>自由商會綜合商隊</label>
    <!-- 綁定到本派系：只有此 FactionDef 的商隊會用 -->
    <faction>MyMod_FreeGuild</faction>
    <stockGenerators>
      <li Class="StockGenerator_Category">
        <categoryDef>FoodMeals</categoryDef>
      </li>
      <li Class="StockGenerator_Tag">
        <tradeTag>WeaponRanged</tradeTag>
        <thingDefCountRange>3~6</thingDefCountRange>
      </li>
      <li Class="StockGenerator_Slaves">
        <countRange>0~0</countRange>
      </li>
    </stockGenerators>
  </TraderKindDef>
</Defs>

另外確保 FactionDef 上有對應的商隊/軌道商人引用欄位（caravanTraderKinds、visitorTraderKinds、baseTraderKinds）。若不指定，系統會回退到 categoryTag 預設的商隊種類。

外交事件：和平談判與共同敵人

玩家透過通訊台請求和平、或任務系統觸發外交轉折時，本質都是改 goodwill。下面示範一個「停火協議」用的小工具方法：

public static class DiplomacyUtil
{
    // 賄賂談和：花 silver 換 goodwill，跨過門檻自動轉中立
    public static bool TryBribeForPeace(Faction target, int silver)
    {
        if (target == null || target.IsPlayer) return false;
        if (target.def.permanentEnemy) return false; // 永久敵不可談和

        // 每 100 銀 +5 好感，但有上限避免一次買到同盟
        int delta = Mathf.Min(silver / 100 * 5, 30);
        return target.TryAffectGoodwillWith(
            Faction.OfPlayer, delta,
            canSendMessage: true,
            canSendHostilityLetter: true,
            reason: null);
    }
}
§6 玩家對派系互動：通訊台、任務、皇室

玩家主動與派系互動的入口集中在通訊台（comms console）。它由 Building_CommsConsole 提供，互動選項是一組 RimWorld.ICommunicable，每個 visible faction 都會生出一條 FactionDialogMaker 產生的對話樹。

通訊台能做什麼

互動條件底層
呼叫貿易船有軌道貿易商在範圍TradeShip / PassingShipManager
請求商隊到訪非敵對且 def.canRequestTradersFactionDialogMaker.RequestTraderOption
請求軍援Ally 且有冷卻額度FactionDialogMaker.RequestMilitaryAidOption
賄賂提升好感非永久敵扣 silver → TryAffectGoodwillWith
自訂派系自動就會出現在通訊台。只要不是 hidden、不是 permanentEnemy（無談判價值），FactionDialogMaker 會自動為它生對話選項——你通常不需要寫任何通訊台程式碼。要新增「自訂選項」才需要 patch FactionDialogMaker.FactionDialogFor。

透過任務給予/扣除 goodwill（銜接第 22 章）

任務系統（Quest）是改派系關係最乾淨的方式，因為它把「獎懲」資料化在 QuestScriptDef 內，無需 Harmony。對應的 QuestNode 是 QuestNode_GiveRewards 與專門的 QuestPart_FactionGoodwillChange：

<!-- QuestScriptDef 片段：任務成功時 +12 好感 -->
<QuestNode Class="QuestNode_End">
  <!-- 先省略 ... 在 success 分支中 -->
</QuestNode>

<!-- 用 QuestNode 改好感（內部建立 QuestPart_FactionGoodwillChange） -->
<li Class="QuestNode_ChangeFactionGoodwill">
  <faction>$askerFaction</faction>
  <change>12</change>
  <canSendLetter>false</canSendLetter>
</li>

在 C# 端，任務獎勵 RimWorld.Reward_Goodwill 也是直接走 TryAffectGoodwillWith。所以無論 XML 或 C#，最終都收斂到同一個 API——這也是為什麼建議你所有改關係的程式碼都統一走它。

皇室（Empire faction，銜接第 33 章）

Royalty DLC 引入的帝國是一個特殊 Faction（FactionDef Empire，由 Faction.OfEmpire 取得）。它的「好感」之外還疊了一層榮譽（royal favor）與頭銜（RoyalTitle）系統，由 Pawn_RoyaltyTracker 管理。重點分工：

  • 對帝國的好感仍是標準 FactionRelation，影響突襲/貿易。
  • 榮譽點數是另一條軸，用來升頭銜、換 permits、psycasts，不等於 goodwill。
  • 授予榮譽的入口是 Pawn.royalty.GainFavor(Faction.OfEmpire, amount)。
別把榮譽當好感改。新手常想用 TryAffectGoodwillWith 去「給玩家榮譽」，那是錯的——榮譽走 Pawn_RoyaltyTracker.GainFavor，且只在 Royalty DLC 啟用時存在。引用前要先 ModsConfig.RoyaltyActive 檢查，否則無 DLC 玩家會 crash。第 33 章會深入。
§7 自訂派系常見需求：定向關係、突襲強度、專屬裝備

需求 A：讓我的派系「一定友好」或「一定敵對」

一定友好（開局盟友）

沒有「正向起始好感」欄位可設——起始好感最高只到 0（中立）。要保證友好，需在世界生成後用 TryAffectGoodwillWith 把好感推過 +75，或掛 GoodwillSituationDef 提供正向 naturalGoodwillOffset。設 mustStartOneEnemy=false、naturalEnemy=false 確保不會開局敵對。

一定敵對（宿敵）

設 permanentEnemy=true（強制敵對、不可談判，起始好感 -100）。若只想「開局敵對但可被拉攏」，改設 naturalEnemy=true（起始 -80），而非 permanentEnemy。

需求 B：調整突襲強度

突襲規模由 parms.points 與 groupMaker 的「成本曲線」共同決定。兩條可調曲線在 FactionDef 上：

欄位型別效果
maxPawnCostPerTotalPointsCurveSimpleCurve限制「單一 pawn 成本佔總 points 的比例上限」隨總點數變化。低點數時禁止出現太貴的單位（避免開局遇到重裝兵）。
raidCommonalityFromPointsCurveSimpleCurve影響此派系被選為突襲來源的頻率隨威脅點數變化（高難度才常派精兵）。
 
<!-- 低點數時封鎖昂貴單位；高點數逐步放行 -->
<maxPawnCostPerTotalPointsCurve>
  <points>
    <!-- (總points, 單pawn成本上限佔比) -->
    <li>(0,    0.25)</li>
    <li>(1000, 0.35)</li>
    <li>(4000, 1.0)</li>
  </points>
</maxPawnCostPerTotalPointsCurve>
更直接的「強度」旋鈕是 PawnKind 的 combatPower。突襲組隊把每個 PawnKindDef 的 combatPower 當「價格」。想讓你的派系整體更兇，與其調曲線，不如在 groupMaker 放更高權重的高 combatPower kind，或自訂專屬高階 PawnKindDef（第 11 章）。

需求 C：派系專屬 apparel / weapon

讓某件裝備只在你的派系出現，組合兩個機制：

  1. 在 ThingDef（武器/服裝）加 tradeTags，並在 groupMaker 的 PawnKindDef 的 apparelTags / weaponTags 引用該 tag——組隊時會優先配發帶對應 tag 的物品。
  2. 用 FactionDef 的 apparelStuffFilter 限制材質，營造「這派系只穿某種材料」的視覺辨識。
<!-- 專屬大衣：只有帶 FreeGuildGear tag 的 kind 會穿 -->
<ThingDef ParentName="ApparelMakeableBase">
  <defName>Apparel_FreeGuildCoat</defName>
  <label>商會大衣</label>
  <tradeTags>
    <li>FreeGuildGear</li>
  </tradeTags>
  <!-- 其餘 apparel 欄位省略 -->
</ThingDef>

<!-- PawnKind 透過 apparelTags 拉進這件裝備 -->
<PawnKindDef>
  <defName>FreeGuild_Trooper</defName>
  <apparelTags>
    <li>FreeGuildGear</li>
  </apparelTags>
  <apparelRequired>
    <li>Apparel_FreeGuildCoat</li>
  </apparelRequired>
</PawnKindDef>
tradeTags 的雙重作用。同一個 tag 既能被 PawnKind 用來「配發」，也能被 TraderKindDef 的 StockGenerator_Tag 用來「上架販售」。一個 tag 就能讓專屬裝備既出現在敵人身上、又能在該派系商隊買到。
§8 陷阱與最佳實務

派系顏色衝突

多個 FactionDef 用相近 colorSpectrum 會讓世界地圖與外交頁難以分辨。給每個自訂派系一段明確區隔的色相，並避開內建派系常用色（海盜紅、部落綠、外地人藍）。

permanentEnemy 蓋過其他起始旗標

起始好感由 TryMakeInitialRelationsWith 推導：permanentEnemy 直接回 -100 並蓋過 naturalEnemy。想「開局敵對但可挽回」，不要用 permanentEnemy，改設 naturalEnemy=true（起始 -80，可被拉回中立／同盟）。

舊存檔相容

不要 patch 既有 FactionDef 的關鍵欄位來「改造」內建派系——會破壞舊檔。新增獨立 FactionDef + §4 的後補生成，是相容性最佳解。XML 引用其他 mod 的 def 時包 MayRequire，避免缺 mod 時 def 載入失敗。

hidden faction 別進外交頁

任何「敵我語意但不該被外交」的派系（蟲族、無主動物、純突襲來源）必設 hidden=true。否則玩家會在外交頁看到「機械蟲」並嘗試談判，造成混亂與潛在 null 問題。

MayRequire 範例

<!-- 只有當某 DLC/mod 啟用時，才把這段 groupMaker 套進來 -->
<options>
  <Town_Guard>10</Town_Guard>
  <Mechanoid_Centipede MayRequire="Ludeon.RimWorld.Biotech">2</Mechanoid_Centipede>
</options>

遍歷派系時的安全寫法

// ✗ 危險：AllFactions 含玩家與隱藏派系
foreach (Faction f in Find.FactionManager.AllFactions) { ... }

// ✓ 外交/突襲邏輯應排除玩家與隱藏
foreach (Faction f in Find.FactionManager.AllFactionsVisible)
{
    if (f.IsPlayer || f.defeated) continue;
    // 安全處理可互動的派系
}
defeated 與 temporary 派系也要過濾。被消滅的派系（f.defeated）與臨時派系（任務生成、f.temporary）仍可能留在 AllFactions 內。突襲來源、貿易來源、外交對象的篩選都要排除它們，否則會選到「已不存在」的派系而事件靜默失敗。
