測試與 CI

在一個沒有 headless 模式、緊綁 Unity 執行期的遊戲裡，如何建立可信賴的測試與自動化驗證流程。

Chapter 16 · Testing & CI
§1 為什麼 RimWorld modding 很難「測試」

對習慣了 xUnit／TDD／CI 綠燈的資深 C# 工程師而言，RimWorld modding 最違反直覺的一點是：幾乎沒有任何「正規」的測試入口。原因不是 mod 作者不想測，而是執行環境的本質使然。

Unity 執行期依賴

絕大多數 Verse / RimWorld 型別在 static 建構或屬性存取時就會觸碰 Unity 物件（GameObject、Texture2D、Material、AudioClip）。在 Unity runtime 之外 new 一個 Pawn 或讀 ThingDef.graphic，通常直接擲出例外或回傳 null。

無 headless 模式

RimWorld 沒有 -batchmode -nographics 之類能在 CI 跑完整邏輯的官方開關。遊戲必須開圖形視窗、載入完整 Def 資料庫、跑 mod static ctor 才會進入可用狀態，無法在無顯示器的 runner 上「跑一局」。

全域可變狀態

Find.CurrentMap、Current.Game、DefDatabase<T> 全是 static singleton。沒有 DI 容器、沒有可注入的時鐘或 RNG（Rand 也是 static）。單一測試很容易污染下一個測試。

啟動成本極高

即使能在遊戲內測，從點擊到進入地圖數十秒起跳。回饋迴圈遠比一般後端專案慢，逼得測試策略必須「分層」：能離線測的盡量離線測，剩下的才進遊戲。

核心結論 不要試圖在遊戲內「跑單元測試」，也不要試圖把整個 Verse 搬進測試專案。正確策略是分層：純邏輯離線用 NUnit/xUnit 測；觸碰 Verse 的部分用 Dev mode + DebugAction 手動驗證；端到端用遊戲啟動時的自我檢查腳本。下面逐層展開。
§2 純邏輯抽離：可測試性的第一原則

能進 NUnit/xUnit 的，只有不 import Verse / RimWorld / UnityEngine 命名空間的純 C# 邏輯。因此最重要的設計原則就是：把「決策／計算」與「對遊戲狀態的讀寫」徹底切開。

反模式：邏輯與 Verse 糾纏

// ✗ 無法測試：函式同時讀 Pawn、算公式、又寫回 Need
public void ApplyMoodDecay(Pawn pawn)
{
    Need_Mood mood = pawn.needs.mood;
    float outdoorBonus = pawn.Position.Roofed(pawn.Map) ? 0f : 0.05f;
    float decay = 0.1f - outdoorBonus + (pawn.health.hediffSet.PainTotal * 0.2f);
    mood.CurLevel = Mathf.Clamp01(mood.CurLevel - decay);
}

正模式：抽出純函式

把公式抽成一個只吃基本型別 / DTO、只回傳結果的 static class。它不認識 Pawn，因此可在任何 runner 上跑。薄薄的 Verse 包裝層只負責「取值 → 呼叫純函式 → 寫值」。

// MyMod.Core.dll —— 不引用 Assembly-CSharp.dll，純邏輯
namespace MyMod.Core
{
    public readonly struct MoodDecayInput
    {
        public readonly bool IsRoofed;
        public readonly float PainTotal;     // 0..1
        public readonly float CurrentMood;   // 0..1

        public MoodDecayInput(bool isRoofed, float painTotal, float currentMood)
        {
            IsRoofed = isRoofed;
            PainTotal = painTotal;
            CurrentMood = currentMood;
        }
    }

    public static class MoodMath
    {
        public const float BaseDecay = 0.1f;
        public const float OutdoorBonus = 0.05f;
        public const float PainWeight = 0.2f;

        /// <summary>回傳衰減後的新心情值（0..1）。純函式，無副作用。</summary>
        public static float NextMood(in MoodDecayInput input)
        {
            float bonus = input.IsRoofed ? 0f : OutdoorBonus;
            float decay = BaseDecay - bonus + (input.PainTotal * PainWeight);
            return Clamp01(input.CurrentMood - decay);
        }

        private static float Clamp01(float v) => v < 0f ? 0f : (v > 1f ? 1f : v);
    }
}
// MyMod.dll —— 薄包裝，引用 Assembly-CSharp.dll + MyMod.Core.dll
public void ApplyMoodDecay(Pawn pawn)
{
    var input = new MoodDecayInput(
        isRoofed:    pawn.Position.Roofed(pawn.Map),
        painTotal:   pawn.health.hediffSet.PainTotal,
        currentMood: pawn.needs.mood.CurLevel);

    pawn.needs.mood.CurLevel = MoodMath.NextMood(input);  // ← 純函式，已被測過
}
專案結構建議 開兩個 .csproj：MyMod.Core（無遊戲依賴，net472/netstandard2.0）與 MyMod（引用 Assembly-CSharp.dll 與 Core）。測試專案只引用 MyMod.Core。兩個 DLL 都丟進 mod 的 Assemblies/ 即可被遊戲載入。
§3 完整範例：抽離邏輯 + NUnit 測試

承上節的 MoodMath。測試專案 MyMod.Tests 只引用 MyMod.Core 與 NUnit，完全不碰遊戲 DLL，所以能在任意 CI runner 上 dotnet test。

using NUnit.Framework;
using MyMod.Core;

namespace MyMod.Tests
{
    [TestFixture]
    public class MoodMathTests
    {
        private const float Eps = 1e-4f;

        [Test]
        public void Roofed_NoPain_DecaysByBase()
        {
            var input = new MoodDecayInput(isRoofed: true, painTotal: 0f, currentMood: 0.8f);
            float result = MoodMath.NextMood(input);
            Assert.That(result, Is.EqualTo(0.7f).Within(Eps)); // 0.8 - 0.1
        }

        [Test]
        public void Outdoor_GetsBonus_DecaysLess()
        {
            var input = new MoodDecayInput(isRoofed: false, painTotal: 0f, currentMood: 0.8f);
            float result = MoodMath.NextMood(input);
            Assert.That(result, Is.EqualTo(0.75f).Within(Eps)); // 0.8 - (0.1 - 0.05)
        }

        [Test]
        public void HighPain_AcceleratesDecay()
        {
            var input = new MoodDecayInput(isRoofed: true, painTotal: 0.5f, currentMood: 0.8f);
            float result = MoodMath.NextMood(input);
            // 0.8 - (0.1 + 0.5*0.2) = 0.8 - 0.2 = 0.6
            Assert.That(result, Is.EqualTo(0.6f).Within(Eps));
        }

        [TestCase(0.05f, 1f,  0f)]   // 大量疼痛 → clamp 到下限 0
        [TestCase(1f,    0f,  0.9f)] // 室內無痛 → 0.9
        public void Clamps_To_ValidRange(float mood, float pain, float expected)
        {
            var input = new MoodDecayInput(isRoofed: true, painTotal: pain, currentMood: mood);
            float result = MoodMath.NextMood(input);
            Assert.That(result, Is.EqualTo(expected).Within(Eps));
            Assert.That(result, Is.InRange(0f, 1f));
        }
    }
}

跑法與一般 .NET 專案無異：

# 整個解決方案只測 Core，秒級回饋，可進 pre-commit hook
dotnet test MyMod.Tests/MyMod.Tests.csproj -c Release
xUnit 亦可 若團隊慣用 xUnit，把 [Test] 換成 [Fact]、[TestCase] 換成 [Theory]+[InlineData]、斷言改 Assert.Equal(expected, actual, precision) 即可。重點不在框架，而在「被測對象不依賴 Verse」。
§4 [DebugAction] 與 Dev mode 手動測試

觸碰 Verse 的程式碼無法離線測，但可以用 Dev mode 的 Debug Actions Menu 把它變成「一鍵可重現的手動測試」。在 Options → Development mode 開啟後，右上工具列的小蟲圖示會展開選單，你的 [DebugAction] 會出現在指定分類下。

using LudeonTK;   // 1.5+ DebugAction 已移到 LudeonTK 命名空間
using Verse;
using RimWorld;

public static class MyModDebugActions
{
    [DebugAction("MyMod", "Spawn test colonist (mood=50%)",
                 allowedGameStates = AllowedGameStates.PlayingOnMap)]
    private static void SpawnTestColonist()
    {
        Map map = Find.CurrentMap;
        Pawn p = PawnGenerator.GeneratePawn(PawnKindDefOf.Colonist, Faction.OfPlayer);
        GenSpawn.Spawn(p, UI.MouseCell(), map);
        p.needs.mood.CurLevel = 0.5f;
        Log.Message($"[MyMod] spawned {p.LabelShort}, mood={p.needs.mood.CurLevel:F2}");
    }

    // DebugActionType.ToolMapForPawns：點選 pawn 後對其執行，最適合驗證單體效果
    [DebugAction("MyMod", "Apply mood decay tick",
                 actionType = DebugActionType.ToolMapForPawns,
                 allowedGameStates = AllowedGameStates.PlayingOnMap)]
    private static void DecayMood(Pawn p)
    {
        float before = p.needs.mood.CurLevel;
        new MoodComp().ApplyMoodDecay(p);
        Log.Message($"[MyMod] {p.LabelShort}: {before:F2} -> {p.needs.mood.CurLevel:F2}");
    }
}
  • DebugActionType.Action：一般一鍵動作（預設）。
  • DebugActionType.ToolMap / ToolMapForPawns：進入「點選工具」模式，對地圖格或 pawn 執行。
  • allowedGameStates：限制只在 PlayingOnMap、Entry（主選單）等狀態出現，避免在錯誤情境崩潰。
1.4 vs 1.5 命名空間變動 DebugAction、DebugOutput 等開發工具在 1.5 被搬進 LudeonTK 命名空間。跨版本維護時這是常見的編譯失敗點——詳見 §9 版本回歸。
§5 QuickTest 啟動參數與 scenario 自動載入

每次手動點「New colony → 選 scenario → 等地圖生成」太慢。RimWorld 提供命令列參數讓你直接跳進一局，把回饋迴圈從分鐘級壓到數十秒，特別適合搭配 §4 的 DebugAction。

參數作用
-quicktest跳過主選單，用內建測試 scenario 直接生成一張小地圖並進入。
-savedatafolder=PATH指定獨立的設定／存檔資料夾，讓測試環境與正式遊戲隔離（測試 mod 清單不污染日常存檔）。
-disableglobalfaststartup停用部分快取，確保每次都重新跑 Def 載入流程——驗證 mod 載入路徑時有用。
-popupwindow / -screen-width / -screen-heightUnity 通用參數，控制視窗化與解析度，方便側開遊戲邊改邊看。

建立一個本機測試捷徑（PowerShell 為例）：

# test-rimworld.ps1 —— 啟用測試 mod 清單，直接進地圖
& "D:\Steam\steamapps\common\RimWorld\RimWorldWin64.exe" `
  -quicktest `
  -savedatafolder="D:\rw-test-profile" `
  -logfile="D:\rw-test-profile\latest.log"
搭配自我檢查腳本 -quicktest + §7 的 GameComponent 自我檢查 = 一條命令就能「啟動 → 進地圖 → 跑斷言 → 把結果寫進 log」。再配 -logfile 指定固定路徑，CI 之外的「半自動煙霧測試」就成型了。
§6 Mock RimWorld 型別：為什麼通常不值得

來自後端世界的直覺是「介面化 + Moq」。但 Verse 幾乎沒有為此設計，硬 mock 的代價極高：

幾乎沒有介面

Pawn、Thing、Map 都是具象 class，欄位多為 public field 而非 virtual property。Moq 只能 mock virtual/interface 成員，對這些型別幾乎無從下手。

建構即副作用

new Pawn() 看似可行，但任何進一步存取（needs、health、def）都依賴尚未初始化的 comp 樹與 DefDatabase，NullReferenceException 滿天飛。

static 無法注入

Rand、Find、GenDate 全 static。Mock 它們需要 shim 級框架（如 Microsoft Fakes / Pose），授權與維護成本高，且仍可能在 type 初始化時觸雷。

務實替代

與其 mock Pawn，不如把它需要的資料抽成 DTO（§2 的 MoodDecayInput）。「在邊界提取基本型別」幾乎總是比「mock 整個遊戲物件」便宜且更穩定。

策略適用代價
DTO 萃取（§2）純計算邏輯低 首選
對自己的 interface 用 Moq你自己寫的服務層低
Reflection 手刻假物件少數非測不可的 Verse 互動中高 脆弱
Fakes/Pose shim static幾乎不建議高 授權+觸雷
§7 整合測試：啟動時自我檢查腳本

要驗證「在真實遊戲環境裡邏輯正確」，最實用的做法是掛一個 GameComponent（或 StaticConstructorOnStartup），在遊戲進入特定狀態時自動跑一串斷言，把結果寫進 log。這不是傳統意義的單元測試，而是環境內的端到端煙霧測試。

using System;
using System.Collections.Generic;
using Verse;
using RimWorld;

// 只在 -quicktest 或定義環境變數時啟用，避免污染正式遊戲
public class SelfCheckGameComponent : GameComponent
{
    private bool _ran;

    public SelfCheckGameComponent(Game game) { }

    public override void GameComponentTick()
    {
        if (_ran || Find.TickManager.TicksGame < 60) return; // 等地圖穩定一秒
        _ran = true;
        if (Environment.GetEnvironmentVariable("MYMOD_SELFCHECK") != "1") return;
        RunSelfCheck();
    }

    private void RunSelfCheck()
    {
        var failures = new List<string>();

        Check(failures, "Def loaded",
            DefDatabase<ThingDef>.GetNamedSilentFail("MyMod_Gizmo") != null);

        Pawn p = PawnGenerator.GeneratePawn(PawnKindDefOf.Colonist, Faction.OfPlayer);
        p.needs.mood.CurLevel = 0.8f;
        new MoodComp().ApplyMoodDecay(p);
        Check(failures, "mood decayed below 0.8", p.needs.mood.CurLevel < 0.8f);

        if (failures.Count == 0)
            Log.Message("[MyMod][SELFCHECK] ALL PASSED");
        else
            foreach (var f in failures)
                Log.Error($"[MyMod][SELFCHECK] FAILED: {f}"); // ← 見 §10
    }

    private static void Check(List<string> fails, string name, bool ok)
    {
        if (!ok) fails.Add(name);
    }
}

用 GameComponent 而非 StaticConstructorOnStartup 的好處：前者在「遊戲已載入、地圖已存在」時才跑，可以安全觸碰 Find.CurrentMap 與生成 pawn；後者在 Def 載入階段就執行，地圖還不存在。

務必加開關 自我檢查會生成 pawn、寫狀態，絕不能在玩家正式存檔裡跑。用環境變數、命令列旗標或一個只在 dev build 啟用的 #if DEBUG 條件包起來。
§8 CI 設想：GitHub Actions 編譯驗證

因為跑不了遊戲，CI 的務實定位是「編譯閘門 + 純邏輯測試」，而非端到端。它能擋住的問題已經很多：Verse API 簽章變動、命名空間搬遷、語法錯誤、純邏輯回歸。

關鍵問題：CI 上沒有遊戲 DLL

Assembly-CSharp.dll 受版權保護，不能 commit 進 repo。兩種務實解法：

  1. NuGet 參考組件：用社群維護的 Krafs.Rimworld.Ref 套件（剝離 IL、僅含公開簽章的 reference assembly），合法且可放心進 CI，讓 MyMod 主專案也能在 runner 上編譯。
  2. 只測 Core：若不想引入參考組件，CI 至少能跑 MyMod.Core 的單元測試（§3），它根本不需要遊戲 DLL。
# .github/workflows/ci.yml
name: ci
on: [push, pull_request]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'

      # Core 引用 Krafs.Rimworld.Ref（NuGet），無需真正的遊戲 DLL
      - name: Restore
        run: dotnet restore

      - name: Build (warnings as errors)
        run: dotnet build -c Release --no-restore -warnaserror

      - name: Unit tests (pure logic)
        run: dotnet test MyMod.Tests/MyMod.Tests.csproj -c Release --no-build

      # 可選：把編好的 mod DLL 當 artifact 上傳，供手動煙霧測試下載
      - uses: actions/upload-artifact@v4
        with:
          name: mod-assemblies
          path: '**/Assemblies/*.dll'
CI 在 Linux 上即可編譯 Mod DLL 目標多為 net472，靠 NuGet 參考組件即可在 ubuntu runner 上用 SDK 編譯，毋需 Windows 也毋需安裝遊戲。把 -warnaserror 打開能讓 API 棄用警告變成硬失敗，提早抓到版本問題。
§9 版本回歸測試：多 RimWorld 版本編譯

RimWorld 每個大版本（1.3 / 1.4 / 1.5 …）都會改 API。維護「同時支援多版本」的 mod，最有價值的回歸測試就是對每個目標版本各編一次。CI 矩陣化非常合適。

多 target + 條件編譯

<!-- MyMod.csproj：用 MSBuild property 切換參考組件版本 -->
<PropertyGroup>
  <RimWorldVersion Condition="'$(RimWorldVersion)'==''">1.5</RimWorldVersion>
  <DefineConstants Condition="'$(RimWorldVersion)'=='1.5'">RW_1_5</DefineConstants>
  <DefineConstants Condition="'$(RimWorldVersion)'=='1.4'">RW_1_4</DefineConstants>
</PropertyGroup>

<ItemGroup>
  <PackageReference Include="Krafs.Rimworld.Ref"
                    Version="1.5.4297" Condition="$(RW_1_5)" />
</ItemGroup>
// 用 #if 隔離版本差異，例如 §4 提到的命名空間搬遷
#if RW_1_5
using LudeonTK;     // DebugAction 1.5 起在這
#else
using Verse;        // 1.4 及以前
#endif

CI 矩陣對每個版本各跑一次 build，任一版本壞了立刻紅燈：

strategy:
  matrix:
    rwver: ['1.4', '1.5']
steps:
  - run: dotnet build -c Release -p:RimWorldVersion=${{ matrix.rwver }} -warnaserror
輸出到對應資料夾 多版本 mod 的慣例是把不同版本 DLL 放進 1.4/Assemblies/、1.5/Assemblies/，再由 LoadFolders.xml 指定。CI 可把每個矩陣產物 publish 到對應路徑，直接打包成可發布結構。
§10 log-based assertion：用 Log.Error 當失敗信號

遊戲內沒有測試 runner，但有 log。把「Log.Error 出現 = 測試失敗」當成約定，就能讓外部腳本只靠 grep log 判定半自動測試結果——這正是 §5 + §7 串起來的最後一塊。

偵測「非預期錯誤」本身就是一種測試

RimWorld 在 Dev mode 會把例外、紅字錯誤都寫進 log。許多 mod bug 不會崩潰、只會默默 Log.Error。因此「跑一局後 log 裡有沒有錯誤」本身就是極有價值的回歸信號。

# run-smoketest.ps1 —— 啟動 → 跑自我檢查 → 解析 log 判定退出碼
$env:MYMOD_SELFCHECK = "1"
$log = "D:\rw-test-profile\latest.log"
if (Test-Path $log) { Remove-Item $log }

$proc = Start-Process -PassThru `
  "D:\Steam\...\RimWorldWin64.exe" `
  -ArgumentList "-quicktest", "-logfile=$log"

Start-Sleep -Seconds 90           # 給它時間進地圖跑自我檢查
Stop-Process -Id $proc.Id -Force

$text = Get-Content $log -Raw
if ($text -match "\[MyMod\]\[SELFCHECK\] ALL PASSED" `
    -and $text -notmatch "\[MyMod\]\[SELFCHECK\] FAILED") {
    Write-Host "SMOKE TEST PASSED"; exit 0
} else {
    Write-Host "SMOKE TEST FAILED"; exit 1
}

對應的斷言輔助：把失敗統一導向 Log.Error 並打上可 grep 的前綴標記。

public static class RuntimeAssert
{
    private const string Prefix = "[MyMod][ASSERT]";

    public static void True(bool cond, string msg)
    {
        if (!cond)
            Log.Error($"{Prefix} FAILED: {msg}");   // 失敗 → 紅字 → 被 grep 抓到
    }

    public static void Equal<T>(T expected, T actual, string ctx)
    {
        if (!EqualityComparer<T>.Default.Equals(expected, actual))
            Log.Error($"{Prefix} FAILED: {ctx} expected={expected} actual={actual}");
    }
}
注意 log 噪音與時序 其他 mod 也會 Log.Error，所以一定要用專屬前綴過濾自己的訊息，別用泛泛的 "error" 比對。Start-Sleep 等固定秒數很脆弱——更穩健是輪詢 log 直到出現 ALL PASSED 或逾時。此法是「半自動煙霧測試」，不是可進 GitHub Actions 的全自動 CI（runner 上跑不了遊戲）。

底層 · 純邏輯單元測試
