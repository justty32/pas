機械師與機械族（Biotech DLC）

Biotech 的機械師讓殖民者能透過頻寬（bandwidth）控制一群機械。本章拆解 Mechanitor, 頻寬, 機械 PawnKind、製造與召回，以及自訂機械。

Chapter 34 · Mechanitor
心智模型：四個彼此正交的子系統

機械師（mechanitor）玩法在反編譯後並不是單一龐然大物，而是四個幾乎正交的子系統疊起來的。 把它們拆開理解，後面所有 API 就有了座標：

身份層

誰是機械師？由 RimWorld.Pawn_MechanitorTracker（掛在 pawn.mechanitor）與一個 mechlink Hediff 決定。沒有它，pawn 連控制機械的「插槽」都沒有。

容量層

能控制多少機械？由 bandwidth（StatDefOf.MechBandwidth 統計值）決定：機械師本身提供基礎頻寬，band node（CompBandNode）與身體部件再加成。每隻機械吃掉 StatDefOf.BandwidthCost。

實體層

機械本身是 Verse.Pawn，其 RaceProperties.IsMechanoid 為真、mechWeightClass 分級。它隸屬玩家 faction，但行為由 overseer 與 work mode 驅動。

生產層

機械從哪來？mech gestator（孵化槽）＋ RecipeDef ＋ mech recharger（充電器）。製造、餵食、休眠回充都在這一層。

反編譯定位 用 dnSpy 開 Assembly-CSharp.dll，主要鎖定 namespace RimWorld 下的 Pawn_MechanitorTracker、MechanitorUtility、CompMechCarrier、 Building_MechGestator（孵化器，孵化盒 comp 為 CompMechGestatorTank）、CompBandNode（頻寬節點 comp）、 MechWorkModeDefOf，以及 Verse 下的 RaceProperties、Pawn。 所有 def 型別（MechWeightClassDef、MechWorkModeDef）也在 RimWorld。
§1 · DLC 守門與總覽

機械師的一切都依賴 Biotech DLC。沒裝 DLC 時，相關型別仍存在於 Assembly-CSharp.dll （它們是核心組件的一部分），但對應的 Def（PawnKind、ThingDef、Recipe）不會載入。 因此你的 mod 必須做兩件事：XML 端用 MayRequire 條件載入、C# 端先檢查 ModLister.BiotechInstalled 再碰任何機械相關邏輯。

C# 守門

using Verse;
using RimWorld;

public static class MechGate
{
    // 在任何讀寫 pawn.mechanitor / 機械 Def 之前都先過這一關
    public static bool Available => ModLister.BiotechInstalled;

    public static void TryDoMechStuff(Pawn pawn)
    {
        if (!Available)
        {
            // 沒裝 DLC：直接安靜返回，不要丟例外、不要碰 mechanitor
            return;
        }
        var tracker = pawn.mechanitor;   // 只有確定有 DLC 才安全存取
        if (tracker == null) return; // 此 pawn 不是機械師
        // ... 真正的機械邏輯
    }
}
最常見的崩潰 在沒裝 Biotech 的存檔裡，若你的 Harmony patch 無條件呼叫 pawn.mechanitor.ControlledPawns， pawn.mechanitor 多半是 null → NullReferenceException → 存檔載入崩潰且難以歸因。每一條碰機械的程式路徑都要先過 BiotechInstalled。

XML 守門：MayRequire

在 def 的任一節點上加 MayRequire="Ludeon.RimWorld.Biotech"， 若 DLC 未啟用，該節點（連同它包住的子樹）會被靜默丟棄，不會報錯。 整份 def 檔若只在 DLC 下才有意義，可放在獨立資料夾，再用 LoadFolders.xml 條件載入： 

<!-- LoadFolders.xml -->
<loadFolders>
  <v1.5>
    <li>Common</li>
    <!-- 只有啟用 Biotech 時才載入這個資料夾 -->
    <li IfModActive="Ludeon.RimWorld.Biotech">Biotech</li>
  </v1.5>
</loadFolders>

單一節點層級的條件載入（適合一份 def 裡只有某段需要 DLC）：

<ThingDef ParentName="BuildingBase">
  <defName>MyMod_Workshop</defName>
  <!-- 只有 Biotech 在時，才賦予這個建築 mech 加成的 comp -->
  <comps>
    <li Class="CompProperties_BandNode" MayRequire="Ludeon.RimWorld.Biotech">
      <hediff>BandNode</hediff>
    </li>
  </comps>
</ThingDef>
守門方式用途失效行為
ModLister.BiotechInstalledC# 執行期判斷回傳 false，由你決定降級
MayRequire="..."（節點屬性）單一 def 節點條件載入該節點被丟棄，其餘照載
MayRequireAnyOf="A,B"多 DLC 任一即可全缺才丟棄
LoadFolders.xml + IfModActive整資料夾條件載入整夾跳過
機械≠殖民者 先建立直覺：機械是 Pawn、屬玩家 faction，但它不是「殖民者」。 它沒有心情、需求（除了能量）、無法被招募成人類意義上的成員，也通常不計入「殖民者人口」這個用於事件點數的指標。 §8 會詳述這個判斷的精確 API。
§2 · 機械師身份：mechlink、tracker 與 overseer 綁定

一個普通 pawn 要成為機械師，必須先擁有 mechlink——一個植入式 Hediff （def 名 MechlinkImplant，hediff 類別 Hediff_Mechlink；由可用物品 Mechlink 透過 CompUseEffect_InstallImplantMechlink 植入，而非外科手術 recipe）。 mechlink 一旦存在，遊戲就會為該 pawn 配置一個 RimWorld.Pawn_MechanitorTracker， 之後可由 pawn.mechanitor 取得。

取得身份

使用物品 Mechlink 植入 mechlink，或角色生成時即帶有。MechanitorUtility.IsMechanitor(pawn) 是最權威的判斷。

tracker 物件

Pawn_MechanitorTracker 管理「這位機械師控制了哪些機械、頻寬用了多少、可控上限多少」。它隨 mechlink 出現、移除 mechlink 時清理。

tracker 的關鍵成員

成員型別意義
ControlledPawnsList<Pawn>實際生效（頻寬足以涵蓋）的機械子集；超出頻寬的會被排除
TotalBandwidthint可用頻寬上限（基礎 + 建築 + 部件加成）
UsedBandwidthint已被 ControlledPawns 占用的頻寬（各機械 bandwidthCost 加總）
CanControlMechsAcceptanceReport是否處於可下令狀態（昏迷／在世界圖／囚禁／精神崩潰會回傳失敗原因字串，可隱式轉 bool）
OverseenPawnsList<Pawn>此機械師所有已指派的機械（由 Overseer relation 計算）；ControlledPawns 是其中受頻寬上限約束、實際生效的子集

判斷與列舉機械師控制的機械

using System.Linq;
using Verse;
using RimWorld;

public static class MechanitorReads
{
    // 最可靠的「是不是機械師」判斷
    public static bool IsMechanitor(Pawn p)
        => MechanitorUtility.IsMechanitor(p);

    public static int FreeBandwidth(Pawn mechanitor)
    {
        var t = mechanitor.mechanitor;
        if (t == null) return 0;
        return t.TotalBandwidth - t.UsedBandwidth;
    }

    public static void DumpControlled(Pawn mechanitor)
    {
        var t = mechanitor.mechanitor;
        if (t == null) return;
        foreach (Pawn mech in t.ControlledPawns)
        {
            Log.Message($"{mech.LabelShort}  cost={mech.GetStatValue(StatDefOf.BandwidthCost)}  weight={mech.def.race.mechWeightClass}");
        }
    }
}

overseer 綁定：機械「歸誰管」

機械與機械師之間的關係不是 Pawn.relations（那是人際關係/家族）， 而是一條專屬的 overseer 指派。機械側用 mech.GetOverseer()（擴充方法，定義於 MechanitorUtility） 取得目前主人；指派／解除則走 tracker 的方法或 MechanitorUtility 的工具方法。 

// 取得某隻機械目前的 overseer（可能為 null＝無主，會 feral 化）
Pawn overseer = mech.GetOverseer();

// 指派關係的底層是一條 Overseer 的 PawnRelation（PawnRelationDefOf.Overseer），
// 機械師→機械方向；遊戲據此計算 OverseenPawns 與頻寬占用（見 §8）
mech.relations.AddDirectRelation(PawnRelationDefOf.Overseer, mechanitor);

// 實務上更高階：把機械指派進機械師的某個控制群（內部會設好關係與 work mode）
mechanitor.mechanitor.AssignPawnControlGroup(mech);

// 機械端的指派狀態由 CompOverseerSubject 持有，經屬性 mech.OverseerSubject 取得
CompOverseerSubject subject = mech.OverseerSubject;
命名因版本而異 指派／解除 overseer 的高階入口在版本間動過（控制群 AssignPawnControlGroup / MechanitorControlGroup.Assign 等）。不要硬背高階方法名，實際開 dnSpy 看你的目標版本。 穩定不變的是底層機制：指派以 PawnRelationDefOf.Overseer 這條 PawnRelation 表示， 機械側由 CompOverseerSubject（屬性 mech.OverseerSubject）持有狀態、機械師側由 mechanitor tracker 管理。
§3 · 機械 PawnKind / ThingDef：分級、頻寬成本與工種

一隻機械需要兩個 Def：一個 race 的 ThingDef（定義它「是什麼」——血量、移動、外觀、 RaceProperties），以及一個 PawnKindDef（定義「怎麼生成這個個體」——裝備、貼圖變體）。 機械的特性集中在 ThingDef.race 這個 RaceProperties 上。

機械分級：MechWeightClassDef

分級影響召回、攜帶、AI 群體行為與許多上限判斷。香草四級：

MechWeightClassDef典型機械特性
LightMilitor, Lifter, Cleansweeper可被 CompMechCarrier 攜帶；低頻寬成本
MediumScyther, Lancer, Constructoid主力，無法被攜帶
HeavyCentipede 系列高頻寬、高血量
UltraHeavyWar queen, Diabolus（特殊）多半為敵對/特殊；玩家難控

RaceProperties 上的機械欄位

欄位意義
intelligence機械通常為 ToolUser（能做部分工作）；純戰鬥機械可 Animal 級
mechEnabledWorkTypes此機械可承擔的 WorkTypeDef 列表（如 Cleaning、Hauling）。空＝純戰鬥
mechWeightClass指向 MechWeightClassDef
mechFixedSkillLevel機械做工作時固定的技能等級（無法成長）
fleshType應為 Mechanoid（影響傷害、修復而非治療）
needsRest / foodType機械不睡眠、不吃食物；以「能量」need 取代（見 §4）

注意：頻寬成本不是 RaceProperties 欄位，而是 Stat StatDefOf.BandwidthCost（XML 寫在 <statBases> 的 <BandwidthCost>，C# 端以 pawn.GetStatValue(StatDefOf.BandwidthCost) 讀取）。

自訂一隻新機械：race ThingDef

<!-- Biotech/Defs/ThingDefs_Races/Mech_MyMod.xml -->
<Defs>

  <ThingDef ParentName="BaseMechanoid" MayRequire="Ludeon.RimWorld.Biotech">
    <defName>Mech_MyHarvester</defName>
    <label>採收機</label>
    <description>輕型農用機械，專責收割與搬運作物。</description>
    <statBases>
      <MarketValue>900</MarketValue>
      <MoveSpeed>3.6</MoveSpeed>
      <ToxicEnvironmentResistance>1</ToxicEnvironmentResistance>
      <!-- 頻寬成本是 Stat（StatDefOf.BandwidthCost），非 race 欄位 -->
      <BandwidthCost>1</BandwidthCost>
    </statBases>

    <race>
      <intelligence>ToolUser</intelligence>
      <fleshType>Mechanoid</fleshType>
      <needsRest>false</needsRest>
      <foodType>None</foodType>

      <!-- 機械師相關（bandwidthCost 不在這裡，見上方 statBases） -->
      <mechWeightClass>Light</mechWeightClass>
      <mechFixedSkillLevel>8</mechFixedSkillLevel>

      <!-- 可承擔的工作（空則純戰鬥） -->
      <mechEnabledWorkTypes>
        <li>PlantCutting</li>
        <li>Hauling</li>
      </mechEnabledWorkTypes>

      <body>Mech_Lifter</body>     <!-- 沿用輕型機械的身體結構 -->
      <lifeStageAges>
        <li><def>MechanoidFullyFormed</def><minAge>0</minAge></li>
      </lifeStageAges>
    </race>

    <!-- 能量 comp：取代睡眠/吃飯（見 §4） -->
    <comps>
      <li Class="CompProperties_Rechargeable">
        <!-- Rechargeable 只有 ticksToRecharge（充電所需 tick），能量本身由 Need_MechEnergy 管 -->
        <ticksToRecharge>10000</ticksToRecharge>
      </li>
      <li Class="CompProperties_OverseerSubject">
        <delayUntilFeralCheck>60000</delayUntilFeralCheck> <!-- 失去 overseer 後多久檢查失控 -->
      </li>
    </comps>
  </ThingDef>

</Defs>

§4 · 製造機械：孵化槽、配方與能量回充

玩家製造機械的流程：在 mech gestator（孵化槽，類別 Building_MechGestator，繼承自 Building_WorkTableAutonomous）下達一個 gestation cycle（孵化配方，RecipeDef），消耗材料與電力， 經過數個週期後（bill 為 Bill_Mech），從關聯的孵化盒（gestation processor，comp CompMechGestatorTank）產出一隻機械。

① 孵化槽

建築 ThingDef，thingClass 為 Building_MechGestator（孵化盒 comp 為 CompProperties_MechGestatorTank）。耗電、需機械師站位輸入原料。

② 孵化配方

RecipeDef 的 ingredients 定義所需材料（鋼鐵、塑鋼、subcore）；products 是要生出的機械 PawnKind。

③ 能量回充

產出後機械以 CompProperties_Rechargeable 管理能量；耗盡會 self-shutdown，需到 recharger 充電。

孵化配方（RecipeDef）

<!-- 機械孵化配方：在孵化槽列為可下令的選項 -->
<RecipeDef MayRequire="Ludeon.RimWorld.Biotech">
  <defName>GestateMyHarvester</defName>
  <label>孵化採收機</label>
  <jobString>正在孵化採收機。</jobString>

  <!-- 機械配方的識別：產出一隻 mech 而非 thing -->
  <mechResurrection>false</mechResurrection>
  <gestationCycles>3</gestationCycles>   <!-- 需幾個孵化週期 -->

  <ingredients>
    <li>
      <filter><thingDefs><li>Steel</li></thingDefs></filter>
      <count>60</count>
    </li>
    <li>
      <filter><thingDefs><li>SignalChip</li></thingDefs></filter>
      <count>1</count>
    </li>
  </ingredients>

  <!-- 要孵出的機械 -->
  <products>
    <Mech_MyHarvester>1</Mech_MyHarvester>
  </products>

  <!-- 哪些孵化槽能執行 -->
  <recipeUsers>
    <li>MechGestator</li>
  </recipeUsers>

  <!-- 解鎖研究（串第 14 章） -->
  <researchPrerequisite>BasicMechtech</researchPrerequisite>
</RecipeDef>
香草配方的權威來源 不要憑空寫機械配方的欄位。打開 Data/Biotech/Defs/RecipeDefs/Recipes_Gestation.xml， 照香草 GestateMilitor / GestateLifter 的結構抄。欄位（如 gestationCycles、 mechResurrection）在版本間偶有調整，以你目標版本的香草檔為準。

能量回充與 self-shutdown

機械不吃食物——它有「能量」（由 CompProperties_Rechargeable 或機械版的能量 need 管理）。 能量耗盡時，機械進入 self-shutdown：它會掛上一個「電量耗盡」Hediff、停止行動， 直到被搬到 mech recharger（充電器，Building_MechCharger）回充。 在 recharge work mode（見 §5）下，機械會自動走去充電。 

// C# 端讀取/操作機械能量（comp 名稱以反編譯為準）
using Verse;
using RimWorld;

public static class MechEnergy
{
    public static float EnergyPercent(Pawn mech)
    {
        // 機械的能量在 1.5 走 Need_MechEnergy（掛在 pawn.needs）
        var need = mech.needs?.TryGetNeed<Need_MechEnergy>();
        return need?.CurLevelPercentage ?? 0f;
    }

    public static bool IsSelfShutdown(Pawn mech)
    {
        // 自我關機並非以 Hediff 表現（HediffDefOf 只暴露 MechlinkImplant，無「電量耗盡」項）。
        // 用核心擴充方法 Pawn.IsSelfShutdown()（RestUtility），內部判斷 needs.energy 的 SelfShutdown job。
        return mech.IsSelfShutdown();
    }
}
充電器電力不足 recharger 需要持續電力（串第 18 章）。若殖民地電網跳閘，機械無法回充 → 全體 self-shutdown → 工作與防禦全停。設計機械隊伍時，務必把充電負載算進電力預算，並考慮備援電池。
§5 · 控制群與指令：work mode、攜帶與頻寬節點

玩家不會逐隻微操機械，而是給每隻機械指定一個 work mode（MechWorkModeDef）， 決定它在沒有明確命令時的預設行為。香草四種模式：

MechWorkModeDef行為適用
Work自動承擔 mechEnabledWorkTypes 內的工作勞動機械（lifter、cleansweeper）
Recharge走去充電器回充，待命能量低、暫不需用時
Escort跟隨機械師移動並戰鬥戰鬥機械（militor、scyther）
SelfShutdown就地關機，不消耗能量長期封存、節能

查 RimWorld.MechWorkModeDefOf 取得型別安全的引用，避免硬編字串：

using Verse;
using RimWorld;

public static class MechCommands
{
    // work mode 掛在「控制群」而非單一機械的 comp 上：
    // 取得機械所屬控制群（MechanitorControlGroup）再呼叫 SetWorkMode
    public static void SetMode(Pawn mech, MechWorkModeDef mode)
    {
        var group = mech.GetMechControlGroup();   // MechanitorUtility 擴充方法
        group?.SetWorkMode(mode);
    }

    public static void SendToWork(Pawn mech)
        => SetMode(mech, MechWorkModeDefOf.Work);

    public static void ParkAndRecharge(Pawn mech)
        => SetMode(mech, MechWorkModeDefOf.Recharge);
}

攜帶與召回：CompMechCarrier

某些機械（如 war queen）能在體內「攜帶」一群小型機械（如 war urchin）， 作戰時釋放、戰後回收，回收時消耗材料補充庫存。這由 CompProperties_MechCarrier / CompMechCarrier 實作。 

<!-- 在母體機械 ThingDef 的 comps 加入攜帶器 -->
<li Class="CompProperties_MechCarrier" MayRequire="Ludeon.RimWorld.Biotech">
  <fixedIngredient>Steel</fixedIngredient>   <!-- 補充一隻所需材料 -->
  <costPerPawn>10</costPerPawn>
  <maxPawnsToSpawn>6</maxPawnsToSpawn>
  <spawnPawnKind>Mech_WarUrchin</spawnPawnKind>
  <maxIngredientCount>60</maxIngredientCount>   <!-- 庫存材料上限（非 maxToFill） -->
  <startingIngredientCount>60</startingIngredientCount>
</li>
// 程式化釋放／回收（gizmo 背後就是這些）
var carrier = warQueen.GetComp<CompMechCarrier>();
if (carrier != null && carrier.CanSpawn.Accepted)   // CanSpawn 是 AcceptanceReport，非 bool
{
    carrier.TrySpawnPawns();   // 釋放小型機械（受庫存與成本限制）
}

頻寬節點：CompBandNode

機械師的基礎頻寬有限。band node 是耗電建築（香草 ThingDef BandNode， comp 為 CompBandNode / CompProperties_BandNode，沒有專屬 thingClass）。 它不是「自動全域加成」：機械師必須先把節點 調諧（tune）給自己（CompBandNode.tunedTo）。 每一座「已調諧且供電」的節點，會在該機械師身上掛一個 Hediff_BandNode，透過 StatDefOf.MechBandwidth 的 statOffset 固定 +1 頻寬（每座 +1，非可調的 bandwidthGain）。串第 18 章——斷電即失去加成，可能立刻造成頻寬超載。 

<!-- 建築：頻寬節點（沿用香草 BandNode 的 comp 結構） -->
<ThingDef ParentName="BuildingBase" MayRequire="Ludeon.RimWorld.Biotech">
  <defName>MyMod_BandNodeMk2</defName>
  <!-- 無專屬 thingClass：行為由 CompBandNode 提供 -->
  <comps>
    <li Class="CompProperties_Power">
      <compClass>CompPowerTrader</compClass>
      <basePowerConsumption>400</basePowerConsumption>
    </li>
    <li Class="CompProperties_BandNode">
      <!-- 每座已調諧節點固定 +1 MechBandwidth；加成量在 Hediff_BandNode 內寫死，無 bandwidthGain 欄位 -->
      <hediff>BandNode</hediff>
      <retuneDays>3</retuneDays>
      <tuneSeconds>5</tuneSeconds>
    </li>
  </comps>
</ThingDef>
頻寬 = 殖民地的 mech 上限旋鈕 若你的 mod 想讓玩家養更大的機械軍團，調整方向有三：降低機械的 BandwidthCost stat、 提高 band node hediff（BandNode）的 MechBandwidth statOffset（band node 本身每座固定 +1，要加碼得改 hediff 的 stage statOffset）、 或加入提升 MechBandwidth 基礎值的身體部件 Hediff。最後一種最「劇情友善」（升級 mechlink）。
§6 · 機械叢集（敵對玩法）與作為 incident

機械師讓玩家「擁有」機械；硬幣的另一面是 mech cluster（機械叢集）—— 由 RimWorld.MechClusterGenerator 生成的一坨敵對機械結構，含休眠機械、防禦建築（盾發生器、迫擊砲、 機械繁殖器）與喚醒條件。它與機械師玩法是同一套機械 race，但 faction 為 Mechanoid（敵對）， 且初始多半休眠。

叢集組成

MechClusterGenerator.GenerateClusterSketch(points, map, ...) 依威脅點數產生一份藍圖（sketch），包含建築與休眠機械的擺放。

喚醒條件

叢集帶 CompWakeUpDormant / proximity / 倒數計時等觸發器；玩家靠近或攻擊即喚醒整群，轉為主動攻擊。

把叢集包成自訂 incident（串第 12 章）

using Verse;
using RimWorld;

public class IncidentWorker_MyMechCluster : IncidentWorker
{
    protected override bool CanFireNowSub(IncidentParms parms)
    {
        // DLC 守門：沒 Biotech 就不該觸發
        if (!ModLister.BiotechInstalled) return false;
        return base.CanFireNowSub(parms) && parms.target is Map;
    }

    protected override bool TryExecuteWorker(IncidentParms parms)
    {
        Map map = (Map)parms.target;
        float points = parms.points > 0 ? parms.points : 400f;

        // 產生叢集藍圖
        var sketch = MechClusterGenerator.GenerateClusterSketch(points, map);

        // 找落點並實體化
        IntVec3 center = MechClusterUtility.FindClusterPosition(map, sketch, 100, 0.5f);
        MechClusterUtility.SpawnCluster(center, map, sketch, dropInPods: true);

        SendStandardLetter(parms, new TargetInfo(center, map));
        return true;
    }
}
不要重造輪子 MechClusterUtility / MechClusterGenerator 已封裝藍圖生成、落點搜尋與實體化。 自訂叢集 99% 的需求是「調整組成與威脅曲線」，而非重寫生成器。 參考香草 IncidentWorker_MechCluster 的反編譯，照它的呼叫順序走最安全。
§7 · C# API：tracker、MechanitorUtility、overseer 與狀態

把前面散落的 API 收攏成一張速查表，這是你寫 Harmony patch 或 GameComponent 時的工具箱。

API用途
pawn.mechanitor取得 Pawn_MechanitorTracker（非機械師為 null）
MechanitorUtility.IsMechanitor(pawn)權威判斷是否為機械師
MechanitorUtility.EverControllable(Pawn mech)判斷某機械是否可被機械師控制（內部即 mech.OverseerSubject != null）
mech.GetOverseer()取得機械目前的 overseer（機械師），無則 null
tracker.ControlledPawns該機械師控制的機械清單
tracker.TotalBandwidth / UsedBandwidth頻寬上限／已用
tracker.CanControlMechs是否處於可下令狀態
mech.GetComp<CompMechCarrier>()攜帶器（釋放/回收小型機械）
MechWorkModeDefOf.Work / Recharge / Escort型別安全的 work mode 引用
MechClusterUtility.SpawnCluster(...)生成敵對叢集

一個務實的 GameComponent：監看頻寬超載

using System.Linq;
using Verse;
using RimWorld;

public class GameComponent_BandwidthWatch : GameComponent
{
    private int nextCheckTick;

    public GameComponent_BandwidthWatch(Game game) { }

    public override void GameComponentTick()
    {
        // DLC 守門 + 低頻檢查（每 2500 tick ≈ 1 小時）
        if (!ModLister.BiotechInstalled) return;
        if (Find.TickManager.TicksGame < nextCheckTick) return;
        nextCheckTick = Find.TickManager.TicksGame + 2500;

        var map = Find.CurrentMap;
        if (map == null) return;

        foreach (Pawn p in map.mapPawns.FreeColonists)
        {
            if (!MechanitorUtility.IsMechanitor(p)) continue;
            var t = p.mechanitor;
            if (t.UsedBandwidth > t.TotalBandwidth)
            {
                Messages.Message(
                    $"{p.LabelShort} 的機械頻寬超載：{t.UsedBandwidth}/{t.TotalBandwidth}",
                    p, MessageTypeDefOf.NegativeEvent);
            }
        }
    }
}

記得在 defs/MainButtonDefs 旁不需註冊；GameComponent 只要型別存在即被自動實例化，無需 def。

§8 · 陷阱與最佳實務

未裝 DLC 崩潰

任何碰 pawn.mechanitor、機械 Def 的程式路徑，第一行就先 ModLister.BiotechInstalled。Harmony patch 尤其危險，因為它會在所有存檔執行。

頻寬超載 → feral

機械失去 overseer 或頻寬超載，會在倒數（delayUntilFeralCheck）後 feral 化——脫離玩家控制、變敵對。指派機械前務必確認 FreeBandwidth > bandwidthCost。

機械無 overseer 的行為

剛生成、或機械師死亡後，機械可能短暫無主。不要假設 GetOverseer() 永遠非 null——它常為 null，需先判空再用。

recharge 站電力不足

充電器斷電 → 機械無法回充 → 全體 self-shutdown。把充電負載納入電力預算，必要時加電池備援（串第 18 章）。

weight class 與成本失衡

自訂機械若 bandwidthCost 太低又能力太強，會破壞玩法平衡（無限軍團）。Light 級通常 1、Medium 2~3、Heavy 4+。對齊香草曲線。

機械算不算「殖民者」

影響事件點數、人口統計與襲擊規模。判斷要用對的 API，下面詳述。

「機械算不算殖民者人口」的精確判斷

RimWorld 區分數種 pawn 集合，機械落在哪個集合直接影響事件點數（storyteller wealth/population）與 襲擊規模。常見誤用是把機械混進 FreeColonists 做人口統計：

集合 / 判斷機械是否計入說明
map.mapPawns.FreeColonists否純人類殖民者；機械不在此
map.mapPawns.FreeColonistsAndPrisoners否同上不含機械
map.mapPawns.SpawnedColonyMechs是玩家機械專用集合
pawn.IsColonyMech—判斷「是不是玩家機械」的最直接屬性
StorytellerUtility 人口點數部分機械有自己的 wealth/threat 計算，非按「殖民者數」
// 正確分流：人類殖民者 vs 玩家機械
int humanColonists = map.mapPawns.FreeColonists.Count;          // 不含機械
int colonyMechs    = map.mapPawns.SpawnedColonyMechs.Count;    // 只有機械

// 對單一 pawn 的判斷
bool isMech = p.RaceProps.IsMechanoid;
bool isPlayerMech = p.IsColonyMech;   // 機械 + 屬玩家 faction
最易踩雷：把機械當殖民者計入點數 若你的 mod 自訂事件點數公式時用了 FreeColonists.Count + allMechs， 會在機械軍團龐大時讓襲擊強度暴衝（或反之被遊戲重複計算）。 機械的威脅貢獻由香草透過 wealth/combat power 另算，不要再把機械「人頭」加進殖民者數。

最佳實務清單

  1. 每條機械程式路徑第一行先 if (!ModLister.BiotechInstalled) return;。
  2. XML 用 MayRequire="Ludeon.RimWorld.Biotech" 或 LoadFolders.xml 條件載入機械 def。
  3. 自訂機械 race 從香草 BaseMechanoid 繼承，配方照 Recipes_Gestation.xml 抄。
  4. bandwidthCost 對齊 weight class（Light 1 / Medium 2-3 / Heavy 4+）。
  5. 指派 overseer 前檢查 FreeBandwidth >= bandwidthCost，避免 feral。
  6. 讀 GetOverseer() 一律先判 null。
  7. 把充電器電力負載納入第 18 章的電力預算。
  8. 人口統計用 FreeColonists（人類）與 SpawnedColonyMechs（機械）分流，勿混算。
  9. 所有方法/欄位名以你目標版本的 dnSpy 反編譯為準，不硬背跨版本可能改名的成員。
