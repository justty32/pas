研究與生產鏈

從 ResearchProjectDef 解鎖路徑，到 RecipeDef / Bill / WorkGiver 的完整生產流水線——一條龍打通 RimWorld 的科技與製造系統。

Chapter 14 · Research & Production
§1 ResearchProjectDef：研究專案定義

ResearchProjectDef（命名空間 RimWorld）是科技樹的最小節點。它描述「要花多少研究點、需要哪些前置、要在哪種研究台研究、解鎖後讓哪些 Def 變得可用」。研究本身不直接「解鎖」任何東西——解鎖是反向的：ThingDef/RecipeDef 透過 researchPrerequisites 指向研究專案。

核心欄位一覽

欄位型別說明
baseCost float 研究所需點數（未經 techprint / 難度縮放）。原版範圍約 100（早期）至 4000（後期）
prerequisites List<ResearchProjectDef> 必須先完成的研究專案，全部完成後此節點才可研究
techLevel TechLevel 科技時代（Neolithic / Medieval / Industrial / Spacer…），低於玩家派系科技時有成本懲罰
requiredResearchBuilding ThingDef 研究時必須在範圍內存在的建築（如 HiTechResearchBench）
requiredResearchFacilities List<ThingDef> 必須連接的設施（如多重分析儀 MultiAnalyzer）
researchViewX / researchViewY float 在研究樹視圖上的座標（格為單位，X 通常與 tier 對應）
tab ResearchTabDef 所屬分頁，未指定時歸入 Main
techprintCount / techprintMarketValue int / float （Royalty）需先取得的科技藍圖數量與價值
hiddenPrerequisites List<ResearchProjectDef> 不在樹上畫線、但仍必須完成的隱藏前置

完整 XML 範例

<!-- Defs/ResearchProjectDefs/Research_MyMod.xml -->
<Defs>

  <!-- 第一層：基礎能量學 -->
  <ResearchProjectDef>
    <defName>MyMod_BasicEnergetics</defName>
    <label>basic energetics</label>
    <description>解鎖能量電池與基礎導管。</description>
    <baseCost>500</baseCost>
    <techLevel>Industrial</techLevel>
    <tab>MyMod_Tab</tab>
    <researchViewX>0</researchViewX>
    <researchViewY>1.0</researchViewY>
  </ResearchProjectDef>

  <!-- 第二層：進階導能（需要基礎 + 高科技台 + 多重分析儀） -->
  <ResearchProjectDef>
    <defName>MyMod_AdvancedConduction</defName>
    <label>advanced conduction</label>
    <description>解鎖高效導管與我的工作台。</description>
    <baseCost>2000</baseCost>
    <techLevel>Spacer</techLevel>
    <tab>MyMod_Tab</tab>
    <prerequisites>
      <li>MyMod_BasicEnergetics</li>
      <li>MicroelectronicsBasics</li> <!-- 原版專案 -->
    </prerequisites>
    <requiredResearchBuilding>HiTechResearchBench</requiredResearchBuilding>
    <requiredResearchFacilities>
      <li>MultiAnalyzer</li>
    </requiredResearchFacilities>
    <researchViewX>2</researchViewX>
    <researchViewY>1.0</researchViewY>
  </ResearchProjectDef>

</Defs>
techLevel 的成本縮放 實際研究成本 = baseCost × ResearchManager 的難度倍率 × techLevel 倍率。當研究的 techLevel 高於玩家派系科技時，每差一級成本乘以 0.5 的倒數（亦即更貴）；計算邏輯位於 ResearchProjectDef::CostFactor(TechLevel researcherTechLevel)。Spacer 級研究讓 Industrial 派系研究時會顯著變貴。
§2 研究樹佈局與 ResearchTabDef

研究視窗（MainTabWindow_Research）以分頁（tab）區隔不同領域。原版只有 Main 與 Anomaly（DLC）。自訂大量研究時，建立專屬 ResearchTabDef 可避免塞爆主分頁。

定義一個 ResearchTabDef

<!-- Defs/ResearchTabDefs/ResearchTabs_MyMod.xml -->
<Defs>
  <ResearchTabDef>
    <defName>MyMod_Tab</defName>
    <label>my mod tech</label>
    <generalTitle>My Mod Technologies</generalTitle>
  </ResearchTabDef>
</Defs>

佈局座標的心智模型

researchViewX → 水平層級

對應科技的「深度」。前置在左、後續在右。原版慣例：每個 tier 約間隔 1～2 格。X 必須大於所有 prerequisites 的 X，否則連線會往回畫，視覺混亂。

researchViewY → 垂直分流

同一層的並列研究用不同 Y 分開避免重疊。Y 沒有語意，純為排版。若兩節點 X 相同且 Y 太近，連線箭頭會打結。

用除錯模式校正佈局 開啟 Dev Mode 後在研究視窗右上有座標輔助；也可用 Debug Actions → Research → Finish all 快速驗證解鎖鏈。佈局重疊時，RimWorld 不會報錯，只會把節點疊在一起——務必開遊戲目視檢查。
§3 從 C# 查詢與解鎖研究

執行期的研究狀態由 Find.ResearchManager（ResearchManager 型別，掛在 Game 上）管理。每個 ResearchProjectDef 也提供便捷的查詢屬性。

查詢狀態

using RimWorld;
using Verse;

ResearchProjectDef proj = DefDatabase<ResearchProjectDef>.GetNamed("MyMod_BasicEnergetics");

// 是否已完成（屬性，內部查 ResearchManager 的進度表）
bool done = proj.IsFinished;

// 目前進度（0~1）與已累積點數
float pct      = proj.ProgressPercent;          // 0.0 ~ 1.0
float apparent = proj.ProgressApparent;         // 已投入的研究點

// 前置是否全部滿足（可以開始研究嗎）
bool canStart = proj.PrerequisitesCompleted;

// 1.5 起部分查詢需要傳入派系/分析等級，安全寫法用屬性即可

強制解鎖（FinishProject）

// 立即完成整個研究專案——常用於事件獎勵、作弊指令、測試
Find.ResearchManager.FinishProject(proj);

// 1.5 的完整簽章：FinishProject(project, doCompletionDialog, researcher, doCompletionLetter)
Find.ResearchManager.FinishProject(
    proj,
    doCompletionDialog: false,   // 不彈完成對話框
    researcher:         null,
    doCompletionLetter: true);    // 仍寄出解鎖信件

增加部分進度（不直接完成）

// 給研究專案加點（例如「研究情報」事件給予一次性進度）
// 1.5：AddProgress(project, amount, source)；舊版可能為 amount 直接累加。
Find.ResearchManager.AddProgress(proj, 300f);

// 設定當前正在研究的專案（相當於玩家點選）
Find.ResearchManager.SetCurrentProject(proj);

遍歷所有研究 / 反查解鎖內容

// 列出所有尚未完成、但前置已滿足的可研究專案
var available = DefDatabase<ResearchProjectDef>.AllDefsListForReading
    .Where(p => !p.IsFinished && p.PrerequisitesCompleted);

// 反查：某研究解鎖了哪些 ThingDef（用 researchPrerequisites 過濾）
var unlockedThings = DefDatabase<ThingDef>.AllDefsListForReading
    .Where(t => t.researchPrerequisites != null
             && t.researchPrerequisites.Contains(proj));
FinishProject 不會回溯前置 直接對後段研究呼叫 FinishProject 不會自動完成它的 prerequisites。若你的事件要「解鎖一整條鏈」，需自行遞迴走 prerequisites 逐一完成，否則玩家會看到一個「已完成但前置未完成」的怪異狀態（研究樹連線顯示錯亂）。
§4 RecipeDef 深入

RecipeDef（命名空間 RimWorld）描述「一筆生產工單做什麼」：吃什麼原料、產出什麼、花多少工、需要什麼技能。它本身不綁定工作台——綁定關係由 recipeUsers（在 RecipeDef 上）或 recipes（在 ThingDef 上）建立（見 §10 的坑）。

核心欄位一覽

欄位型別說明
ingredients List<IngredientCount> 輸入原料清單，每項含數量 + ThingFilter（見 §5）
fixedIngredientFilter ThingFilter 限制玩家在工單 UI 可勾選的原料集合（白名單）
products List<ThingDefCountClass> 產出物與數量
workAmount float 基礎工作量（work ticks 等價值，未除以工人工作速度）。-1 表示由原料量推算
workSpeedStat StatDef 決定工作快慢的 Stat（如 GeneralLaborSpeed、SmithingSpeed）
workSkill / workSkillLearnFactor SkillDef / float 影響品質與經驗的技能
skillRequirements List<SkillRequirement> 最低技能門檻，不達標的 Pawn 無法接此工單
recipeUsers List<ThingDef> 哪些工作台可執行此配方（反向綁定）
researchPrerequisite(s) ResearchProjectDef / List<…> 解鎖此配方所需研究（單數 / 複數兩個欄位）
workerClass Type 處理生產完成回呼的 C# 類別，預設 RecipeWorker（見 §8）
effectWorking / soundWorking EffecterDef / SoundDef 製作時的特效與循環音效

完整 XML 範例

<!-- Defs/RecipeDefs/Recipes_MyMod.xml -->
<Defs>
  <RecipeDef>
    <defName>MyMod_MakeEnergyCell</defName>
    <label>make energy cell</label>
    <description>用鋼鐵與元件製作能量電池。</description>
    <jobString>Making energy cell.</jobString>
    <workAmount>1200</workAmount>
    <workSpeedStat>GeneralLaborSpeed</workSpeedStat>
    <workSkill>Crafting</workSkill>
    <effectWorking>Smith</effectWorking>
    <soundWorking>Recipe_Smith</soundWorking>

    <ingredients>
      <li>
        <filter>
          <thingDefs>
            <li>Steel</li>
          </thingDefs>
        </filter>
        <count>50</count>
      </li>
      <li>
        <filter>
          <thingDefs>
            <li>ComponentIndustrial</li>
          </thingDefs>
        </filter>
        <count>2</count>
      </li>
    </ingredients>

    <!-- UI 白名單：限制可勾選原料（避免顯示無關物資） -->
    <fixedIngredientFilter>
      <thingDefs>
        <li>Steel</li>
        <li>ComponentIndustrial</li>
      </thingDefs>
    </fixedIngredientFilter>

    <products>
      <MyMod_EnergyCell>1</MyMod_EnergyCell>
    </products>

    <skillRequirements>
      <Crafting>6</Crafting>
    </skillRequirements>

    <researchPrerequisite>MyMod_BasicEnergetics</researchPrerequisite>

    <!-- 反向綁定到工作台 -->
    <recipeUsers>
      <li>MyMod_AssemblyBench</li>
    </recipeUsers>
  </RecipeDef>
</Defs>
workAmount 的單位 workAmount 是「總工作 tick 數」的基準值，但實際耗時要除以工人的 workSpeedStat 與工作台的 WorkTableWorkSpeedFactor。workAmount = 1200 對技能與設備皆為 1.0 的工人約 1200 ticks ≈ 20 遊戲秒。設為 -1 時 RimWorld 會以原料總量推算（罕用，多數配方明確指定）。
§5 IngredientCount 與 ThingFilter

每筆 ingredients li 反序列化成一個 IngredientCount，內含一個 filter（ThingFilter）與一個 count。ThingFilter 是 RimWorld 中極為通用的「物品集合描述器」，同時用於儲存區、工單、貿易、配方原料。

ThingFilter 的篩選維度

thingDefs

逐一列舉允許的具體 ThingDef。最精確的白名單。

categories

以 ThingCategoryDef 整類納入，如 Textiles、MeatRaw、Foods。

stuffCategories

按材質類別篩選（Metallic、Woody、Fabric…），用於「任意金屬」這種配方。

disallowed* / 品質 / 血量

disallowedThingDefs、allowedQualities、allowedHitPointsPercents 可排除特定項或設定品質/耐久區間。

「任意肉類 75 份」配方原料寫法

<ingredients>
  <li>
    <filter>
      <categories>
        <li>MeatRaw</li>
      </categories>
      <disallowedThingDefs>
        <li>Meat_Human</li> <!-- 排除人肉 -->
      </disallowedThingDefs>
    </filter>
    <count>75</count>
  </li>
</ingredients>

<!-- 配套：UI 白名單同步限制可勾選範圍 -->
<fixedIngredientFilter>
  <categories>
    <li>MeatRaw</li>
  </categories>
  <disallowedThingDefs>
    <li>Meat_Human</li>
  </disallowedThingDefs>
</fixedIngredientFilter>

從 C# 動態檢查 ThingFilter

// 判斷某個 Thing 是否符合配方第一個原料的篩選
IngredientCount ing = recipe.ingredients[0];
bool ok = ing.filter.Allows(someThing);

// 該原料總共需要的數量（已考量 nutrition 類特殊計數）
float need = ing.GetBaseCount();

// 程式建立一個 ThingFilter（如動態配方）
var filter = new ThingFilter();
filter.SetAllow(ThingDefOf.Steel, true);
filter.SetAllow(ThingCategoryDefOf.Manufactured, true);
filter.RecalculateDisplayRootCategory();   // 重建 UI 樹
ingredients filter vs fixedIngredientFilter ingredients[i].filter 決定「該原料槽位接受什麼」；fixedIngredientFilter 決定「玩家在工單 UI 能勾選哪些細項」。兩者必須一致或前者更寬，否則會出現「配方允許但 UI 不給選」或反之的矛盾。漏寫 fixedIngredientFilter 時 RimWorld 會自動以 ingredients 的聯集推導，但顯示可能凌亂。
§6 工作台 ThingDef（billGiver / WorkGiver）

工作台是一個 thingClass = Building_WorkTable 的 ThingDef。Building_WorkTable 實作 IBillGiver，因此它擁有一個 BillStack，玩家可在其上排工單。Pawn 端則由 WorkGiver_DoBill 掃描所有 billGiver、找到可做的工單並指派工作。

工作台 ThingDef 範例

<!-- Defs/ThingDefs_Buildings/Buildings_MyMod.xml -->
<Defs>
  <ThingDef ParentName="BenchBase">
    <defName>MyMod_AssemblyBench</defName>
    <label>assembly bench</label>
    <thingClass>Building_WorkTable</thingClass>
    <category>Building</category>
    <altitudeLayer>Building</altitudeLayer>
    <passability>PassThroughOnly</passability>
    <useHitPoints>true</useHitPoints>
    <designationCategory>Production</designationCategory>

    <size>(3,1)</size>
    <fillPercent>0.5</fillPercent>

    <statBases>
      <MaxHitPoints>180</MaxHitPoints>
      <WorkToBuild>3000</WorkToBuild>
      <!-- 工作台自身的工作速度加成 -->
      <WorkTableWorkSpeedFactor>1.0</WorkTableWorkSpeedFactor>
    </statBases>

    <costList>
      <Steel>100</Steel>
      <ComponentIndustrial>2</ComponentIndustrial>
    </costList>

    <!-- 需要電力：加上 CompPowerTrader -->
    <comps>
      <li Class="CompProperties_Power">
        <compClass>CompPowerTrader</compClass>
        <basePowerConsumption>200</basePowerConsumption>
      </li>
    </comps>

    <!-- 解鎖此建築的研究 -->
    <researchPrerequisites>
      <li>MyMod_AdvancedConduction</li>
    </researchPrerequisites>

    <!-- 正向綁定配方（也可只用 RecipeDef.recipeUsers 反向綁，見 §10） -->
    <recipes>
      <li>MyMod_MakeEnergyCell</li>
    </recipes>
  </ThingDef>
</Defs>

WorkGiver_DoBill 的運作鏈

// 概念流程（無需自己寫，Building_WorkTable 已內建）
WorkGiver_DoBill.JobOnThing(pawn, billGiver)
  → 取得 billGiver.BillStack
  → 逐一檢查每張 Bill：ShouldDoNow() && PawnAllowedToStartAnew()
  → 找到原料（TryFindBestBillIngredients）
  → 產生 JobDriver_DoBill：移動 → 取料 → 在工作台製作 → 產出
billGiver 不一定是工作台 任何實作 IBillGiver 的 ThingDef 都能掛工單——例如裝設屍體火化的「火葬電爐」、手術用的「醫療床」（手術其實也是 RecipeDef + Bill）。判斷一個建築能不能排工單，看它的 thingClass 是否實作 IBillGiver，而非它叫不叫「bench」。
§7 Bill 系統（Bill_Production / BillStack）

玩家排的每一筆「做 X 直到 Y」就是一個 Bill_Production（繼承 Bill）。所有工單收在 IBillGiver.BillStack（BillStack 型別）裡。

關鍵型別關係

型別角色
BillStack掛在 billGiver 上的工單容器，含 List<Bill> bills
Bill_Production標準生產工單：含 repeatMode、targetCount、ingredientSearchRadius、儲存篩選
Bill_ProductionWithUft產出「未完成品」(UnfinishedThing) 的工單，如雕像、神經訓練器
RecipeWorkerCounter計算「做到 N 個」時的現有數量（見下）

從 C# 程式化新增工單

using RimWorld;
using Verse;

Building_WorkTable bench = ...;      // 目標工作台
RecipeDef recipe = DefDatabase<RecipeDef>.GetNamed("MyMod_MakeEnergyCell");

// 用 RecipeDef.MakeNewBill() 確保套用 workerClass 對應的 Bill 型別
Bill_Production bill = (Bill_Production)recipe.MakeNewBill();

// 設定「重複做到庫存達 10 個」
bill.repeatMode  = BillRepeatModeDefOf.TargetCount;
bill.targetCount = 10;

// 原料搜尋半徑（格）；999 表示整張地圖
bill.ingredientSearchRadius = 30f;

// 加入工作台的 BillStack
bench.billStack.AddBill(bill);

RecipeWorkerCounter：自訂「做到 N 個」的計數

「做到庫存有 N 個」時，RimWorld 需要知道「現在有幾個」。預設用 products[0] 計數，但若你的配方產出抽象成果（如把幾種物品算成同一類），需自訂 RecipeWorkerCounter。

public class RecipeWorkerCounter_EnergyCells : RecipeWorkerCounter
{
    // 計算地圖上「已存在」多少個目標物（決定是否還要繼續做）
    public override int CountProducts(Bill_Production bill)
    {
        Map map = bill.billStack.billGiver.Map ?? Find.CurrentMap;
        int total = 0;
        total += map.resourceCounter.GetCount(MyModDefOf.MyMod_EnergyCell);
        // 把「高階電池」也計入，2 顆抵 1 顆高階
        total += map.resourceCounter.GetCount(MyModDefOf.MyMod_EnergyCellHi) * 2;
        return total;
    }

    // 計數對象的可讀標籤（顯示在工單 UI）
    public override string ProductsDescription(Bill_Production bill)
        => "energy cells (incl. hi-cap)";
}
<!-- 在 RecipeDef 指定 counter -->
<RecipeDef>
  ...
  <workerCounterClass>MyMod.RecipeWorkerCounter_EnergyCells</workerCounterClass>
</RecipeDef>
§8 自訂 RecipeWorker

RecipeWorker（workerClass，預設 RecipeWorker）是配方的「邏輯處理器」。當你需要在生產完成時插入額外行為（給經驗、觸發事件、變更產出、消耗額外資源），覆寫它最乾淨——比 Harmony patch GenRecipe.MakeRecipeProducts 安全得多。

常見覆寫點

Notify_IterationCompleted

每完成「一次製作迭代」呼叫。最常用——在此給技能經驗、觸發成就、隨機掉落副產物、消耗 Hediff 資源。

MakeRecipeProducts

覆寫產出生成本身。回傳 IEnumerable<Thing>，可依工人技能、原料品質動態決定產出種類與數量。

ConsumeIngredients

覆寫原料消耗邏輯（少見）。例如保留部分原料、或依機率不消耗。

AvailableOnNow / GetIngredientCount

條件式可用性（如僅夜間）、或動態調整所需原料量。

完整範例：完成時給額外經驗 + 機率副產物

using System.Collections.Generic;
using RimWorld;
using Verse;

public class RecipeWorker_EnergyCell : RecipeWorker
{
    // 每完成一次製作迭代後呼叫
    public override void Notify_IterationCompleted(
        Pawn billDoer, List<Thing> ingredients)
    {
        base.Notify_IterationCompleted(billDoer, ingredients);
        if (billDoer == null) return;

        // ① 額外給 Intellectual 經驗（製作高科技品的副學習）
        billDoer.skills?.Learn(SkillDefOf.Intellectual, 60f);

        // ② 20% 機率掉落一個「廢料」副產物
        if (Rand.Chance(0.2f))
        {
            Thing scrap = ThingMaker.MakeThing(MyModDefOf.MyMod_Scrap);
            scrap.stackCount = 3;
            GenPlace.TryPlaceThing(
                scrap, billDoer.Position, billDoer.Map,
                ThingPlaceMode.Near);
        }

        // ③ 觸發世界統計 / 自訂事件
        Find.World.GetComponent<WorldComp_MyStats>()
            ?.RegisterEnergyCellMade();
    }

    // 動態調整產出：技能 ≥ 12 的工人額外多產 1 個
    public override IEnumerable<Thing> MakeRecipeProducts(
        RecipeDef recipeDef, Pawn worker,
        List<Thing> ingredients,
        Thing dominantIngredient, IBillGiver billGiver,
        Precept_ThingStyle precept = null,
        ThingStyleDef style = null, int? overrideGraphicIndex = null)
    {
        foreach (Thing t in base.MakeRecipeProducts(
            recipeDef, worker, ingredients, dominantIngredient,
            billGiver, precept, style, overrideGraphicIndex))
        {
            yield return t;
        }

        if (worker?.skills?.GetSkill(SkillDefOf.Crafting).Level >= 12)
        {
            Thing bonus = ThingMaker.MakeThing(MyModDefOf.MyMod_EnergyCell);
            yield return bonus;
        }
    }
}
<!-- 在 RecipeDef 掛上 workerClass -->
<RecipeDef>
  <defName>MyMod_MakeEnergyCell</defName>
  <workerClass>MyMod.RecipeWorker_EnergyCell</workerClass>
  ...
</RecipeDef>
MakeRecipeProducts 的簽章隨版本變動 此方法的參數列在 1.3 → 1.4 → 1.5 多次新增（precept、style、overrideGraphicIndex 都是後加的）。覆寫時務必對照你目標版本反編譯的真實簽章，否則 override 不會生效（變成多載新方法、原方法照跑 base）。若要跨版本，用 #if 切換簽章或只覆寫穩定的 Notify_IterationCompleted。
§9 StatDef 與生產效率

生產耗時 = workAmount ÷ (工人 workSpeedStat × 工作台 WorkTableWorkSpeedFactor)。要做出「升級版工作台更快」「特定 Pawn 製作更快」的效果，重點都在操作 StatDef。

生產相關的關鍵 StatDef

StatDef作用
WorkSpeedGlobalPawn 的全域工作速度（受傷、心情、義體影響）
GeneralLaborSpeed泛用勞動速度，多數工作台配方的 workSpeedStat
SmithingSpeed / TailoringSpeed / DrugSynthesisSpeed 等特定工種的速度乘數
WorkTableWorkSpeedFactor工作台自身的速度倍率（建築 statBase）
FoodPoisonChance / ButcheryFleshEfficiency產出品質/效率類 Stat（非速度）

自訂 StatDef + StatPart：依電力狀態調整速度

StatPart 讓你在「基礎值」之上加入動態修正。下例：工作台通電時速度 +20%，斷電（用備援）時 -50%。

<!-- Defs/StatDefs/Stats_MyMod.xml：可選的自訂 Stat -->
<StatDef>
  <defName>MyMod_AssemblySpeed</defName>
  <label>assembly speed</label>
  <category>BasicsPawn</category>
  <defaultBaseValue>1.0</defaultBaseValue>
  <minValue>0.1</minValue>
  <toStringStyle>PercentZero</toStringStyle>
  <parts>
    <li Class="MyMod.StatPart_PowerBonus">
      <poweredFactor>1.2</poweredFactor>
      <unpoweredFactor>0.5</unpoweredFactor>
    </li>
  </parts>
</StatDef>
using RimWorld;
using Verse;

public class StatPart_PowerBonus : StatPart
{
    public float poweredFactor   = 1f;
    public float unpoweredFactor = 1f;

    // 修改最終數值
    public override void TransformValue(StatRequest req, ref float val)
    {
        if (!req.HasThing) return;
        CompPowerTrader power = req.Thing.TryGetComp<CompPowerTrader>();
        if (power == null) return;
        val *= power.PowerOn ? poweredFactor : unpoweredFactor;
    }

    // 滑鼠懸停時的解釋文字
    public override string ExplanationPart(StatRequest req)
    {
        if (!req.HasThing) return null;
        CompPowerTrader power = req.Thing.TryGetComp<CompPowerTrader>();
        if (power == null) return null;
        return power.PowerOn
            ? $"通電加成: x{poweredFactor:F2}"
            : $"斷電懲罰: x{unpoweredFactor:F2}";
    }
}

從 C# 讀取生產速度

// 取 Pawn 的某項生產速度（含所有 StatPart 與 Hediff 修正）
float speed = pawn.GetStatValue(StatDefOf.GeneralLaborSpeed);

// 取建築的工作台速度倍率
float tableFactor = bench.GetStatValue(StatDefOf.WorkTableWorkSpeedFactor);

// 估算一筆工單的實際耗時（ticks）
float ticks = recipe.WorkAmountTotal(stuffDef: null)
            / (speed * tableFactor);
StatPart vs 直接 statBases 靜態固定值用 statBases（如建築固定 1.2x）；需要依執行期狀態（電力、天氣、Hediff、附近建築）動態變化時才用 StatPart。StatPart 是官方提供、相容性最佳的動態 Stat 機制，遠優於 Harmony patch StatWorker.GetValue。
§10 常見坑

坑 1：recipeUsers vs ThingDef.recipes 的雙向綁定

配方與工作台的關聯有兩種寫法，兩者等價但作用方向相反：

RecipeDef.recipeUsers

在配方上列出可用它的工作台。適合「我新增配方，要塞進別人（含原版）的工作台」——無需碰原版 ThingDef。

ThingDef.recipes

在工作台上列出它能做的配方。適合「我新增工作台，掛上自己的配方」。

把配方加進「原版工作台」務必用 recipeUsers 或 PatchOperation 若你想讓配方出現在原版 ElectricSmelter 上，不要直接改寫原版 ThingDef 的 recipes（會與其他 mod 衝突、且需整段覆蓋）。正確做法是在你的 RecipeDef 寫 <recipeUsers><li>ElectricSmelter</li></recipeUsers>，或用 PatchOperationAdd 對原版 ThingDef 的 recipes 節點追加。兩種綁定都會被收進同一份運行期清單（ThingDef.AllRecipes），不會重複觸發，但同一配方同時寫進兩邊會在某些版本造成清單出現兩次——擇一即可。

坑 2：研究前置循環 / 跨檔載入順序

prerequisites 形成環會導致整條鏈永遠無法研究 A 的前置是 B、B 的前置又是 A，RimWorld 不會在載入時報明確錯誤，只會讓兩者都顯示為「鎖定」且永遠無法開始。hiddenPrerequisites 也計入循環判定。設計多分支科技樹時，務必確保是有向無環圖（DAG）。
跨檔引用不受檔名順序影響，但 defName 必須存在 RimWorld 先載入所有 Def 再解析交叉引用，所以 prerequisites 指向哪個檔案的研究都行。但若 defName 拼錯或指向不存在的 Def，會在啟動日誌出現 Could not resolve cross-reference，該欄位變 null——研究會直接「沒有前置」而非報錯卡住，容易漏看。發布前務必檢查紅字日誌。

坑 3：解鎖了配方卻沒解鎖工作台

研究只解鎖「掛了該 researchPrerequisite 的 Def」 常見錯誤是：研究完成，配方解鎖了，但放置工作台的研究忘了設定，導致玩家「能做卻沒地方做」。確認三者前置一致：① 工作台 ThingDef 的 researchPrerequisites；② 配方 RecipeDef 的 researchPrerequisite；③ 產物 ThingDef（若需單獨解鎖）。三者通常指向同一個研究專案。

坑 4：fixedIngredientFilter 漏寫導致 UI 空白

用 categories 當原料時，記得同步 fixedIngredientFilter 若 ingredients 用 categories 但 fixedIngredientFilter 只列了部分 thingDefs，玩家在工單詳情會看到「允許的原料」清單與實際可用原料對不上，甚至勾不到該勾的項目。最穩做法：fixedIngredientFilter 覆蓋範圍 ≥ ingredients 各槽 filter 的聯集。
