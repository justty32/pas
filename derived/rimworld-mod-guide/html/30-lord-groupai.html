群體 AI 的全貌

第 29 章講的是「一個 pawn 怎麼決定下一個 Job」——那是個體層級的 Verse.AI.ThinkTree。 但很多行為是群體層級的：突襲隊要先在地圖邊緣集結、一起推進、損失過半就一起撤退；訪客團要一起走進殖民地、待一陣子、再一起離開；典禮要所有參與者圍著篝火坐定。 這些「一群人共享一個劇本、彼此協調狀態」的需求，個體思維樹根本表達不出來——它沒有「全隊」這個概念。

RimWorld 用 Verse.AI.Group.Lord 解決這個問題。Lord 是一個掛在地圖上的隱形指揮官（沒有實體、不佔格、不會被攻擊）， 內部是一台狀態機。它管理一群 pawn，並在每個狀態下對每個成員下達 PawnDuty（職責）。 個體思維樹再透過 ThinkNode_Duty 把這個 duty 翻譯成具體的 Job。換句話說：

Lord（指揮官）

群體狀態機的執行期實例，掛在 Map.lordManager.lords。持有成員清單、當前 LordToil、Faction。

LordJob（劇本）

整段群體行為的定義 + 狀態圖。覆寫 CreateGraph() 產出一台 StateGraph。

LordToil（狀態）

群體當下處於的一個狀態，負責對每個成員指派 DutyDef。

關鍵命名空間 Lord、LordJob、LordToil、LordMaker、StateGraph、Transition、Trigger 全部在 Verse.AI.Group； 個別的內建 LordJob（如 LordJob_AssaultColony）與 LordToil（如 LordToil_AssaultColony）大多在 RimWorld 命名空間，但仍繼承自 Verse.AI.Group 的基底。 PawnDuty 與 DutyDef 在 Verse.AI；pawn.mindState.duty 是 pawn 與 Lord 之間的橋。
§1 為何需要 Lord：個體思維樹的天花板

回顧第 29 章：每個 pawn 每隔一段時間（或被打斷時）會跑一次 Pawn_JobTracker.DetermineNextJob， 由 ThinkTreeDef 決定下一個 Job。這個過程是純個體的——pawn A 不知道 pawn B 此刻在想什麼，更不可能讓十個人「同時」決定撤退。 你當然可以在每個 pawn 的思維樹裡塞條件，但要做到下面這些事，個體思維樹會立刻爆炸：

  • 共享的群體階段：整隊要先「集結」到位才能「進攻」，這個「到位了沒」必須有人統一判斷。
  • 群體層級的轉移條件：「死傷過半就全隊逃」——這需要一個能看見全隊存活率的觀察者。
  • 角色分工：典禮中有人是主持、有人是觀眾、有人演奏；突襲中有人攻擊、有人放火、有人守橋。
  • 外部事件響應：建築被破壞、有成員受傷、時間到了——這些 signal 要能改變整群人的行為。

Lord 把上面這些全部收進一台狀態機。下面用表格把兩種 AI 的責任邊界劃清楚：

面向個體 AI（第 29 章）群體 AI（Lord，本章）
核心型別ThinkTreeDef / ThinkNode / JobGiverLordJob / StateGraph / LordToil
作用範圍單一 pawn 的下一個 Job一群 pawn 共享的階段與職責
掛載點pawn.thinker（每個 pawn 一份）map.lordManager.lords（每群一個 Lord）
狀態幾乎無狀態（每次重新評估）有狀態（當前 LordToil，可記憶）
協調能力無——pawn 互不知情有——統一指派 duty、統一轉移
轉移條件思維樹節點的 priority / 條件Transition + Trigger
輸出直接是 Job是 PawnDuty（再由思維樹轉成 Job）
典型用途吃飯、睡覺、工作、戰鬥反應突襲、撤退、訪客、商隊、典禮、逃難
銜接點：duty 是兩者的接縫 Lord 不直接寫 pawn.jobs.StartJob(...)。它只設 pawn.mindState.duty， 然後依靠該 pawn 思維樹裡的 ThinkNode_Duty 把 duty 展開成 Job。所以群體 AI 與個體 AI 是分層協作， 不是互相取代——詳見 §5 與第 29 章 §6。

一張地圖上同時存在很多 Lord

不要把 Lord 想成「整張地圖一個」。突襲隊一個、第二波增援又一個、來訪商隊一個、正在舉行的典禮一個—— 它們各自獨立、互不干涉，全部掛在 Map.lordManager 底下。一個 pawn 在任一時刻最多屬於一個 Lord （以 lord.ownedPawns 記錄成員，pawn 端可用 pawn.GetLord() 反查）。 

// 取得某 pawn 當前所屬的 Lord（沒有則回傳 null）
Lord lord = pawn.GetLord();   // 擴充方法，等價於 map.lordManager.LordOf(pawn)

// 列出地圖上所有 Lord
foreach (Lord l in map.lordManager.lords)
{
    Log.Message($"Lord faction={l.faction?.Name} job={l.LordJob.GetType().Name} pawns={l.ownedPawns.Count}");
}
§2 三大角色：LordJob / LordToil / Transition + Trigger

理解 Lord 系統，關鍵是看清三個型別怎麼分工。下面先用一句話定位，再逐個展開。

LordJob ＝ 劇本

整段行為的定義。它的 CreateGraph() 回傳一台 StateGraph，裡面裝著所有 LordToil 與 Transition。一個 Lord 持有一個 LordJob。

LordToil ＝ 當下狀態

狀態圖的一個節點。進入時 Init()、停留時每 tick LordToilTick()，並負責 UpdateAllDuties() 給每個成員指派 duty。

Transition ＋ Trigger ＝ 轉移

Transition(from, to) 連接兩個 LordToil；掛在它上面的 Trigger 判斷「什麼時候要轉」（受傷、時間到、損失過半…）。

LordJob

LordJob 是抽象基底（Verse.AI.Group.LordJob）。你要做的群體行為，就是寫一個子類，覆寫 CreateGraph()。 Lord 建立時會呼叫一次 CreateGraph()，把回傳的 StateGraph 存起來，之後就靠它運轉。重要成員：

成員說明
StateGraph CreateGraph()必覆寫。建立並回傳狀態圖（含 LordToil 與 Transition）。
Lord lord反指回持有它的 Lord（基底已注入，子類可直接用 lord.ownedPawns 等）。
bool LostImportantReferenceDuringLoading存檔載入時若關鍵引用遺失（如目標已不存在），回傳 true 讓 Lord 被清掉。
void ExposeData()序列化自己的欄位（攻擊目標、計時器等）。狀態圖不存檔，載入時重建——見 §8。
bool ShouldRemovePawn(...)判斷某 pawn 是否該被踢出此 Lord（預設依死亡/離圖等）。
bool CanOpenAnyDoor / AddFleeToil常見可覆寫的旗標，影響成員開門權、是否自動補一個逃跑狀態。

LordToil

LordToil（注意是 toil，借用 Job 系統的詞）代表群體當下的一個狀態。它最重要的職責是 UpdateAllDuties()——在進入狀態時、以及成員變動時被呼叫，為每個 lord.ownedPawns 設定 duty。生命週期：

方法時機典型用途
Init()剛進入此 toil（一次）初始化計時器、播放音效、記錄起始狀態
UpdateAllDuties()進入時 + 成員增減時為每個 pawn 設 mindState.duty（核心）
LordToilTick()停留期間每 tick持續性邏輯（如逐 tick 檢查聚集度、推進進度）
Cleanup()離開此 toil釋放資源、停止音效
data (LordToilData)—需要存檔的 toil 狀態放這裡（會被 ExposeData）

Transition 與 Trigger

Transition 是一條有向邊：new Transition(sourceToil, targetToil)。 你把一個或多個 Trigger 加到它身上（transition.AddTrigger(...)）；任一 Trigger 觸發，Lord 就從 source 切到 target。 還可加 preActions（轉移瞬間執行的動作，如清空計時、廣播訊息）。Trigger 是判斷邏輯，常見幾種會在 §7 列表，這裡先看連線形狀： 

StateGraph graph = new StateGraph();

// 兩個狀態
LordToil stage  = new LordToil_DefendPoint(stageLoc);  // 直接 new 一個內建 LordToil
graph.AddToil(stage);
LordToil assault = new LordToil_AssaultColony();
graph.AddToil(assault);

// 一條轉移：聚集完成 → 進攻
Transition t = new Transition(stage, assault);
t.AddTrigger(new Trigger_TicksPassed(2500));        // 等待約 40 秒
t.AddPreAction(new TransitionAction_Message("突襲隊開始進攻！"));
graph.AddTransition(t);
狀態圖是「資料」，不是執行緒 StateGraph 只是節點 + 邊的集合；真正驅動它的是 Lord.LordTick()（由 LordManager 每 tick 呼叫）。 每 tick，Lord 會檢查當前 toil 出發的所有 Transition 的 Trigger，命中就轉移。
§3 建立與管理 Lord（LordMaker / AddPawn / Notify_PawnLost）

Lord 不是直接 new 出來的，而是透過工廠 Verse.AI.Group.LordMaker。最常用的入口： 

public static Lord MakeNewLord(
    Faction faction,          // 這群人屬於哪個派系（可 null，如野生動物群、機械群）
    LordJob lordJob,          // 劇本（你的子類）
    Map map,                  // 掛在哪張地圖
    IEnumerable<Pawn> startingPawns = null);  // 初始成員（可後補）

它會建立 Lord、呼叫 lordJob.CreateGraph()、把 Lord 註冊進 map.lordManager，並把 startingPawns 一一 AddPawn。 典型用法（自己 spawn 一隊人然後組隊）： 

using Verse;
using Verse.AI.Group;
using RimWorld;

// 假設 pawns 已經 GenSpawn 到 map 上、屬於 enemyFaction
List<Pawn> pawns = ...;
Faction enemyFaction = ...;
Map map = ...;

// 1) 建立 LordJob（你的子類或內建）
LordJob lordJob = new LordJob_AssaultColony(enemyFaction);

// 2) 建立 Lord 並把整隊塞進去
Lord lord = LordMaker.MakeNewLord(enemyFaction, lordJob, map, pawns);

// 此後這群 pawn 的群體行為就由 lord 接管

成員的加入與離開

Lord 的成員清單是 lord.ownedPawns。動態增減用下面這組方法——它們會自動觸發當前 toil 的 UpdateAllDuties()，讓新人立刻拿到 duty：

方法作用
lord.AddPawn(pawn)把 pawn 加入此 Lord；會設 pawn.mindState.duty 並呼叫 UpdateAllDuties。
lord.AddPawns(IEnumerable<Pawn>)批次加入（內部逐一 AddPawn）。
lord.Notify_PawnLost(pawn, PawnLostCondition)通知 Lord 某成員「失去」（死亡、被俘、離圖、叛變…）。Lord 會移出清單並可能觸發轉移。
lord.RemovePawn(pawn)單純移出清單，不帶「失去原因」語意（較少直接用）。
lord.CurLordToil讀當前 toil。
lord.GotoToil(toil)強制切換到指定 toil（繞過 Transition，少用）。

PawnLostCondition（Verse.AI.Group 列舉）描述「怎麼失去的」，影響 Lord 是否視同傷亡、是否計入撤退判斷：

值語意
Killed死亡——計入「死傷」類 Trigger。
Incapped倒地失能。
MadePrisoner被俘。
ExitedMap正常離圖（撤退完成、訪客離開）。
ChangedFaction叛變/招降。
LeftVoluntarily自願脫隊。
不要手動改 ownedPawns 請一律走 AddPawn / Notify_PawnLost。直接動 lord.ownedPawns.Remove(...) 會跳過 duty 重算與轉移檢查， 導致 pawn 卡在舊 duty 或 Lord 算錯存活率。RimWorld 內部各處（戰死、被俘、離圖）都是呼叫 Notify_PawnLost 來通知 Lord 的。

Lord 與 Faction 的關係

lord.faction 決定這群人對殖民地的敵友態度與 pawn 思維樹的解讀。三種常見組合：

敵對派系

突襲、圍攻、機械叢集。faction 與玩家為敵；成員思維樹會把「攻擊殖民地」類 duty 正常執行。

友好/中立派系

訪客、商隊、求援團。faction 友好；duty 多為移動、社交、交易、離開。

無派系（null）

野生動物遷徙、瘋狂動物群、某些異常實體。Lord 仍可協調移動，但無外交語意。

生成敵對 pawn 通常配合 RimWorld.PawnGroupMakerUtility.GeneratePawns(...)（見第 11 章 Pawn 生成）， 再用 GenSpawn.Spawn 放到地圖邊緣，最後 MakeNewLord。整個 raid 流程的核心就是這三步——這也是 §7 要講的 IncidentWorker 內幕。

§4 寫一個自訂 LordJob：集結 → 攻擊 → 死傷過半則逃

現在把前面拼起來，寫一個完整可用的自訂 LordJob 骨架。劇本：突襲隊先到一個集結點待命，等聚齊或時間到就轉入「攻擊據點」， 若損失達到一定比例就全隊「離圖逃跑」。這展示了 CreateGraph() 的標準寫法、Transition 的串接、以及 ExposeData 的存檔處理。

骨架程式碼

using System.Collections.Generic;
using Verse;
using Verse.AI;
using Verse.AI.Group;
using RimWorld;

public class LordJob_StageThenAssault : LordJob
{
    // 需要存檔的欄位（攻擊集結點、損失閾值）
    private IntVec3 stageLoc;
    private float retreatThreshold = 0.5f;

    // 無參數建構子是「存檔載入」用的——必須有！
    public LordJob_StageThenAssault() { }

    public LordJob_StageThenAssault(IntVec3 stageLoc, float retreatThreshold = 0.5f)
    {
        this.stageLoc = stageLoc;
        this.retreatThreshold = retreatThreshold;
    }

    public override StateGraph CreateGraph()
    {
        StateGraph graph = new StateGraph();

        // === 狀態 1：集結待命（圍著 stageLoc 防守，不主動進攻）===
        LordToil_DefendPoint stage = new LordToil_DefendPoint(stageLoc);
        graph.AddToil(stage);   // 第一個加入的 toil 預設為 StartingToil

        // === 狀態 2：進攻殖民地 ===
        LordToil_AssaultColony assault = new LordToil_AssaultColony();
        graph.AddToil(assault);

        // === 狀態 3：全隊離圖逃跑 ===
        LordToil_ExitMap exit = new LordToil_ExitMap(LocomotionUrgency.Jog, canDig: false);
        graph.AddToil(exit);

        // --- 轉移 A：集結完成（時間到）→ 進攻 ---
        Transition toAssault = new Transition(stage, assault);
        toAssault.AddTrigger(new Trigger_TicksPassed(2500));   // 約 40 秒
        toAssault.AddPreAction(new TransitionAction_Message(
            "敵軍開始進攻！", MessageTypeDefOf.ThreatBig));
        graph.AddTransition(toAssault);

        // --- 轉移 B：死傷達閾值 → 逃跑（從「集結」與「進攻」都可觸發）---
        foreach (LordToil from in new[] { stage, assault })
        {
            Transition toFlee = new Transition(from, exit);
            toFlee.AddTrigger(new Trigger_FractionPawnsLost(retreatThreshold));
            toFlee.AddPreAction(new TransitionAction_Message(
                "敵軍損失慘重，開始撤退！", MessageTypeDefOf.NeutralEvent));
            graph.AddTransition(toFlee);
        }

        return graph;
    }

    // 存檔：只存資料欄位，不存 StateGraph（載入後 CreateGraph 重建）
    public override void ExposeData()
    {
        Scribe_Values.Look(ref stageLoc, "stageLoc");
        Scribe_Values.Look(ref retreatThreshold, "retreatThreshold", 0.5f);
    }
}
無參數建構子是強制的 存檔載入時，Scribe 會先用無參數建構子 new 出 LordJob，再呼叫 ExposeData() 還原欄位， 最後重新呼叫 CreateGraph() 重建狀態圖。若你只寫了帶參數的建構子，載入舊存檔會直接拋例外（MissingMethodException / 反射失敗）。

StartingToil

graph.AddToil(toil) 加入的第一個 toil 自動成為 StartingToil（Lord 建立後的初始狀態）。 若要明確指定，可設 graph.StartingToil = someToil;（setter 會把該 toil 移到 lordToils 的索引 0）。 要在外部 signal 或 preAction 裡切換到特定狀態，自己保留 toil 變數的參照，再用 lord.GotoToil(toil) 跳轉即可。 

// 明確指定起始狀態（不依賴加入順序）
graph.StartingToil = stage;

// 「全域逃跑」toil 由引擎自動掛：當 LordJob.AddFleeToil 屬性為 true（預設）、
// 且派系會逃時，Lord.SetJob 會自動補一個 LordToil_PanicFlee 與對應轉移
善用內建 LordToil 拼裝 你很少需要從零寫 LordToil。LordToil_DefendPoint、LordToil_AssaultColony、LordToil_ExitMap、 LordToil_Travel、LordToil_Stage 等都是現成的——多數自訂 LordJob 只是把這些既有 toil 用不同的 Transition 串起來。 先看內建 job 怎麼拼（§6），再決定要不要自己寫 toil。
§5 LordToil 與 Duty：UpdateAllDuties / DutyDef / ThinkNode_Duty

LordToil 真正「指揮」pawn 的方式是設定 pawn.mindState.duty。 這個 duty 是一個 Verse.AI.PawnDuty，包著一個 DutyDef 與一些參數（焦點目標、半徑等）。 個體思維樹裡的 ThinkNode_Duty 看到這個 duty，就跳進對應的子思維樹去產生 Job。

UpdateAllDuties()：核心

每個 LordToil 覆寫 UpdateAllDuties()，遍歷 lord.ownedPawns 給每人指派 duty。最小自訂 toil 長這樣： 

public class LordToil_GatherAroundFire : LordToil
{
    private IntVec3 fireSpot;

    public LordToil_GatherAroundFire(IntVec3 fireSpot)
    {
        this.fireSpot = fireSpot;
    }

    public override void UpdateAllDuties()
    {
        foreach (Pawn p in lord.ownedPawns)
        {
            // 給每個成員「在火堆附近閒晃 / 圍坐」的 duty
            p.mindState.duty = new PawnDuty(DutyDefOf.Idle, fireSpot, radius: 6f);
        }
    }
}

PawnDuty 常用建構參數與欄位：

欄位型別說明
defDutyDef職責類型（決定走哪段思維樹）。
focus / focusSecondLocalTargetInfo主要/次要焦點目標（一個格子、一個 thing）。null 會出事——見 §8。
radiusfloat圍繞 focus 的活動半徑（如防守、閒晃範圍）。
locomotionLocomotionUrgency移動急迫度（Walk / Jog / Sprint）。
canDigDutyDef 設定是否允許挖牆（圍攻常開）。

常見內建 DutyDef

DutyDef 定義在 XML（RimWorld.DutyDefOf 提供常用的強型別引用）。每個 DutyDef 內含 thinkNode——指向一段子思維樹，那才是 duty 真正展開成 Job 的地方。速查：

DutyDef行為典型 LordToil
DutyDefOf.Defend守在 focus 附近，反擊來犯者LordToil_DefendPoint
DutyDefOf.AssaultColony主動尋敵、攻擊殖民地目標LordToil_AssaultColony
DutyDefOf.ExitMapBest走最近/最安全的地圖邊離開LordToil_ExitMap
DutyDefOf.ExitMapBestAndDefendSelf離圖但被打會還手撤退/驚走
DutyDefOf.Idle原地/小範圍閒晃集結、典禮觀眾、訪客待機
DutyDefOf.TravelOrLeave朝目標移動，否則離開商隊、過境
DutyDefOf.Escort護衛某個 focus pawn護衛任務
DutyDefOf.Steal偷竊竊盜團

duty 怎麼被思維樹消費（銜接第 29 章 §6）

pawn 的 ThinkTreeDef 裡通常很靠前就有一個 ThinkNode_Duty。它讀 pawn.mindState.duty.def.thinkNode， 把控制權交給該 DutyDef 自帶的子思維樹。也就是說：Lord 設 duty → 思維樹的 ThinkNode_Duty 攔截 → 跑 DutyDef.thinkNode → 產生 Job。 下面是一個 DutyDef 的 XML 形狀（簡化）： 

<DutyDef>
  <defName>MyMod_GatherAroundFire</defName>
  <thinkNode Class="ThinkNode_Priority">
    <subNodes>
      <!-- 在 focus 附近找格子閒晃 -->
      <li Class="JobGiver_WanderNearDutyLocation">
        <ticksBetweenWandersRange>300~600</ticksBetweenWandersRange>
      </li>
    </subNodes>
  </thinkNode>
</DutyDef>
分層的好處 因為 duty 只是「意圖」、思維樹才負責「怎麼做」，所以同一個 DutyDef 在不同 pawn（人類 vs 機械 vs 動物）身上會用各自的思維樹展開， 自動尊重各 pawn 的能力差異。你寫 LordToil 時只要表達「我要他幹嘛」，不必管「他怎麼幹」。
UpdateAllDuties 不是每 tick 跑 它只在進入 toil、成員增減時被呼叫。若你的 duty focus 需要隨時間移動（如追著玩家跑的集結點）， 要嘛在 LordToilTick() 裡定期手動重設 duty，要嘛用會自己追蹤目標的 DutyDef。別預期 UpdateAllDuties 會自動重跑。
§6 內建可複用的 LordJob 範例

動手寫之前，先盤點 RimWorld 內建的 LordJob——大多數需求都能直接用或子類化。下面挑最常被 mod 重用的幾個（皆在 RimWorld 命名空間，繼承 Verse.AI.Group.LordJob）。

LordJob_AssaultColony

標準突襲。成員直接 AssaultColony，可設定是否能搶東西、是否能綁架、損失過半是否撤退。建構子有多個 bool 旗標控制行為。

LordJob_StageThenAttack

先在某點集結一段時間再進攻——增援、伏擊常用。等同 §4 骨架的官方版。

LordJob_VisitColony

友好訪客團：走進殖民地、待一陣子（社交/閒晃）、再離開。可設停留時間與集合點。

LordJob_Ritual

Ideology 典禮的驅動者。依 RitualBehaviorDef 指派每個參與者的角色 duty（主持、觀眾、表演）。見第 32 章。

LordJob_TradeWithColony

商隊：抵達交易點、開放交易、結束後離開。

LordJob_DefendBase / DefendPoint

固守據點型（如出生點營地、機械叢集核心），被攻擊才反應。

直接重用

// 突襲：能搶能綁，損失 50% 撤退（旗標依版本略有出入，請於 dnSpy 核對建構子簽章）
LordJob assault = new LordJob_AssaultColony(
    assaulterFaction: enemyFaction,
    canKidnap: true,
    canTimeoutOrFlee: true,
    sappers: false,
    useAvoidGridSmart: true,
    canSteal: true,
    breachers: false,
    canPickUpOpportunisticWeapons: true);

Lord lord = LordMaker.MakeNewLord(enemyFaction, assault, map, pawns);

子類化擴充

若內建行為八成符合，只差一兩個轉移，最划算的做法是繼承並在 CreateGraph() 裡先 base.CreateGraph()、再補轉移。 但注意：很多內建 LordJob 的 CreateGraph 不是設計成可乾淨擴充的，常見做法反而是整段複製改寫。 

public class LordJob_AssaultColony_BurnFirst : LordJob_AssaultColony
{
    public LordJob_AssaultColony_BurnFirst() { }
    public LordJob_AssaultColony_BurnFirst(Faction f) : base(f, true, true) { }

    public override StateGraph CreateGraph()
    {
        StateGraph graph = base.CreateGraph();
        // 在 graph 上補一個「先放火」前置狀態與轉移……
        // （實務上常因 base graph 的內部 toil 難以引用而改為整段重寫）
        return graph;
    }
}
dnSpy 是你最好的範本庫 想做某種群體行為？先在 dnSpy 搜 LordJob_，找最接近的那個，讀它的 CreateGraph。 你會學到該用哪些 LordToil、哪些 Trigger、怎麼串。RimWorld 的群體 AI 幾乎全是這套積木拼出來的，照抄改參數即可。
§7 觸發器與事件：Trigger 速查 / Signal / IncidentWorker

Transition 何時觸發，全看掛在它上面的 Trigger。Trigger 有兩類判斷模型： 一是狀態輪詢（每 tick 由 Lord 問「現在該轉了嗎」），二是signal 驅動（外部事件丟一個 TriggerSignal 進來）。

常用 Trigger 速查

Trigger觸發條件模型
Trigger_TicksPassed進入當前 toil 後經過 N tick輪詢
Trigger_TicksPassedAndNoRecentHarm時間到且最近沒受傷輪詢
Trigger_FractionPawnsLost損失成員比例 ≥ 閾值輪詢
Trigger_PawnHarmed有成員受到傷害signal
Trigger_PawnLost有成員死亡/離隊signal
Trigger_PawnCannotReachMapEdge有成員找不到出口（被圍困）輪詢
Trigger_BecameNonHostileToPlayer派系關係轉為非敵對（招安）輪詢
Trigger_Memo收到指定字串的 memo signalsignal
Trigger_OnClamor聽到喧鬧（爆炸、開門被吵醒）signal
Trigger_UrgentlyHungry成員飢餓緊急（長期駐留群）輪詢

Signal 機制：外部事件如何影響 Lord

Signal 驅動的 Trigger 不會自己輪詢，而是等別人把訊號送進來。送法有兩種：

  • 系統級通知：傷害、死亡、開門等由引擎自動廣播。例如某 pawn 受傷時，Pawn / DamageWorker 路徑會呼叫 lord.Notify_PawnDamaged(...) / lord.Notify_PawnLost(...)，Lord 再轉成 TriggerSignal 餵給每條 Transition 的 Trigger。
  • 自訂 memo：你可以從任何地方呼叫 lord.ReceiveMemo("MyEvent")，配合 Trigger_Memo("MyEvent") 觸發自訂轉移—— 這是 mod 讓「按鈕、研究完成、建築被毀」等任意事件改變群體行為的標準手段。
// 在任意位置丟一個 memo，讓掛了 Trigger_Memo("EnrageNow") 的轉移立刻觸發
lord.ReceiveMemo("EnrageNow");

// 對應的轉移（在 CreateGraph 裡）
Transition enrage = new Transition(passive, berserk);
enrage.AddTrigger(new Trigger_Memo("EnrageNow"));
graph.AddTransition(enrage);

建築相關事件常見 lord.Notify_BuildingDamaged(...) / Notify_BuildingLost(...)——例如圍攻方的迫擊砲被打掉、 防守方的核心建築被破壞時，引擎會通知對應 Lord，讓「設施被毀就改打法」這類轉移生效。

IncidentWorker：raid 的內幕就是 spawn + MakeNewLord

第 12 章講過 RimWorld.IncidentWorker。一個敵襲事件（如 IncidentWorker_RaidEnemy）做的事，拆開來就是本章的東西：

  1. 用 PawnGroupMakerUtility.GeneratePawns(...) 依預算生成一隊敵人。
  2. 選定進場方式（地圖邊緣步行、空投、出生點）並 GenSpawn.Spawn 到地圖。
  3. 依 RaidStrategyDef 決定要用哪個 LordJob（突襲、圍攻、伏擊…）。
  4. 呼叫 LordMaker.MakeNewLord(faction, lordJob, map, pawns) 把他們組成一個 Lord。
  5. 送出 letter / message 通知玩家，事件結束。

也就是說，RaidStrategyDef（RimWorld.RaidStrategyDef）的 workerClass 內部會 MakeLords(...)，決定生成哪個 LordJob。 要做「全新的敵襲打法」，你新增一個 RaidStrategyDef + 一個 LordJob 子類即可，無需 Harmony patch incident 主流程。

最小自訂 raid 配方 新 LordJob 子類（決定打法）→ 新 RaidStrategyDef（XML，指向你的 LordJob 或自訂 worker）→ （可選）新 IncidentDef / PawnGroupKindDef 調整生成。整條鏈純資料 + 一個小 C# 類，是擴充敵襲最乾淨的路徑。
§8 除錯與存檔重建

Dev mode 的 Lord 視覺化

開啟 Dev mode 後，最有用的是 Debug 繪圖開關。在 Debug 設定（齒輪）面板裡有幾個關鍵項：

開關看到什麼
Draw duties每個 pawn 頭上畫出當前 duty 與 focus 連線——一眼看出誰在守哪、誰在攻誰。
Draw lords把 Lord 的成員與當前 toil 標出來，確認分組是否正確。
Log duty changes / lord toil在 toil 切換時輸出 log，追蹤狀態機是否如預期轉移。

程式碼層面，lord.DebugString()（或在 inspect 面板）可印出當前 toil、成員、各 Transition 的 Trigger 狀態，是排查「為什麼不轉移」的第一站。

常見故障與對策

群體卡在某 toil 不轉移

檢查該 toil 出發的 Transition 是否真的有 Trigger、Trigger 條件是否永遠為 false（如 Trigger_TicksPassed 的 tick 從未被重置、FractionPawnsLost 閾值寫成 0 反而立刻轉走或寫成 1 永不觸發）。

pawn 一直 Idle / 站著不動

多半是 duty 的 focus 為 null，或 focus 指向不可達/已消失的目標，導致對應 JobGiver 回傳 null，pawn 退回 idle。給 duty 一律設有效 focus。

UpdateAllDuties 沒重跑

記得它只在進 toil / 成員增減時跑。需要隨時間更新的 duty（移動目標）要在 LordToilTick() 裡手動重設。

載入存檔後 Lord 消失/報錯

多為 LordJob 缺無參數建構子，或 ExposeData 漏存關鍵欄位導致 LostImportantReferenceDuringLoading 為 true 被清掉。

存檔：StateGraph 不序列化，必須能重建

這是寫 LordJob 最容易踩的坑。RimWorld 不存 StateGraph——它只存 LordJob 的資料欄位（透過 ExposeData）， 載入時用無參數建構子 new 出 LordJob、ExposeData 還原欄位、再呼叫 CreateGraph() 重建整張圖。所以：

  • CreateGraph 必須是純函式：只能依賴已 ExposeData 還原的欄位，不能依賴執行期才有的暫態資料。
  • 需要跨存檔記住的東西（攻擊目標、計時器基準、階段索引）必須放進欄位並 ExposeData，否則載入後歸零。
  • 引用 Thing/Pawn 用 Scribe_References；若該引用可能在載入時已不存在，於 LostImportantReferenceDuringLoading 回傳 true，讓 Lord 被乾淨清除而非崩潰。
private Thing attackTarget;

public override bool LostImportantReferenceDuringLoading
{
    get { return attackTarget == null; }   // 目標沒了就讓 Lord 被移除
}

public override void ExposeData()
{
    Scribe_References.Look(ref attackTarget, "attackTarget");
}

LordToil 本身需要記憶的狀態（如「已經放了幾把火」）也不能直接放欄位——因為 toil 是 CreateGraph 重建出來的新實例。 要存的 toil 狀態放進 LordToilData 子類掛在 toil.data，它會被 Lord 一併 ExposeData，並在重建時按順序還原。

測試一定要含「存檔→讀檔」這一步 很多 Lord bug 只在讀檔後才現形（圖重建出錯、欄位歸零、引用遺失）。 請務必在 Dev mode 觸發你的群體行為 → 中途存檔 → 讀回 → 確認狀態機接續正確。第 16 章的測試流程適用於此。
