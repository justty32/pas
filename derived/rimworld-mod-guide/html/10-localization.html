在地化 Localization

Languages/ 資料夾結構、Keyed 翻譯、DefInjected 注入、Strings 清單、.Translate() 與 placeholder、文法解析器 (Grammar Resolver)、複數/性別處理、缺漏偵測與命名規範。

RimWorld 1.6 · Verse.Translator · LanguageDatabase
1. 在地化心智模型

RimWorld 的在地化系統不是傳統的「key → value 平面字典」。它由三條獨立通道組成，各自對應不同的字串來源。 理解這三條通道的分工，是寫出可維護翻譯的前提。

Keyed

程式碼裡用字串字面量當 key，呼叫 "Key".Translate() 取回譯文。對應 Languages/X/Keyed/*.xml。由 C# 主動觸發。

DefInjected

把 Def 裡的 label / description 等欄位「就地覆寫」成譯文。對應 Languages/X/DefInjected/<DefType>/*.xml。由引擎自動套用，程式碼無感。

Strings

純文字清單檔（每行一筆），供名字產生器、隨機訊息、文法規則讀取。對應 Languages/X/Strings/**。由規則系統消費。

核心類別位置 翻譯入口擴充方法 Translate() 定義於 Verse.Translator；語言資料庫為 Verse.LanguageDatabase， 目前啟用的語言由 LanguageDatabase.activeLanguage 持有，可透過 Find.ActiveLanguageWorker 取得文法處理器。
2. Languages/ 資料夾結構

翻譯檔放在 Mod 根目錄下的 Languages/<語言資料夾名>/。語言資料夾名稱必須與遊戲認得的 LanguageFolderName 完全一致（區分大小寫）。常見對照：

語言資料夾名稱原生名稱
繁體中文ChineseTraditional繁體中文
簡體中文ChineseSimplified简体中文
英文EnglishEnglish
日文Japanese日本語
韓文Korean한국어

English 是回退基準 (fallback) 無論玩家用哪種語言，引擎都會先載入 English 作為基底，再以使用者語言覆蓋。 因此 英文翻譯應視為「完整母本」：所有 Keyed key、所有 DefInjected 欄位都該在 English 裡有定義。 其他語言缺漏的條目會自動回退到英文，不會留空。

LanguageInfo.xml（選用）

若你發布的是「翻譯包」而非單純功能 Mod，可在語言資料夾根放一份 LanguageInfo.xml 標註譯者資訊與文法設定。

<LanguageInfo>
  <friendlyNameNative>繁體中文</friendlyNameNative>
  <friendlyNameEnglish>Chinese (Traditional)</friendlyNameEnglish>
  <canBeApostropheConjugated>false</canBeApostropheConjugated>
  <hasGenders>false</hasGenders>
</LanguageInfo>
3. Keyed Translations

Keyed 翻譯用於程式碼產生的字串：訊息、按鈕標籤、letter 內容、tooltip 等任何由 C# 主動顯示的文字。 檔案放在 Languages/X/Keyed/，可任意命名（建議帶 Mod 前綴），引擎會載入該資料夾下所有 .xml 並合併。

XML 格式

<!-- Languages/English/Keyed/YourMod_Messages.xml -->
<?xml version="1.0" encoding="utf-8"?>
<LanguageData>

  <!-- 簡單字串 -->
  <YourMod_ButtonApply>Apply</YourMod_ButtonApply>
  <YourMod_DialogTitle>Configuration</YourMod_DialogTitle>

  <!-- 帶 placeholder（{0}、{1} …） -->
  <YourMod_RitualComplete>{0} has completed the ritual at {1}.</YourMod_RitualComplete>

  <!-- 帶具名參數（NamedArgument） -->
  <YourMod_PawnGreeting>{PAWN_nameDef} greets the colony.</YourMod_PawnGreeting>

</LanguageData>
<!-- Languages/ChineseTraditional/Keyed/YourMod_Messages.xml -->
<?xml version="1.0" encoding="utf-8"?>
<LanguageData>
  <YourMod_ButtonApply>套用</YourMod_ButtonApply>
  <YourMod_DialogTitle>設定</YourMod_DialogTitle>
  <YourMod_RitualComplete>{0} 已於 {1} 完成儀式。</YourMod_RitualComplete>
  <YourMod_PawnGreeting>{PAWN_nameDef} 向殖民地問候。</YourMod_PawnGreeting>
</LanguageData>

C# 取用

using Verse;

// 最基本：key 字面量 .Translate()，回傳 TaggedString
TaggedString title = "YourMod_DialogTitle".Translate();

// 直接當 string 用（TaggedString 有 implicit 轉 string）
Widgets.Label(rect, "YourMod_ButtonApply".Translate());

// 帶位置參數：依序填入 {0} {1} …
Messages.Message(
    "YourMod_RitualComplete".Translate(pawn.LabelShort, location.GetUniqueLoadID()),
    MessageTypeDefOf.PositiveEvent);
TaggedString 不是 string Translate() 回傳 Verse.TaggedString，內含富文字標記（顏色、連結 hyperlink）。 多數情況可隱式轉成 string，但若你要做字串比較或拼接，建議顯式取 .Resolve() 或 .RawText，避免標記殘留造成比對失敗。
4. DefInjected — 翻譯 Def 欄位

Def 的 label、description 等欄位在主 XML（Defs/）裡通常寫英文。 DefInjected 讓你不動原始 Def，在 Languages/X/DefInjected/<DefType>/ 用「Def 名稱 + 欄位路徑」覆寫成譯文。 路徑結構：DefInjected/<DefType>/<任意檔名>.xml，子節點名稱為 <defName>.<fieldPath>。

原始 Def（不翻譯，維持英文）

<!-- Defs/ThingDefs_Weapons.xml -->
<ThingDef>
  <defName>YourMod_PlasmaRifle</defName>
  <label>plasma rifle</label>
  <description>A rifle that fires superheated plasma bolts.</description>
  <comps>
    <li Class="CompProperties_Quality">
      <label>plasma core</label>
    </li>
  </comps>
</ThingDef>

DefInjected 翻譯

<!-- Languages/ChineseTraditional/DefInjected/ThingDef/Items_Weapons.xml -->
<?xml version="1.0" encoding="utf-8"?>
<LanguageData>

  <!-- <defName>.<欄位> -->
  <YourMod_PlasmaRifle.label>電漿步槍</YourMod_PlasmaRifle.label>
  <YourMod_PlasmaRifle.description>發射過熱電漿彈的步槍。</YourMod_PlasmaRifle.description>

  <!-- 巢狀清單以索引定位：comps.0.label -->
  <YourMod_PlasmaRifle.comps.0.label>電漿核心</YourMod_PlasmaRifle.comps.0.label>

</LanguageData>

欄位路徑語法規則

情境路徑寫法說明
頂層欄位defName.label最常見
巢狀物件欄位defName.parent.child用 . 逐層深入
清單元素（無 key）defName.list.0.label用 0-based 索引
清單元素（有 li 字串）defName.list.0純字串清單，直接指到索引本身
handle 鎖定的清單元素defName.list.handleName節點若有 <li handle="x"> 可用名稱代替索引 建議
索引脆弱性 用數字索引（comps.0.label）定位清單時，一旦你在原始 Def 的清單中間插入新元素，所有後續索引就會錯位、翻譯套到錯的條目。 對需要翻譯的清單元素，在原 Def 加 handle 屬性（<li handle="plasmaCore" ...>）， DefInjected 改用 defName.comps.plasmaCore.label，即可解除對順序的依賴。

純字串清單範例（如 rulesStrings、verbs 等）

<!-- 原 Def 片段 -->
<tradeTags>
  <li>WeaponRanged</li>
</tradeTags>

<!-- 注意：tradeTags 是邏輯標籤，不該翻譯。下面僅示範語法 -->
<!-- 一個「該翻譯的純字串清單」如 RulePack 的 rulesStrings： -->
<YourMod_SomeRulePack.rulesStrings.0>譯文第一條</YourMod_SomeRulePack.rulesStrings.0>
<YourMod_SomeRulePack.rulesStrings.1>譯文第二條</YourMod_SomeRulePack.rulesStrings.1>
自動產生 DefInjected 樣板 開發者模式下開啟 Options → Mod options / Translation，使用 「Save translation report / 偵錯選單的 Output translation related」 工具， RimWorld 會掃描所有 Def 並產生「應翻譯但尚未翻譯」的條目清單，直接複製成 DefInjected 骨架，省去手動列舉每個欄位路徑。
5. Strings/ 資料夾

Strings/ 存放純文字清單（.txt，每行一筆，# 開頭為註解）， 由文法/規則系統與名字產生器讀取。最常見用途是動物/人物名字池與訊息變體。

動物名字檔

# Languages/ChineseTraditional/Strings/Names/Animal/Cat.txt
小花
咪咪
布丁
橘子
# 以 # 開頭的行會被忽略
雪球

程式端透過 RulePackDef 或 NameBank 引用對應的 Strings 路徑。引擎依當前語言尋找該相對路徑， 找不到才回退英文。檔案以 UTF-8（無 BOM）儲存，避免首行出現隱形字元污染第一筆資料。

在規則文法中引用 Strings

<!-- 透過 RulePack 從 Strings 檔載入候選字串 -->
<RulePackDef>
  <defName>YourMod_GreetingPack</defName>
  <rulePack>
    <rulesStrings>
      <li>greeting->Hello</li>
      <li>greeting->Hi there</li>
    </rulesStrings>
  </rulePack>
</RulePackDef>
Strings vs Keyed 的界線 固定、單一的 UI 字串 → Keyed。需要「從候選池隨機抽一條」或「依文法規則組合」的字串（名字、口號、隨機事件描述）→ Strings + RulePack。
6. .Translate() 與 placeholder 替換

Verse.Translator 提供一組擴充方法。下表為資深工程師最常碰到的幾個：

方法回傳用途
"Key".Translate()TaggedString標準翻譯；key 缺漏時回傳帶錯誤標記的 key 本身 最常用
"Key".Translate(args...)TaggedString帶參數，填入 {0}/{1} 或具名參數
"Key".TranslateSimple()string不做文法解析、不處理參數，純查表回傳。效能略高，適合無參數的純靜態字串
"Key".CanTranslate()bool查詢 key 是否存在於當前語言資料庫，用於條件式 fallback
"Key".TranslateWithBackup("Key2")TaggedString主 key 缺漏時改用備援 key
str.Formatted(args...)TaggedString對「已是譯文的字串」再做參數替換，常與 DefInjected 取回的 label 搭配

位置參數替換

// XML: <YourMod_Kills>{0} killed {1} raiders.</YourMod_Kills>
string msg = "YourMod_Kills".Translate(pawn.LabelShort, count);
// → "Randy killed 5 raiders."

// 直接對字面字串做 Formatted（少見，但合法）
string s = "{0} of {1}".Formatted(3, 10);

CanTranslate 條件式回退

string key = $"YourMod_Stage_{stageIndex}";
string text = key.CanTranslate()
    ? key.Translate()
    : "YourMod_Stage_Default".Translate();
TranslateSimple 的取捨 TranslateSimple() 跳過文法解析與參數插值，所以遇到含 {...} 或文法標記的字串會原樣輸出、不替換。 只在你確定該字串是「無參數、無文法」的純靜態文字時使用；其餘一律用 Translate()。
7. 帶參數翻譯與文法解析 (Grammar Resolver)

位置參數 {0} 足以應付簡單句，但碰到「依性別變化的代名詞」「依數量變化的單複數」等情況就需要 文法解析器 (GrammarResolver) 與具名參數 (NamedArgument)。 引擎會把參數連同其關聯的文法規則一起交給 GrammarResolver.Resolve 展開。

NamedArgument — 具名參數

用 obj.Named("PAWN") 把一個物件綁到名稱 PAWN，譯文裡就能用 {PAWN_nameDef}、{PAWN_pronoun}、{PAWN_possessive} 等「子欄位」存取它的文法屬性。 

using Verse;
using Verse.Grammar;

// XML:
// <YourMod_Heal>{HEALER_nameDef} healed {PATIENT_objective}. {PATIENT_pronoun} is grateful.</YourMod_Heal>

string text = "YourMod_Heal".Translate(
    healer.Named("HEALER"),
    patient.Named("PATIENT"));

// 英文輸出："Randy healed her. She is grateful."
// pronoun / objective 等由引擎依 patient 的性別自動帶入

常見文法子欄位（Pawn 為例）

子欄位含義英文範例（依性別）
{X_nameDef}名字（含 def 解析）Randy
{X_pronoun}主格代名詞he / she / they
{X_objective}受格代名詞him / her / them
{X_possessive}所有格his / her / their
{X_gender}性別字串male / female
{X_labelShort}短標籤Randy

直接使用 GrammarResolver（進階）

當你要組合一段由多條規則動態生成的描述（如事件 letter 內文），可手動建立規則集並呼叫 resolver：

using Verse.Grammar;

var request = new GrammarRequest();
request.Includes.Add(YourMod_DefOf.YourMod_EventTextPack); // RulePackDef
request.Rules.AddRange(GrammarUtility.RulesForPawn("PAWN", pawn, request.Constants));

string resolved = GrammarResolver.Resolve(
    "root",           // 起始 symbol
    request,
    debugLabel: "YourMod event");

ActiveLanguageWorker — 語言層級的文法操作

Find.ActiveLanguageWorker 回傳目前語言的 LanguageWorker，封裝了該語言特有的文法規則 （單複數、所有格 's、序數詞、片語連接等）。需要在程式裡手動套用語言規則時透過它： 

var worker = Find.ActiveLanguageWorker;

// 數字 → 該語言的序數詞（English: "1st", "2nd"…）
string ordinal = worker.OrdinalNumber(rank);

// 套用所有格（English: "Randy" → "Randy's"）
string poss = worker.PostProcessed(rawText);

// 以「逗號 + and」連接片語清單，依語言規則
string joined = worker.WithIndefiniteArticle(noun);
為何不要在 C# 裡硬拼文法 像 name + "'s sword" 或 "You killed " + n + " raiders" 這種拼接， 在中文不需要 's、在某些語言複數規則不同——一旦譯者拿到的是被英文文法綁死的片段就無法正確在地化。 把整句交給 Translate() + 具名參數，讓每個語言的 XML 自己決定詞序與文法。
8. 複數與性別處理

性別分歧（pronoun 自動帶入）

只要參數是 Pawn 並以 .Named() 傳入，{X_pronoun} 等子欄位會依該 pawn 性別自動選字， 譯者不必手動分岔。但若某語言代名詞無性別差異（如中文口語常不區分），譯文直接寫固定字即可： 

<!-- English：依性別變化 -->
<YourMod_Rest>{PAWN_nameDef} is resting. {PAWN_pronoun} needs recovery.</YourMod_Rest>

<!-- ChineseTraditional：代名詞不分性別，直接寫「他」即可（或省略） -->
<YourMod_Rest>{PAWN_nameDef} 正在休息，需要時間恢復。</YourMod_Rest>

單複數（依數量切換）

RimWorld 文法支援以 條件規則 / plural 標記 依數值切換詞形。 常見作法是在 RulePack 內用 (p=...) 之類的條件，或在 Keyed 字串中以 NamedArgument 搭配文法子欄位。 最務實的跨語言寫法是「把數字與名詞都當參數，讓各語言 XML 自行決定要不要加複數」： 

// C#
string msg = "YourMod_RaidersIncoming".Translate(count.Named("COUNT"));
<!-- English：用文法 plural 規則（依 COUNT 是否 > 1 切換 raider/raiders） -->
<YourMod_RaidersIncoming>{COUNT} {COUNT_pluralRaider}</YourMod_RaidersIncoming>

<!-- ChineseTraditional：中文無單複數變化，直接量詞 -->
<YourMod_RaidersIncoming>來襲：{COUNT} 名襲擊者</YourMod_RaidersIncoming>
原則：把「決策權」留給 XML C# 端只負責提供「原料」（數字、pawn、物件），絕不在程式碼裡判斷單複數或性別再拼字。 所有形態變化都應該是「某個語言的 XML 文法規則」可覆寫的，這樣新增一種語言時完全不需動 C#。

LanguageWorker 的複數鉤子

語言層級的單複數規則由各語言的 LanguageWorker 子類實作（例如英文有 LanguageWorker_English）。 若你做的是「新語言翻譯包」且該語言有特殊複數邏輯，可繼承 LanguageWorker 覆寫對應方法； 一般功能 Mod 不需碰這層。

9. 翻譯缺漏偵測

RimWorld 內建翻譯偵錯工具，開發時務必啟用，確保發布前沒有遺漏的 key 或 DefInjected 欄位。

  1. 開啟 Dev Mode：Options → 一般 → Development mode 打勾，頂部出現偵錯工具列。
  2. 啟用缺漏記錄：在偵錯設定中勾選 「Log translation library / Show translation in real time」 相關選項，缺漏的 key 會即時寫入 Log。
  3. 產生翻譯報告：偵錯選單 Output → Translation 類別下，執行 「Save translation report」，RimWorld 掃描全部 Def 與 Keyed 引用，輸出：
    • 未翻譯的 DefInjected 欄位（含完整路徑，可直接複製成骨架）
    • 當前語言缺漏、回退到英文的 Keyed key
    • 原始 Def 已不存在、但翻譯檔仍殘留的「孤兒」條目
    • 重複定義的 key
  4. 檢查啟動 Log：載入 Mod 時，引擎會對「程式碼呼叫了 .Translate() 但語言庫查無此 key」發出 Translation key not found 警告（紅字），逐一補齊。
缺漏 key 的行為 "NoSuchKey".Translate() 不會丟例外，而是回傳帶錯誤標記的 key 字面量本身 （遊戲內會看到原始 key 字串而非譯文）。這代表缺漏在 UI 上「靜默退化」，唯一可靠的發現途徑就是 Dev Mode 的 Log 與翻譯報告——上線前一定要跑一遍。
EnglishKeyword 對照檢查 翻譯報告也能比對「英文母本有、其他語言沒有」的差集。把英文當完整 source， 其餘語言的工作就是消化這份差集，不會有「不知道還缺什麼」的盲區。
10. 最佳實踐

玩家可見字串一律走 Translate

任何會顯示給玩家的字（按鈕、訊息、letter、tooltip、設定項）都必須是 Keyed 或 DefInjected， 絕不硬編碼字面量。「之後再翻」幾乎等於永遠不翻。

Key 加命名空間前綴

所有 Keyed key 以 ModName_ 開頭（如 YourMod_ButtonApply）， 避免與基礎遊戲或其他 Mod 撞 key。撞 key 會造成後載入者覆蓋前者的隱性 bug。

英文是完整母本

English 資料夾必須涵蓋全部條目，作為唯一 fallback。其他語言可不完整——缺漏自動回退英文， 但英文缺漏就會露出原始 key。

用具名參數而非位置參數

多參數時 {PAWN_nameDef} 比 {0} 可讀性高、不易因詞序調整而錯位， 且能存取文法子欄位。位置參數只留給單參數簡單句。

DefInjected 用 handle 不用索引

需翻譯的清單元素在原 Def 加 handle，DefInjected 用名稱定位。 免除「插入元素導致索引全錯位」的脆弱性。

文法/形態決策留給 XML

C# 只提供原料（數字、pawn），不在程式裡判斷單複數或性別拼字。 所有形態變化都該是各語言 XML 可覆寫的，新增語言零改 C#。

發布前檢查清單

項目檢查方式
所有 .Translate() 的 key 都有 English 定義啟動 Log 無 key not found 警告
所有 Def 的 label/description 都有 DefInjected翻譯報告無 missing DefInjected 條目
無孤兒/重複 key翻譯報告 orphaned / duplicate 區段為空
檔案編碼為 UTF-8 無 BOM避免首筆資料含隱形字元
語言資料夾名稱正確（大小寫）遊戲內語言列表能正確套用
沒有硬編碼的玩家可見字串全文搜尋 C# 中的中英文字串字面量複查
遞交給譯者的友善設計 把 Keyed 檔依功能拆分（YourMod_UI.xml、YourMod_Letters.xml…）、為每組 key 加 XML 註解說明出現情境、 在含參數的 key 旁註明各參數含義。譯者不必讀你的程式碼就能正確翻譯，回頭修正的成本最低。
