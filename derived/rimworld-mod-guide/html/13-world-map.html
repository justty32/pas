§1 World / WorldGrid / Tile 結構

RimWorld 的世界是一顆球面網格（goldberg polyhedron），由數千個多邊形 Tile 鋪滿。整個行星層由 RimWorld.Planet.World（Find.World）統管，幾何資料則在 WorldGrid。每個 Tile 不是 class 實例，而是以 tileID（int）為索引存進平行陣列——這是為了在 6,000+ 個 Tile 時維持記憶體與遍歷效率。

World

行星層根物件。持有 grid、factionManager、worldObjects、components（WorldComponent 集合）等。透過 Find.World 取得。

WorldGrid

幾何與地理資料容器。以 tileID 索引：grid[tileID] 回傳 Tile。提供 TraversalDistanceBetween、GetTileNeighbors、GetTileCenter 等。

Tile

單一格的地理屬性：biome、hilliness、temperature、elevation、rainfall、swampiness、rivers、roads。

從 C# 讀取 Tile 屬性

using RimWorld.Planet;
using Verse;

// tileID 通常來自 WorldObject.Tile、Caravan.Tile、Settlement.Tile 等
int tileID = caravan.Tile;

Tile tile = Find.WorldGrid[tileID];   // indexer，等同 Find.World.grid[tileID]

BiomeDef biome     = tile.biome;          // 生態域（決定動植物、地圖生成）
Hilliness hill     = tile.hilliness;      // enum：Flat / SmallHills / LargeHills / Mountainous / Impassable
float temp         = tile.temperature;    // 當前季節氣溫（°C，會隨季節/緯度變動）
float elevation    = tile.elevation;      // 海拔（公尺，負值代表海洋）
float rainfall     = tile.rainfall;       // 年降雨量（mm）
float swampiness   = tile.swampiness;     // 沼澤度 0~1

// 平均/極端溫度需用 GenTemperature（season-aware）
float avgTemp = GenTemperature.AverageTemperatureAtTileForTwelfth(
    tileID, Twelfth.Undefined);

// 是否為海洋 / 可定居
bool isOcean = tile.WaterCovered;       // elevation <= 0
bool canBuild = !tile.WaterCovered
    && tile.hilliness != Hilliness.Impassable;

遍歷與鄰接查詢

// 取得某 Tile 的所有相鄰 Tile（球面網格多為 6 鄰，部分為 5）
var neighbors = new List<int>();
Find.WorldGrid.GetTileNeighbors(tileID, neighbors);

// 兩 Tile 間的「格數距離」（非 km；用於概略判斷遠近）
int hops = Find.WorldGrid.TraversalDistanceBetween(tileA, tileB);

// 真實地表距離（km），用 SphericalDistance 換算
Vector3 ca = Find.WorldGrid.GetTileCenter(tileA);
Vector3 cb = Find.WorldGrid.GetTileCenter(tileB);
float km = Find.WorldGrid.ApproxDistanceInTiles(tileA, tileB) * 8.8f;
tileID 為何是 int 而非物件引用 Tile 是值資料容器，整張地圖以「結構陣列」形式存放。Mod 程式碼幾乎一律以 int tileID 傳遞與儲存位置（WorldObject、Caravan、Settlement 都用 Tile 屬性回傳 int），存檔時也只存 int。需要屬性時才用 Find.WorldGrid[tileID] 即時查表。
§2 WorldObject 與 WorldObjectDef

WorldObject 是「站在世界地圖某個 Tile 上的東西」——殖民地、敵對基地、商隊、墜毀船骸、臨時遭遇地點，全都是 WorldObject 的子類別。它由 WorldObjectDef 定義靜態屬性（材質、是否可選取、對應的 C# class），由 Find.WorldObjects 管理。

原版主要 WorldObject 類別

類別WorldObjectDef用途
SettlementWorldObjects.Settlement派系常駐基地（玩家殖民地或 NPC 聚落）
CaravanWorldObjects.Caravan玩家在世界地圖移動的商隊
Site各種 Site*任務地點（廢墟、隊伍、出土遺跡等），由 SitePart 組成
MapParent—抽象基底：所有「可以生成出 Map 的 WorldObject」
TravelingTransportPodsWorldObjects.TravelingTransportPods飛行中的運輸艙群
RoutePlannerWaypoint—玩家路徑規劃用的路點

WorldObjectDef XML 結構

<!-- Defs/WorldObjectDefs/MyWorldObjects.xml -->
<Defs>

    <defName>MyAncientBeacon</defName>
    <label>古代信標</label>
    <!-- 對應的 C# class（繼承 WorldObject 或其子類別）-->
    <worldObjectClass>MyMod.WorldObject_AncientBeacon</worldObjectClass>
    <!-- 世界地圖上的材質 -->
    <texture>World/WorldObjects/AncientBeacon</texture>
    <expandingIcon>true</expandingIcon>          <!-- 縮小時顯示放大圖示 -->
    <expandingIconTexture>World/WorldObjects/Expanding/AncientBeacon</expandingIconTexture>
    <expandingIconPriority>50</expandingIconPriority>
    <canHaveFaction>true</canHaveFaction>
    <selectable>true</selectable>
    <neverMultipleOnTile>true</neverMultipleOnTile>
    <!-- 滑鼠移上去的色彩 -->
    <rotateGraphicWhenTraveling>false</rotateGraphicWhenTraveling>
    <!-- 可附加的 WorldObjectComp（官方擴充點，免 Harmony）-->

WorldObjectComp 是首選擴充點 和地圖上的 ThingComp 一樣，WorldObjectComp 讓你不必繼承整個 WorldObject 就能附加行為（計時消失、所有權、隨機事件等）。原版 TimeoutComp、EnterCooldownComp 即是範例。優先考慮 Comp，再考慮繼承。
§3 繼承 WorldObject 自訂世界物件

當 Comp 不足以表達你的邏輯（例如需要自訂 Gizmo、自訂選取資訊、或要生成地圖），就繼承 WorldObject（純世界物件）或 MapParent（要能進入產生 Map）。

純世界物件（不可進入）

using System.Collections.Generic;
using RimWorld.Planet;
using Verse;

namespace MyMod
{
    public class WorldObject_AncientBeacon : WorldObject
    {
        private int chargeLeft = 5;

        // 每 ~1 世界 tick(=遊戲 tick) 呼叫；用 Tick / TickInterval 控制
        public override void Tick()
        {
            base.Tick();
            // 世界物件也用 ShouldTickNow 分散負載
            if (this.IsHashIntervalTick(2500))   // 每 ~1 小時遊戲時間
            {
                // ... 週期性邏輯
            }
        }

        // 滑鼠停留 / 選取時左下角的資訊文字
        public override string GetInspectString()
        {
            return $"剩餘充能：{chargeLeft}";
        }

        // 選取時右下角的指令按鈕
        public override IEnumerable<Gizmo> GetGizmos()
        {
            foreach (var g in base.GetGizmos()) yield return g;

            yield return new Command_Action
            {
                defaultLabel = "啟動信標",
                defaultDesc  = "消耗一次充能觸發效果。",
                icon         = ContentFinder<UnityEngine.Texture2D>.Get("UI/Commands/Activate"),
                action       = Activate,
                // 充能耗盡則禁用
                Disabled     = chargeLeft <= 0,
                disabledReason = "充能已耗盡。"
            };
        }

        private void Activate()
        {
            chargeLeft--;
            Messages.Message("信標已啟動。",
                this, MessageTypeDefOf.PositiveEvent);
            if (chargeLeft <= 0)
                Find.WorldObjects.Remove(this);   // 自我移除
        }

        // ★ 必須序列化自訂欄位，否則存讀檔後歸零
        public override void ExposeData()
        {
            base.ExposeData();
            Scribe_Values.Look(ref chargeLeft, "chargeLeft", 5);
        }
    }
}

從 C# 生成並放置 WorldObject

// 用 WorldObjectMaker 工廠建立，再加入 WorldObjects 管理器
int tile = TileFinder.RandomStartingTile();   // 或自選 tileID

var beacon = (WorldObject_AncientBeacon)WorldObjectMaker.MakeWorldObject(
    MyWorldObjectDefOf.MyAncientBeacon);
beacon.Tile = tile;
beacon.SetFaction(Faction.OfPlayer);   // 若 canHaveFaction
Find.WorldObjects.Add(beacon);

// 用 DefOf 安全引用（避免字串拼錯）
[DefOf]
public static class MyWorldObjectDefOf
{
    public static WorldObjectDef MyAncientBeacon;
    static MyWorldObjectDefOf() => DefOfHelper.EnsureInitializedInCtor(typeof(MyWorldObjectDefOf));
}
務必呼叫 base.ExposeData() 與 base.GetGizmos() WorldObject 的基底 ExposeData 會序列化 tile、faction、comps 等核心狀態；忘了呼叫會導致存檔後物件位置/陣營丟失。同理 GetGizmos 不轉發 base 會吃掉原版（如 Settlement 的攻擊按鈕）的指令。
§4 FactionDef 結構

FactionDef 定義一個派系的「種類」（部落、海盜、帝國、機械……），決定其科技等級、外觀、敵友傾向、聚落生成規則。注意：FactionDef 是模板，Faction 才是執行期實例——同一個 FactionDef 可生成多個 Faction 實例。

關鍵欄位

欄位型別說明
isPlayerbool是否為玩家派系（殖民地）
permanentEnemybool永久敵對，goodwill 鎖死在 -100、無法談和（如機械族、蟲族）
naturalColonyGoodwillIntRange對玩家殖民地的初始好感隨機區間（如 -100~-80 為天生敵對）
techLevelTechLevel科技等級：Animal/Neolithic/Medieval/Industrial/Spacer/Ultra/Archotech，影響交易物與襲擊裝備
apparelStuffFilterThingFilter該派系 Pawn 服飾可用的材質白名單（部落用皮/布，工業用合成纖維等）
pawnGroupMakersList襲擊/交易/和平訪客的隊伍組成規則
maxConfigurableAtWorldCreationint世界生成時此 Def 最多可建立幾個 Faction 實例
requiredCountAtGameStartint開局保證生成的數量
hiddenbool不顯示於派系列表（如「無派系野生動物」「永久荒野」）
humanlikeFactionbool是否由類人 Pawn 組成（決定能否談判/交易）

permanentEnemy 與 naturalColonyGoodwill 的交互 permanentEnemy=true 會強制 goodwill 鎖在 -100 並忽略 naturalColonyGoodwill，且 TryAffectGoodwillWith 對其無效（§6）。若你只想「開局敵對但仍可外交」，請用 permanentEnemy=false 搭配偏負的 naturalColonyGoodwill。
§5 Faction 關係（goodwill / hostility / FactionRelationKind）

每一對 Faction 之間維護一個 FactionRelation，核心是 goodwill（-100 ~ +100 的 int）與由其推導的 FactionRelationKind 列舉。關係是對稱的：A 對 B 的 goodwill 等於 B 對 A 的。

Hostile

goodwill ≤ -75（門檻 GoodwillThresholdToBecomeHostile）。會發動襲擊、互相攻擊。

Neutral

介於敵對與盟友之間。可交易、可訪問，但不會無條件協防。

Ally

goodwill ≥ +75（GoodwillThresholdToBecomeAlly）。可呼叫軍援、共同對敵。

查詢關係

Faction player = Faction.OfPlayer;
Faction other  = /* 某 NPC 派系 */;

// 當前好感（int，-100~100）
int gw = player.GoodwillWith(other);

// 關係種類列舉
FactionRelationKind kind = player.RelationKindWith(other);
// → Hostile / Neutral / Ally

// 便捷布林
bool hostile = player.HostileTo(other);
bool ally    = kind == FactionRelationKind.Ally;

// 取得 FactionRelation 物件本身（含 baseGoodwill 等）
FactionRelation rel = player.RelationWith(other, allowNull: true);
baseGoodwill vs goodwill FactionRelation 內部其實是 baseGoodwill（基礎值）；最終 goodwill 還會受臨時修正（如近期被你救援）影響。改動關係請一律走 TryAffectGoodwillWith 或 SetRelationDirect，不要直接寫欄位，否則對稱性與事件鉤子會失效。
§6 從 C# 建立 Faction 與改變 Goodwill

執行期動態生成一個新 Faction

using RimWorld;
using RimWorld.Planet;
using Verse;

public static Faction SpawnRaiderClan()
{
    FactionDef def = MyFactionDefOf.MyRaiderClan;

    // FactionGenerator 負責隨機命名、配色、leader 等
    Faction faction = FactionGenerator.NewGeneratedFaction(
        new FactionGeneratorParms(def));

    // 與所有既有派系建立初始關係（依各 def 的 naturalGoodwill）
    foreach (Faction existing in Find.FactionManager.AllFactionsListForReading)
    {
        faction.TryMakeInitialRelationsWith(existing);
    }

    // 登記到 FactionManager（★ 不做這步派系不會被遊戲承認）
    Find.FactionManager.Add(faction);

    // （選用）在世界上生成它的基地
    Settlement s = (Settlement)WorldObjectMaker.MakeWorldObject(
        WorldObjectDefOf.Settlement);
    s.SetFaction(faction);
    s.Tile = TileFinder.RandomSettlementTileFor(faction);
    s.Name = SettlementNameGenerator.GenerateSettlementName(s);
    Find.WorldObjects.Add(s);

    return faction;
}

改變好感：TryAffectGoodwillWith

Faction player = Faction.OfPlayer;
Faction other  = /* 目標派系 */;

// 相對調整 goodwill（會自動 clamp、觸發敵對/結盟事件、彈通知）
bool changed = other.TryAffectGoodwillWith(
    player,                       // 與誰的關係
    goodwillChange: 25,           // 增減量（正=友好）
    canSendMessage: true,         // 是否彈出右上角通知
    canSendHostilityLetter: true, // 跨越敵對門檻時是否寄信
    reason: HistoryEventDefOf.DeliveredQuestReward, // 原因（影響 UI 文字）
    lookTarget: null);

// 直接設定到絕對值（少用；繞過 reason 系統）
player.SetRelationDirect(other, FactionRelationKind.Ally);
// 或設 baseGoodwill 後重算
player.SetRelation(new FactionRelation(other, FactionRelationKind.Neutral));
TryAffectGoodwillWith 才是「正路」 它會：① 對稱更新雙方；② 自動 clamp 到 [-100,100]；③ 跨過 ±75 門檻時切換 RelationKind 並觸發 Notify_RelationKindChanged（讓 NPC 立刻轉為攻擊或停火）；④ 依 reason 顯示「因為你完成委託，好感 +25」之類的訊息。直接寫 goodwill 欄位會跳過這一切。
permanentEnemy 派系無法被影響 對 permanentEnemy=true 的派系呼叫 TryAffectGoodwillWith 會回傳 false 且毫無作用——機械族、蟲族、（部分 DLC）異虫不能談和。設計委託獎勵「提升好感」時要排除這類派系，否則玩家會誤以為功能失效。
§7 Caravan 系統與 WorldPath

Caravan 是玩家在世界地圖上移動的 Pawn 群（含載具、物資）。它是 WorldObject 子類別，移動依賴世界尋路系統 WorldPathFinder，沿著 Tile 間的「移動難度」找最短路徑。

Caravan 的關鍵成員

成員說明
caravan.pawnsThingOwner<Pawn>：商隊內所有 Pawn（含俘虜/動物）
caravan.PawnsListForReading只讀 List，遍歷成員用
caravan.Tile目前所在 tileID
caravan.patherCaravan_PathFollower：移動狀態機，持有當前 WorldPath
caravan.TicksPerMove移動速度（受成員、地形、負重影響）
caravan.AllThings商隊攜帶的所有物品（背包+載具）

建立商隊並下達移動指令

// 由一群 Pawn 組成商隊（通常離開地圖時呼叫）
var members = new List<Pawn> { colonist1, colonist2, packAnimal };

Caravan caravan = CaravanMaker.MakeCaravan(
    members,
    Faction.OfPlayer,
    startingTile,
    addToWorldPawnsIfNotAlready: true);

// 下令前往目標 Tile（pather 會自行尋路）
caravan.pather.StartPath(destinationTile, repathImmediately: true);

// 也可指定到達後的「到站動作」（進入聚落、攻擊等）
caravan.pather.StartPath(targetSettlement.Tile, new CaravanArrivalAction_VisitSettlement(
    targetSettlement), repathImmediately: true);

WorldPath 與世界尋路

// 手動計算一條世界路徑（評估距離/天數，不實際移動）
using (WorldPath path = Find.WorldPathFinder.FindPath(
    startTile, destTile, traveler: caravan))
{
    if (path != WorldPath.NotFound)
    {
        float totalCost = path.TotalCost;          // 總移動成本（ticks 概念）
        List<int> nodes = path.NodesReversed;       // tileID 序列（終點→起點）

        // 估計旅程天數
        int ticks = CaravanArrivalTimeEstimator.EstimatedTicksToArrive(
            startTile, destTile, caravan);
        float days = ticks / (float)GenDate.TicksPerDay;
    }
}   // ★ WorldPath 實作 IDisposable，務必 using 釋放回 pool
WorldPath 是池化資源，記得 Dispose Find.WorldPathFinder.FindPath 回傳的 WorldPath 來自物件池，用完必須 Dispose()（用 using 區塊最保險）歸還，否則池耗盡會拋例外。Caravan 的 pather 內部已自管路徑，只有你「手動評估路徑」時才需要自己 FindPath。
§8 BiomeDef 與 Biome 動植物分布

BiomeDef 定義一個生態域：能長什麼植物、出沒什麼動物、地圖生成的密度、可否定居、移動難度。每個 Tile 都綁定一個 BiomeDef（tile.biome），它同時影響世界層（移動成本）與地圖層（生成內容）。

動植物分布的權重機制

BiomeDef 內以 wildPlants / wildAnimals 列出 ThingDef → commonality（出現權重）。地圖生成時依權重隨機抽選；權重越高、密度（plantDensity / animalDensity）越大，該生態域就越「茂密」或「多獸」。

<!-- Defs/BiomeDefs/MyBiome.xml -->
<Defs>
  <BiomeDef>
    <defName>AshenWastes</defName>
    <label>灰燼荒原</label>
    <description>火山灰覆蓋的貧瘠地帶。</description>
    <workerClass>BiomeWorker_Desert</workerClass>  <!-- 決定哪些 Tile 成為此 biome -->

    <canBuildBase>true</canBuildBase>
    <canAutoChoose>true</canAutoChoose>
    <allowRoads>true</allowRoads>
    <allowRivers>false</allowRivers>

    <!-- 整體密度（影響地圖生成的植物/動物總量）-->
    <plantDensity>0.08</plantDensity>
    <animalDensity>0.6</animalDensity>
    <diseaseMtbDays>60</diseaseMtbDays>
    <movementDifficulty>2</movementDifficulty>  <!-- 世界層商隊移動難度 -->
    <forageability>0.2</forageability>

    <!-- 野生植物：ThingDef → commonality -->
    <wildPlants>
      <Plant_Bush>2.0</Plant_Bush>
      <Plant_TreeDrago>0.3</Plant_TreeDrago>
    </wildPlants>

    <!-- 野生動物：ThingDef(animal) → commonality -->
    <wildAnimals>
      <Hare>1.0</Hare>
      <Boomrat>0.8</Boomrat>
      <Megasloth>0.05</Megasloth>
    </wildAnimals>

    <!-- 可額外用 PatchOperation 把動物塞進原版 biome（見下方 callout）-->
  </BiomeDef>
</Defs>

從 C# 查詢 Biome 的動植物

BiomeDef biome = tile.biome;

// 取得此 biome 所有可生成的動物（含權重）
foreach (PawnKindDef animal in biome.AllWildAnimals)
{
    float commonality = biome.CommonalityOfAnimal(animal);
    // ...
}

// 植物
foreach (ThingDef plant in biome.AllWildPlants)
{
    float c = biome.CommonalityOfPlant(plant);
}
給原版 Biome 加動物：用 PatchOperationAdd 與其複製整個 BiomeDef，不如用 Patch 把你的動物插進原版 biome 的 wildAnimals：
<xpath>/Defs/BiomeDef[defName="TemperateForest"]/wildAnimals</xpath> + PatchOperationAdd。這樣相容性最佳、不覆蓋他人改動。詳見「常見模式」章的 PatchOperations。
§9 SettlementDef 與 SitePartDef

聚落與任務地點是兩條不同的世界物件管線：Settlement 是派系的長駐基地（用 WorldObjectDef 為 Settlement 的 def），其防禦/外觀由派系的 settlementGenerationParams 與地圖生成器決定；Site 是臨時任務點，由一個或多個 SitePart 拼裝而成，每個 SitePart 對應一個 SitePartDef。

Settlement（常駐基地）

WorldObject 子類 Settlement。綁定 Faction，可被攻擊/交易/拜訪。進入時由 WorldObjectDef 與派系的生成參數共同產生地圖（含建築、防禦、駐軍）。

Site（任務地點）

WorldObject 子類 Site，內含 List<SitePart>。每個 SitePart 由 SitePartDef + SitePartWorker 定義地圖生成內容與威脅（如「叛軍隊伍」「精密廢墟」）。

SitePartDef XML 與從 C# 生成 Site

<!-- Defs/SitePartDefs/MySitePart.xml -->
<SitePartDef>
  <defName>AshenCache</defName>
  <label>灰燼藏匿點</label>
  <workerClass>SitePartWorker_ItemStash</workerClass>  <!-- 控制地圖內容 -->
  <tags>
    <li>ItemStash</li>
  </tags>
  <defaultHidden>false</defaultHidden>
</SitePartDef>
// 用 SiteMaker 組裝一個含多個 SitePart 的 Site
var parts = new List<SitePartDefWithParams>
{
    new SitePartDefWithParams(
        MySitePartDefOf.AshenCache,
        new SitePartParams { threatPoints = 300f })
};

Site site = SiteMaker.MakeSite(
    parts,
    tile: TileFinder.RandomStartingTile(),
    faction: enemyFaction);

site.GetComponent<TimeoutComp>()?.StartTimeout(15 * GenDate.TicksPerDay); // 15 天後消失
Find.WorldObjects.Add(site);
Site 通常由 QuestNode 生成 在 1.3+ 的 Quest 系統中，Site 一般透過 QuestNode_GenerateSite 在委託腳本內生成並掛上獎勵/威脅，而非手動 SiteMaker。若你的世界物件是「委託目標」，優先寫成 QuestScriptDef + QuestNode（見「事件敘事」章）。手動 SiteMaker 適用於非委託的自發性世界事件。
§10 常見坑

世界生成時序：別在錯的時間點動世界

世界與遊戲的初始化有嚴格順序。最常見的錯誤是在「世界還沒建好」時就嘗試讀 Find.World，或在 StaticConstructorOnStartup（遊戲啟動、尚無 World）中存取 Find.WorldGrid。

鉤子此時可用適合做
[StaticConstructorOnStartup]Defs、材質；無 World/GameHarmony patch、預載材質、建 DefOf 快取
GameComponent.FinalizeInitWorld、Maps 均就緒讀世界資料、初始化每局狀態
WorldComponent.FinalizeInitWorld 就緒（Map 可能尚未）世界層持久化資料初始化
GenStep / WorldGenStep正在生成中插入世界/地圖生成步驟
 
// ✗ 錯誤：StaticConstructorOnStartup 時 Find.World 為 null
[StaticConstructorOnStartup]
public static class BadInit
{
    static BadInit()
    {
        var t = Find.WorldGrid[0];  // NullReferenceException！
    }
}

// ✓ 正確：用 GameComponent 在每局載入完成後才動世界
public class MyGameComponent : GameComponent
{
    public MyGameComponent(Game game) { }

    public override void FinalizeInit()
    {
        // 此刻 Find.World / Find.Maps 已就緒
        var tile = Find.WorldGrid[Find.CurrentMap?.Tile ?? 0];
    }
}

WorldComponent 持久化：忘了 ExposeData 就遺失資料

世界層的自訂狀態（派系陰謀進度、全域旗標、跨地圖追蹤）應放在 WorldComponent。它隨世界存讀檔，但你必須自己實作 ExposeData 序列化欄位，且要在 XML 註冊或直接靠反射自動載入。

using RimWorld.Planet;
using Verse;

public class WorldComponent_ClanTracker : WorldComponent
{
    private int totalRaidsSurvived;
    private Dictionary<Faction, int> betrayalCount = new Dictionary<Faction, int>();

    // 建構子簽章固定為 (World world)
    public WorldComponent_ClanTracker(World world) : base(world) { }

    // 世界層的週期更新（注意是 World tick，不是 Map tick）
    public override void WorldComponentTick()
    {
        if (Find.TickManager.TicksGame % 60000 == 0)
        {
            // 每遊戲天結算一次
        }
    }

    // ★ 持久化：漏寫此方法 = 存讀檔後資料全失
    public override void ExposeData()
    {
        base.ExposeData();
        Scribe_Values.Look(ref totalRaidsSurvived, "totalRaidsSurvived", 0);
        Scribe_Collections.Look(ref betrayalCount, "betrayalCount",
            LookMode.Reference, LookMode.Value);

        if (Scribe.mode == LoadSaveMode.PostLoadInit)
            betrayalCount ??= new Dictionary<Faction, int>();
    }
}

// 取用（RimWorld 會自動為每個 WorldComponent 子類建立實例）
var tracker = Find.World.GetComponent<WorldComponent_ClanTracker>();
用 Scribe_References 存 Faction，別用 Scribe_Deep Faction 由 FactionManager 擁有，存自訂資料時對 Faction 一律用 Scribe_References.Look（或字典的 LookMode.Reference）。若誤用 Deep 會在讀檔時複製出孤兒 Faction，造成關係系統與地圖派系標記錯亂。同理，存 tileID 用 int（Scribe_Values），別存 Tile 物件。
WorldComponent 會被自動建立——但只在「新世界」 RimWorld 在世界生成時掃描所有 WorldComponent 子類並各建一個實例。若玩家用舊存檔（你的 Mod 是後加的）載入，新版本的 WorldComponent 仍會被補建（建構子被呼叫），但ExposeData 讀不到對應節點 → 欄位保持預設值。設計遷移時要容忍「首次出現、資料為空」的情況。
