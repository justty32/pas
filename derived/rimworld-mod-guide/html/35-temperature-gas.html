溫度、氣體、屋頂與天氣

房間溫度、毒氣與煙霧、屋頂遮蔽、天氣事件——這些環境系統決定了殖民地的宜居性。本章拆解 GenTemperature、Room、Gas 與 WeatherDef。

Chapter 35 · Temperature & Gas
本章導覽

RimWorld 的環境系統由四個相互糾纏卻又各自獨立的子系統構成：房間 / 區域模型負責把地圖切成可交換熱量的單位、溫度系統每 tick 在這些單位之間流動熱量、氣體網格在地圖上擴散毒氣與煙霧、而屋頂與天氣則決定哪些格子算室內、降雨多寡、日照與命中懲罰。要寫出一個會暖房的柴爐、一個自訂的毒霧發生器、或一個專屬的暴風天氣事件，都得先弄懂這幾層怎麼疊在一起。

§3 玩法影響

反編譯定位本章引用的型別主要落在 Verse（Region、Room、GenTemperature、GasGrid、RoofGrid、WeatherDef、WeatherManager）與 RimWorld（CompTempControl、CompHeatPusher、Building_Vent、Building_Cooler、GameConditionManager、GameCondition_*）兩個命名空間。建議在 dnSpy 開啟 Assembly-CSharp.dll，照本章標註的型別名直接搜尋。版本以 1.5 為準，部分污染相關 API 屬 Biotech DLC。
§1 房間與區域模型（Region → Room）

溫度不是逐格獨立模擬的——那樣大地圖會直接卡死。RimWorld 先把地圖切成區域 (Region)，再把相連的區域聚合成房間 (Room)，溫度系統真正操作的單位是 Room。理解這條鏈是理解整個溫度系統的前提。

Region：地圖的空間索引單位

Verse.Region 是地圖空間分割的基本塊（最大 12×12），由 Verse.RegionAndRoomUpdater 維護。它原本是為了 pathfinding 與 reachability 服務的，但 Room 正是建立在 Region 之上：每個 Region 屬於一個 Room，相鄰且無「隔斷」（牆、門關閉等）的 Region 會被合併到同一個 Room。當你蓋一道牆、開一扇門、或拆掉一塊地板，RegionAndRoomUpdater 會把受影響的 Region 標記為 dirty，並在下個適當時機重算房間歸屬。

Region

≤12×12 的空間塊，pathfinding / reachability / room 聚合的基礎。由 RegionGrid 持有。

Room

由相連 Region 聚合而成的「房間」。持有溫度、Region 列表、RoomRoleDef、是否使用室外溫度等。

District

1.x 後引入的中間層 Verse.District，介於 Region 與 Room 之間（門會切割 District 但仍屬同 Room）。多數 mod 不需直接碰。

Room：溫度交換的單位

Verse.Room 的關鍵成員：

成員型別說明
Temperaturefloat (get/set)房間目前溫度（攝氏）。set 時會寫回房間群組的溫度暫存。室外房間 get 會回傳地圖室外溫度。
UsesOutdoorTemperaturebool房間是否沿用室外溫度（未密閉、或屬於室外的「假房間」時為 true）。
OpenRoofCountint房間內未被屋頂覆蓋的格數。>0 通常意味著漏天、會被當室外處理。
CellCountint房間格數，溫度的「熱容量」與此相關（大房間升溫慢）。
PsychologicallyOutdoorsbool心理上是否算室外（影響 thought，串第20章）。
RoleRoomRoleDef房間角色（臥室、餐廳、監獄…），由 RoomRoleWorker 推斷。

取得一個格子所在房間最常用的入口是擴充方法 RegionAndRoomQuery.RoomAt(cell, map)，或更口語的 cell.GetRoom(map)（Verse.GridsUtility.GetRoom）。Thing 上則有 thing.GetRoom()。

// 取得某格 / 某 pawn 所在房間，並讀其溫度
Room room = cell.GetRoom(map);             // Verse.GridsUtility.GetRoom
if (room != null && !room.UsesOutdoorTemperature)
{
    float t = room.Temperature;        // 室內封閉房間的實際溫度
}

Room pawnRoom = pawn.GetRoom();          // Verse.RegionAndRoomQuery / Pawn 擴充

密閉判定：roofed + walled

一個房間要被視為「室內、可獨立控溫」，直覺上需要同時滿足：四周封閉（沒有缺口連到室外）且完全有屋頂。在程式碼層面，RimWorld 用「房間是否與室外連通」與 OpenRoofCount 來判定，最終體現在 UsesOutdoorTemperature：

  • 房間若連到地圖邊緣或一個「fog/室外」假房間 → UsesOutdoorTemperature == true。
  • 房間若有任何格子無屋頂（OpenRoofCount > 0）→ 視為與室外熱交換、實際也跟著室外走。
  • 真正密閉（四面牆 + 全屋頂、無缺口）才會擁有獨立的房間溫度暫存，控溫設備才有意義。
「房間」的最大格數上限RimWorld 對房間有大小上限（Room.RegionCount / 細胞數過大會被當室外處理，常見約 300+ 格的開放空間會 fallback 成室外溫度）。把暖氣放在一個沒封頂或過大的「房間」裡，溫度會死死貼著室外溫度不動——這幾乎是新手控溫 mod 第一個踩到的坑。

GenTemperature.GetTemperatureForCell — 逐格溫度查詢

很多時候你手上只有一個 IntVec3，想知道「這格現在幾度」。Verse.GenTemperature.GetTemperatureForCell(c, map) 會做正確的分流：若該格屬於密閉房間，回房間溫度；否則回室外溫度。

// GenTemperature.GetTemperatureForCell 的行為（概念重述，非逐行原始碼）
public static float GetTemperatureForCell(IntVec3 c, Map map)
{
    Room room = c.GetRoom(map);
    if (room != null && !room.UsesOutdoorTemperature)
        return room.Temperature;
    return map.mapTemperature.OutdoorTemp;   // 室外溫度（含季節 + 緯度）
}
最佳實務幾乎所有「這格幾度」的查詢都該走 GenTemperature.GetTemperatureForCell 或 GenTemperature.TryGetDirectAirTemperatureForCell，而不要自己 cell.GetRoom().Temperature——前者已經幫你處理好室內/室外 fallback、null room 等邊界情況。
§2 溫度 API 與調節（GenTemperature / CompTempControl / CompHeatPusher）

溫度的「資料層」是 Room.Temperature，「行為層」則由一群 ThingComp 每 tick 往房間注入或抽走熱量。要寫控溫 mod，你會在這兩端之間工作。

GenTemperature：靜態工具集

方法用途
GetTemperatureForCell(c, map)逐格溫度，含室內/室外 fallback（§1）。
TryGetDirectAirTemperatureForCell(c, map, out temp)取得「直接空氣溫度」，回傳是否成功。被火、設備等局部熱源呼叫。
ControlTemperatureTempChange(cell, map, energyLimit, targetTemp)計算這 tick 應施加的溫度變化量，是 CompTempControl 的核心數學。考慮房間熱容量與能量上限。
PushHeat(cell, map, energy)往該格所在房間注入 energy（>0 升溫，<0 降溫）。Building_Cooler、火、CompHeatPusher 都呼叫它。
OutdoorTemp / SeasonalTemp由 MapTemperature 持有的室外溫度（緯度 + 季節 + 天氣修正）。
// 直接往一格所在房間推熱：正值升溫、負值降溫
GenTemperature.PushHeat(this.Position, this.Map, 21f);

// 朝 targetTemperature 控溫，energyLimit 限制每 tick 能投入的能量
float change = GenTemperature.ControlTemperatureTempChange(
    cell, map, energyLimit: 21f, targetTemperature: 21f);
room.Temperature += change;

CompTempControl：暖氣 / 冷氣的核心

RimWorld.CompTempControl 是「有目標溫度」的設備共用 comp（暖氣、冷氣、地下溫控、自動門簾…）。它持有 targetTemperature 與 operatingAtHighPower，並在 UI 上提供「升溫/降溫到 X 度」的 gizmo。它本身不直接推熱——具體的 Building_Heater / Building_Cooler 每 tick 讀 CompTempControl 的目標值，再呼叫 GenTemperature 施加變化。

<ThingDef ParentName="BuildingBase">
  <defName>MyHeater</defName>
  <thingClass>RimWorld.Building_Heater</thingClass>
  <comps>
    <li Class="CompProperties_Power">
      <compClass>CompPowerTrader</compClass>
      <basePowerConsumption>175</basePowerConsumption>
    </li>
    <li Class="CompProperties_TempControl">
      <energyPerSecond>21</energyPerSecond>
    </li>
    <li Class="CompProperties_HeatPusher">
      <compClass>CompHeatPusherPowered</compClass>
      <heatPerSecond>21</heatPerSecond>
      <heatPushMaxTemperature>28.5</heatPushMaxTemperature>
    </li>
  </comps>
</ThingDef>
原版暖氣其實是 HeatPusher原版「暖氣 (Heater)」用的就是 CompHeatPusherPowered + CompProperties_TempControl 組合；「冷氣 (Cooler)」才是真正 Building_Cooler（一面進熱一面排熱、需貼牆）。要做純升溫設備，沿用 HeatPusher 路線最省事；要做雙面溫差設備，才走 Building_Cooler。

CompHeatPusher：無腦推熱

Verse.CompHeatPusher（與帶電版 Verse.CompHeatPusherPowered）每 tick 對所在房間推固定熱量，直到房間溫度達 heatPushMaxTemperature 為止。柴爐、營火、太陽燈的副熱、化學燃料發電機都用它。它沒有目標溫度概念，只會單向加熱（heatPerSecond 設負值即可單向降溫，但極少這樣用）。

欄位說明
heatPerSecond每秒推入的熱量（換算成 tick）。正升溫負降溫。
heatPushMaxTemperature達此溫度後停止推熱（防止無限加熱）。
heatPushMinTemperature降溫用時的下限。

Building_Vent / Building_Cooler 模式

Building_Vent（通風口）

RimWorld.Building_Vent 連通兩側房間溫度，使其趨於平均。開關狀態由 CompFlickable 控制；關閉時 FlickableComp 切斷溫度傳遞。本質是「在牆上開一個讓熱量流通的洞」。

Building_Cooler（冷氣）

RimWorld.Building_Cooler 有方向性：背面（intake）抽熱、正面（exhaust）排熱。必須卡在牆上，兩側分別屬於不同房間。CompTempControl 給目標溫度，效率隨外側溫度上升而衰減（過熱會結霜停機）。

實作：一個自訂的「會推熱的供暖建築」

串第18章的 CompPowerTrader 與 PlaceWorker，做一個只在通電且房間封閉時才有效的暖爐。重點是覆寫 TickRare（每 250 tick 一次，溫度系統不需要每 tick）並自己控制推熱條件。

using RimWorld;
using Verse;

public class Building_SmartHeater : Building
{
    private CompPowerTrader powerComp;
    private CompTempControl tempControl;

    public override void SpawnSetup(Map map, bool respawningAfterLoad)
    {
        base.SpawnSetup(map, respawningAfterLoad);
        powerComp   = GetComp<CompPowerTrader>();
        tempControl = GetComp<CompTempControl>();
    }

    public override void TickRare()
    {
        base.TickRare();
        if (powerComp != null && !powerComp.PowerOn) return;

        Room room = this.GetRoom();
        // 開放空間 / 漏天房間：推熱無意義，省下運算
        if (room == null || room.UsesOutdoorTemperature) return;

        float target = tempControl.targetTemperature;
        if (room.Temperature >= target) return;        // 已達標就停手

        // 用 ControlTemperatureTempChange 取得本 tick 應變化量（energyPerSecond 由 props 帶入）
        float energy = tempControl.Props.energyPerSecond * 4.16667f; // TickRare ≈ 4.17s
        float change = GenTemperature.ControlTemperatureTempChange(
            this.Position, this.Map, energy, target);
        room.Temperature += change;

        // 可調整耗電顯示：滿載 vs 待機
        powerComp.PowerOutput = -powerComp.Props.PowerConsumption;
    }
}
用 TickRare 不用 Tick溫度變化緩慢，TickRare（每 250 tick）對溫控完全足夠，且省下大量逐 tick 計算。原版 CompTempControl 與 CompHeatPusher 也都在 rare/medium tick 層級工作。把溫控邏輯塞進 Tick 是常見的性能浪費（串第36章）。
§3 溫度對玩法的影響（舒適區間 / 凍傷中暑 / 腐敗 / 種植）

溫度本身只是一個數字，真正影響玩法的是消費這個數字的各個系統。下面四條是 mod 最常需要掛鉤的路徑。

(1) 舒適溫度區間：StatDef + StatPart

每個 pawn 有 ComfyTemperatureMin / ComfyTemperatureMax 兩個 StatDef（RimWorld.StatDefOf）。最終值由 race base + 衣物保暖度（Insulation_Cold / Insulation_Heat）+ hediff + 基因等 StatPart 疊加。要讓自訂裝備或基因改變耐寒耐熱，正規做法是寫 StatPart 或在 ThingDef/GeneDef 上掛 statOffsets（串第14、17、21章）。

// 取得 pawn 的舒適溫度上下限與「目前是否過冷/過熱」
float min = pawn.GetStatValue(StatDefOf.ComfyTemperatureMin);
float max = pawn.GetStatValue(StatDefOf.ComfyTemperatureMax);
float ambient = pawn.AmbientTemperature;   // Verse.Thing.AmbientTemperature
bool tooCold = ambient < min;
bool tooHot  = ambient > max;
// 自訂耐寒披風：直接在 ThingDef 上加 statOffsets（無需 C#）
<statOffsets>
  <Insulation_Cold>28</Insulation_Cold>   <!-- 攝氏，向下擴張舒適下限 -->
  <Insulation_Heat>4</Insulation_Heat>
</statOffsets>

(2) 凍傷與中暑：Hediff

過冷產生 HediffDefOf.Hypothermia（失溫）、過熱產生 Heatstroke（中暑）。它們由 Pawn_HealthTracker 下的環境檢查每隔一段時間根據 AmbientTemperature 與舒適區間的偏差量推進嚴重度（severity）。要做「自訂材質造成特殊溫度傷害」，正規做法是定義新的 HediffDef + HediffGiver（或 HediffGiver_Heat / HediffGiver_Hypothermia 的衍生），掛在 race 的 RaceProperties.hediffGiverSets 上。

別直接每 tick 加 severity失溫/中暑的嚴重度推進是「偏離舒適區越遠、推進越快；回到舒適區會自動回復」。若你用 Harmony 硬塞 severity，要記得也處理回復段，否則 pawn 進了室內仍持續凍傷。優先複用既有 HediffGiver 的數學。

(3) 食物腐敗：CompRottable

RimWorld.CompRottable 是讓物品會腐爛的 comp，其腐敗進度受所在格溫度影響：低於 TemperatureThreshold（約 0°C）幾乎不腐、低於 RotProgressFreezeThreshold（約 −10°C，冷凍）完全凍結。冷凍庫 mod 的本質就是「把一個密閉房間用冷氣壓到 0°C 以下」。

溫度區間CompRottable 行為
> 0°C正常腐敗，越熱越快（線性加速）。
−10°C ~ 0°C腐敗大幅減速。
< −10°C完全凍結，rotProgress 不前進（冷凍保存）。
// 查詢某堆食物還能放多久（受目前溫度影響）
CompRottable rot = thing.TryGetComp<CompRottable>();
if (rot != null)
{
    float daysLeft = rot.PropsRot.TicksToRotStart != 0
        ? (rot.PropsRot.TicksToRotStart - rot.RotProgress) / 60000f
        : 0f;
    RotStage stage = rot.Stage;   // Fresh / Rotting / Dessicated
}

(4) 種植溫度區間

植物（RimWorld.Plant / ThingDef.plant）有可生長的溫度窗。原版硬編在 Plant.GrowthRateFactor_Temperature：低於約 0°C 或高於約 58°C 不長，最適區間約 6°C–42°C。要自訂耐寒作物，通常透過自訂 PlantProperties（如 growMinGlow、生長季）配合並用 Harmony patch GrowthRateFactor_Temperature，因為溫度窗本身未完全 def 化（串第19章）。

StatPart

改舒適溫度、保暖度的乾淨做法（§14）。

HediffGiver

自訂溫度傷害，複用回復數學（§28）。

CompRottable

冷凍庫＝把房間壓到 −10°C 以下。

§4 氣體系統（GasType / GasGrid / 毒氣 Hediff / 污染）

RimWorld 1.6 的地圖氣體用一張統一網格管理。三種「環境氣體」共享同一個 Verse.GasGrid，逐 tick 在格子間擴散、消散。

GasType：三種氣體

GasType來源效果
BlindSmoke煙幕彈、機械蟲噴霧降低視野/命中（致盲煙）。
ToxGas毒氣彈、自爆機械、毒霧發生器對 pawn 累積毒性負荷（tox buildup hediff）。
RotStink腐爛物、屍體堆產生負面 thought（惡臭），不直接傷害。

GasGrid：擴散與密度

Verse.GasGrid（透過 map.gasGrid）以一張 byte 陣列存每格各氣體的密度（0–255）。每 tick 在開放格之間平均擴散、並隨時間衰減；牆與關閉的門會阻擋擴散。常用 API：

方法說明
AnyGasAt(cell)該格是否有任何氣體。
DensityAt(cell, gasType)該格某種氣體密度（byte）。
AddGas(cell, gasType, amount, canOverflow)在該格注入氣體（會觸發鄰格擴散）。
SetDirect(index, ...)低階直接寫，少用。
using Verse;

// 在建築周圍持續釋放毒氣（毒霧發生器的核心）
public class CompToxEmitter : ThingComp
{
    public override void CompTickRare()
    {
        Map map = parent.Map;
        if (map == null) return;
        IntVec3 c = parent.Position;
        // 注入 ToxGas，數量上限 255；canOverflow 讓鄰格分擔
        map.gasGrid.AddGas(c, GasType.ToxGas, amount: 100, canOverflow: true);
    }
}
GasGrid 與便捷工具三種環境氣體的散佈與密度全部由 Verse.GasGrid（map.gasGrid）統一處理，逐格操作走 AddGas / DensityAt / AnyGasAt。要在地圖一片區域瞬間鋪滿毒氣（如毒氣彈爆炸），原版用 GenExplosion.DoExplosion 帶 postExplosionGasType（並由 postExplosionGasRadiusOverride / postExplosionGasAmount 控制範圍與量）的路徑，比手動逐格 AddGas 更省事。

爆炸後留毒氣：純 XML

<!-- 一個爆炸後在落點周圍噴 ToxGas 的彈藥 ProjectileDef -->
<projectile>
  <damageDef>Bomb</damageDef>
  <explosionRadius>2.9</explosionRadius>
  <postExplosionGasType>ToxGas</postExplosionGasType>
</projectile>

毒氣對 pawn：tox buildup hediff

當 pawn 處在 ToxGas 中且未戴防毒裝備（gas mask / 全罩面具，由 StatDefOf.ToxicResistance 與 apparel 的 gas 防護決定），會累積 HediffDefOf.ToxicBuildup 的嚴重度。達一定門檻會造成器官傷害甚至死亡。pawn 的 GetStatValue(StatDefOf.ToxicResistance) 與所在格的 ToxGas 密度共同決定每 tick 的累積速率（由 Pawn_HealthTracker 環境檢查推進）。

Biotech 污染（pollution，軟依賴）

Biotech 引入的地圖污染是另一套系統（與 GasGrid 無關）：地形被污染後 TerrainDef 切換為 polluted 變體，污染由 map.pollutionGrid 管理；wastepack（污染包）會洩漏污染、毒性落塵 (toxic fallout) 是覆蓋全圖的 GameCondition。要在 mod 中觸碰這些，務必用軟依賴，因為玩家不一定有 Biotech。

<!-- 用 MayRequire 讓欄位只在 Biotech 啟用時載入，避免缺 DLC 報錯 -->
<ThingDef>
  <defName>MyReactor</defName>
  <comps>
    <li Class="CompProperties_PolluteOverTime"
        MayRequire="Ludeon.RimWorld.Biotech">
      <cellsToPollutePerDay>3</cellsToPollutePerDay>
    </li>
  </comps>
</ThingDef>
C# 端的 Biotech 軟依賴若 C# 直接引用 Biotech 專屬型別（如污染相關 class），在無 DLC 的玩家機器上 JIT 到該方法就會 TypeLoadException。把 Biotech 專屬程式碼隔離進獨立方法，並先用 ModsConfig.BiotechActive 判斷後才呼叫；必要時整個 patch class 加 [StaticConstructorOnStartup] 條件註冊。
§5 屋頂與遮蔽（RoofDef / RoofGrid / 塌陷）

屋頂同時影響溫度（密閉判定）、降雨遮蔽、日照（室內光）與礦山挖掘（厚岩頂）。它存在一張獨立的 Verse.RoofGrid，與地形、建築網格分離。

RoofDef：三種屋頂

RoofDefdefName特性
薄頂RoofConstructed殖民者建造的屋頂；可拆除；不擋掉落物（drop pod 可穿）。
厚岩頂RoofRockThick山體內的厚頂；無法手動拆除，需挖掉整塊岩石；擋空投與軌道砲。
薄岩頂RoofRockThin山體邊緣的薄岩頂；可被掘穿。

RoofDef 關鍵欄位：isThickRoof（是否厚頂，決定能否阻擋軌道打擊與 drop pod）、isNatural（自然生成）、collapseLeavingThingDef（塌陷後留下的瓦礫/岩塊）、VanishOnCollapse。

RoofGrid：查詢與設定

using Verse;

RoofGrid roofs = map.roofGrid;
bool hasRoof   = roofs.Roofed(cell);            // 該格是否有屋頂
RoofDef def    = roofs.RoofAt(cell);            // 屋頂型別（null = 露天）

// 設定屋頂（會影響光照、降雨、之後的 region/room 重算）
roofs.SetRoof(cell, RoofDefOf.RoofConstructed);
roofs.SetRoof(cell, null);                    // 拆除屋頂

屋頂建造 / 拆除 area

玩家不是逐格蓋屋頂，而是劃定「建造屋頂區」(BuildRoofArea)與「拆除屋頂區」(NoRoofArea)兩張 area（RimWorld.Area_BuildRoof / RimWorld.Area_NoRoof，存於 map.areaManager）。殖民者在有牆支撐的範圍內把標記區內的格子蓋上 RoofConstructed。要在 mod 中程式化指定屋頂區：

Area_BuildRoof buildRoof = map.areaManager.BuildRoof;
buildRoof[cell] = true;          // 標記此格「應蓋屋頂」

Area_NoRoof noRoof = map.areaManager.NoRoof;
noRoof[cell] = true;             // 標記此格「應拆屋頂」

屋頂塌陷：RoofCollapseUtility

屋頂需要鄰近的支撐物（牆、柱、岩石）。當支撐被拆除/摧毀，無支撐的屋頂會塌陷砸傷下方一切。支撐距離由 Verse.RoofCollapseUtility.RoofMaxSupportDistance（原版約 6 格）決定——任何屋頂格若在該距離內找不到支撐就會被列入塌陷。判定由 RoofCollapseCellsFinder / RoofCollapserImmediate 執行。

型別 / 成員用途
RoofCollapseUtility.RoofMaxSupportDistance屋頂能離支撐最遠的距離（≈6）。
RoofCollapseUtility.WithinRangeOfRoofHolder(c, map)該格是否在某支撐物範圍內。
RoofCollapseCellsFinder.RemoveBulkCollapsingRoofs(...)計算並移除一批失去支撐的屋頂。
RoofCollapserImmediate.DropRoofInCells(cells, map)立即令指定格屋頂塌陷（含砸傷）。
塌陷會砸死人用 DropRoofInCells 或拆除支撐牆時，塌陷會對下方 pawn/物品造成 DamageDefOf.Crush（厚岩頂塌陷幾乎必殺）。製作「炸塌山頂壓敵人」之類機制時務必先確認落點，並避免砸到自己人。厚頂塌陷後留下 collapseLeavingThingDef（岩塊），會永久佔格。

屋頂對溫度/降雨/日照的影響

  • 溫度：屋頂是房間密閉的必要條件之一。漏一格頂，整個房間 UsesOutdoorTemperature 變 true（§1）。
  • 降雨/落塵：有頂的格子不會被雨淋濕、不沾毒性落塵的污染，pawn 在頂下也不會直接淋雨掉心情。
  • 日照：薄頂格室內光偏暗（影響太陽能板＝戶外才發電、室內植物需人工照明，串第18、19章）。厚岩頂下完全無天光。
§6 天氣與環境事件（WeatherDef / WeatherManager / GameConditionDef）

天氣 (Weather) 與遊戲條件 (GameCondition) 是兩個容易混淆的系統：天氣是地圖當下的「天空狀態」（晴/雨/霧/雪），由 WeatherManager 在 WeatherDef 間平滑過渡；GameCondition是疊加在天氣之上、有明確開始/結束的「事件狀態」（毒性落塵、火山冬天、極光、熱浪、寒流…）。一個 GameCondition 可以強制覆蓋天氣，但兩者資料結構獨立。

WeatherDef：天氣定義

欄位說明
rainRate降雨/降雪率（>0 會淋濕露天格、滅火、生長加成等）。
windSpeedFactor風速倍率（影響風力發電、火勢蔓延）。
accuracyMultiplier命中率倍率（霧/雨降低遠程命中，串第15章）。
moveSpeedMultiplier露天移動速度倍率（暴雪減速）。
overlayClasses畫面覆蓋特效（雨幕、雪幕、霧）的 SkyOverlay 類別清單（List<Type>）。
skyColorsNightMid / Day / Dusk各時段天空配色（SkyColorSet），決定畫面氛圍與環境光。
commonalityRainfallFactor / temperatureRange降雨量加權曲線（SimpleCurve）與此天氣可出現的溫度區間（生物群系/季節篩選）。
<WeatherDef>
  <defName>AshStorm</defName>
  <label>灰燼風暴</label>
  <description>火山灰遮天蔽日，能見度極差。</description>
  <rainRate>0</rainRate>
  <windSpeedFactor>2.2</windSpeedFactor>
  <accuracyMultiplier>0.5</accuracyMultiplier>
  <moveSpeedMultiplier>0.85</moveSpeedMultiplier>
  <perceivePriority>1</perceivePriority>
  <windSpeedOffset>1.2</windSpeedOffset>
  <overlayClasses>
    <li>WeatherOverlay_Fog</li>
  </overlayClasses>
  <skyColorsNightMid>
    <sky>(0.30, 0.26, 0.24)</sky>
    <shadow>(0.40, 0.36, 0.34)</shadow>
    <overlay>(0.55, 0.50, 0.46)</overlay>
    <saturation>0.7</saturation>
  </skyColorsNightMid>
  <weatherThought>null</weatherThought>
</WeatherDef>

WeatherManager：當前天氣與過渡

RimWorld.WeatherManager（map.weatherManager）持有 curWeather / lastWeather 與兩者間的混合度 curWeatherAge。天氣切換不是瞬間的——它在兩個 WeatherDef 間平滑插值天空配色、降雨幕等。下一個天氣由 WeatherDecider 根據生物群系的 baseWeatherCommonalities（WeatherCommonalityRecord.commonality）再乘上 commonalityRainfallFactor 與當前條件（季節、降雨量、溫度範圍）加權隨機選出。

// 讀取與強制天氣
WeatherDef now = map.weatherManager.curWeather;
float rain = map.weatherManager.RainRate;     // 已混合的實際降雨率

// 強制立刻切到指定天氣（debug / 事件用）
map.weatherManager.TransitionTo(DefDatabase<WeatherDef>.GetNamed("AshStorm"));

GameConditionDef：覆蓋全圖的事件狀態

Verse.GameConditionDef + 對應的 RimWorld.GameCondition 子類（RimWorld.GameCondition_*）構成「有時限的全圖環境效果」。它由 GameConditionManager（每張地圖一個 + 一個世界層級的）管理。許多事件（Incident，串第12章）的本體就是「註冊一個 GameCondition」：

GameCondition 子類效果
GameCondition_ToxicFallout毒性落塵：露天累積毒性、植物死亡。
GameCondition_VolcanicWinter火山冬天：日照與溫度下降。
GameCondition_Aurora極光：夜晚正面 thought。
GameCondition_HeatWave / ColdSnap熱浪 / 寒流：偏移室外溫度。
GameCondition_NoSunlight遮蔽天光（如機械族篩網事件）。

自訂一個簡單的 GameCondition（溫度偏移）

using RimWorld;
using Verse;

public class GameCondition_ChillFog : GameCondition
{
    private const float TempOffset = -12f;

    // 偏移室外溫度（每張受影響地圖會問所有 active condition）
    public override float TemperatureOffset()
    {
        return TempOffset;
    }

    // 覆蓋天空：讓畫面變冷色調的霧
    public override List<SkyOverlay> SkyOverlays(Map map)
    {
        return overlays;
    }
    private static readonly List<SkyOverlay> overlays =
        new List<SkyOverlay> { new WeatherOverlay_Fog() };
}
<!-- 對應的 GameConditionDef -->
<GameConditionDef>
  <defName>ChillFog</defName>
  <conditionClass>MyMod.GameCondition_ChillFog</conditionClass>
  <label>寒霧</label>
  <description>濃重的冷霧壓低了氣溫。</description>
  <endMessage>寒霧散去了。</endMessage>
  <canBePermanent>false</canBePermanent>
</GameConditionDef>

要由事件觸發它，寫一個 IncidentWorker（串第12章）在 TryExecuteWorker 內 RegisterCondition（見 §7），或在既有 IncidentWorker_MakeGameCondition 的 IncidentDef 上指定 gameCondition 欄位即可，後者完全免 C#。

commonality 不是出現機率本身生物群系 baseWeatherCommonalities 中的 commonality / WeatherDecider 的權重會被當下的季節溫度範圍、降雨量需求過濾後才生效。你的自訂天氣若 temperatureRange 設得很窄或與生物群系不符，可能永遠抽不到。debug 時用 map.weatherManager.TransitionTo 直接驗證視覺與數值，再回頭調權重。

讓自訂建築參與溫度交換：per-tick 注入 tempChange

核心模式：每（rare）tick 判斷房間是否封閉，再用 ControlTemperatureTempChange 取得本 tick 變化量寫回 room.Temperature，或直接 PushHeat。下面是一個「太陽燈式」恆溫推熱 comp，可掛在任何 ThingDef 上。

using RimWorld;
using Verse;

public class CompProperties_RoomHeater : CompProperties
{
    public float heatPerSecond = 10f;
    public float maxTemperature = 25f;
    public CompProperties_RoomHeater() { compClass = typeof(CompRoomHeater); }
}

public class CompRoomHeater : ThingComp
{
    CompProperties_RoomHeater Props => (CompProperties_RoomHeater)props;

    public override void CompTickRare()
    {
        Map map = parent.Map;
        if (map == null) return;

        // 需通電才生效（若有 CompPowerTrader）
        CompPowerTrader power = parent.TryGetComp<CompPowerTrader>();
        if (power != null && !power.PowerOn) return;

        Room room = parent.GetRoom();
        // 開放空間：CompHeatPusher 在這裡也會直接無效（熱量被當室外吸收）
        if (room == null || room.UsesOutdoorTemperature) return;
        if (room.Temperature >= Props.maxTemperature) return;

        // TickRare ≈ 4.1667 秒；換算成本 tick 的熱量再推
        float energy = Props.heatPerSecond * 4.1667f;
        GenTemperature.PushHeat(parent.Position, map, energy);
    }
}
PushHeat vs 直接寫 room.TemperaturePushHeat 會把能量正確分攤到房間熱容量（大房間升溫慢、小房間快），是物理上正確的做法；直接 room.Temperature += x 則忽略熱容量、所有房間升幅一樣。除非你刻意要「瞬間設定房間溫度」（如事件），否則一律用 PushHeat / ControlTemperatureTempChange。
§8 陷阱與最佳實務

房間未密閉 → UsesOutdoorTemperature

缺一塊屋頂、漏一格牆、或房間過大都會讓 UsesOutdoorTemperature 變 true，溫度死貼室外。控溫前永遠先 if (room == null || room.UsesOutdoorTemperature) return;。

CompHeatPusher 在開放空間無效

HeatPusher 推的熱在室外/漏天房間會被即時吸收，看起來「沒作用」。柴爐放戶外不會暖——這是設計而非 bug。提示玩家需封閉空間。

毒氣網格效能

GasGrid 每 tick 對全圖開放格做擴散；大地圖（如 400×400）上瘋狂 AddGas 會拖垮 tick。用 CompTickRare 釋放、限制範圍、達上限後停止注入（串第36章）。

屋頂塌陷誤傷

拆掉承重牆或厚岩頂塌陷會 Crush 下方一切。程式化塌陷前先掃落點 pawn，避免砸到友軍；厚頂塌陷還會留岩塊永久佔格。

GameCondition 與 WeatherDef 重疊

GameCondition 可覆蓋天氣與天空（SkyOverlays/TemperatureOffset），多個條件的溫度偏移會疊加。自訂時注意別跟毒性落塵/火山冬天衝突造成溫度疊到離譜。

跨 DLC 的 pollution 軟依賴

污染、wastepack、polluted terrain 屬 Biotech。XML 用 MayRequire="Ludeon.RimWorld.Biotech"；C# 先判 ModsConfig.BiotechActive 再進隔離方法，避免無 DLC 玩家 TypeLoadException。

其他易錯點

  • 溫度單位：API 一律攝氏。玩家設定可能顯示華氏，但程式內部與 def 一律 °C，別在 def 寫成華氏數值。
  • Region/Room 重算時機：改屋頂或牆後，房間不一定立刻更新；別在同一 tick 內既 SetRoof 又依賴新的 GetRoom 結果，必要時等 RegionAndRoomUpdater 處理完。
  • map 可能為 null：物品在背包、運輸中、未 spawn 時 parent.Map == null。所有溫度/氣體/屋頂查詢前先判 map。
  • 太陽能/室內光：屋頂遮天會讓太陽能板不發電、戶外作物不長——這是溫度章節之外但常一起踩的環境耦合（串第18、19章）。
  • DefOf 快取：頻繁用到的 RoofDef/GasType/GameConditionDef 用 [DefOf] 靜態快取，別每 tick GetNamed（字串查表慢，串第36章）。
除錯小抄開發者模式下：God mode + 溫度工具可直接畫溫度；map.weatherManager.TransitionTo 強切天氣；debug action「Do incident → 各 GameCondition」可立即觸發毒性落塵/火山冬天驗證 TemperatureOffset 疊加；查房間用 debug 的 Room 高亮工具確認 UsesOutdoorTemperature。
