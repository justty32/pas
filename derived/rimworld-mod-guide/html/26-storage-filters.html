儲存、分類與篩選

ThingCategoryDef 樹狀分類、ThingFilter 與 StorageSettings、繼承 Building_Storage 自訂儲存建築、物品堆疊、ThingRequest + ListerThings 查詢，以及 SpecialThingFilterDef 與常見坑。

RimWorld 1.5 · Verse / RimWorld · Storage, Categories & Filters
1. 心智模型：分類樹 → 篩選器 → 儲存設定

RimWorld 的儲存系統由三層概念疊起來，務必先分清楚誰是誰： ThingCategoryDef 是靜態的「物品分類樹」（編譯/載入期決定）， ThingFilter 是一份可變的「哪些物品被允許」的勾選狀態（玩家在 UI 裡勾的就是它）， StorageSettings 則在 ThingFilter 之外再加上「優先度」並把它掛到一個儲存載體上。

ThingCategoryDef

靜態分類樹的節點，有 parent 與 childCategories，葉節點掛著一堆 ThingDef。這是篩選器 UI 樹狀展開的骨架。

ThingFilter

執行期的「允許集合」。內含 allowedDefs（攤平後的最終結果）與設定來源（categories / specialFilters / 品質 / hitpoints 範圍）。

StorageSettings

= 一個 ThingFilter + 一個 Priority，並持有指向其擁有者（IStoreSettingsParent）的參照。儲存格、貨架、傾印區用的都是它。

關鍵分野 ThingCategoryDef 是「世界上有哪些分類」（共享、唯讀、全 mod 共用一棵樹）；ThingFilter 是「這個容器接受哪些東西」（每個容器一份、可被玩家改）。 新手最常把兩者混為一談——你不會為了「我的箱子只收木頭」去改分類樹，而是去設定那個箱子的 ThingFilter。

三層關係

ThingCategoryDef（樹，全域共享、靜態）
 └─ 提供 UI 樹的結構 + 「整類勾選」的展開來源
        ↓ 被引用
ThingFilter（每容器一份，可變）
 ├─ categories / disallowedCategories  ← 來源設定
 ├─ specialFiltersToAllow/Disallow     ← SpecialThingFilterDef
 ├─ allowedQualities / allowedHitPointsPercents
 └─ allowedDefs（HashSet<ThingDef>）    ← 最終攤平結果，查詢就看它
        ↓ 被包進
StorageSettings（ThingFilter + Priority + owner）
        ↓ 掛在
IStoreSettingsParent（Building_Storage、Zone_Stockpile…）
2. ThingCategoryDef 階層

ThingCategoryDef（Verse 命名空間）以 parent 構成單一父鏈、自動回填 childCategories 與 childThingDefs， 形成一棵全域共享的分類樹。根節點是內建的 Root。樹是在 def 載入後由 ThingCategoryDef.ResolveReferences / DefDatabase 後處理階段建立雙向連結的。

ThingCategoryDef 重要欄位

欄位型別說明
defNamestring唯一名，ThingDef 的 thingCategories 以此引用
labelstringUI 顯示名（篩選器樹上的文字）
parentThingCategoryDef父分類；不填則只能掛在 Root 下 建議填
childCategoriesList<ThingCategoryDef>執行期自動回填，勿手寫
childThingDefsList<ThingDef>執行期自動回填：所有 thingCategories 指向本類的 ThingDef
childSpecialFiltersList<SpecialThingFilterDef>掛在此分類下的特殊篩選（如「允許腐敗」）
resourceReadoutRootbool是否作為左下資源面板的彙總根

新增一個分類（掛到內建分類樹下）

<Defs>
  <!-- 在內建 ResourcesRaw 下新增子分類 -->
  <ThingCategoryDef>
    <defName>MyMod_ExoticOres</defName>
    <label>exotic ores</label>
    <!-- 關鍵：parent 指向已存在的分類 defName -->
    <parent>ResourcesRaw</parent>
  </ThingCategoryDef>
</Defs>
不要手寫 childCategories / childThingDefs 這兩個 List 由引擎在載入後雙向回填：分類的父子靠 parent 建立，分類與物品的關係靠 ThingDef 端的 thingCategories 建立。 手寫只會被覆蓋或造成重複，永遠從「子指向父」的方向宣告。

常見內建分類（作為 parent 的落點）

defName用途
Root整棵樹的根，一般不直接掛
ResourcesRaw原物料（木材、鋼鐵、布料…）
Manufactured加工/製成品（零件、藥物…）
Foods / FoodMeals / FoodRaw食物相關
Weapons / Apparel武器 / 衣物
Items雜項物品的常見落點
3. ThingDef.thingCategories — 把物品歸類

物品要出現在篩選器樹的某個分類下，方向是反過來的：不是在分類裡列出物品，而是在 ThingDef 上用 <thingCategories> 列出它屬於哪些分類。引擎會據此回填到對應 ThingCategoryDef 的 childThingDefs。 

<ThingDef ParentName="ResourceBase">
  <defName>MyMod_Voidstone</defName>
  <label>voidstone</label>
  <stackLimit>75</stackLimit>
  <!-- 一個物品可同時屬於多個分類；篩選器樹中會出現在每一處 -->
  <thingCategories>
    <li>MyMod_ExoticOres</li>
    <li>ResourcesRaw</li>
  </thingCategories>
</ThingDef>
最常見坑：thingCategories 沒掛 → 物品在篩選器裡「不存在」 若一個 ThingDef 完全沒有 thingCategories（且其 ParentName 也沒帶），它就不屬於任何分類節點， 於是無法被任何 StorageSettings 的「整類勾選」收納，也不會出現在貯藏區/貨架的篩選器樹上。 症狀：物品生得出來、能撿、但永遠被標為「不允許貯藏」殖民者不搬。第 10 節有完整排查。
透過 ParentName 繼承分類 多數情況你會 ParentName="ResourceBase" 之類繼承一個 abstract base，base 已帶 thingCategories。 若你在子 def 重寫 <thingCategories>，預設是整段取代（非合併）；要追加用 <thingCategories>...<li>X</li></thingCategories> 並自行把 base 的也列上，或用 Inherit="False" 控制。
4. ThingFilter — 允許集合的核心

ThingFilter（RimWorld 命名空間）是真正決定「哪些東西被允許」的物件。 它對外暴露多種「設定維度」，但最終都會解析（resolve）成一個攤平的 allowedDefs（HashSet<ThingDef>）。查詢時呼叫 Allows(thing) 即可。

ThingFilter 設定維度

維度API / 欄位說明
整類允許SetAllow(ThingCategoryDef, bool)勾整個分類（含子類遞迴展開）
單品允許SetAllow(ThingDef, bool)勾單一物品
特殊篩選SetAllow(SpecialThingFilterDef, bool)如「允許腐敗 / 允許汙染」
品質範圍allowedQualities (QualityRange)武器/衣物等有品質者
耐久範圍allowedHitPointsPercents (FloatRange)0~1，過濾破損品
最終結果AllowedThingDefs / allowedDefs攤平後的允許集合（唯讀枚舉）

常用 ThingFilter API

using Verse;
using RimWorld;

ThingFilter f = new ThingFilter();

// 1) 整類允許（會遞迴納入該分類底下所有 childThingDefs）
f.SetAllow(ThingCategoryDefOf.Foods, true);

// 2) 在整類基礎上排除單品
f.SetAllow(ThingDefOf.MealNutrientPaste, false);

// 3) 特殊篩選：不允許腐敗中的食物
f.SetAllow(SpecialThingFilterDefOf.AllowRotten, false);

// 4) 品質 / 耐久範圍（對有品質的物品才有意義）
f.allowedHitPointsPercents = new FloatRange(0.5f, 1f);

// 5) 查詢：某 Thing 是否被允許
bool ok = f.Allows(someThing);          // 綜合所有維度
bool okDef = f.Allows(someThing.def);     // 只看 def 層級

// 6) 全勾 / 全清（常用於初始化）
f.SetAllowAll(parentFilter: null);          // 全允許（受 parent 限制）
f.SetDisallowAll();                          // 全不允許
parentFilter（父篩選器）：合法範圍的上界 多數 ThingFilter 解析時會帶一個 parentFilter——它界定「這個容器理論上可能接受的最大集合」。 例如某種專收武器的貨架，其 parentFilter 只含武器類，玩家就無法在 UI 裡勾出食物。 SetAllowAll(parentFilter) 的「全勾」其實是「勾滿 parentFilter 容許的範圍」。

ThingFilter 的 XML 形態

ThingFilter 也能直接寫在 def 裡（例如 fixedStorageSettings、配方的 fixedIngredientFilter）。XML 結構對應上述維度：

<filter>
  <!-- 整類允許 -->
  <categories>
    <li>ResourcesRaw</li>
  </categories>
  <!-- 在整類中額外排除的分類 -->
  <disallowedCategories>
    <li>Wools</li>
  </disallowedCategories>
  <!-- 額外點名允許的單品（不在上述分類也會被收） -->
  <thingDefs>
    <li>MyMod_Voidstone</li>
  </thingDefs>
  <!-- 點名排除的單品 -->
  <disallowedThingDefs>
    <li>Steel</li>
  </disallowedThingDefs>
  <!-- 特殊篩選 -->
  <specialFiltersToDisallow>
    <li>AllowRotten</li>
  </specialFiltersToDisallow>
</filter>
5. StorageSettings 與 IStoreSettingsParent

StorageSettings = ThingFilter filter + StoragePriority priority + 指向擁有者的 owner（IStoreSettingsParent）。所有可貯藏載體——貯藏區（Zone_Stockpile）、 貨架/箱子（Building_Storage）、傾印區——都透過實作 IStoreSettingsParent 來持有它。

IStoreSettingsParent 介面

public interface IStoreSettingsParent
{
    // 這個容器目前生效的設定（玩家可改的那份）
    StorageSettings GetStoreSettings();

    // 上界：父設定，限制玩家在 UI 裡能勾的最大範圍（可回 null）
    StorageSettings GetParentStoreSettings();

    // 1.5+：UI 是否顯示「優先度」下拉（傾印區之類可能不顯示）
    bool StorageTabVisible { get; }

    // 設定變更後的回呼（讓搬運工作重算）
    void Notify_SettingsChanged();
}

StoragePriority 與查詢

值說明
Unstored未貯藏（地上散落），最低
Low / Normal / Preferred / Important / Critical由低到高；殖民者把物品從低優先度搬向高優先度的合法格
// 找「能收這個 thing、且優先度高於現處」的最佳貯藏格
IntVec3 cell;
bool found = StoreUtility.TryFindBestBetterStoreCellFor(
    thing, carrier, map,
    StoreUtility.CurrentStoragePriorityOf(thing),
    carrier.Faction, out cell);

// 某格目前的貯藏設定是否允許某 thing
bool ok = thing.Position.GetSlotGroup(map)
              ?.Settings.AllowedToAccept(thing) ?? false;
SlotGroup：把「格子」與「設定」綁起來的中介 每個 IStoreSettingsParent 對應一個 SlotGroup（thing.Position.GetSlotGroup(map) 取得）， 由 map.haulDestinationManager 統一管理。搬運 AI 找落點時走的是 SlotGroup，不是直接問建築。
6. 自訂儲存建築（繼承 Building_Storage）

想做「貨架 / 保險箱 / 專用容器」最省事的做法是繼承 Building_Storage——它已實作 IStoreSettingsParent、自帶 Storage gizmo、會把自身格子註冊成 SlotGroup。 你只需在 XML 給它 building.fixedStorageSettings（上界）與 defaultStorageSettings（預設勾選）。

XML：貨架類建築定義

<ThingDef ParentName="BuildingBase">
  <defName>MyMod_OreRack</defName>
  <label>ore rack</label>
  <thingClass>MyMod.Building_OreRack</thingClass>
  <category>Building</category>
  <building>
    <!-- 標記為可貯藏建築；多格貨架 -->
    <fixedStorageSettings>
      <filter>
        <!-- 上界：這個架子最多只能設成收原物料 -->
        <categories>
          <li>ResourcesRaw</li>
        </categories>
      </filter>
    </fixedStorageSettings>
    <defaultStorageSettings>
      <priority>Important</priority>
      <filter>
        <!-- 預設勾：玩家放下時就生效的初始選擇 -->
        <categories>
          <li>ResourcesRaw</li>
        </categories>
      </filter>
    </defaultStorageSettings>
  </building>
  <!-- 多格、可被踏入的貨架（依需求）-->
  <size>(1,2)</size>
  <passability>PassThroughOnly</passability>
</ThingDef>

C#：擴充 Building_Storage

using Verse;
using RimWorld;

public class Building_OreRack : Building_Storage
{
    // 物品被放入此貯藏格時觸發（含搬運送達）
    public override void Notify_ReceivedThing(Thing newItem)
    {
        base.Notify_ReceivedThing(newItem);
        // 例：放入即標記、播效果、更新貼圖層數…
    }

    // 物品離開此格時觸發
    public override void Notify_LostThing(Thing newItem)
    {
        base.Notify_LostThing(newItem);
    }

    // 可選：對接受條件再加自訂限制（如只在通電時收）
    public override bool Accepts(Thing t)
    {
        if (!base.Accepts(t)) return false;
        var power = GetComp<CompPowerTrader>();
        return power == null || power.PowerOn;   // 沒電就不收
    }
}
Building_Storage 已幫你做掉的事 Storage gizmo（含 Copy/Paste 設定按鈕）、SlotGroup 註冊、與搬運 AI 的對接、設定持久化（存檔）。 你幾乎只需處理「收到/失去物品的副作用」與「額外接受條件（Accepts override）」。
多格 vs 單格貨架的容量 原版貯藏建築每格仍受該物品 stackLimit 限制。要「一格放超量」需另寫邏輯（如 LWM Deep Storage 的做法：override 容量計算與堆疊顯示）。單純繼承 Building_Storage 不會自動突破 stackLimit。
7. 物品堆疊：stackLimit 與堆疊合併

堆疊上限由 ThingDef 的 <stackLimit> 決定（資源類常見 75 或 100，餐點常為 10，武器/衣物為 1 不可疊）。 合併、拆分由 Thing.stackCount 與 TryAbsorbStack / SplitOff 處理。 可疊與否並非靠額外 Comp，而是直接看 stackLimit > 1。

堆疊相關欄位 / API

成員說明
def.stackLimit單堆最大數量；1 表示不可疊
thing.stackCount當前這堆的數量
thing.TryAbsorbStack(other, respect)嘗試把 other 併入本堆（受 stackLimit 約束）
thing.SplitOff(count)從本堆拆出 count 個成為新 Thing
thing.CanStackWith(other)兩者能否合併（同 def、同品質/材質/腐敗度…）
// 從一堆木頭拆 30 出來，生成獨立 Thing
Thing wood = ...;                 // stackCount = 100
Thing portion = wood.SplitOff(30);  // portion=30, wood=70

// 把兩堆合併（若同 def 且未達 stackLimit）
if (a.CanStackWith(b))
    a.TryAbsorbStack(b, respectStackLimit: true);
「CompStackable」其實不是原版類別 原版 RimWorld 的堆疊能力內建於 Thing/ThingDef（看 stackLimit），並無一個叫 CompStackable 的官方 Comp。 某些 mod 會自製類似 Comp 來擴充「不可疊物品也能疊」或「巨量堆疊」。若你看到 CompStackable，它幾乎都來自第三方框架， 請勿假設核心 API 存在它。要改堆疊行為，調 stackLimit 或 override 相關方法才是原版路徑。
品質/材質會阻止合併 CanStackWith 嚴格：不同材質（鋼鐵 vs 木製）、不同品質、不同腐敗進度、被汙染與否都會讓兩堆無法合併。 做「自動整理」類功能時別假設同 defName 就能併。
8. ThingRequest 與 ListerThings — 查詢地圖物品

要在地圖上找「所有某類物品」，不要自己掃格子，用 map.listerThings。它以 ThingRequest 為查詢鍵，內部維護多種預先分組的清單（依 def、依 group），查詢是 O(結果數) 而非 O(全圖)。

ThingRequest 三種建構

建構回傳備註
ThingRequest.ForDef(ThingDef)該 def 的所有實例最精確、最快
ThingRequest.ForGroup(ThingRequestGroup)整個語意群組如 HaulableEver、FoodSource、Weapon
ThingRequest.ForUndefined()無分組查詢少用

ListerThings 查詢

using System.Collections.Generic;
using Verse;
using RimWorld;

Map map = ...;

// A) 依 def：取地圖上所有鋼鐵堆
List<Thing> allSteel =
    map.listerThings.ThingsOfDef(ThingDefOf.Steel);

// B) 依群組：所有「曾可搬運」的物品
List<Thing> haulables =
    map.listerThings.ThingsInGroup(ThingRequestGroup.HaulableEver);

// C) 用 ThingRequest 物件（給需要 request 參數的 API）
ThingRequest req = ThingRequest.ForGroup(ThingRequestGroup.FoodSource);

// D) 配合 GenClosest 找最近且通過篩選的一個
Thing nearestFood = GenClosest.ClosestThingReachable(
    pawn.Position, map, req,
    PathEndMode.ClosestTouch,
    TraverseParms.For(pawn),
    maxDistance: 9999f,
    validator: t => myFilter.Allows(t) && !t.IsForbidden(pawn));
不要 cache ThingsOfDef 回傳的 List ThingsOfDef / ThingsInGroup 回的是 lister 內部的活清單，會隨生成/銷毀即時變動。 若你要在迴圈中刪改物品，先 .ToList() 複製一份再迭代，否則會「集合在迭代中被修改」例外或漏掉元素。
常用 ThingRequestGroup HaulableEver / HaulableAlways、FoodSource、Weapon、Apparel、 Plant、Corpse、BuildingArtificial、Pawn。群組由引擎在物品生成時自動歸入（依 def 屬性），查詢免維護。
9. ThingDefCountClass — 成本 / 產出列表

凡是「一個 def + 一個數量」的清單（建築造價、配方產出、貿易包內容）都用 ThingDefCountClass（欄位 thingDef 與 count）。它的 XML 寫法很特別： 用 屬性式 <li>ThingDefName>Count</li> 形式（自訂解析），而非巢狀子節點。

XML：建築造價

<ThingDef>
  <defName>MyMod_OreRack</defName>
  ...
  <costList>
    <!-- 格式：<li>ThingDef>數量</li> -->
    <li>Steel>75</li>
    <li>ComponentIndustrial>2</li>
  </costList>
</ThingDef>

C#：建構與讀取

using System.Collections.Generic;
using Verse;
using RimWorld;

// 程式化建立一張產出/成本表
List<ThingDefCountClass> products = new()
{
    new ThingDefCountClass(ThingDefOf.Steel, 15),
    new ThingDefCountClass(ThingDefOf.ChunkSlagSteel, 1),
};

// 例：依表生成物品到某格
foreach (ThingDefCountClass tc in products)
{
    Thing made = ThingMaker.MakeThing(tc.thingDef);
    made.stackCount = tc.count;
    GenPlace.TryPlaceThing(made, cell, map, ThingPlaceMode.Near);
}

// 讀取建築的造價（cost）
List<ThingDefCountClass> cost = myDef.costList;
相關的「def + 數量」型別 ThingDefCountClass（含 count，給造價/產出）；IngredientCount（配方材料，含 filter + 數量，較複雜）； ThingDefCount（struct 版，無 XML 解析、純值傳遞用）。寫配方 RecipeDef 用的是 ingredients(IngredientCount) 與 products(ThingDefCountClass)。
10. SpecialThingFilterDef — 特殊篩選

有些「篩選條件」不是分類、也不是單品，而是一種動態謂詞（依物品執行期狀態判斷）， 例如「允許腐敗中的食物」「允許汙染品」「允許無利刃武器」。這類由 SpecialThingFilterDef 定義， 搭配一個 SpecialThingFilterWorker 子類實作判斷邏輯。

內建範例

defName含意
AllowRotten是否允許腐敗（含腐爛進行中）的食物/屍體
AllowFresh是否允許新鮮（未腐）品
AllowDeadmansApparel是否允許死人衣
AllowBiocodedWeapons是否允許生物編碼武器

自訂 SpecialThingFilterDef + Worker

<SpecialThingFilterDef>
  <defName>MyMod_AllowCharged</defName>
  <label>allow charged cells</label>
  <description>Allow energy cells that still hold charge.</description>
  <parentCategory>Items</parentCategory>   <!-- 掛到分類樹的哪個節點 -->
  <allowedByDefault>true</allowedByDefault>
  <workerClass>MyMod.SpecialThingFilterWorker_Charged</workerClass>
</SpecialThingFilterDef>
using Verse;

public class SpecialThingFilterWorker_Charged : SpecialThingFilterWorker
{
    // 回傳「這個 thing 是否被本特殊篩選視為『匹配』」
    public override bool Matches(Thing t)
    {
        var comp = t.TryGetComp<CompCharge>();
        return comp != null && comp.ChargePercent > 0f;
    }

    // 此特殊篩選對哪些 def 「可能適用」（決定它是否出現在該 def 的篩選 UI）
    public override bool CanEverMatch(ThingDef def)
    {
        return def.HasComp(typeof(CompCharge));
    }
}
Matches vs CanEverMatch CanEverMatch(def) 是靜態可能性（決定 UI 要不要顯示這個勾選項給該類物品）； Matches(thing) 是執行期實際判定（這一件具體物品此刻算不算）。兩者都要實作，否則 UI 不顯示或判定全錯。
11. 常見坑與排查清單

thingCategories 未掛載

ThingDef 沒任何 thingCategories（自身與 ParentName 都沒），物品不屬於任何分類節點，篩選器找不到、整類勾選收不到、殖民者不搬。最高頻坑。

parent 指向不存在的分類

ThingCategoryDef 的 parent 或 ThingDef 的 thingCategories 寫了不存在的 defName，啟動 log 會報 cross-reference 解析失敗，該節點被丟棄。

filter 階層 / parentFilter 矛盾

容器的 fixedStorageSettings（上界）不含某分類，玩家就永遠勾不出它。「我的貨架怎麼勾不到木頭」常是 fixedStorageSettings 把它擋在上界外。

手寫 childThingDefs / childCategories

這兩個是引擎回填欄位，手寫會被覆蓋或重複。分類關係只能從「子→父」(parent / thingCategories) 宣告。

誤以為 CompStackable 是原版

原版堆疊內建於 ThingDef.stackLimit，無官方 CompStackable。看到它多半是第三方框架，別假設核心存在。

cache 了 ListerThings 活清單

ThingsOfDef/ThingsInGroup 回傳內部活 List，物品增刪即變。迭代中要改物品先 .ToList() 複製。

「物品無法被貯藏 / 搬運工不搬」排查步驟

  1. 開 dev mode，啟動 log 是否有此 ThingDef / ThingCategoryDef 的 cross-reference 解析錯誤。
  2. 確認該 ThingDef（含 ParentName 繼承）至少有一個 thingCategories 條目。
  3. 在遊戲內打開任一貯藏區的篩選器樹，確認物品出現在預期分類下；找不到 → 回步驟 2。
  4. 確認目標容器的 fixedStorageSettings（上界）涵蓋該分類，否則玩家勾不到。
  5. 確認 defaultStorageSettings 或玩家手動勾選有實際允許它（filter.Allows(thing) 為 true）。
  6. 確認物品屬於 HaulableEver 群組（def.EverHaulable），否則搬運 AI 根本不考慮它。
  7. 用 map.listerThings.ThingsOfDef(yourDef) 確認物品確實存在於地圖、未被 forbidden。
快速自測一行 在 dev 動作裡對某物件呼叫 Log.Message(thing.def.thingCategories?.Count.ToString() ?? "NULL"); 若印出 NULL 或 0，分類沒掛——這就是它消失在篩選器與搬運邏輯之外的根因。
