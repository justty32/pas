1. Transpiler 深入：IL 指令操作

Prefix / Postfix 只能在方法前後切入；當你必須改寫方法內部的某段邏輯 （而不能整個取代）時，Transpiler 是唯一選擇。Transpiler 接收原方法的 IL 指令串 （IEnumerable<CodeInstruction>），回傳改寫後的指令串。本質上你是在編譯後的 CIL 層級做手術。

Transpiler 是最後手段 它對目標方法的 IL 結構高度敏感，任何遊戲更新都可能讓你的指令比對失效。能用 Prefix/Postfix 解決就別用 Transpiler。 務必加上「找不到目標就拋例外或記錄並原樣回傳」的防呆。

CodeInstruction 與 OpCodes 對應 C#

每個 CodeInstruction 包含一個 opcode（System.Reflection.Emit.OpCode）與選用的 operand。常見對應如下：

C# 概念典型 OpCodeoperand 內容
載入區域變數 / 參數Ldloc, Ldloc_0..3, Ldarg, Ldarg_0區域變數索引 / 參數索引
儲存到區域變數Stloc, Stloc_0..3區域變數索引
載入整數常數Ldc_I4, Ldc_I4_S, Ldc_I4_0..8整數值（短型用 _S）
呼叫方法Call, CallvirtMethodInfo
讀 / 寫欄位Ldfld, Stfld, Ldsfld, StsfldFieldInfo
比較Cgt, Clt, Ceq無
條件跳躍Brfalse, Brtrue, Beq, Ble, BgtLabel
回傳Ret無
查 IL 的工具 用 dnSpy / ILSpy 開啟 Assembly-CSharp.dll，切到「IL」檢視模式，逐條對照目標方法的指令， 就能寫出對應的比對與替換邏輯。沒看過實際 IL 而硬寫 Transpiler 幾乎一定錯。

定位指令模式

典型做法是把指令串轉成 List，用迴圈掃描特徵指令（anchor），再以其為基準插入 / 刪除 / 替換。

static IEnumerable<CodeInstruction> Transpiler(IEnumerable<CodeInstruction> instructions)
{
    var codes = new List<CodeInstruction>(instructions);

    for (int i = 0; i < codes.Count; i++)
    {
        // 找出「呼叫某個特定方法」的位置作為錨點
        if (codes[i].opcode == OpCodes.Call
            && codes[i].operand is MethodInfo mi
            && mi.Name == "SomeTargetMethod")
        {
            FileLog.Log($"[MyMod] 在 index {i} 找到 SomeTargetMethod");
            // ... 在此插入 / 刪除 / 替換 ...
            break;
        }
    }
    return codes;
}

插入 / 刪除 / 替換

// ① 插入：在錨點「前」插入兩條指令（載入 this 與呼叫自訂 hook）
codes.InsertRange(i, new[]
{
    new CodeInstruction(OpCodes.Ldarg_0),
    new CodeInstruction(OpCodes.Call,
        AccessTools.Method(typeof(MyHooks), nameof(MyHooks.OnBeforeCall))),
});

// ② 刪除：移除錨點當條指令（例如砍掉一次多餘呼叫）
codes.RemoveAt(i);

// ③ 替換：把「呼叫 A」改成「呼叫 B」(保留 operand 以外其他資訊)
codes[i].operand = AccessTools.Method(typeof(MyHooks), nameof(MyHooks.Replacement));
Label 與 ExceptionBlocks 不要弄丟 若有跳躍指令的目標 Label 或 try/catch 區塊掛在你要刪除 / 替換的那條指令上， 必須把 labels / blocks 移轉到新指令（newInstr.labels.AddRange(old.labels)）， 否則 IL 會驗證失敗、整個遊戲崩潰。安全做法是「替換 operand」而非「整條換掉」。

完整範例：把 if (x > 5) 改成 if (x > 10)

假設原始 C# 是 if (x > 5) { ... }。編譯後常見為： ldloc x → ldc.i4.5 → cgt → brfalse ...。 我們只要把那條 ldc.i4.5 的常數從 5 改成 10。 

// 目標：將比較門檻常數 5 → 10
static IEnumerable<CodeInstruction> Transpiler(IEnumerable<CodeInstruction> instructions)
{
    var codes = new List<CodeInstruction>(instructions);
    bool patched = false;

    for (int i = 0; i < codes.Count; i++)
    {
        // 比對「載入常數 5 後面緊接 cgt」這個特徵，避免誤改其他 5
        bool isLoad5 =
            (codes[i].opcode == OpCodes.Ldc_I4_5) ||
            (codes[i].opcode == OpCodes.Ldc_I4_S && Convert.ToInt32(codes[i].operand) == 5) ||
            (codes[i].opcode == OpCodes.Ldc_I4   && Convert.ToInt32(codes[i].operand) == 5);

        bool nextIsCgt = i + 1 < codes.Count && codes[i + 1].opcode == OpCodes.Cgt;

        if (isLoad5 && nextIsCgt)
        {
            // 直接把這條改成「載入 10」（用通用的 Ldc_I4 + operand 最穩）
            codes[i].opcode  = OpCodes.Ldc_I4;
            codes[i].operand = 10;
            patched = true;
            FileLog.Log($"[MyMod] 門檻 5 → 10 已替換 @ index {i}");
            break;
        }
    }

    if (!patched)
        FileLog.Log("[MyMod] 警告：未找到 (x > 5) 模式，Transpiler 未生效！");

    return codes;
}
用 FileLog.Log 除錯 Transpiler Transpiler 在「方法首次被 patch」時執行一次，此時遊戲 log 系統未必方便觀察。 Harmony 內建的 HarmonyLib.FileLog.Log(string) 會寫到桌面的 harmony.log.txt， 最適合驗證「有沒有找到錨點、改了哪個 index」。除錯完記得移除或設旗標關閉。
2. 性能最佳化

RimWorld 是 tick-based 模擬，TickRate 預設每秒 60 ticks，大型殖民地可有上千個 Thing 同時 tick。 Mod 在 hot path 上的任何浪費都會被放大成卡頓，性能意識是進階 modder 的基本功。

避免在 Tick() 重運算

Thing.Tick() / TickRare() 是每幀（或每 250 ticks）呼叫的 hot path。 把不會每幀變的結果快取在欄位，只在輸入變動時重算，不要每 tick 做反射、LINQ、字串格式化或集合配置。

用偶發 tick 分攤負載

不需要每幀精度的邏輯改用 TickRare()（每 250 ticks ≈ 4.16 秒）或 TickLong()（每 2000 ticks）。 或自行用 Find.TickManager.TicksGame 配合 GenTicks.TicksGame % N == offset 把工作分散到不同幀。

GenTicks 偶發 tick：把負載打散

public override void Tick()
{
    base.Tick();

    // 每 60 ticks（約 1 秒）才做一次昂貴檢查，
    // 並以 thingIDNumber 當 offset，讓不同物件落在不同幀，避免同幀尖峰
    if ((GenTicks.TicksGame + thingIDNumber) % 60 == 0)
    {
        DoExpensiveScan();
    }
}
HashOffsetTicks 慣例 RimWorld 自身大量使用 this.HashOffsetTicks() / this.IsHashIntervalTick(interval) 來實作「分散式間隔 tick」。優先用 this.IsHashIntervalTick(250) 取代自行寫 % 250，語意更清楚且已內建 hash 打散。

快取反射結果（MethodInfo / FieldInfo）

AccessTools.Method / AccessTools.Field 內部做字串查找與反射，絕不可在 hot path 重複呼叫。一律在靜態欄位快取一次。

public static class Reflect
{
    // 一次性快取：型別載入時解析一次，之後零反射查找開銷
    public static readonly FieldInfo  HungerField =
        AccessTools.Field(typeof(Need_Food), "curLevelInt");

    public static readonly MethodInfo RecacheMethod =
        AccessTools.Method(typeof(Pawn_NeedsTracker), "Notify_NeedAdded");
}

// 使用（無反射查找開銷，僅一次 GetValue/Invoke）
float hunger = (float)Reflect.HungerField.GetValue(pawn.needs.food);
更進一步：委派快取 連 FieldInfo.GetValue 的反射裝箱開銷都想省，可用 Harmony 的 AccessTools.FieldRefAccess<TInstance, TField>(name) 取得強型別存取委派， 或用 MethodInfo.CreateDelegate 把 MethodInfo 編成委派，hot path 上接近原生呼叫速度。

HashSet vs List：選對容器

操作List<T>HashSet<T>建議
包含判斷 ContainsO(n) 線性掃描O(1) 雜湊頻繁查存在性 → HashSet 快
循序走訪快、cache 友善較慢、無序只迭代不查存在 → List
索引存取 [i]O(1)不支援需索引 → List
去重需手動天生去重需唯一集合 → HashSet
// 反例：在 Tick 裡對大 List 反覆 Contains → O(n) 每幀
if (trackedThings.Contains(thing)) { ... }   // ❌ List 上是線性掃描

// 正解：存在性查詢改用 HashSet
private readonly HashSet<Thing> _tracked = new HashSet<Thing>();
if (_tracked.Contains(thing)) { ... }   // ✓ O(1)
Unity Profiler 抓真正瓶頸 別憑感覺優化。RimWorld 用 Unity 引擎，可掛 Unity Profiler（或 Dev mode 下的內建除錯計量、社群的 Dubs Performance Analyzer mod）找出實際吃 CPU 的方法。 先量測、再優化；把時間花在 self-time 最高的呼叫上，而不是看起來「應該很慢」的地方。
3. Scribe 存檔系統深入

RimWorld 的存讀檔由 Scribe 系統驅動，採「同一份 ExposeData() 同時負責存與讀」的對稱設計。 任何要進存檔的自訂類別實作 IExposable 並覆寫 ExposeData()。 Scribe.mode 指示當前是 Saving、LoadingVars 等階段。

Scribe_* 方法對照

方法適用型別說明
Scribe_Values.Look(ref v, "label", default)值型別 / string / enum基本值；提供 default 在缺欄位時回退 最常用
Scribe_Defs.Look(ref def, "label")任何 Def以 defName 序列化，讀檔時查 DefDatabase 還原
Scribe_References.Look(ref thing, "label")ILoadReferenceable（Thing/Pawn 等）存「引用」而非物件本體；需該物件由別處 Deep 存下 注意順序
Scribe_Deep.Look(ref obj, "label", ctorArgs)IExposable把物件內容整個存進去；擁有者用此存其子物件
Scribe_Collections.Look(ref list, "label", LookMode.X)List / Dictionary / HashSet須指定 LookMode（Value/Def/Reference/Deep）
Deep vs Reference 的關鍵差異 Scribe_Deep 存的是「誰擁有這個物件」——把內容完整寫入，讀檔時重建。 Scribe_References 存的是「指向某個別處已存的物件」。 同一個物件只能有一個 Deep 擁有者，其他持有者一律用 Reference，否則讀檔會產生重複實例或 null。

完整範例：IExposable 自訂類別

public class MyWorldComponentData : IExposable
{
    public int            counter;
    public ThingDef       favouredDef;          // Def 引用
    public Pawn           owner;                // 物件引用（別處 Deep 存）
    public List<int>      tickStamps = new List<int>();
    public Dictionary<Pawn, int> scores = new Dictionary<Pawn, int>();
    public SubData        sub = new SubData();   // 由本類別擁有 → Deep

    public void ExposeData()
    {
        Scribe_Values.Look(ref counter, "counter", 0);
        Scribe_Defs.Look(ref favouredDef, "favouredDef");
        Scribe_References.Look(ref owner, "owner");
        Scribe_Deep.Look(ref sub, "sub");

        // 集合：list of value
        Scribe_Collections.Look(ref tickStamps, "tickStamps", LookMode.Value);

        // Dictionary：key = Reference(Pawn)、value = Value(int)
        Scribe_Collections.Look(ref scores, "scores",
            LookMode.Reference, LookMode.Value);

        // 讀檔後集合可能為 null（存檔當時為空），補回空集合避免 NRE
        if (Scribe.mode == LoadSaveMode.PostLoadInit)
        {
            tickStamps ??= new List<int>();
            scores     ??= new Dictionary<Pawn, int>();
        }
    }
}

存檔版本升級：欄位遷移

當你新版改了欄位名稱或結構，舊存檔仍寫著舊欄位。遷移邏輯放在 Scribe.mode == LoadSaveMode.LoadingVars 階段——此時 XML 已可讀但物件尚未 PostLoad， 可同時嘗試讀新舊兩個 key，把舊資料搬到新欄位。 

public int newThreshold;          // v2 新欄位
// 舊版叫 "oldLimit"，新版改名 "newThreshold"

public void ExposeData()
{
    Scribe_Values.Look(ref newThreshold, "newThreshold", -1);

    if (Scribe.mode == LoadSaveMode.LoadingVars && newThreshold == -1)
    {
        // 新欄位讀不到（== default）→ 嘗試讀舊欄位做遷移
        int legacy = 0;
        Scribe_Values.Look(ref legacy, "oldLimit", 0);
        newThreshold = legacy * 10;   // 套用任何轉換規則
        Log.Message($"[MyMod] 遷移舊存檔 oldLimit={legacy} → newThreshold={newThreshold}");
    }
}
LoadSaveMode 階段 Saving（寫出）→ LoadingVars（讀入原始值，可做遷移）→ ResolvingCrossRefs（解析 Reference 指向）→ PostLoadInit（全部就緒，補 null、重建快取）。遷移放在 LoadingVars，重建衍生狀態放在 PostLoadInit。
4. 物件引用陷阱

C# 工程師最容易踩的雷：把 Pawn / Thing 引用長期存進自己的欄位或靜態集合。 在 RimWorld 中物件會死亡、被銷毀、被卸載地圖；你的引用會變成「孤兒」——指向一個已 Destroyed 或不再屬於任何地圖的物件， 不但邏輯錯誤，還會讓 GC 無法回收，造成記憶體洩漏與存檔膨脹。

持有 Pawn 引用的三大風險 ① 孤兒引用洩漏：Pawn 已死但你的靜態 List 還抓著它，永遠不被 GC。 ② 存檔污染：用 Scribe_References 存一個沒被任何 Deep 擁有者保存的 Pawn，讀檔時得到 null 或殘骸。 ③ 跨地圖失效：Pawn 隨 caravan 離開地圖後，原地圖的快取引用就指向「不在這張地圖」的物件。

正解：存 ID，用時重查

不要長期持有物件本身，改存一個穩定識別子，需要時透過查詢重新取得「當下有效」的引用。

// ❌ 危險：長期持有 Pawn 引用
private static List<Pawn> _watchedPawns = new List<Pawn>();

// ✓ 安全：只存 thingIDNumber（穩定、跨存檔一致）
private static HashSet<int> _watchedIds = new HashSet<int>();

// 登記時存 ID
_watchedIds.Add(pawn.thingIDNumber);

// 使用時重查，並驗證仍然有效
foreach (Pawn p in PawnsFinder.AllMaps_FreeColonistsAndPrisoners)
{
    if (_watchedIds.Contains(p.thingIDNumber) && !p.Dead && p.Spawned)
    {
        ProcessWatched(p);   // 保證是當下有效的活體
    }
}

thingIDNumber 作為 key

每個 Thing 有唯一且存檔穩定的 thingIDNumber（int）。用它當字典 key / 集合元素， 既不阻擋 GC，又能在讀檔後仍對得上同一個物件。

def + position 重查

對「地圖上某格的某種建築」，可存 def + IntVec3，需要時用 map.thingGrid.ThingsAt(cell) 重新定位，避免抓著可能被拆除的舊引用。

真要持有引用時，配合銷毀事件清理 若設計上不得不快取引用，務必在物件失效時主動移除：覆寫 Thing.Destroy() / Pawn.Kill() 的對應 Postfix， 或每隔一段 tick 掃描 !Spawned || Destroyed || Dead 的項目並從集合剔除。
5. 多版本相容（1.4 / 1.5）

RimWorld 每個大版本都可能改動 API 簽名、Def 結構或 IL。維護一份同時支援多版本的 Mod，需要在 About.xml、組件 DLL 與目錄結構三個層面協調。

About.xml 宣告支援版本

<!-- About/About.xml -->
<ModMetaData>
  <packageId>yourname.mymod</packageId>
  <supportedVersions>
    <li>1.4</li>
    <li>1.5</li>
  </supportedVersions>
</ModMetaData>

多版本目錄結構

RimWorld 載入時會優先讀取與當前遊戲版本同名的子資料夾（1.4/、1.5/），其內各有獨立的 Assemblies/ 與 Defs/。共用內容放在根目錄。

MyMod/
├── About/About.xml
├── Defs/            # 兩版共用的 Def
├── Textures/        # 共用資源
├── 1.4/
│   ├── Assemblies/MyMod.dll   # 針對 1.4 編譯的 DLL
│   └── Defs/                   # 1.4 專屬 Def 覆寫
└── 1.5/
    ├── Assemblies/MyMod.dll   # 針對 1.5 編譯的 DLL
    └── Defs/

#if 條件編譯（csproj DefineConstants）

用一份 .cs 原始碼、兩個 build 組態，靠前置處理符號切出不同程式碼路徑。在 csproj 為各組態定義 RW14 / RW15。

<!-- MyMod.csproj：為不同組態定義常數並輸出到對應資料夾 -->
<PropertyGroup Condition="'$(Configuration)' == 'RW15'">
  <DefineConstants>RW15</DefineConstants>
  <OutputPath>$(SolutionDir)\1.5\Assemblies\</OutputPath>
</PropertyGroup>
<PropertyGroup Condition="'$(Configuration)' == 'RW14'">
  <DefineConstants>RW14</DefineConstants>
  <OutputPath>$(SolutionDir)\1.4\Assemblies\</OutputPath>
</PropertyGroup>
// 程式碼中以符號切換版本差異的 API
public Vector3 GetDrawPos(Pawn pawn)
{
#if RW15
    // 1.5 改了簽名 / 新增參數的假想範例
    return pawn.DrawPosHeld(Rot4.North);
#else // RW14
    return pawn.DrawPos;
#endif
}
執行期偵測版本：VersionControl.CurrentVersion 若差異很小、不值得為它分兩份 DLL，可在執行期判斷： VersionControl.CurrentMajor / VersionControl.CurrentMinor（如 1 / 5），或 VersionControl.CurrentVersion（System.Version）。 但注意：若某 API 在某版本根本不存在，編譯就會失敗，那種情況仍須走 #if 條件編譯——執行期判斷只能處理「兩版都編得過、但行為要不同」的場景。
6. 和其他 Mod 互動

成熟的 Mod 生態裡，相容與整合是常態。核心原則是軟相依：偵測對方是否存在，存在才啟用整合， 不存在也不該崩潰；且不要在編譯期硬連結對方 DLL，以免對方未安裝時連載入都失敗。

偵測 Mod 是否啟用

// 用對方的 packageId 判斷是否啟用（不需引用其 DLL）
if (ModsConfig.IsActive("ceteam.combatextended"))
{
    Log.Message("[MyMod] 偵測到 Combat Extended，啟用相容處理");
    ApplyCombatExtendedPatches();
}

動態取得型別：AccessTools.TypeByName

當你需要對另一個 Mod 的型別做 Harmony patch，但又不能硬引用它，用字串動態取得 Type，再交給 Harmony。

Type ceType = AccessTools.TypeByName("CombatExtended.CompAmmoUser");
if (ceType != null)
{
    MethodInfo target = AccessTools.Method(ceType, "TryStartReload");
    if (target != null)
    {
        harmony.Patch(target,
            postfix: new HarmonyMethod(
                AccessTools.Method(typeof(MyCePatches), nameof(MyCePatches.AfterReload))));
    }
}
用 [HarmonyPatch] 的條件式版本 若改用屬性式 patch，可在 patch 類別放靜態 bool Prepare() 回傳是否套用，並用 [HarmonyPatch] 搭配 static MethodBase TargetMethod() 動態回傳目標方法（內部用 TypeByName 解析）。 對方未安裝時 Prepare() 回 false，整個 patch 安全跳過。

建立 Mod API：Interface + 共享 DLL

若你希望別的 Mod能擴充你的功能，正式做法是發佈一個只含介面的「API DLL」： 你的 Mod 定義 interface，第三方引用該 API DLL 實作介面，你在執行期反射收集所有實作者並呼叫。 雙方都不需硬引用對方的主 DLL。 

// ── MyMod.API.dll（公開、可被第三方引用）──
namespace MyMod.Api
{
    public interface IMyModExtension
    {
        string ExtensionName { get; }
        void OnColonyTick(Map map);
    }
}

// ── 你的主 Mod：啟動時掃描所有實作者 ──
[StaticConstructorOnStartup]
public static class ExtensionRegistry
{
    public static readonly List<IMyModExtension> Extensions = new List<IMyModExtension>();

    static ExtensionRegistry()
    {
        // 掃描所有已載入型別，找實作介面的具象類別並實例化
        foreach (Type t in GenTypes.AllTypes)
        {
            if (typeof(IMyModExtension).IsAssignableFrom(t)
                && !t.IsAbstract && !t.IsInterface)
            {
                Extensions.Add((IMyModExtension)Activator.CreateInstance(t));
                Log.Message($"[MyMod] 載入擴充：{t.FullName}");
            }
        }
    }
}

// 主 Mod 在適當時機廣播給所有擴充
foreach (var ext in ExtensionRegistry.Extensions)
    ext.OnColonyTick(map);
API DLL 要穩定 一旦發佈 API 介面，第三方就會依賴它。後續盡量只「新增」成員、避免改動既有簽名； 若非破壞不可，沿用版本目錄或新介面（IMyModExtension2）並行，給依賴者過渡空間。
7. RimWorld 資料結構速查

進階開發最常用的「拿到我要的東西」入口。以下都假設 using Verse; / using RimWorld;。

取用方式回傳說明 / 用途
Find.CurrentMap Map 玩家當前正在檢視的地圖。注意：世界視圖或多地圖時可能非你要操作的地圖，UI 以外的邏輯應顯式傳入目標 Map 慎用
PawnsFinder.AllMaps_FreeColonists List<Pawn> 所有地圖上的自由殖民者。另有 AllMaps_FreeColonistsAndPrisoners、AllMapsCaravansAndTravelingTransporters_Alive 等變體
Find.WorldPawns.AllPawnsAlive IEnumerable<Pawn> 世界層級（離開地圖但仍存活）的所有 Pawn，例如已離隊但未死的角色
GenClosest.ClosestThingReachable(root, map, req, peMode, traverseParams, ...) Thing 從某格出發、考慮可達性，找最近符合條件的 Thing。Job / AI 找目標的核心 API
GenGrid.Walkable(IntVec3, Map) bool 該格是否可行走（無不可通過建築 / 地形）。路徑與放置判斷常用
map.thingGrid.ThingsAt(IntVec3) IEnumerable<Thing> 取得某格上的所有 Thing。另有 ThingsListAt（回 List，零配置走訪更快）
FloodFiller（map.floodFiller.FloodFill(root, passCheck, processor)） void 從起點做廣度優先洪水填充，可用來找連通區域、房間、可達範圍

組合範例：找最近的可行走、可達的指定物品

// 從 pawn 位置找最近、可達、未被預訂的鋼鐵堆
Thing steel = GenClosest.ClosestThingReachable(
    root:           pawn.Position,
    map:            pawn.Map,
    thingReq:       ThingRequest.ForDef(ThingDefOf.Steel),
    peMode:         PathEndMode.ClosestTouch,
    traverseParams: TraverseParms.For(pawn),
    maxDistance:    9999f,
    validator:      t => !t.IsForbidden(pawn) && pawn.CanReserve(t));

if (steel != null && GenGrid.Walkable(steel.Position, pawn.Map))
{
    // 進一步：列出那一格上所有東西
    foreach (Thing t in pawn.Map.thingGrid.ThingsAt(steel.Position))
        Log.Message($"同格物件：{t.LabelCap} x{t.stackCount}");
}
找連通房間用 FloodFiller 要計算「從某格出發、所有可步行相連的格子」（例如自訂的密室 / 區域偵測），用 map.floodFiller.FloodFill(root, c => c.Walkable(map), c => visited.Add(c))。 RimWorld 房間系統 RegionAndRoomUpdater 內部也是同類概念。
延伸資源

本指南到此告一段落。以下是持續精進 RimWorld modding 的常用參考來源：

RimWorld Wiki — Modding 區

官方授權的社群 Wiki，涵蓋 XML Def 欄位、modding 教學與版本變更紀錄。查 Def 結構的第一站。

官方 Discord — Modding channel

RimWorld 官方 Discord 伺服器的 modding 頻道，聚集大量資深 modder。疑難雜症、API 變更討論最即時的去處。

Harmony 官方文件

pardeike/Harmony 的官方文件站，Prefix / Postfix / Transpiler / Finalizer、AccessTools、patch 順序與優先級的權威說明。

HugsLib

廣為使用的 modding 函式庫，提供設定面板、update news、log 發佈、便捷的 mod 生命週期 hook，省去大量樣板程式碼。

RimWorld 反編譯參考

用 dnSpy / ILSpy 反編譯 Assembly-CSharp.dll 是讀懂遊戲內部行為、寫 patch 與 Transpiler 的終極依據；社群亦有定期同步的反編譯原始碼倉庫可供查閱。

心法 RimWorld 沒有對外保證穩定的 modding API——遊戲本體的所有 internal / private 成員都可能隨版本變動。 養成「先反編譯讀原始碼、再動手 patch」的習慣，並用軟相依與防呆把版本風險降到最低，才是長期維護 Mod 的正道。
