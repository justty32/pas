性能剖析與確定性

一個 mod 能不能上線，常取決於它在 200 隻小人、巨型地圖下還順不順。本章談 tick 預算、剖析工具、常見熱點，以及多人 mod 相容所需的確定性。

Chapter 36 · Performance & Determinism
§1 tick 模型與預算

RimWorld 的核心是一個 tick-based 模擬迴圈。Verse.TickManager 在每個遊戲 tick 推進一次世界狀態， 正常速度（speed 1）下大約是每秒 60 tick。當玩家把速度切到 2× / 3×（甚至開發模式的超速）時， 引擎會在同一個現實秒內跑更多次 TickManager.DoSingleTick()，因此每個 tick 的成本會被速度倍率「線性放大」。 你在 Tick() 裡多花 0.05 ms，在 3× 速度、200 個實例下，就是每秒額外 0.05 × 3 × 200 × 60 ≈ 1800 ms 的純額外負擔——遠超過一秒，遊戲必然掉速。

TickManager 並不會「每個 tick 都呼叫每一個 Thing」。為了攤平負載，它維護了三條 Verse.TickList： Normal（每 tick）、Rare（每 250 tick）、Long（每 2000 tick）。 每個 Thing 在 spawn 時，依其 def.tickerType（Verse.TickerType 列舉）被登記到對應的 TickList。 Rare / Long 清單內部還會把成員打散到不同的「子桶」，讓不同實例在不同 tick 觸發，而不是同一個 tick 一次全算。 

// Verse.TickerType（節錄概念）
public enum TickerType
{
    Never,   // 完全不 tick（純裝飾、家具、岩石等）
    Normal,  // 每 tick：Thing.Tick()
    Rare,    // 每 250 tick：Thing.TickRare()
    Long     // 每 2000 tick：Thing.TickLong()
}

對 Verse.ThingWithComps 而言，這三個進入點會分別把呼叫往下傳遞給每個 comp： Tick() → comp 的 CompTick()、TickRare() → CompTickRare()、TickLong() → CompTickLong()。 因此你寫 ThingComp 時，選對覆寫哪一個方法，是性能的第一道防線。 

public class CompMyHeater : ThingComp
{
    // 每 tick：只給真的需要逐 tick 精度的東西（投射物、移動、即時溫度交換）
    public override void CompTick() { }

    // 每 250 tick（約 4.2 秒）：絕大多數「狀態檢查」應該放這裡
    public override void CompTickRare()
    {
        var power = parent.TryGetComp<CompPowerTrader>();
        if (power != null && power.PowerOn)
            GenTemperature.PushHeat(parent.Position, parent.Map, 21f);
    }

    // 每 2000 tick（約 33 秒）：低頻維護、長期衰減、統計
    public override void CompTickLong() { }
}
注意：TickRare 不等於「每 250 tick 全部一起算」 引擎把 Rare 清單成員分散到不同 tick 觸發，所以你無法假設所有同類實例在同一 tick 收到 TickRare。 如果你的邏輯需要「全體同步在某個整點計算」，請改用 Find.TickManager.TicksGame % interval == 0 自行判斷，而不是依賴 TickRare 的時序。

各 tick 類型該放什麼

類型頻率覆寫方法適合放的邏輯反例（不該放）
Never—無純裝飾、靜態建築、礦脈任何需要隨時間變化的狀態
Normal60/秒Tick / CompTick投射物飛行、即時 mote、戰鬥逐 tick 判定「電力夠不夠」「附近有沒有敵人」這類查詢
Rare每 250 tickTickRare / CompTickRare狀態檢查、資源消耗、目標重新評估、緩慢加熱需要逐格平滑移動的物件
Long每 2000 tickTickLong / CompTickLong腐爛/老化推進、長期統計、低頻自我修復玩家會盯著看的即時數值

預設往 Rare 靠

九成的 comp 邏輯其實只需要每幾秒檢查一次。先用 CompTickRare，只有確定要逐 tick 精度時才升級到 CompTick。這是最便宜的優化，也最不容易出錯。

tickerType 在 XML 設定

Thing 的 ticker 類型由 ThingDef.tickerType 決定，而不是你覆寫了哪個方法。XML 設了 Never，即使你寫了 CompTickRare 也不會被呼叫。兩邊要對得上。

<ThingDef ParentName="BuildingBase">
  <defName>MyHeater</defName>
  <!-- 沒寫的話預設 Normal，記得依需求降到 Rare -->
  <tickerType>Rare</tickerType>
  <comps>
    <li Class="CompProperties_Power"> ... </li>
  </comps>
</ThingDef>
§2 降低每 tick 的成本

就算你選了正確的 ticker 類型，Tick() 內部的程式碼仍可能太貴。降低 tick 成本有四個核心技巧： 用 HashOffset 攤平負載、快取昂貴查詢、避免每 tick 配置物件 / 閉包、用遊戲 tick 計數而非現實計時器。

① HashOffset 攤平負載

Verse.Thing.HashOffset() 回傳一個基於該 Thing 唯一 ID 的穩定整數。 把它加進「整點判斷」，能讓不同實例落在不同 tick 執行重邏輯——也就是引擎在 TickList 內部做的同一件事， 只是你在 Tick() 裡自己手動做。這能把「200 個實例在同一 tick 一起算」變成「每 tick 只算約 1 個」。 

public override void CompTick()
{
    // 壞：每個實例都在「tick 能被 250 整除」時一起算 → 尖峰負載
    if (Find.TickManager.TicksGame % 250 == 0)
        DoExpensiveCheck();

    // 好：用 HashOffset 把同類實例攤到不同 tick
    if ((Find.TickManager.TicksGame + parent.HashOffset()) % 250 == 0)
        DoExpensiveCheck();
}
GenTicks.TickRareInterval / TickLongInterval 是常數 別自己硬寫 250 / 2000，用 Verse.GenTicks.TickRareInterval、GenTicks.TickLongInterval， 語意清楚且不會因引擎調整而失準。需要「每 N 秒」時用 GenTicks.TicksPerRealSecond（= 60）換算。

② 快取昂貴查詢，別每 tick 跑 LINQ

最常見的性能殺手是「在 Tick() 裡每次都跑一段 LINQ 或全地圖掃描」。LINQ 每次呼叫都會配置 enumerator/閉包物件， 在 hot path 上累積成嚴重的 GC 壓力；全地圖掃描本身又是 O(n)。正確做法是把結果快取，只在失效時重算。 

public class CompNearbyEnemySensor : ThingComp
{
    private int _lastScanTick = -99999;
    private bool _enemyNearbyCached;

    private const int ScanInterval = 120; // 2 秒重算一次就夠

    public bool EnemyNearby
    {
        get
        {
            int now = Find.TickManager.TicksGame;
            if (now - _lastScanTick >= ScanInterval)
            {
                _lastScanTick = now;
                // 用引擎現成的空間查詢，避免自己掃全地圖的 LINQ
                _enemyNearbyCached = parent.Map.attackTargetsCache != null
                    && GenHostility.AnyHostileActiveThreatTo(parent.Map, parent.Faction);
            }
            return _enemyNearbyCached;
        }
    }

    public override void CompTickRare()
    {
        if (EnemyNearby) { /* ... */ }
    }
}
絕對不要這樣寫 Map.mapPawns.AllPawnsSpawned.Where(p => p.HostileTo(Faction)).Any() 放在每 tick 的迴圈裡—— 每個 Thing 每 tick 都重掃一次全地圖人口、配置一串 enumerator。200 個實例 × 60 tick 就是每秒上萬次全地圖 LINQ，這是教科書級的卡頓來源。

③ 避免每 tick new 物件與閉包

C# 的 GC 在遊戲中是「無聲的卡頓製造機」：每 tick 配置的小物件會堆積成 Gen0 垃圾，觸發 GC 時造成 spike（畫面一頓）。 在 hot path 上要刻意避免 new List<>()、lambda 捕捉、裝箱（boxing）、字串串接、以及隱含配置的 LINQ。

每 tick 配置（壞）

void Tick()
{
    var tmp = new List<IntVec3>();
    cells.ForEach(c => tmp.Add(c));
    // 每 tick 一個 List + 一個 lambda 閉包
}

重用緩衝（好）

static readonly List<IntVec3> _buf = new();
void Tick()
{
    _buf.Clear();
    for (int i = 0; i < cells.Count; i++)
        _buf.Add(cells[i]);
    // 零配置；用 for 取代 ForEach 也省一個閉包
}
RimWorld 內建的物件池 引擎自己也用這招：Verse.SimplePool<T> 可借還暫存物件；許多計算用的 tmpList 都是 static 欄位重用的。 模仿引擎慣例即可，但記得 static 緩衝不可跨呼叫保留狀態，用前一定 Clear()，更要避免在多執行緒情境下共用（見 §6）。

④ 用 TicksGame 比較，而非現實計時器

計時邏輯一律以 Find.TickManager.TicksGame（遊戲 tick 計數）為基準，記錄「下次觸發的 tick」並比較大小。 千萬別用 System.DateTime.Now 或 Stopwatch 來驅動遊戲邏輯——那會隨速度倍率與機器效能而漂移，且在多人模式直接導致 desync（見 §6）。

§3 剖析工具：先量再改

性能優化的鐵律是先剖析、再優化。憑直覺猜熱點十有八九會猜錯，把時間花在不痛的地方。下面是由內到外的幾種量測手段。

① 開發模式內建剖析

開啟 Dev mode 後，左下角工具列的 View Settings 面板裡有 debug profiling 相關開關（顯示 ms/tick、TPS、各系統耗時的覆蓋層）。 它能快速告訴你「現在的瓶頸是 tick（模擬）還是 frame（畫面）」——這是分流的第一步。

症狀多半是觀察點
速度切 3× 上不去、TPS 掉到 60 以下tick（模擬）瓶頸ms/tick 飆高，但 FPS 正常
畫面拖影、滑鼠卡，但時間照跑frame（渲染 / UI）瓶頸FPS 掉，但 TPS 維持
每隔幾秒固定一頓GC spike 或定時重算規律性 spike，配 §2 找配置點
分清 tick 卡頓 vs UI 卡頓 若暫停遊戲（speed 0）後畫面依然卡，問題出在 OnGUI / 渲染（每 frame 跑的東西，例如 Map.MapUpdate、overlay、過量 mote、每 frame 的 GUI 計算）。 若暫停後就順了，問題出在 Tick。兩者的修法完全不同，先分清楚再動手。

② 社群剖析器（觀念層）

Dubs Performance Analyzer

社群最常用的逐方法剖析器。它能列出每個 patch / 每個方法的 ms 佔比，直接點名「哪個 mod 的哪個方法吃掉了你的 tick」。開發期掛上它跑一輪，是定位他人 mod 衝突與自己熱點的利器。

RocketMan

偏「自動優化」取向，會快取與節流大量引擎呼叫。它更適合玩家端加速，但開發期可暫時開來觀察「哪些呼叫被它判定為高頻」，反推自己程式的熱點分佈。

本指南只介紹觀念，不教安裝。重點是：它們的核心都是 Harmony patch + 計時打點，理解原理後你也能在自己程式碼裡手動打點。

③ 在自己程式碼打點：DeepProfiler 與 Stopwatch

Verse.DeepProfiler 是引擎內建的階層式計時器，用 Start(label) / End() 包夾一段程式碼， 結果會進到引擎的剖析輸出。它本身有開銷，只在開發期排查時使用，發布版要移除或用旗標關閉。 

// 用 DeepProfiler 包夾可疑區段（成對 Start/End，建議 try/finally 保證配對）
DeepProfiler.Start("MyMod.RecalculateGrid");
try
{
    RecalculateGrid();
}
finally
{
    DeepProfiler.End();
}

若只想要一個粗略的毫秒數，System.Diagnostics.Stopwatch 最直接。注意它只能用來量測， 量到的數字寫進 log，絕不可拿來驅動遊戲邏輯（不確定性，見 §6）。 

// 一次性量測某個重操作的成本，輸出到 log
var sw = Stopwatch.StartNew();
DoExpensiveThing();
sw.Stop();
Log.Message($"[MyMod] DoExpensiveThing 花了 {sw.Elapsed.TotalMilliseconds:F3} ms");
量測本身會干擾結果 在每 tick 的 hot path 上塞 Stopwatch.StartNew()（會配置物件）或 DeepProfiler，量到的數字會被工具本身的開銷污染。 量單一重操作用 Stopwatch；量「整個 tick 的分佈」交給 Dubs Performance Analyzer，別自己在每 tick 打一堆點。
§4 常見熱點與對應修法

以下是 mod 性能問題的「常客」，按出現頻率排列，每一項都附上修法方向。

熱點為何貴修法
每 tick LINQ / 正則配置 enumerator/閉包、O(n) 掃描、正則編譯快取結果（§2）、改 for 迴圈、正則用 RegexOptions.Compiled 且只編一次
頻繁 PathFinder 呼叫A* 尋路是最貴的單一操作之一快取路徑、用 reachability 先做廉價可達性判斷、降低重算頻率
每 tick 取 Find.X屬性鏈與空檢查累積在 spawn / 首次使用時取一次存成欄位
大量 Mote / overlay每 frame 渲染，且 mote 也要 tick降密度、用 ThrowMetaIcon 節流、靜態 overlay 改用快取材質
Harmony Prefix 跑在 hot path每次目標方法被呼叫都多一層提早 return、或改 Transpiler（串第 3 章）
未快取的 DefDatabase 查詢GetNamed 走字典查找＋字串用 static readonly 欄位快取 Def 引用
每次呼叫的 reflectionGetMethod/GetField 每次都掃 metadata快取 MethodInfo/FieldInfo，最好做成 delegate

Harmony Prefix 在 hot path：提早 return

第 3 章談過 Harmony。這裡的性能重點是：patch 一個高頻方法時，prefix 的第一件事應該是廉價的「要不要管」判斷，不符就立刻放行。 別把昂貴的判斷或 LINQ 放在每次都會跑的 prefix 開頭。 

[HarmonyPatch(typeof(Pawn), "Tick")]  // Pawn.Tick 是 protected，須用字串名而非 nameof()
static class Pawn_Tick_Patch
{
    static void Postfix(Pawn __instance)
    {
        // 第一行就用最廉價的條件擋掉九成情況，避免每 tick 都跑下面的重邏輯
        if (!__instance.IsColonist) return;
        if ((Find.TickManager.TicksGame + __instance.HashOffset()) % 250 != 0) return;

        // 真正的重邏輯：每 250 tick、僅殖民者才跑
        DoExpensiveColonistLogic(__instance);
    }
}
不要 patch Pawn.Tick 然後在裡面跑重邏輯 Pawn.Tick 是全遊戲最熱的方法之一。如果你的 postfix 每 tick 對每隻 pawn 都做事，等同於替整個遊戲加裝拖油瓶。 要嘛提早 return 攤平（如上），要嘛根本不要 patch Tick——改 patch 一個只在事件發生時呼叫的方法（見 §7 事件驅動）。

快取 DefDatabase 與 reflection

public static class MyDefOf
{
    // 用 [DefOf] 讓引擎在載入後自動填入，零執行期查找
    [DefOf]
    public static ThingDef MyHeater;

    static MyDefOf() => DefOfHelper.EnsureInitializedInCtor(typeof(MyDefOf));
}

// reflection 一定要快取——別在 hot path 重複 GetMethod
static readonly FieldInfo _privateField =
    AccessTools.Field(typeof(Pawn_HealthTracker), "cachedPain");

// 更進階：把 FieldInfo 編成 delegate，呼叫成本接近直接存取
static readonly AccessTools.FieldRef<Pawn_HealthTracker, float> _painRef =
    AccessTools.FieldRefAccess<Pawn_HealthTracker, float>("cachedPain");
用 [DefOf] 取代 DefDatabase.GetNamed [DefOf] 欄位在遊戲載入後由引擎一次性解析、之後直接欄位讀取，是零成本的。 DefDatabase<T>.GetNamed("...") 每次都走字典＋字串雜湊，放在 hot path 是浪費。只有「defName 在執行期才決定」時才用 GetNamed，且要快取。
§5 記憶體與載入成本

性能不只是 tick——載入時間與記憶體佔用同樣影響體驗，而且這兩者的成本應該與遊戲執行期分離： 載入期可以多花時間做一次性的準備（建快取、編 delegate），換取執行期的便宜。

StaticConstructorOnStartup 與載入期工作

// 標了這個 attribute 的類別，其 static 建構子會在遊戲載入完成後執行一次
[StaticConstructorOnStartup]
public static class MyMod
{
    // 載入期建立 Harmony patch、預載材質、建快取——一次性成本
    public static readonly Material OverlayMat =
        MaterialPool.MatFrom("MyMod/Overlays/Glow", ShaderDatabase.Transparent);

    static MyMod()
    {
        var h = new Harmony("author.mymod");
        h.PatchAll();
    }
}
Texture / Material 只在 StaticConstructorOnStartup 載 ContentFinder<Texture2D>.Get、MaterialPool.MatFrom 等只能在主執行緒、且在內容載入完成後呼叫（串第 8 章）。 別在 tick 或建構子裡每次 MatFrom——它有快取，但查找與字串仍有成本；把 Material 存成 static readonly 一次取好。

避免靜態大集合洩漏

static 集合的生命週期跟整個程序一樣長。如果你往 static Dictionary/List 塞 Thing/Pawn 卻不在它們銷毀時移除， 這些物件永遠不會被 GC 回收——既洩漏記憶體，又可能讓已 despawn 的物件被誤用。讀檔 / 換地圖時更要清空。

洩漏（壞）

static Dictionary<Pawn, Data> 在 SpawnSetup 加入，卻沒在 DeSpawn/Destroy 移除，也沒在 GameComponent 換檔時清空 → 跨存檔累積、舊 Pawn 永久存活。

正確（好）

用 ThingComp / HediffComp 把資料「掛在物件身上」，隨物件一起被 Scribe 與 GC 管理；真要用 static 快取，務必在物件銷毀與讀檔時清。

存檔膨脹：Scribe 大資料

存檔大小與讀寫時間，正比於你用 Verse.Scribe 序列化的資料量（串第 7 章）。常見地雷是「每 tick 累積、卻從不清理的歷史資料」被一路存進檔案。 把可重算的衍生資料標為「不存」，只存真正的權威狀態。 

public override void PostExposeData()
{
    // 存：真正的權威狀態
    Scribe_Values.Look(ref _charge, "charge", 0f);

    // 不存：可由 charge 算出的衍生快取（讀檔後重算即可，避免存檔膨脹）
    // _displayColor、_cachedGlowRadius 這類一律不寫進 Scribe
}
階段成本特性策略
Mod 載入期一次性，玩家願意等建 patch、預載材質、建 delegate 快取、解析 DefOf
地圖生成 / 讀檔偶發、可接受短暫卡重建衍生快取、清空 static 集合
遊戲執行期（tick/frame）高頻、必須便宜只讀快取、零配置、攤平負載
§6 確定性與多人相容

RimWorld 的 Multiplayer mod 採用「鎖步同步（lockstep）」：伺服器只同步玩家指令，每個客戶端各自跑完全相同的模擬。 只要任一處計算在不同客戶端產生不同結果，世界狀態就會分岔（desync），導致對不上、最終斷線。 因此你的 mod 必須是確定性的：給定相同輸入與相同 tick，永遠算出相同結果。

亂數一律走 Verse.Rand

確定性最常見的破口就是亂數。Verse.Rand 的亂數狀態是世界狀態的一部分、會被同步與存檔。 所有遊戲邏輯的隨機都要用 Rand.Value、Rand.Range、Rand.Chance、Rand.Element 等， 絕不可用 System.Random（未種子）或 UnityEngine.Random。 

// 對：走 Verse.Rand，狀態被引擎同步
if (Rand.Chance(0.25f)) SpawnBonusLoot();
int dmg = Rand.Range(8, 14);
var pick = things.RandomElement(); // 走 Rand

// 錯：自己的 Random，各客戶端種子不同 → desync
var rng = new System.Random();
int bad = rng.Next(8, 14);

非確定性區段用 PushState / PopState 包夾

有時你需要「為了視覺或預覽」算一段隨機，但不想干擾主世界的亂數序列（推進全域 Rand 狀態本身會造成 desync—— 因為某客戶端多抽了一次）。做法是用 Rand.PushState(seed) 開一個獨立的、以固定 seed 起始的亂數區段，用完 Rand.PopState() 還原。 

// 用固定 seed 包夾：區段內的抽樣不影響外部 Rand 序列，且各客戶端結果一致
int seed = Gen.HashCombineInt(parent.thingIDNumber, Find.TickManager.TicksGame);
Rand.PushState(seed);
try
{
    var tint = new Color(Rand.Value, Rand.Value, Rand.Value);
    ApplyVisualTint(tint);
}
finally
{
    Rand.PopState(); // 一定要還原，否則之後的 Rand 序列被你改寫
}
PushState / PopState 必須成對 用 try/finally 確保即使中途拋例外也會 PopState。漏掉 PopState 會讓 Rand 狀態堆疊失衡，連鎖污染後續所有亂數—— 這種 bug 在單人下可能無感，到多人模式才爆，極難回溯。

確定性殺手清單

禁用 / 危險原因正確替代
DateTime.Now / Environment.TickCount依現實時間，各客戶端不同Find.TickManager.TicksGame
System.Guid.NewGuid()本質隨機、不可重現引擎的 thingIDNumber / GenText 種子 ID
未種子 System.Random / UnityEngine.Random不在同步範圍內Verse.Rand
依賴 Dictionary / HashSet 迭代順序順序未保證、跨執行可能不同排序後迭代，或用 List 維持插入序
浮點數平台差異 / float 累積誤差不同 CPU / JIT 可能有極小差避免長串浮點累積；關鍵判定盡量用整數 / Rand
多執行緒寫共享狀態競態 → 不可重現世界邏輯留在主執行緒（串太吾筆記同類陷阱）
Dictionary 迭代順序是隱形地雷 foreach (var kv in someDict) 若用迭代結果去抽 Rand 或決定行為，等於把「不保證的順序」餵進確定性流程。 需要穩定順序時，先 OrderBy 一個確定的鍵（如 thingIDNumber）再迭代。

此外，Multiplayer mod 提供 ITweakField 與各種同步 API（SyncMethod、SyncField）來讓「玩家觸發的動作」被正確地廣播。 若你的 mod 有玩家可點的按鈕會改變世界狀態，必須讓那個動作走同步管道，而不是在本地直接改——否則只有按按鈕的客戶端會生效，其餘客戶端 desync。

§7 設計原則：讓「快」成為預設

真正可規模化的性能來自架構，而非微優化。下面幾條原則應該在動筆寫 C# 之前就先想清楚。

昂貴計算移出 hot path

能在事件發生時算一次、就別在每 tick 重算。把結果快取，hot path 只讀不算。

事件驅動取代輪詢

別每 tick 問「狀態變了嗎」，改在狀態改變的源頭通知（串第 12 章事件敘事）。輪詢是 O(tick)，事件是 O(變化次數)。

能用資料就別寫程式

能用 StatDef / StatPart / XML 表達的數值修正，就別在 C# 每 tick 算（串第 14 章）。引擎會替你快取 stat 結果。

用 StatPart 取代每 tick 計算

RimWorld.StatPart 是引擎「把某個修正套進 stat 計算」的擴充點，而 stat 結果本身是會被快取的（多數 stat 不是每 tick 重算）。 把你的「依條件加成」寫成 StatPart，既乾淨又便宜，遠勝於 patch 一個 hot 方法每 tick 改數值。 

public class StatPart_MyAura : StatPart
{
    public override void TransformValue(StatRequest req, ref float val)
    {
        if (req.Thing is Pawn p && HasAura(p))
            val *= 1.2f;
    }

    public override string ExplanationPart(StatRequest req)
    {
        if (req.Thing is Pawn p && HasAura(p))
            return "MyMod_AuraBonus".Translate() + ": x1.2";
        return null;
    }
}
<!-- 在 StatDef 裡掛上 StatPart，引擎自動套用並快取 -->
<StatDef>
  <defName>WorkSpeedGlobal</defName>
  <parts>
    <li Class="MyMod.StatPart_MyAura" />
  </parts>
</StatDef>
TickRare 足夠時，永遠別用 Tick 問自己：「玩家真的能察覺這個狀態 4 秒的延遲嗎？」九成情況答案是否。把 CompTick 降級成 CompTickRare， 是投報率最高的單一改動——成本直接除以 250。能降 Long 就更好。
§8 量測導向：一個完整的優化案例

最後用一個貼近真實的小案例串起全章。假設你做了一個「警報塔」comp：附近有敵人時點亮 overlay 並廣播訊息。 玩家回報「放幾十座之後遊戲在 3× 卡爆」。我們按量測流程逐步修。

  1. 剖析定位：掛 Dubs Performance Analyzer，發現 CompAlarmTower.CompTick 佔了 tick 時間的一大塊；ms/tick 高但 FPS 正常 → 確定是 tick 瓶頸（§3）。
  2. 看初版程式碼：每 tick、每座塔都跑一段全地圖 LINQ 找敵人，還每 tick new 一個結果 List（§2、§4 的雙重地雷）。
  3. 降頻 + 攤平：把檢查降到每 250 tick，並用 HashOffset() 讓幾十座塔分散到不同 tick，消掉尖峰（§2 ①）。
  4. 快取查詢：把「附近有無敵人」改成走引擎 GenHostility，結果快取到欄位，避免每次全地圖 LINQ（§2 ②、§4）。
  5. 再量測：回到 Analyzer 確認該方法已從榜首消失。沒退步才算數，不靠感覺。
// ── 初版（壞）：每 tick + 全地圖 LINQ + 每 tick new ──
public override void CompTick()
{
    var hostiles = parent.Map.mapPawns.AllPawnsSpawned
        .Where(p => p.HostileTo(parent.Faction))
        .ToList();                 // 每 tick 配置 + 全掃
    _lit = hostiles.Any(p =>
        p.Position.DistanceTo(parent.Position) < 20f);
}

// ── 修正版（好）：降頻 + HashOffset 攤平 + 廉價可達性 + 快取 ──
private bool _lit;
private const int Interval = 250;
private const float RadiusSq = 20f * 20f;

public override void CompTick()
{
    if ((Find.TickManager.TicksGame + parent.HashOffset()) % Interval != 0)
        return;                       // 攤平：每 tick 只有極少數塔真的算

    // 先用最廉價的全圖敵意判斷擋掉「整張圖根本沒敵人」的常態
    if (!GenHostility.AnyHostileActiveThreatTo(parent.Map, parent.Faction))
    {
        _lit = false;
        return;
    }

    // 只有真有威脅時，才做局部距離檢查；用平方距離省一次開根號
    _lit = false;
    var pawns = parent.Map.mapPawns.AllPawnsSpawned;
    for (int i = 0; i < pawns.Count; i++)   // for 取代 LINQ，零配置
    {
        var p = pawns[i];
        if (p.HostileTo(parent.Faction)
            && (p.Position - parent.Position).LengthHorizontalSquared < RadiusSq)
        {
            _lit = true;
            break;                      // 找到一個就夠，提早跳出
        }
    }
}
確定性同樣免費搭車 上面的修正版沒有用到任何 System.Random、DateTime、或 Dictionary 迭代順序，計時也全走 TicksGame。 它在多人模式下天然確定。提早遵守 §6 規範，比上線後才抓 desync 划算太多——確定性 bug 難重現，是最折磨人的一類。

本章重點回顧

  • 選對 ticker：預設往 TickRare / TickLong 靠，能不每 tick 就別每 tick（§1、§7）。
  • 攤平與快取：HashOffset() 消尖峰、快取查詢、hot path 零配置（§2）。
  • 先量再改：用內建剖析分清 tick / frame 瓶頸，社群 Analyzer 點名熱點，改完一定回量（§3、§8）。
  • 避開常客熱點：每 tick LINQ、頻繁尋路、未快取的 DefDatabase / reflection、hot path 上的 Prefix（§4）。
  • 分離成本：載入期一次性、執行期便宜；別讓 static 集合洩漏，別讓 Scribe 存可重算的快取（§5）。
  • 確定性是底線：亂數走 Rand、必要時 PushState/PopState 包夾，遠離 DateTime/Guid/未種子 Random/字典順序（§6）。
一句話帶走 性能優化是一個量測閉環：發現熱點 → HashOffset 攤平 → 快取查詢 → 再量測。確定性則是一條紀律：提早遵守，永不後悔。
