能力系統的全貌

RimWorld 的「能力 (Ability)」是一套統一的主動技能框架——pawn 透過 Pawn_AbilityTracker（pawn.abilities）持有一組 Ability 實例， 每個 Ability 由一個 AbilityDef 驅動，UI 上以 Gizmo（指令按鈕）呈現。 Royalty DLC 的超能 (Psycast) 與 Anomaly DLC 的實體能力都是建立在同一套 Ability 基礎上，差別只在於是否消耗精神專注 (psyfocus)、是否掛在 PsycasterPathDef 上。

AbilityDef

能力的定義（XML）：冷卻、verb、效果 comps、目標需求。

Ability

能力的執行期實例：綁定 pawn，負責 Activate / CanCast / Gizmo。

CompAbilityEffect

能力命中時的實際效果：發射投射物、施加 Hediff、傳送等。

關鍵命名空間 Ability、AbilityDef、AbilityComp 系列都在 RimWorld 命名空間； Pawn_AbilityTracker 在 Verse。Psycast 相關（Hediff_Psylink、Pawn_PsychicEntropyTracker）在 RimWorld。
1 · AbilityDef 結構

AbilityDef 是能力的 XML 定義。最核心的四個面向：冷卻 (cooldownTicksRange)、施放動作 (verbProperties)、效果 (comps)、是否需要目標 (targetRequired)。

欄位速查

欄位型別說明
cooldownTicksRangeIntRange冷卻範圍（60 ticks = 1 秒）。實作層常見 cooldownTicksRange；舊資料/部分能力用單一 cooldownTicks
verbPropertiesVerbProperties施放動作（射程、warmup、音效、是否需要 LoS）。verbClass 通常為 Verb_CastAbility
compsList<CompProperties>效果 comp 清單；命中時依序套用
targetRequiredbool是否必須選目標（false = 自我施放型，如增益光環）
targetWorldCellbool目標是地格而非 thing（如砲擊、傳送落點）
jobDefJobDef施放時執行的 Job，預設 CastAbilityOnThing / CastAbilityOnWorldTile
hostilebool是否為敵對行為（影響 AI / 關係）
aiCanUseboolAI pawn 是否會主動使用
warmupMote / iconPathstring蓄力特效 / Gizmo 圖示

完整 XML 範例：一個範圍治療能力

<?xml version="1.0" encoding="utf-8"?>
<Defs>
  <AbilityDef>
    <defName>MyMod_HealingPulse</defName>
    <label>治療脈衝</label>
    <description>對目標區域釋放治療能量，移除流血並恢復少量傷害。</description>
    <iconPath>UI/Abilities/HealingPulse</iconPath>
    <cooldownTicksRange>1800~1800</cooldownTicksRange>  <!-- 30 秒 -->
    <hostile>false</hostile>
    <targetRequired>true</targetRequired>
    <targetWorldCell>true</targetWorldCell>  <!-- 點地格而非單體 -->

    <!-- 施放動作：射程、蓄力 -->
    <verbProperties>
      <verbClass>Verb_CastAbility</verbClass>
      <drawAimPie>false</drawAimPie>
      <range>24.9</range>
      <warmupTime>1.0</warmupTime>
      <requireLineOfSight>true</requireLineOfSight>
      <targetParams>
        <canTargetLocations>true</canTargetLocations>
        <canTargetPawns>true</canTargetPawns>
        <canTargetSelf>true</canTargetSelf>
      </targetParams>
    </verbProperties>

    <!-- 命中時套用的效果（見第 2 節）-->
    <comps>
      <li Class="CompProperties_AbilityGiveHediff">
        <compClass>CompAbilityEffect_GiveHediff</compClass>
        <hediffDef>MyMod_HealingBuff</hediffDef>
        <onlyApplyToSelf>false</onlyApplyToSelf>
      </li>
    </comps>
  </AbilityDef>
</Defs>
cooldownTicksRange vs cooldownTicks 實際施放後的冷卻是寫進 Ability.StartCooldown()，讀的是 def.cooldownTicksRange。 若你只寫 <cooldownTicks> 而程式碼讀 range，會得到 0 冷卻（無冷卻）。請以 cooldownTicksRange 為準，固定值寫成 1800~1800。
2 · Ability comps（CompProperties_AbilityEffect 系列）

能力的「效果」由掛在 <comps> 下的一組 comp 組成。每個 comp 由 CompProperties_AbilityXxx（資料）+ CompAbilityEffect_Xxx（行為）配對。 命中時 Ability.Activate 會依序對每個目標呼叫 comp 的 Apply。

常用內建效果 comp

Properties ClassComp Class效果
CompProperties_AbilityLaunchProjectileCompAbilityEffect_LaunchProjectile朝目標發射 ThingDef 投射物
CompProperties_AbilityGiveHediffCompAbilityEffect_GiveHediff對目標施加指定 HediffDef
CompProperties_AbilityTeleportCompAbilityEffect_Teleport把 caster / 目標傳送到目標地格
CompProperties_AbilityGoodwillImpactCompAbilityEffect_GoodwillImpact調整派系關係
CompProperties_AbilityStunCompAbilityEffect_Stun暈眩目標
CompProperties_AbilityResurrectCompAbilityEffect_ResurrectMech 等復活類

範例：LaunchProjectile + GiveHediff 組合

  <comps>
    <!-- 1. 對目標發射一枚投射物 -->
    <li Class="CompProperties_AbilityLaunchProjectile">
      <compClass>CompAbilityEffect_LaunchProjectile</compClass>
      <projectileDef>MyMod_Projectile_Frostbolt</projectileDef>
    </li>

    <!-- 2. 同時對目標施加減速 Hediff -->
    <li Class="CompProperties_AbilityGiveHediff">
      <compClass>CompAbilityEffect_GiveHediff</compClass>
      <hediffDef>MyMod_Chilled</hediffDef>
      <severity>1.0</severity>
      <onlyApplyToSelf>false</onlyApplyToSelf>
      <applyToSelfOnTarget>false</applyToSelfOnTarget>
    </li>
  </comps>

範例：Teleport（自我傳送 / 閃現）

  <targetWorldCell>true</targetWorldCell>
  <comps>
    <li Class="CompProperties_AbilityTeleport">
      <compClass>CompAbilityEffect_Teleport</compClass>
      <destination>Selected</destination>      <!-- 傳到選定地格 -->
      <range>0</range>
      <destClamping>true</destClamping>
    </li>
  </comps>
找正確欄位名的最快路徑 用 dnSpy / ILSpy 打開 Assembly-CSharp.dll，搜尋對應的 CompProperties_Ability* class， 其 public 欄位就是 XML 可填的標籤。RimWorld 的 XML loader 直接以反射對應欄位名（大小寫敏感）。
3 · 自訂 CompAbilityEffect（覆寫 Apply）

內建效果不夠用時，繼承 CompAbilityEffect 並覆寫 Apply(LocalTargetInfo target, LocalTargetInfo dest)。 若要自訂欄位，同時繼承 CompProperties_AbilityEffect 並在建構子指定 compClass。

Properties + Comp 配對

using RimWorld;
using Verse;

// ── 1. Properties：定義 XML 可填欄位 ──────────────────────
public class CompProperties_AbilityDrainHealth : CompProperties_AbilityEffect
{
    public float drainAmount = 10f;       // 吸取血量
    public HediffDef healCasterHediff;       // 給施法者的治療 Hediff

    public CompProperties_AbilityDrainHealth()
    {
        // 把 Properties 綁定到行為 class
        compClass = typeof(CompAbilityEffect_DrainHealth);
    }
}

// ── 2. Comp：實際行為 ─────────────────────────────────────
public class CompAbilityEffect_DrainHealth : CompAbilityEffect
{
    // 強型別存取 Properties
    public new CompProperties_AbilityDrainHealth Props =>
        (CompProperties_AbilityDrainHealth)props;

    /// <summary>
    /// 能力命中每個目標時呼叫一次。
    /// target = 主要目標；dest = 次要目標（多數能力相同）。
    /// </summary>
    public override void Apply(LocalTargetInfo target, LocalTargetInfo dest)
    {
        base.Apply(target, dest); // 保留基底行為（可省，但建議呼叫）

        Pawn victim = target.Pawn;
        Pawn caster = parent.pawn; // parent = Ability，pawn = 施法者
        if (victim == null || caster == null) return;

        // 對目標造成傷害
        var dinfo = new DamageInfo(
            DamageDefOf.Psychic, Props.drainAmount, instigator: caster);
        victim.TakeDamage(dinfo);

        // 回饋給施法者一個治療 Hediff
        if (Props.healCasterHediff != null)
        {
            Hediff h = HediffMaker.MakeHediff(Props.healCasterHediff, caster);
            caster.health.AddHediff(h);
        }
    }

    // 可選：能力是否能對某目標生效（影響 UI 灰階與右鍵提示）
    public override bool CanApplyOn(LocalTargetInfo target, LocalTargetInfo dest)
    {
        return target.Pawn != null && !target.Pawn.Dead;
    }

    // 可選：滑鼠懸停目標時顯示的提示文字
    public override string ExtraLabelMouseAttachment(LocalTargetInfo target)
    {
        return target.Pawn != null ? "將吸取生命" : null;
    }
}

對應 XML

  <comps>
    <li Class="YourNamespace.CompProperties_AbilityDrainHealth">
      <!-- compClass 已在建構子設定，不必再寫 -->
      <drainAmount>15</drainAmount>
      <healCasterHediff>MyMod_LifeStolen</healCasterHediff>
    </li>
  </comps>
Apply 的兩個關鍵入口 多目標能力（AoE）會對範圍內每個目標各呼叫一次 Apply(target, dest)； 若需要「整體一次性」邏輯（如生成單一爆炸），覆寫 Apply(GlobalTargetInfo) 或在 Ability.Activate 層處理較合適。parent 是 AbilityComp.parent（型別 Ability）。
4 · 從 C# 給 Pawn 能力

pawn 的能力存在 Pawn_AbilityTracker（pawn.abilities）。 用 GainAbility(AbilityDef) 新增，RemoveAbility 移除，GetAbility 取得實例。 

using RimWorld;
using Verse;

public static class AbilityGrantUtility
{
    public static Ability GrantHealingPulse(Pawn pawn)
    {
        // pawn.abilities 在某些 pawn（如動物）可能為 null，先確認
        if (pawn.abilities == null)
        {
            // 必要時手動建立 tracker（極少需要；人形 pawn 通常已有）
            Log.Warning($"{pawn} 沒有 AbilityTracker，無法授予能力");
            return null;
        }

        var def = DefDatabase<AbilityDef>.GetNamed("MyMod_HealingPulse");

        // 已擁有就不重複授予
        if (pawn.abilities.GetAbility(def) != null)
            return pawn.abilities.GetAbility(def);

        // 授予；第二參數 sendLetter 控制是否跳出「習得能力」信件
        pawn.abilities.GainAbility(def);

        return pawn.abilities.GetAbility(def);
    }

    public static void RevokeHealingPulse(Pawn pawn)
    {
        var def = DefDatabase<AbilityDef>.GetNamed("MyMod_HealingPulse");
        pawn.abilities?.RemoveAbility(def);
    }
}

常見授予時機

  • 透過 Hediff 的 HediffComp_GiveAbility（裝植入物時自動給）
  • GeneDef（Biotech 基因）的 abilities 欄位
  • TraitDef 的 degreeData → abilities
  • 劇情事件 / 自訂 Comp 中手動 GainAbility

來源管理（重要）

若能力由 Hediff/Gene 提供，移除來源時要記得 RemoveAbility，否則 能力會殘留。透過 HediffComp_GiveAbility 等內建機制可自動處理生命週期。

透過 Hediff 自動給能力（推薦） 與其在 C# 硬塞，更穩健的做法是定義一個帶 HediffCompProperties_GiveAbility 的 Hediff， 讓「裝上植入物 → 自動獲得能力 / 拆除 → 自動移除」的生命週期由引擎托管。
5 · 施放能力（Activate / CanCast / Gizmo）

Ability 實例封裝了施放的全部邏輯。正常流程是玩家點 Gizmo → 進入目標選擇 → 產生 Job → Job 末端呼叫 Activate。 你也可以在 C# 中直接驅動。

核心 API

成員回傳說明
CanCastbool屬性：目前是否可施放（冷卻、psyfocus、條件皆通過）
GizmosForPawn(...)IEnumerable<Command>產生 UI 指令按鈕
Activate(target, dest)bool立即執行能力效果（跳過目標選擇 UI）
QueueCastingJob(target, dest)—排入 Job，pawn 走過去/蓄力後再施放（正常路徑）
StartCooldown(ticks)—手動進入冷卻
CooldownTicksRemainingint剩餘冷卻
GicCanCast / CanCast +
GetReason		—含原因字串的版本，用於 UI 灰階提示

直接施放（程式驅動，無 UI）

public static void ForceCast(Pawn caster, LocalTargetInfo target)
{
    var def = DefDatabase<AbilityDef>.GetNamed("MyMod_HealingPulse");
    Ability ability = caster.abilities?.GetAbility(def);
    if (ability == null) return;

    // CanCast：冷卻中 / psyfocus 不足 / 目標不合法都會擋下
    if (!ability.CanCast)
    {
        Messages.Message("無法施放（冷卻或條件不符）",
            MessageTypeDefOf.RejectInput, historical: false);
        return;
    }

    // 路徑 A：立即生效（不走 pawn 移動 / 蓄力）
    ability.Activate(target, target);

    // 路徑 B（較自然）：排入 Job，pawn 自己處理移動與蓄力
    // ability.QueueCastingJob(target, target);
}

覆寫 Gizmo（自訂按鈕外觀 / 行為）

Gizmo 由 Ability.GizmosForPawn(...) 產生，內部建立 Command_Ability。 要自訂時，最乾淨的做法是繼承 Ability 並在 AbilityDef.abilityClass 指定你的子類。 

using System.Collections.Generic;
using RimWorld;
using Verse;

public class Ability_ChargedHeal : Ability
{
    // AbilityDef 需指定 <abilityClass>YourNamespace.Ability_ChargedHeal</abilityClass>
    public Ability_ChargedHeal(Pawn pawn, AbilityDef def) : base(pawn, def) { }

    public override IEnumerable<Command> GetGizmos()
    {
        foreach (var g in base.GetGizmos())
        {
            if (g is Command_Ability cmd)
            {
                // 自訂：低血量時按鈕變紅，提示更醒目
                if (pawn.health.summaryHealth.SummaryHealthPercent < 0.3f)
                    cmd.defaultLabel = def.label + " (緊急!)";
            }
            yield return g;
        }
    }

    // 覆寫額外的施放條件（疊加在 CanCast 之上）
    public override bool GizmoDisabled(out string reason)
    {
        if (pawn.InMentalState)
        {
            reason = "精神崩潰中無法施放";
            return true;
        }
        return base.GizmoDisabled(out reason);
    }
}
Activate vs QueueCastingJob Activate 立即套用效果並進入冷卻，不會讓 pawn 移動或播放蓄力——適合腳本化、瞬發或除錯。 正常玩家路徑請用 QueueCastingJob，它會處理走位、warmup、verb 動畫、psyfocus 消耗時序。
6 · Psycast 與 PsycasterPathDef

超能 (Psycast) 是 Royalty DLC 引入、建立在 Ability 之上的體系。 pawn 需先有 psylink（Hediff_Psylink，提供 psychic sensitivity 與 psylink level）， 超能本身仍是 AbilityDef，但歸屬於 PsycasterPathDef（超能路徑/流派）， 並透過 AbilityDef.level 與 psylink 等級門檻解鎖。

PsycasterPathDef

一條超能路徑（如「Voidsent」），定義可學的 abilities、解鎖順序與所需 psylink level。

Hediff_Psylink

賦予 psylink level 的 Hediff，提升可施放的超能等級與 psyfocus 容量。

AbilityDef（超能）

普通 AbilityDef + <level> + 帶 psyfocus / entropy 消耗的 verb / comp。

PsycasterPathDef XML 骨架

<PsycasterPathDef>
  <defName>MyMod_FrostPath</defName>
  <label>霜寒之道</label>
  <description>操控冰霜的超能流派。</description>
  <iconPath>UI/PsycastPaths/Frost</iconPath>
  <!-- 各能力在此路徑面板中的座標與所需 psylink level -->
  <abilities>
    <li>MyMod_Frostbolt</li>
    <li>MyMod_IceWall</li>
  </abilities>
</PsycasterPathDef>

超能 AbilityDef（標註 psyfocus / entropy 消耗）

<AbilityDef>
  <defName>MyMod_Frostbolt</defName>
  <label>霜彈</label>
  <level>1</level>                  <!-- 所需 psylink level -->
  <cooldownTicksRange>600~600</cooldownTicksRange>

  <!-- 超能消耗：psyfocus（藍條）與 entropy（紅條 / 失序）-->
  <statBases>
    <Ability_EntropyGain>15</Ability_EntropyGain>     <!-- 增加的失序 -->
    <Ability_PsyfocusCost>0.05</Ability_PsyfocusCost> <!-- 消耗的精神專注 -->
  </statBases>

  <verbProperties>
    <verbClass>Verb_CastAbility</verbClass>
    <range>24.9</range>
    <warmupTime>0.5</warmupTime>
    <targetParams><canTargetPawns>true</canTargetPawns></targetParams>
  </verbProperties>

  <comps>
    <li Class="CompProperties_AbilityLaunchProjectile">
      <compClass>CompAbilityEffect_LaunchProjectile</compClass>
      <projectileDef>MyMod_Projectile_Frostbolt</projectileDef>
    </li>
  </comps>
</AbilityDef>
Royalty vs Anomaly 的差異 Royalty 的 psycast 走 psyfocus + entropy 模型；Anomaly 的「實體能力」（如某些怪物/植入物能力） 多半是普通 Ability，不消耗 psyfocus，但可能有自己的資源（如黑暗能量）或純冷卻。 兩者底層都是 Ability，所以 §1–§5 的 API 對兩者皆適用。
7 · 精神專注 (psyfocus) / 失序 (entropy) / Hediff_Psylink

超能的兩種資源由 Pawn_PsychicEntropyTracker（pawn.psychicEntropy）管理：

Psyfocus（精神專注，藍條）

施放超能的「彈藥」。施放消耗、靜心 (meditation) 回復。容量與回復受 psylink level、psychic sensitivity 影響。

Entropy（失序，紅條）

施放的「副作用熱量」。累積過高 → 過載 (overload) → 暈眩甚至死亡。會隨時間自然衰退。

讀寫資源（C#）

using RimWorld;
using Verse;

public static void InspectPsyche(Pawn pawn)
{
    Pawn_PsychicEntropyTracker psy = pawn.psychicEntropy;
    if (psy == null) return; // 無 psylink 的 pawn 可能為 null / 無作用

    float focus   = psy.CurrentPsyfocus;       // 0..1
    float entropy = psy.EntropyValue;          // 當前失序
    float maxEnt  = psy.MaxEntropy;            // 失序上限

    // 手動消耗 psyfocus / 增加 entropy（一般交給能力系統，這裡示範 API）
    psy.TryAddEntropy(20f);                  // 回傳 bool：是否成功（會否過載）
    psy.OffsetPsyfocusDirectly(-0.1f);       // 直接扣 psyfocus

    Log.Message($"focus={focus:P0} entropy={entropy}/{maxEnt}");
}

授予 psylink（Hediff_Psylink）

public static void GrantPsylink(Pawn pawn, int levels = 1)
{
    // 標準工具：自動建立 / 升級 Hediff_Psylink
    pawn.ChangePsylinkLevel(levels, sendLetter: true);

    // 取得 psylink hediff 實例
    Hediff_Psylink link = pawn.GetMainPsylinkSource();
    if (link != null)
        Log.Message($"psylink level = {link.level}");
}
能力的消耗從哪來 超能 AbilityDef 的 Ability_PsyfocusCost / Ability_EntropyGain 兩個 stat 會在 Ability.Activate 內被讀出並套用到 pawn.psychicEntropy。自訂能力若想消耗這些資源， 只要設好 statBases 即可，不必在 comp 內手動扣。
8 · AbilityGroupDef 與冷卻共用

AbilityGroupDef 讓多個能力共用冷卻——施放群組中任一能力後，整組一起進入冷卻。 常用於「同一組互斥的姿態/變身」或「同類技能不可連發」。

定義群組

<AbilityGroupDef>
  <defName>MyMod_StanceGroup</defName>
  <cooldownTicks>1200</cooldownTicks>  <!-- 群組共用冷卻 -->
  <disableGizmosWhileUndrafted>false</disableGizmosWhileUndrafted>
</AbilityGroupDef>

能力綁定群組

<AbilityDef>
  <defName>MyMod_AggressiveStance</defName>
  <groupAbility>MyMod_StanceGroup</groupAbility> <!-- 指向群組 -->
  <!-- ... 其餘欄位 ... -->
</AbilityDef>

<AbilityDef>
  <defName>MyMod_DefensiveStance</defName>
  <groupAbility>MyMod_StanceGroup</groupAbility> <!-- 同群組 → 共用冷卻 -->
</AbilityDef>
冷卻的兩層 能力本身有 cooldownTicksRange（個別冷卻），群組另有 AbilityGroupDef.cooldownTicks（群組冷卻）。 施放後兩者同時計時，CanCast 會取兩者中較嚴格者。可在 C# 用 ability.def.groupAbility 取得所屬群組、用 group 的冷卻狀態判斷。
9 · 目標選擇（TargetingParameters / AbilityTargetCategory）

目標的「合法性」由 verbProperties.targetParams（型別 TargetingParameters）描述—— 可選 pawn / 物件 / 地格 / 自我，以及陣營過濾。能力的目標類別則由 AbilityDef.targetRequired + targetWorldCell + comp 的 CanApplyOn 共同決定。

TargetingParameters 常用欄位

欄位型別說明
canTargetPawnsbool可選 pawn
canTargetBuildingsbool可選建築
canTargetLocationsbool可選空地格
canTargetSelfbool可選施法者自己
canTargetAnimalsbool可選動物
onlyTargetFactionsList<Faction>限定陣營
validatorPredicate<TargetInfo>自訂過濾（C# 設定）

C# 中自訂目標過濾

若要在執行期動態決定合法目標，覆寫 Ability.GetTargetingParameters() 並在 validator 加入條件。 

public class Ability_OnlyWounded : Ability
{
    public Ability_OnlyWounded(Pawn pawn, AbilityDef def) : base(pawn, def) { }

    public override TargetingParameters GetTargetingParameters()
    {
        var tp = base.GetTargetingParameters();
        tp.canTargetPawns = true;
        tp.canTargetLocations = false;

        // 只允許選「受傷的友方 pawn」
        tp.validator = (TargetInfo t) =>
        {
            Pawn p = t.Thing as Pawn;
            return p != null
                && !p.Dead
                && p.Faction == pawn.Faction
                && p.health.summaryHealth.SummaryHealthPercent < 1f;
        };
        return tp;
    }
}

AbilityTargetCategory（多目標 / AoE 形狀）

範圍能力的「波及範圍」常用 verbProperties.range + comp 的半徑欄位（如某些 comp 的 radius）。對全圖/方向型能力，可在自訂 comp 的 Apply 內以 GenRadial.RadialCellsAround 或 GenAdj 枚舉目標格再逐一處理。 

// 在自訂 CompAbilityEffect.Apply 內，對目標周圍 3 格的 pawn 全部生效
public override void Apply(LocalTargetInfo target, LocalTargetInfo dest)
{
    Map map = parent.pawn.Map;
    foreach (IntVec3 c in GenRadial.RadialCellsAround(target.Cell, 3f, useCenter: true))
    {
        if (!c.InBounds(map)) continue;
        foreach (Thing t in c.GetThingList(map))
            if (t is Pawn p && !p.Dead)
                ApplyToPawn(p);
    }
}
10 · 常見坑

坑 1 DLC 依賴未用 MayRequire 標註

Psycast / Anomaly 相關 Def（PsycasterPathDef、特定 comp class）只在對應 DLC 啟用時存在。 若無條件引用，玩家未裝 DLC 時整個 Def 載入失敗。對 XML 欄位或整個 <li> 加上 MayRequire 才能優雅降級。 

<!-- 整個能力僅在 Royalty 存在時載入 -->
<AbilityDef MayRequire="Ludeon.RimWorld.Royalty">
  <defName>MyMod_Frostbolt</defName>
  <!-- ... -->
</AbilityDef>

<!-- 或僅某個欄位/comp 需要 DLC -->
<li Class="CompProperties_AbilityGiveHediff" MayRequire="Ludeon.RimWorld.Anomaly">
  <!-- ... -->
</li>

C# 端對應：用 ModsConfig.RoyaltyActive / ModsConfig.AnomalyActive 守住程式路徑。

坑 2 comp 的 Apply 時序誤解

Apply 在能力「命中」時呼叫，不是點下 Gizmo 當下。中間有 pawn 走位 + warmup， 到實際施放前目標可能已死亡 / 移出地圖 / 變化狀態。Apply 內必須對 target.Pawn == null、Dead、Spawned 做防禦性檢查。 

public override void Apply(LocalTargetInfo target, LocalTargetInfo dest)
{
    Pawn p = target.Pawn;
    // 命中時目標可能已失效
    if (p == null || p.Dead || !p.Spawned) return;
    // ... 安全執行效果 ...
}

此外，多個 comp 的 Apply 依 comps 清單順序執行—— 若 comp A 殺死目標、comp B 想治療同一目標，B 會撲空。注意順序依賴。

坑 3 冷卻：range vs 單值、群組冷卻覆蓋

如 §1 所述，引擎讀 cooldownTicksRange。只寫 cooldownTicks 會得到無冷卻。 另外，若能力屬於 AbilityGroupDef，玩家感受到的冷卻是「群組冷卻」與「個別冷卻」取較大者—— debug 「為什麼冷卻比我設的久」時先檢查群組。

坑 4 pawn.abilities / psychicEntropy 可能為 null

並非所有 pawn 都有 ability tracker 或 psychic entropy tracker（動物、機械、部分特殊 pawn）。 任何 GainAbility / 讀 psyfocus 前都要 null-check，否則崩在 NRE。

坑 5 自訂 comp 的 Props 用 new 而非 override

CompAbilityEffect.props 是基底欄位，子類要存取自己的強型別 Properties 必須用 public new MyProps Props => (MyProps)props;（屬性遮蔽，非覆寫）。 忘了寫或型別轉錯會在第一次 Apply 拋 InvalidCastException。

坑 6 自訂 abilityClass 缺正確建構子

繼承 Ability 時必須提供 (Pawn, AbilityDef)（以及序列化用的無/單參）建構子， 否則 AbilityUtility 反射建立實例時失敗，能力靜默不出現。

能力 debug checklist
  1. 能力沒出現 → 確認 pawn 有 abilities tracker、def 有正確 abilityClass 建構子
  2. Gizmo 灰階 → 印 ability.CanCast 與冷卻 / psyfocus / 群組冷卻
  3. 效果沒發生 → 在 Apply 最前面加 Log.Message，確認有命中、target 未失效
  4. 未裝 DLC 崩潰 → 檢查所有 DLC-only Def / comp 是否加了 MayRequire
  5. 冷卻不對 → 區分 cooldownTicksRange 與 AbilityGroupDef.cooldownTicks
