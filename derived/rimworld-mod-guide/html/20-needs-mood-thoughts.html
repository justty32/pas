心理模型總覽：資料流

RimWorld 的殖民者心理由三層組成。對 C# 工程師而言，最重要的是釐清「誰是狀態的擁有者」與「資料如何向上彙整」：

Need（需求）

連續數值（0–1）。如 Food、Rest、Joy、Mood。掛在 pawn.needs 下，每隔一段 tick 由 NeedInterval() 衰減/更新。

Thought（想法）

對 Mood 施加正/負偏移的「條目」。分記憶型（事件觸發、會過期）與情境型（持續判斷）。彙整於 pawn.needs.mood.thoughts。

MentalState（精神狀態）

當 Mood 跌破閾值，MentalBreaker 觸發 MentalBreakDef，進而啟動 MentalStateDef（暴走、發呆、逃跑等）。

三者的依賴方向是單向的：

Thought（記憶+情境）  ──彙整 MoodOffset──▶  Need_Mood.CurLevel
                                                  │
                                                  ▼
                                      MentalBreaker（比對閾值）
                                                  │
                                                  ▼
                              MentalBreakDef ──▶ MentalStateDef
核心型別所在組件 Need、NeedDef、ThoughtDef、Thought_Memory、MentalBreaker 全在 RimWorld 命名空間（Assembly-CSharp.dll）。 用 dnSpy 開 RimWorld.Need_Mood / RimWorld.MentalBreaker 是理解這套機制最快的方式。
1 · NeedDef 與自訂 Need

一個需求 = NeedDef（XML 定義）+ 一個繼承 Need 的 C# 類別（needClass 指向它）。 NeedDef 描述顯示與閾值；Need 子類別實作衰減邏輯。

XML：NeedDef 定義

<Defs>
  <NeedDef>
    <defName>Caffeine</defName>
    <needClass>MyMod.Need_Caffeine</needClass>  <!-- 指向你的 C# 類別 -->
    <label>咖啡因</label>
    <description>缺乏咖啡因會降低工作效率並影響心情。</description>
    <colonistAndPrisonersOnly>true</colonistAndPrisonersOnly>
    <major>true</major>           <!-- 是否在 Need 面板顯示為主要需求 -->
    <showOnNeedList>true</showOnNeedList>
    <!-- 顯示在 bar 上的門檻刻度（0–1）-->
    <listPriority>40</listPriority>
  </NeedDef>
</Defs>

C#：繼承 Need，覆寫 NeedInterval 與 CurLevel

Need.NeedInterval() 由引擎每 150 ticks（約遊戲內 1 小時/每秒約 2.5 次，視速度）呼叫一次——這是衰減邏輯的放置處。 CurLevel 是 0–1 的當前值；其 setter 通常被覆寫以套用 clamp 與門檻效果。 

using RimWorld;
using Verse;

namespace MyMod
{
    public class Need_Caffeine : Need
    {
        // 每個遊戲日的衰減量（0–1 區間）。150 ticks 呼叫一次 NeedInterval，
        // 一個遊戲日 = 60000 ticks，故一天約 400 次 interval。
        private const float FallPerDay = 0.6f;

        // 建構子簽章固定：Need(Pawn pawn)。XML 用 needClass 反射建立。
        public Need_Caffeine(Pawn pawn) : base(pawn) { }

        // CurLevel 在基底已是 float 自動屬性；如需門檻語意可覆寫 CurCategory 等。
        // 這裡示範三段門檻供 UI 與其他系統查詢。
        public CaffeineCategory CurCategory
        {
            get
            {
                if (CurLevel < 0.15f) return CaffeineCategory.Withdrawal;
                if (CurLevel < 0.40f) return CaffeineCategory.Low;
                return CaffeineCategory.Fine;
            }
        }

        // 在面板 bar 上畫門檻線（選用）
        public override int GUIChangeArrow => -1; // 顯示向下箭頭（持續下降）

        // 核心：每 150 ticks 被引擎呼叫，負責衰減
        public override void NeedInterval()
        {
            // IsFrozen：pawn 被收進容器/世界地圖外時凍結，不衰減
            if (IsFrozen) return;

            // 150 ticks 佔一天的比例 = 150 / 60000
            float tickFraction = 150f / 60000f;
            CurLevel -= FallPerDay * tickFraction;
            // CurLevel setter 已 clamp 到 [0,1]，但顯式 clamp 更安全
            CurLevel = Mathf.Clamp01(CurLevel);
        }

        // 新 pawn / 載入時設定初始值
        public override void SetInitialLevel()
        {
            CurLevel = Rand.Range(0.4f, 0.8f);
        }
    }

    public enum CaffeineCategory { Withdrawal, Low, Fine }
}
讓喝咖啡補充 Need 在 ingestible 的 CompProperties_Ingested / 或自訂 ThingComp 的攝取 hook 裡， 先 var need = pawn.needs.TryGetNeed<Need_Caffeine>(); 再 if (need != null) need.CurLevel += amount;（?. 不能當複合賦值左值）。 TryGetNeed<T>() 在 pawn 未啟用該 need 時回傳 null，務必判空。
2 · 在 PawnKindDef / race 啟用 need

Need 不會自動掛到每個 pawn。Pawn_NeedsTracker.ShouldHaveNeed(NeedDef) 決定某 pawn 是否擁有該 need。 預設規則包含：NeedDef.colonistAndPrisonersOnly、NeedDef.onlyIfCausedByHediff、種族 intelligence 等。 自訂 need 通常需要透過下列其中一種方式納入判定。

方式 A：XML — 用 developmentalStageFilter / 種族條件控制

若你的 need 適用所有 humanlike 殖民者，colonistAndPrisonersOnly=true 加上不額外限制即可被 ShouldHaveNeed 接受。 想限定特定種族，最乾淨的做法是在 race 的 RaceProperties 不直接支援 need 清單，故改用下列 Harmony 或 hediff 方式。

方式 B：綁定到 Hediff（最穩健）

設定 <onlyIfCausedByHediff>，則只有帶該 Hediff 的 pawn 才會有此 need。常用於「植入物 / 異種生理」需求。 

<NeedDef>
  <defName>Caffeine</defName>
  <needClass>MyMod.Need_Caffeine</needClass>
  <onlyIfCausedByHediff>true</onlyIfCausedByHediff>
  <!-- 哪些 Hediff 會「導致」此 need -->
  <causedByHediff>CaffeineDependency</causedByHediff>
</NeedDef>

方式 C：Harmony Postfix 攔截 ShouldHaveNeed

最靈活——可依 PawnKindDef、faction、trait 任意條件決定。針對 Pawn_NeedsTracker.ShouldHaveNeed。 

[HarmonyPatch(typeof(Pawn_NeedsTracker), "ShouldHaveNeed")]
public static class Patch_ShouldHaveNeed
{
    static void Postfix(Pawn ___pawn, NeedDef nd, ref bool __result)
    {
        // ___pawn：Pawn_NeedsTracker 的私有欄位 pawn（三底線存取）
        if (nd != MyNeedDefOf.Caffeine) return;

        // 只給特定 PawnKind（例如某種喝咖啡成癮的種族/職業）
        bool eligible =
            ___pawn.RaceProps.Humanlike &&
            ___pawn.kindDef == MyPawnKindDefOf.CoffeeAddict;

        __result = eligible;
    }
}

[DefOf]
public static class MyNeedDefOf
{
    public static NeedDef Caffeine;
    static MyNeedDefOf() => DefOfHelper.EnsureInitializedInCtor(typeof(MyNeedDefOf));
}
need 清單會被快取 Pawn_NeedsTracker 內部會快取已建立的 need 清單，呼叫 AddOrRemoveNeedsAsAppropriate() 才重新評估。 若你在遊戲中動態改變條件（例如加上 trait），需手動觸發： pawn.needs.AddOrRemoveNeedsAsAppropriate();
3 · ThoughtDef（stages / durationDays / stackLimit / thoughtClass）

ThoughtDef 是想法的藍圖。一個 ThoughtDef 可有多個 stages（階段）， 每個 stage 帶不同 label 與 baseMoodEffect。引擎依條件選擇當前 stage。

欄位說明
thoughtClassC# 類別。預設記憶型用 Thought_Memory，情境型用 Thought_Situational，或你的子類別
workerClass情境型用：ThoughtWorker 子類別，回傳是否啟動與當前 stage
durationDays記憶型才有效：想法存在的天數，到期自動移除
stackLimit同一 ThoughtDef 可堆疊的最大份數（多次觸發累積）
stackedEffectMultiplier第 2 份起的 mood 衰減乘數（避免線性疊加過強）
stages多階段定義，每段含 label / description / baseMoodEffect
nullifyingTraits擁有這些 trait 的 pawn 完全不受此想法影響
nullifyingPrecepts（Ideology）特定 precept 抵銷此想法

XML：記憶型 ThoughtDef（多 stage）

<ThoughtDef>
  <defName>DrankGreatCoffee</defName>
  <durationDays>1</durationDays>        <!-- 1 遊戲日後過期 -->
  <stackLimit>3</stackLimit>            <!-- 最多疊 3 份 -->
  <stackedEffectMultiplier>0.5</stackedEffectMultiplier>
  <!-- 不指定 thoughtClass 時預設 Thought_Memory -->
  <stages>
    <li>
      <label>喝了好咖啡</label>
      <description>那杯咖啡香醇極了，整個人精神奕奕。</description>
      <baseMoodEffect>6</baseMoodEffect>
    </li>
  </stages>
</ThoughtDef>

XML：負面 + 多 stage（依嚴重度切換）

<ThoughtDef>
  <defName>CaffeineWithdrawal</defName>
  <workerClass>MyMod.ThoughtWorker_Caffeine</workerClass>  <!-- 情境型 -->
  <stages>
    <li>
      <label>有點想喝咖啡</label>
      <baseMoodEffect>-3</baseMoodEffect>
    </li>
    <li>
      <label>咖啡因戒斷</label>
      <baseMoodEffect>-10</baseMoodEffect>
    </li>
  </stages>
</ThoughtDef>
baseMoodEffect 的單位 想法的 mood 偏移以「心情點數」計（殖民者 mood 內部是 0–1，但 UI 與想法以 ±N 點顯示，N 約對應 1 個百分點）。 實際生效值還會經過 MoodOffset() 與 trait thoughtOffset 修正（見第 6、9 節）。
4 · Thought_Memory vs Thought_Situational

這是 modding 最關鍵的二分。選錯類型會導致想法不消失、或效能災難。

Thought_Memory（記憶型）

  • 事件驅動：你主動 TryGainMemory() 加入
  • 會被儲存於 MemoryThoughtHandler
  • 依 durationDays 自動過期移除
  • 可堆疊（stackLimit）
  • 適合：一次性事件——吃到好料、目睹死亡、收到禮物

Thought_Situational（情境型）

  • 條件驅動：由 ThoughtWorker 每隔一段時間重新評估
  • 不儲存——條件不成立就自動消失，不需手動移除
  • 無 durationDays（持續存在於條件為真期間）
  • 適合：環境/狀態類——在漂亮房間、太擠、餓肚子、戒斷中
面向Thought_MemoryThought_Situational
加入方式C# 主動呼叫 TryGainMemoryThoughtWorker 自動評估
移除方式durationDays 到期 / 手動 RemoveMemory條件不成立自動消失
是否存檔是（含剩餘 age）否（每次重算）
效能成本低（只在事件時）較高（週期性評估）
typical workerClass無必填
5 · 從 C# 加想法（TryGainMemory）

記憶型想法透過 pawn.needs.mood.thoughts.memories.TryGainMemory(...) 加入。 存取鏈每一層都可能為 null（動物、無 mood need 的 pawn），務必逐層判空。 

using RimWorld;
using Verse;

public static class CoffeeMood
{
    // 安全地給 pawn 加一個記憶型想法
    public static void GainDrankGreatCoffee(Pawn pawn)
    {
        // needs 可能為 null（如未生成的 pawn），mood 只在 humanlike 上存在
        ThoughtHandler thoughts = pawn?.needs?.mood?.thoughts;
        if (thoughts == null) return;

        // 1) 最簡單：直接用 ThoughtDef
        thoughts.memories.TryGainMemory(MyThoughtDefOf.DrankGreatCoffee);
    }

    // 指定 stage / 來源 pawn（社交想法）
    public static void GainSocialThought(Pawn pawn, Pawn otherPawn)
    {
        ThoughtHandler thoughts = pawn?.needs?.mood?.thoughts;
        if (thoughts == null) return;

        // 2) 建立 Thought_Memory 實例，設定 stage 後再加入
        Thought_Memory mem = (Thought_Memory)ThoughtMaker.MakeThought(
            MyThoughtDefOf.DrankGreatCoffee,
            stageIndex: 0);

        // otherPawn 用於社交想法（影響 opinion）
        thoughts.memories.TryGainMemory(mem, otherPawn);
    }

    // 手動移除（記憶型不一定要等過期）
    public static void RemoveCoffeeThought(Pawn pawn)
    {
        ThoughtHandler thoughts = pawn?.needs?.mood?.thoughts;
        thoughts?.memories.RemoveMemoriesOfDef(MyThoughtDefOf.DrankGreatCoffee);
    }
}

[DefOf]
public static class MyThoughtDefOf
{
    public static ThoughtDef DrankGreatCoffee;
    public static ThoughtDef CaffeineWithdrawal;
    static MyThoughtDefOf() => DefOfHelper.EnsureInitializedInCtor(typeof(MyThoughtDefOf));
}
常用 ThoughtHandler / MemoryThoughtHandler API
  • memories.TryGainMemory(ThoughtDef / Thought_Memory, Pawn otherPawn = null)
  • memories.RemoveMemoriesOfDef(ThoughtDef)
  • memories.NumMemoriesOfDef(ThoughtDef) — 已堆疊份數（檢查 stackLimit 用）
  • thoughts.TotalMoodOffset() — 當前所有想法的 mood 總偏移
  • thoughts.GetMoodThoughts(List<Thought>) — 取全部生效想法
情境型想法不要用 TryGainMemory 情境型由 ThoughtWorker 自動產生。對情境型 ThoughtDef 呼叫 TryGainMemory 會塞進 memory handler， 變成「不會自動消失的殭屍想法」。情境型只要寫好 worker，引擎自己會處理。
6 · MoodOffset 與心情崩潰閾值（MentalBreak）

MoodOffset：想法實際生效值

Thought.MoodOffset() 回傳該想法當下對 mood 的實際偏移。基底是 CurStage.baseMoodEffect， 但你的子類別可覆寫以做動態計算（例如依 need 數值線性插值）。 

public class Thought_CaffeineWithdrawal : Thought_Situational
{
    // 依 Caffeine need 的缺乏程度，動態計算 mood 懲罰
    public override float MoodOffset()
    {
        Need_Caffeine need = pawn.needs?.TryGetNeed<Need_Caffeine>();
        if (need == null) return 0f;

        // need 越低，懲罰越大：0 → -15，0.4 → 0
        float deficit = Mathf.Clamp01((0.4f - need.CurLevel) / 0.4f);
        return -15f * deficit;
    }
}

心情崩潰閾值

MentalBreaker（pawn.mindState.mentalBreaker）持有三條閾值，依 mood 跌破哪一條決定崩潰嚴重度：

屬性預設值（mood 0–1）對應崩潰
BreakThresholdMinor~0.35Minor（輕度，如 sad wander）
BreakThresholdMajor~0.20Major（中度，如 binging）
BreakThresholdExtreme~0.05Extreme（重度，如 berserk）

這些閾值受 trait（如 Nerves）、hediff、難度設定影響。 MentalBreaker.MentalBreakThresholdFor(category) 與 CurMood 是查詢入口。 

// 查詢某 pawn 當前距離崩潰多遠
MentalBreaker breaker = pawn.mindState.mentalBreaker;
float mood    = pawn.needs.mood.CurLevel;          // 0–1
float minorAt = breaker.BreakThresholdMinor;        // 0–1
bool  atRisk  = breaker.BreakMinorIsImminent;        // 即將輕度崩潰
bool  extreme = breaker.BreakExtremeIsApproaching;
調整閾值的正確切入點 不要直接寫 MentalBreaker 私有欄位。閾值來自 stat MentalBreakThreshold 與相關修正。 要全域調整可 patch MentalBreaker 的閾值 getter，或新增提供 StatPart 的 hediff/trait。
7 · MentalStateDef 與 MentalBreakDef（精神崩潰）

兩者分工明確：MentalBreakDef = 「在哪個嚴重度可被選中」的觸發包裝， 內含對應的 MentalStateDef = 實際的行為狀態機（pawn 進入後做什麼）。

XML：MentalStateDef（行為狀態）

<MentalStateDef>
  <defName>CaffeineRage</defName>
  <stateClass>RimWorld.MentalState_Berserk</stateClass> <!-- 重用既有狀態類別 -->
  <workerClass>RimWorld.MentalStateWorker</workerClass>
  <category>Aggro</category>
  <beginLetterLabel>咖啡因暴走</beginLetterLabel>
  <beginLetter>{0} 因為咖啡因戒斷而陷入暴怒，開始攻擊周遭的一切。</beginLetter>
  <beginLetterDef>ThreatSmall</beginLetterDef>
  <recoveryMtbDays>0.5</recoveryMtbDays>  <!-- 平均回復時間 -->
  <danger>Deadly</danger>
</MentalStateDef>

XML：MentalBreakDef（觸發包裝）

<MentalBreakDef>
  <defName>Break_CaffeineRage</defName>
  <mentalState>CaffeineRage</mentalState>     <!-- 指向上面的 MentalStateDef -->
  <intensity>Extreme</intensity>            <!-- Minor / Major / Extreme -->
  <baseCommonality>1.0</baseCommonality>      <!-- 在同 intensity 池內被抽中的權重 -->
  <!-- 只有符合此 worker 條件的 pawn 才會抽到這個 break -->
  <workerClass>MyMod.MentalBreakWorker_Caffeine</workerClass>
</MentalBreakDef>

C#：MentalBreakWorker — 限定誰會觸發

using RimWorld;
using Verse;

public class MentalBreakWorker_Caffeine : MentalBreakWorker
{
    // 回傳此 break 對特定 pawn 的觸發機率權重；0 = 永不觸發
    public override float CommonalityFor(Pawn pawn, bool moodCaused = false)
    {
        Need_Caffeine need = pawn.needs?.TryGetNeed<Need_Caffeine>();
        // 只有處於嚴重戒斷的 pawn 才可能咖啡因暴走
        if (need == null || need.CurLevel > 0.1f) return 0f;
        return base.CommonalityFor(pawn, moodCaused);
    }
}

C#：主動讓 pawn 進入精神狀態

// 略過 mood 判定，直接強制進入某精神狀態（事件腳本常用）
bool started = pawn.mindState.mentalStateHandler.TryStartMentalState(
    MyMentalStateDefOf.CaffeineRage,
    reason: "咖啡因戒斷",
    forceWake: true,
    causedByMood: false);

// 或透過 MentalBreakDef 走正常崩潰流程
MyMentalBreakDefOf.Break_CaffeineRage.Worker.TryStart(pawn, "咖啡因戒斷", causedByMood: true);
重用內建 stateClass 多數情況不需要自己寫 MentalState 子類別——RimWorld 已有 MentalState_Berserk、 MentalState_Wander_Sad、MentalState_BingingFood 等十餘種。只在需要全新行為時才繼承 MentalState 並覆寫 MentalStateTick()。
8 · 自訂 ThoughtWorker（情境想法的條件判斷）

情境型想法的核心是 ThoughtWorker。引擎週期性對每個 pawn 呼叫 CurrentStateInternal(pawn)， 由你回傳 ThoughtState：是否啟動、用哪個 stage。 

using RimWorld;
using Verse;

public class ThoughtWorker_Caffeine : ThoughtWorker
{
    // 回傳 ThoughtState：
    //   ThoughtState.Inactive            → 想法不出現
    //   ThoughtState.ActiveAtStage(int)  → 出現，並指定 stage 索引
    protected override ThoughtState CurrentStateInternal(Pawn p)
    {
        // humanlike 且有 mood need 才評估（情境想法只給有 mood 的 pawn）
        if (!p.RaceProps.Humanlike) return ThoughtState.Inactive;

        Need_Caffeine need = p.needs?.TryGetNeed<Need_Caffeine>();
        if (need == null) return ThoughtState.Inactive;

        // 依需求高低選 stage（對應 ThoughtDef 的 stages 索引）
        if (need.CurLevel < 0.15f) return ThoughtState.ActiveAtStage(1); // 戒斷
        if (need.CurLevel < 0.40f) return ThoughtState.ActiveAtStage(0); // 想喝
        return ThoughtState.Inactive;
    }
}

內建可繼承/參考的 ThoughtWorker

  • ThoughtWorker_Hungry / ThoughtWorker_Tired — need 門檻型範本
  • ThoughtWorker_RoomBeauty / ThoughtWorker_Spacious — 環境型
  • ThoughtWorker_Precept（Ideology）— 信念型，可帶參數
ThoughtState 是 readonly struct 不要 new 一個再改欄位。一律用靜態工廠：ThoughtState.Inactive、ThoughtState.ActiveAtStage(i)、 ThoughtState.ActiveWithReason(label, desc)（自訂顯示文字）。
9 · 特質影響想法（nullifiesThoughts / thoughtOffset）

Trait 透過 TraitDegreeData 修改想法效果，兩個主要機制：完全抵銷某想法、或對某想法施加額外 mood 偏移。

XML：在 TraitDef 內宣告

<TraitDef>
  <defName>CaffeineImmune</defName>
  <commonality>0.5</commonality>
  <degreeDatas>
    <li>
      <label>咖啡因免疫</label>
      <degree>0</degree>

      <!-- 注意：TraitDegreeData 沒有 nullifiedThoughts / thoughtOffsets 欄位。 -->
      <!-- 「某 trait 抵銷某想法」一律在 ThoughtDef 端用 nullifyingTraits 宣告（見下）。 -->
    </li>
  </degreeDatas>
</TraitDef>

反向：在 ThoughtDef 指定 nullifyingTraits

也可從想法那端宣告「哪些 trait 會讓我失效」，效果等同。擇一即可，別兩邊重複。

<ThoughtDef>
  <defName>CaffeineWithdrawal</defName>
  <nullifyingTraits>
    <li>CaffeineImmune</li>
  </nullifyingTraits>
  <!-- ... -->
</ThoughtDef>

坑 1 stackLimit 與 NumMemoriesOfDef 的關係

TryGainMemory 不會「自動」拒絕超過 stackLimit 的疊加——它會加入，但生效時引擎只計入前 stackLimit 份的 mood， 多餘份數仍佔記憶槽位、仍各自獨立倒數過期。若你需要「達上限就不再加」，自行檢查： 

var mem = pawn.needs.mood.thoughts.memories;
if (mem.NumMemoriesOfDef(MyThoughtDefOf.DrankGreatCoffee)
        < MyThoughtDefOf.DrankGreatCoffee.stackLimit)
{
    mem.TryGainMemory(MyThoughtDefOf.DrankGreatCoffee);
}

坑 2 記憶想法「不消失」

常見兩個原因：

  • 沒設 durationDays：未指定 duration 的記憶型想法永久存在，除非手動 RemoveMemoriesOfDef。 若你想要永久就 OK；否則務必設定。
  • 把情境型用成記憶型：對情境型 ThoughtDef 呼叫了 TryGainMemory（見第 5 節警告）。 它會被當記憶儲存、沒有 duration、條件改變也不消失，變成殭屍想法。

坑 3 Situational 想法的效能

情境型 worker 由引擎週期性對每個有 mood 的 pawn呼叫。CurrentStateInternal 內若做昂貴運算 （遍歷地圖、LINQ 配置、反射），在大殖民地會成為明顯卡頓來源。

  • 盡早 return ThoughtState.Inactive（先檢查最廉價的排除條件）
  • 避免每次呼叫 new 集合 / 用 LINQ；改快取或用 for 迴圈
  • 昂貴的判定可自行降頻：用 Find.TickManager.TicksGame % N 配合欄位快取結果

坑 4 存取鏈 NullReference

pawn.needs.mood.thoughts.memories 任何一層都可能 null：動物無 mood need、 生成前的 pawn needs 為 null、機械族無心理。一律 ?. 逐層判空，別假設殖民者就一定有。

坑 5 NeedInterval 的 tick 換算

NeedInterval() 不是每 tick 呼叫，而是每 150 ticks。寫衰減時要用「每 interval」而非「每 tick」的量， 否則衰減會慢 150 倍。一天 = 60000 ticks = 400 個 interval。換算公式： perInterval = perDay * 150 / 60000。

坑 6 DefOf 未初始化 / Def 載入順序

[DefOf] 欄位在 Def 全部載入後才被填值。若在更早的時機（如 Mod 建構子）存取會是 null。 把使用 ThoughtDef/NeedDef 的邏輯放在遊戲執行期（如 worker、tick、patch），別放在載入早期的靜態初始化。
