Harmony 修補

在不修改原始碼的前提下，外科手術式地改變遊戲行為。

Harmony 是什麼

Harmony 是 Andreas Pardeike 開發的 .NET 方法修補框架（GitHub）， 核心機制是在 JIT 編譯後、執行時期動態修改方法的 IL 指令流，插入你自己的邏輯。 RimWorld 遊戲本體已內建 0Harmony.dll，所有 mod 共享同一個 Harmony 實例池。

Prefix

在目標方法執行前插入。可存取參數、可選擇中止原方法（回傳 false）。

Postfix

在目標方法執行後插入。可讀取並修改回傳值（ref __result）。

Transpiler

直接操作 IL 指令序列。最強大、最危險——能做到另外兩種無法做到的事。

1 · Harmony 初始化

Harmony patch 必須在遊戲載入期間、Assembly 被注入後盡早執行。 RimWorld 提供 [StaticConstructorOnStartup] attribute——標記這個 class 後， 遊戲在所有 Def 載入完畢、第一次進入主選單前會自動呼叫其靜態建構子。 

using HarmonyLib;
using Verse;

[StaticConstructorOnStartup]
public static class MyModInit
{
    static MyModInit()
    {
        // "com.yourname.yourmod" 必須全局唯一，建議反向域名格式
        var harmony = new Harmony("com.yourname.yourmod");

        // PatchAll 掃描當前 Assembly 中所有帶 [HarmonyPatch] attribute 的 class
        harmony.PatchAll();

        // 或只掃描特定 Assembly：
        // harmony.PatchAll(Assembly.GetExecutingAssembly());

        Log.Message("[MyMod] Harmony patches applied.");
    }
}
為何不在 Mod 建構子裡初始化？ Mod 子類別的建構子在 Def 載入之前就被呼叫（用來讀取 ModSettings）。 此時許多 RimWorld 類別尚未初始化，貿然 patch 可能命中未就緒的靜態欄位，引發難以追蹤的 NRE。 把 Harmony 初始化放在 [StaticConstructorOnStartup] 靜態建構子是社群標準做法。
手動 Patch（不用 PatchAll） 若你想精細控制每個 patch 的時機，可以用 harmony.Patch(original, prefix, postfix) 手動逐一套用。 這在條件式啟用某些 patch 時很有用（例如偵測到特定 mod 才啟用）。
2 · Prefix — 攔截前置

Prefix 在目標方法的「入口」插入你的程式碼。 回傳 bool 時，false 代表「跳過原始方法（及後續同優先級的 Prefix）」； true 或回傳 void 代表繼續執行。

特殊參數名稱

參數名稱型別說明
__instance目標類別型別目標物件的 this（靜態方法無此參數）
__result回傳值型別（ref）搭配 return false 提前設定回傳值
__state任意型別（ref）在 Prefix 與 Postfix 間傳遞自訂狀態
__originalMethodMethodBase目標方法的 MethodInfo 本身
___fieldName對應型別（ref）三底線前綴：存取目標物件的私有欄位
（原方法參數名）原型別（可加 ref）直接按名稱對應；加 ref 可修改傳入值

完整範例：攔截 Pawn_HealthTracker.MakeDowned

目標：當某個 pawn 擁有特定 Hediff 時，完全阻止其被擊倒（直接傳回，原方法不執行）。 

using HarmonyLib;
using Verse;

[HarmonyPatch(typeof(Pawn_HealthTracker), nameof(Pawn_HealthTracker.MakeDowned))]
public static class Patch_MakeDowned
{
    /// <summary>
    /// 回傳 false → 跳過原始 MakeDowned，pawn 不會進入倒地狀態。
    /// 回傳 true  → 繼續執行原始方法。
    /// </summary>
    static bool Prefix(
        Pawn_HealthTracker __instance,   // this
        DamageInfo?        dinfo,          // 原方法第一個參數（按名稱對應）
        HediffDef          hediffDef)      // 原方法第二個參數
    {
        // Pawn_HealthTracker.pawn 是 internal，用 Traverse 或 ___pawn（三底線）存取
        Pawn pawn = __instance.GetField<Pawn>("pawn"); // 說明用；見 AccessTools 節

        // 若 pawn 具有「免疫倒地」的自訂 Hediff，阻止原方法
        if (pawn.health.hediffSet.HasHediff(MyHediffDefOf.ImmuneToDown))
            return false; // 跳過原始 MakeDowned

        return true; // 其他情況正常執行
    }
}

使用 ref 修改參數

[HarmonyPatch(typeof(StatWorker), nameof(StatWorker.GetValue), new[] { typeof(StatRequest), typeof(bool) })]
public static class Patch_StatWorker_GetValue_Pre
{
    static void Prefix(ref StatRequest req)
    {
        // ref 參數：直接修改傳入的 StatRequest 副本
        // 注意：StatRequest 是 struct，不加 ref 則修改無效
        if (req.HasThing && req.Thing is Pawn pawn)
        {
            // 自訂邏輯 ...
        }
    }
}

提前設定回傳值（搭配 return false）

[HarmonyPatch(typeof(FoodUtility), nameof(FoodUtility.WillIngestFromInventoryNow))]
public static class Patch_WillIngest
{
    static bool Prefix(Pawn pawn, ref bool __result)
    {
        if (pawn.def.defName == "MyCustomPawnDef")
        {
            __result = false; // 設定回傳值
            return false;     // 跳過原方法，直接回傳 __result
        }
        return true;
    }
}
多個 Prefix 的「跳過」語意 return false 只跳過「原始方法」，不會阻止其他 mod 的 Prefix 執行（除非它們的優先權更低）。 若要控制跳過範圍，需搭配 HarmonyPriority。
3 · Postfix — 後置修改

Postfix 在目標方法（以及所有 Prefix）執行完畢後才執行。 它是修改回傳值、追加副作用的最安全位置——不需要了解原方法的完整邏輯。

適合 Postfix 的場景

  • 修改／包裝原方法的回傳值
  • 在原方法行為後附加副作用
  • 記錄／Debug 原方法的輸出
  • 讀取原方法結束後的物件狀態

注意

  • 若原方法拋出例外，Postfix 不會執行
  • 不能阻止原方法執行（那是 Prefix 的職責）
  • ref __result 型別必須完全匹配回傳型別

完整範例：修改 StatWorker.GetValue 回傳值

目標：對特定 pawn 的某個 stat 值套用乘數。 

using HarmonyLib;
using RimWorld;
using Verse;

[HarmonyPatch(typeof(StatWorker), nameof(StatWorker.GetValue),
    new[] { typeof(StatRequest), typeof(bool) })]
public static class Patch_StatWorker_GetValue
{
    /// <summary>
    /// Postfix 的簽章：第一個參數仍可寫 __instance、原方法參數，
    /// 然後加上 ref __result 取得（並修改）回傳值。
    /// </summary>
    static void Postfix(
        StatWorker   __instance,
        StatRequest  req,
        ref float    __result)   // GetValue 回傳 float
    {
        // 只對 MoveSpeed 這個 stat 有效
        if (__instance.stat != StatDefOf.MoveSpeed) return;
        if (!req.HasThing || req.Thing is not Pawn pawn) return;

        // pawn 有特定 trait 時，移動速度 +20%
        if (pawn.story?.traits.HasTrait(MyTraitDefOf.Speedster) == true)
        {
            __result *= 1.2f;
        }
    }
}

同時使用 __state（Prefix ↔ Postfix 通訊）

[HarmonyPatch(typeof(JobDriver_Ingest), "MakeNewToils")]
public static class Patch_Ingest_Timing
{
    // __state 型別由你決定；Prefix 設值，Postfix 讀值
    static void Prefix(ref long __state)
    {
        __state = System.Diagnostics.Stopwatch.GetTimestamp();
    }

    static void Postfix(long __state)
    {
        long elapsed = System.Diagnostics.Stopwatch.GetTimestamp() - __state;
        Log.Message($"MakeNewToils 耗時：{elapsed} ticks");
    }
}
4 · Transpiler — IL 層修改 最後手段
先嘗試 Prefix / Postfix Transpiler 直接修改 IL 指令序列，每次遊戲更新都可能讓 patch 失效，且 debug 難度極高。 只有當 Prefix/Postfix 無法表達你的意圖時，才使用 Transpiler。

Transpiler 接收目標方法的 IEnumerable<CodeInstruction>，回傳修改後的序列。 你可以插入、刪除、替換任意 IL 指令。

必要引用

using HarmonyLib;
using System.Collections.Generic;
using System.Reflection;
using System.Reflection.Emit; // OpCodes 在這裡

Transpiler 簽章

static IEnumerable<CodeInstruction> Transpiler(
    IEnumerable<CodeInstruction> instructions,
    ILGenerator                    il,       // 選用：建立新 label/local
    MethodBase                     original) // 選用：目標方法 MethodInfo

完整範例：在條件判斷前插入額外判斷

假設原始方法有一段 if (pawn.Dead) return;， 我們想在這條 return 之前，額外檢查另一個條件——如果自訂狀態為真也提前 return。 

[HarmonyPatch(typeof(Pawn_JobTracker), "TryFindAndStartJob")]
public static class Patch_TryFindAndStartJob
{
    static IEnumerable<CodeInstruction> Transpiler(
        IEnumerable<CodeInstruction> instructions,
        ILGenerator il)
    {
        var codes = new List<CodeInstruction>(instructions);

        // 找目標：ldfld Pawn::Dead 之後緊跟著 brfalse（if !Dead skip）
        var deadField = AccessTools.Field(typeof(Pawn), "Dead");
        var myHelper = AccessTools.Method(typeof(Patch_TryFindAndStartJob), nameof(ShouldSkipJob));

        for (int i = 0; i < codes.Count - 1; i++)
        {
            // 辨識模式：ldarg.0（this） → ldfld pawn → ldfld Dead → brtrue/brfalse
            if (codes[i].LoadsField(deadField))
            {
                // 在這條指令之前插入：ldarg.0（this）、call ShouldSkipJob、brtrue earlyReturn
                var earlyReturn = il.DefineLabel();

                // 在原有 ldfld Dead 之前注入
                codes.InsertRange(i, new[]
                {
                    new CodeInstruction(OpCodes.Ldarg_0),          // this (Pawn_JobTracker)
                    new CodeInstruction(OpCodes.Call, myHelper),   // call ShouldSkipJob
                    new CodeInstruction(OpCodes.Brtrue, earlyReturn) // if true → jump
                });

                // earlyReturn label 掛在原來 Dead 分支的 return 指令上
                // （這裡簡化：找下一個 ret 指令打上 label）
                for (int j = i + 3; j < codes.Count; j++)
                {
                    if (codes[j].opcode == OpCodes.Ret)
                    {
                        codes[j].labels.Add(earlyReturn);
                        break;
                    }
                }
                break; // 只插一次
            }
        }

        return codes;
    }

    // Transpiler 呼叫的靜態輔助方法——必須是靜態的
    static bool ShouldSkipJob(Pawn_JobTracker tracker)
    {
        // 取得 pawn（私有欄位）
        Pawn pawn = Traverse.Create(tracker).Field<Pawn>("pawn").Value;
        return pawn?.health?.hediffSet?.HasHediff(MyHediffDefOf.Paralyzed) ?? false;
    }
}

常用 CodeInstruction 查找方法

方法說明
ci.LoadsField(FieldInfo)是否為 ldfld / ldsfld 指定欄位
ci.StoresField(FieldInfo)是否為 stfld / stsfld 指定欄位
ci.Calls(MethodInfo)是否呼叫指定方法
ci.opcode == OpCodes.Ret是否為 return 指令
ci.Is(OpCode, object)比對 opcode + operand 組合
CodeInstruction.Call(type, "Method")建立 call 指令的便利工廠
除錯技巧 在 Transpiler 最後加上 foreach (var c in codes) Log.Message(c.ToString()); 可在遊戲 log 中印出整段 IL，方便和 dnSpy 反編譯結果比對。完成後記得移除。
5 · TargetMethod — 動態指定目標

[HarmonyPatch] attribute 只能在編譯期靜態指定目標。 當目標是泛型方法、屬性存取子、建構子，或你需要在執行期決定目標時， 改用 [HarmonyTargetMethod] 靜態方法動態回傳 MethodBase。

Attribute 層級宣告（不指定目標）

[HarmonyPatch] // 不填任何參數，交給 TargetMethod 決定
public static class Patch_DynamicTarget
{
    [HarmonyTargetMethod]
    static MethodBase TargetMethod()
    {
        // 範例一：屬性 getter
        return AccessTools.PropertyGetter(typeof(Pawn), "BodySize");
    }
}

[HarmonyPatch]
public static class Patch_Setter
{
    [HarmonyTargetMethod]
    static MethodBase TargetMethod() =>
        AccessTools.PropertySetter(typeof(Building_Bed), "Medical");
}

[HarmonyPatch]
public static class Patch_Ctor
{
    [HarmonyTargetMethod]
    static MethodBase TargetMethod() =>
        AccessTools.Constructor(typeof(Pawn));
}

多目標：[HarmonyTargetMethods]（複數）

[HarmonyPatch]
public static class Patch_MultiTarget
{
    [HarmonyTargetMethods]
    static IEnumerable<MethodBase> TargetMethods()
    {
        // 同時 patch 多個方法（共用同一個 Prefix/Postfix）
        yield return AccessTools.Method(typeof(Pawn), "Kill");
        yield return AccessTools.Method(typeof(Pawn), "Destroy");
    }

    static void Prefix(Pawn __instance)
    {
        Log.Message($"[MyMod] Pawn {__instance.Name} is being removed.");
    }
}

泛型方法

// 泛型方法必須先 MakeGenericMethod 才能 patch
[HarmonyPatch]
public static class Patch_GenericMethod
{
    [HarmonyTargetMethod]
    static MethodBase TargetMethod()
    {
        return AccessTools
            .Method(typeof(SomeClass), "GenericMethod")
            .MakeGenericMethod(typeof(Pawn)); // 指定型別參數
    }
}
6 · AccessTools — 反射助手

AccessTools 是 Harmony 提供的反射工具集，自動繞過 private / protected 限制， 省去手寫 BindingFlags.NonPublic | BindingFlags.Instance 的麻煩。

欄位、方法、屬性查找

using HarmonyLib;

// ── 欄位 ──────────────────────────────────────────────────
FieldInfo  fi = AccessTools.Field(typeof(Pawn), "ageTracker");
FieldInfo  fs = AccessTools.Field(typeof(StaticClass), "someStaticField");

// 強型別的 ref delegate（效能較佳，適合高頻呼叫）
var getAgeTracker = AccessTools.FieldRefAccess<Pawn, Pawn_AgeTracker>("ageTracker");
Pawn_AgeTracker tracker = getAgeTracker(pawn); // 直接讀取，不用 Invoke

// ── 方法 ──────────────────────────────────────────────────
MethodInfo mi = AccessTools.Method(typeof(Pawn), "CheckAcceptArrest");
// 方法多載：指定參數型別陣列
MethodInfo mo = AccessTools.Method(typeof(StatWorker), "GetValue",
    new[] { typeof(StatRequest), typeof(bool) });

// DeclaredMethod：只在宣告型別查找，不往父類別走（避免繼承歧義）
MethodInfo dm = AccessTools.DeclaredMethod(typeof(Building_Bed), "GetInspectString");

// ── 屬性存取子 ────────────────────────────────────────────
MethodInfo getter = AccessTools.PropertyGetter(typeof(Pawn), "BodySize");
MethodInfo setter = AccessTools.PropertySetter(typeof(Building_Bed), "Medical");

// ── 建構子 ────────────────────────────────────────────────
ConstructorInfo ci = AccessTools.Constructor(typeof(Pawn));
ConstructorInfo cs = AccessTools.Constructor(typeof(SomeClass),
    new[] { typeof(int), typeof(string) });

Traverse — 鏈式反射存取

Traverse 提供 fluent API，用來讀寫深層私有欄位或呼叫私有方法， 不需要自己管理 FieldInfo。 

using HarmonyLib;

// 讀取私有欄位
Pawn_AgeTracker tracker =
    Traverse.Create(pawn)
             .Field<Pawn_AgeTracker>("ageTracker")
             .Value;

// 寫入私有欄位
Traverse.Create(pawn)
         .Field("ageTracker")
         .SetValue(newTracker);

// 呼叫私有方法（有回傳值）
string result =
    Traverse.Create(pawn)
             .Method("GetLabel", "arg1", 42)
             .GetValue<string>();

// 存取靜態型別
float val =
    Traverse.Create(typeof(StaticClass))
             .Field<float>("cachedValue")
             .Value;

// 多層鏈式（nested private fields）
int deep =
    Traverse.Create(pawn)
             .Field("health")
             .Field("hediffSet")
             .Field<int>("internalCounter")
             .Value;
效能注意 Traverse 每次呼叫都有反射開銷。在每 tick 都執行的路徑（如 Tick()）， 改用 AccessTools.FieldRefAccess 建立快取 delegate，或直接快取 FieldInfo。
7 · HarmonyPriority — 執行順序

多個 mod 可以 patch 同一個方法。Harmony 依優先值決定執行順序， 預設值為 Priority.Normal（= 400）。

Prefix 執行順序

Priority 值由高到低執行。
High(600) → Normal(400) → Low(200)

Postfix 執行順序

Priority 值由低到高執行（即與 Prefix 相反）。
Low(200) → Normal(400) → High(600)

宣告方式

[HarmonyPatch(typeof(StatWorker), nameof(StatWorker.GetValue),
    new[] { typeof(StatRequest), typeof(bool) })]
[HarmonyPriority(Priority.High)]        // 整個 class 套用
public static class Patch_HighPriority
{
    [HarmonyPriority(Priority.VeryHigh)] // 也可單獨標在方法上覆寫 class 層級
    static void Prefix() { }

    static void Postfix() { } // 沿用 class 層級 Priority.High
}

預定義優先值

常數值建議用途
Priority.VeryHigh800確保在所有 mod 之前執行（慎用）
Priority.High600需要優先設定初始狀態
Priority.Normal400預設值，一般情況
Priority.Low200需要在其他 mod 修改後再調整
Priority.VeryLow100最後執行的兜底邏輯
Priority.Last0絕對最後（比 VeryLow 更低）
跨 mod 協調的實務 RimWorld 的 mod 載入順序（在 modlist 中的位置）也影響 Harmony 的 patch 順序—— priority 相同時，後載入的 mod 的 Prefix 先執行，Postfix 後執行。 若你的 mod 和另一個 mod 有衝突，可在 About.xml 的 <loadAfter> 宣告相依性。
8 · 常見陷阱

陷阱 1 靜態方法沒有 __instance

__instance 只對實例方法有效。若目標是 static 方法， Harmony 不會注入 __instance——宣告了也永遠是 null， 且有時會導致 patch 無法套用。 

// ❌ 錯誤：FoodUtility.GetManhunterOnDamageChance 是靜態方法
static void Prefix(FoodUtility __instance) { } // __instance 永遠是 null！

// ✅ 正確：直接使用原方法參數
static void Prefix(ThingDef def, Pawn attacker) { }

陷阱 2 ref 參數型別必須完全匹配

ref __result 的型別必須和目標方法的回傳型別完全一致，包含可空（nullable）。 型別不符時 patch 會 silent fail（不套用、不報錯）。 

// 目標方法：float GetValue(...)
static void Postfix(ref double __result) { } // ❌ double ≠ float，不會套用
static void Postfix(ref float  __result) { } // ✅

陷阱 3 [HarmonyPatch] 放在錯誤的層級

Harmony 支援把 patch 參數拆成多個 attribute，但所有指定目標的 attribute 必須放在 class 層級， 不能放在方法上（除了少數特殊 attribute 如 [HarmonyPriority]）。 

// ❌ 錯誤：[HarmonyPatch] 放在方法上
public static class MyPatch
{
    [HarmonyPatch(typeof(Pawn), "Kill")]
    static void Prefix() { } // PatchAll 找不到這個
}

// ✅ 正確：[HarmonyPatch] 在 class 上
[HarmonyPatch(typeof(Pawn), "Kill")]
public static class MyPatch
{
    static void Prefix() { }
}

陷阱 4 PatchAll 找不到方法時的 Silent Fail

若 [HarmonyPatch] 指定的方法名稱打錯，或目標方法不存在， Harmony 預設靜默略過這個 patch，不拋例外，遊戲照常執行但你的 patch 完全無效。 

// 開發期間：手動驗證 patch 是否生效
var harmony = new Harmony("com.yourname.yourmod");
harmony.PatchAll();

// 印出所有已套用的 patch（包含 owner）
foreach (var method in harmony.GetPatchedMethods())
{
    var info = Harmony.GetPatchInfo(method);
    Log.Message($"Patched: {method.DeclaringType?.Name}.{method.Name} — "
        + $"Prefix:{info.Prefixes.Count} Postfix:{info.Postfixes.Count}");
}

陷阱 5 多次 Patch 同一個方法

若你的 Harmony 實例（相同 ID）對同一個方法呼叫多次 Patch（非 PatchAll）， Harmony 2.x 會疊加——每次呼叫都新增一個 patch，導致你的邏輯執行多次。

  • 熱重載或重複初始化時特別容易發生。
  • 使用 PatchAll 通常沒問題，因為靜態建構子只執行一次。
  • 手動 patch 時，先呼叫 harmony.UnpatchAll("com.yourname.yourmod") 再重新 patch。
// 安全的手動 patch 模式（避免重複疊加）
var harmony = new Harmony("com.yourname.yourmod");
harmony.UnpatchAll("com.yourname.yourmod"); // 清掉舊的（如有）
harmony.PatchAll();                          // 重新套用

陷阱 6 方法名稱使用字串 vs nameof

公開方法建議用 nameof()，讓 IDE 能夠重構追蹤。 私有方法只能用字串——拼錯時靜默失效，要特別小心。 

// 公開方法：用 nameof（重構安全）
[HarmonyPatch(typeof(Pawn_HealthTracker), nameof(Pawn_HealthTracker.MakeDowned))]

// 私有方法：只能用字串（小心拼錯）
[HarmonyPatch(typeof(Pawn_HealthTracker), "CheckForStateChange")]

快速除錯 Checklist
  1. 用 harmony.GetPatchedMethods() 確認 patch 有套用
  2. 在 Prefix/Postfix 最開頭加 Log.Message("hit") 確認有被呼叫
  3. 用 dnSpy 確認目標方法的完整簽章（參數型別、名稱）
  4. 確認 [StaticConstructorOnStartup] 靜態建構子確實被執行（可加 Log）
  5. 確認 DLL 有正確放入 Assemblies/ 並在 About.xml 列出
快速參考：三種 Patch 比較
類型 執行時機 可阻止原方法 可改回傳值 複雜度 建議優先度
Prefix 原方法之前 ✓（return false） ✓（搭配 return false） 低 首選
Postfix 原方法之後 ✗ ✓（ref __result） 低 首選
Transpiler IL 層（取代原方法） ✓（完全控制） ✓（完全控制） 高 最後手段
