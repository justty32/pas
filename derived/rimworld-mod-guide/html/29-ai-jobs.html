AI 工作系統：Job、Toil、WorkGiver、ThinkTree

RimWorld 的小人不是被「指揮」，而是被一棵思維樹「拉取」工作。理解 ThinkTree → JobGiver → Job → JobDriver → Toil 這條鏈，是寫任何自訂行為的根。

Chapter 29 · AI Jobs
§1 整條鏈總覽：從 tick 到 Toil

RimWorld 沒有「指揮系統」這種東西。每個 Pawn 身上掛著一個 Verse.AI.Pawn_JobTracker（即 pawn.jobs），它在每個 tick 檢查「我現在有沒有事做」。當前一個 Job 結束、被打斷、或根本沒有 Job 時，它會去拉一棵 Verse.ThinkTreeDef 編譯成的思維樹，由樹從上往下「拉取」(pull) 出下一個該做的 Verse.AI.Job。換句話說：工作不是推給小人的，是小人主動從思維樹裡撈出來的。

這條鏈分成五個層次，職責切得很乾淨。先看資料如何流動：

Pawn_JobTracker.JobTrackerTick()  ── 每 tick 檢查是否需要新 Job
   ↓ ThinkResult thinkResult = thinker.MainThinkNodeRoot.TryIssueJobPackage(...)
ThinkTree（思維樹：一堆 ThinkNode 組成的決策樹）
   ↓ 樹由上往下走，遇到能產出 Job 的節點就回傳
JobGiver_Work（思維樹節點，掃描所有啟用的 WorkGiver）
   ↓ WorkGiver_Scanner.JobOnThing(pawn, thing)
Job（資料封裝：def + targetA/B/C + count + ...）
   ↓ Pawn_JobTracker.StartJob(job)
JobDriver（行為：JobDef.driverClass 指定，產生 Toil 序列）
   ↓ JobDriver.MakeNewToils() yield return ...
Toil（最小行為單元：走過去 / 等待 / 拿起 / 套用效果）
   ↓ 一個接一個執行，全部跑完 Job 才結束

決策層

ThinkTreeDef / ThinkNode / JobGiver 決定「現在該做什麼」。它只負責挑工作，不負責怎麼做。

資料層

Job 是一張「工作單」：哪個 JobDef、目標是誰（targetA/B/C）、數量多少。純資料，無行為。

行為層

JobDriver 把工作單翻譯成一連串 Toil。Toil 才是真正會走路、會敲打、會改變世界狀態的東西。

兩條工作來源：WorkGiver vs JobGiver

初學者最容易搞混 WorkGiver 和 JobGiver。它們都是「產生 Job 的東西」，但定位完全不同：

WorkGiver（玩家工作標籤）

對應到右下角「工作」面板那一欄欄可勾選、可排優先序的工作（種田、烹飪、建造……）。玩家能開關與排序。多個 WorkGiver 被 JobGiver_Work 這個單一思維樹節點統一掃描。寫「殖民者平時自動會做的勞動」就用這條。

JobGiver（思維樹節點）

是 ThinkNode_JobGiver 的子類，直接掛在思維樹的某個位置。玩家看不到、不能關。用於非勞動行為：逃跑、進食、社交、發呆、精神崩潰行為、動物覓食。它的優先序由它在樹裡的位置決定。

名詞職責速查表

名詞類型職責典型基類
ThinkTreeDef Def (XML) 整棵決策樹的根；Humanlike、Animal、Mechanoid 各一棵 —
ThinkNode C# class 樹上的一個節點；可以是優先選擇器、子樹、或產生 Job 的葉節點 Verse.AI.ThinkNode
JobGiver C# class 會直接吐 Job 的思維樹葉節點（非玩家可見工作） Verse.AI.ThinkNode_JobGiver
WorkGiver C# + Def 玩家可見的一格工作；由 JobGiver_Work 統一掃描排序 RimWorld.WorkGiver_Scanner
Job C# 資料物件 一張工作單：def + targetA/B/C + count + 其他參數 Verse.AI.Job
JobDef Def (XML) 工作的靜態描述：用哪個 driver、報告字串、可否被打斷 —
JobDriver C# class 執行 Job 的狀態機，產出 Toil 序列、做預約 Verse.AI.JobDriver
Toil C# 物件 最小行為單元；有 initAction / tickAction / 完成條件 Verse.AI.Toil
心智模型把整條鏈想成：ThinkTree 是「大腦的決策迴路」，WorkGiver/JobGiver 是「念頭」，Job 是「待辦清單上的一條」，JobDriver 是「執行這條時的肌肉記憶腳本」，Toil 是「一個動作」。 你要改什麼行為，先問自己：是要新增一種念頭（JobGiver/WorkGiver）、還是新增一套動作流程（JobDriver/Toil）、還是改決策優先序（ThinkTree）。
§2 JobDef：工作的靜態身分證

Verse.JobDef 是純資料 Def，描述一種工作「是什麼」，但不描述「怎麼做」。怎麼做交給 driverClass 指向的 JobDriver。一個 JobDef 通常對應一個 JobDriver。香草遊戲的 JobDef 都集中在 Defs/JobDefs/。

核心欄位一覽

欄位型別說明
driverClass Type (JobDriver) 必填。執行此工作的 JobDriver 子類，例如 JobDriver_FoodFeedPatient
reportString string 選取小人時右下角顯示的字串模板，可用 TargetA 等佔位符（如 Mining at {0}.）
playerInterruptible bool 玩家下達新命令時能否打斷此工作（預設 true；手術、分娩等設 false）
casualInterruptible bool 能否被「隨意」事件打斷（看到敵人、需求警報等）。手術設 false 避免醫生中途跑掉
joyKind JoyKindDef 若此工作屬於娛樂，提供哪種娛樂類型（社交、感官、奢侈……）
joyMaxParticipants int 娛樂工作最多幾人同時參與（撞球桌 2、聚會更多）
joySkill / joyGainRate 各種 娛樂相關：練哪個技能、娛樂回填速度
allowOpportunisticPrefix bool true 時走去做這工作的路上可順手搬運（順路搬運最佳化）
collideWithPawns bool 執行此工作時是否與其他 Pawn 碰撞（戰鬥相關工作常為 true）
suspendable bool true（預設）時遇到緊急狀況可暫停；長手術鏈通常維持可暫停
alwaysShowWeapon bool 執行時是否一直把武器畫出來

最小 JobDef XML 範例

<!-- Defs/JobDefs/Jobs_MyMod.xml -->
<Defs>
  <JobDef>
    <defName>MyMod_RefuelCustomMachine</defName>
    <!-- driverClass 是整個 Def 的核心，沒它 Job 無法執行 -->
    <driverClass>MyMod.JobDriver_RefuelCustomMachine</driverClass>
    <!-- {0} 會被 TargetA 取代（reportString 解析時） -->
    <reportString>Refueling {0}.</reportString>
    <!-- 玩家點別處可打斷，是日常勞動 -->
    <playerInterruptible>true</playerInterruptible>
    <casualInterruptible>true</casualInterruptible>
    <!-- 走過去的路上可順手搬東西 -->
    <allowOpportunisticPrefix>true</allowOpportunisticPrefix>
  </JobDef>
</Defs>
reportString 的佔位符{0} = TargetA、{1} = TargetB、{2} = TargetC。若需要更動態的描述，可在 JobDriver 覆寫 GetReport() 自行組字串，這比 reportString 模板靈活得多（很多香草工作都這樣做）。

建立 Job 物件（C# 端）

有了 JobDef，在 C# 用 Verse.JobMaker.MakeJob 建立實際的 Job 實例。不要直接 new Job(...)——RimWorld 1.x 起一律走 JobMaker 物件池：

using Verse;       // JobMaker
using Verse.AI;    // Job、JobTag
using RimWorld;    // JobDefOf

// 三種常見建法
Job job1 = JobMaker.MakeJob(MyMod_JobDefOf.MyMod_RefuelCustomMachine, targetThing);
Job job2 = JobMaker.MakeJob(JobDefOf.HaulToCell, haulThing, destCell);
Job job3 = JobMaker.MakeJob(JobDefOf.Ingest, foodThing);
job3.count = 3;  // 吃 3 份

// 把 Job 塞給小人去做（強制指定，跳過思維樹）
pawn.jobs.TryTakeOrderedJob(job1, JobTag.Misc);

替自己的 JobDef 寫一個 DefOf 類別，避免到處用字串查找：

[DefOf]
public static class MyMod_JobDefOf
{
    public static JobDef MyMod_RefuelCustomMachine;

    static MyMod_JobDefOf() => DefOfHelper.EnsureInitializedInCtor(typeof(MyMod_JobDefOf));
}
§3 寫一個自訂 JobDriver

Verse.AI.JobDriver 是整個系統的心臟。它是一個由 Toil 序列驅動的協程狀態機。你主要覆寫兩個方法：

TryMakePreToilReservations

在 Job 開始前做預約。回傳 bool：能成功預約目標就 true，否則 false（Job 取消）。典型呼叫 pawn.Reserve(target, job, ...)。沒預約好就動工會被別的小人搶，或產生 "could not reserve" 紅字。

MakeNewToils

回傳 IEnumerable<Toil>，用 yield return 一個個吐出 Toil。這是個迭代器：Toil 會依序執行，全部完成 Job 才結束。可在中間插入失敗條件、進度條、自訂 action。

常用 Toil 工廠

不要每個 Toil 都手刻。RimWorld 提供一票工廠類別產生常見動作：

工廠方法產出的 Toil
Toils_Goto.GotoThing(ind, peMode) 走到某個 target（以 TargetIndex 指定）；peMode 決定停在哪（Touch / ClosestTouch / InteractionCell）
Toils_Goto.GotoCell(ind, peMode) 走到某個格子
Toils_General.Wait(ticks) 原地等待 N tick（60 tick = 1 秒），常配 WithProgressBar
Toils_Reserve.Reserve(ind) 在 Toil 鏈中途預約目標（除了 PreToil 預約外的補強）
Toils_Haul.StartCarryThing(ind) 把 target 拿到手上（變成攜帶物）
Toils_Haul.CarryHauledThingToCell(ind) 把手上的東西搬到指定格
Toils_Haul.PlaceHauledThingInCell(ind, ...) 把手上的東西放到地上 / 容器
Toils_Ingest.* 進食相關（吃下、扣食物量、回填飢餓需求）
Toils_Combat.* 戰鬥相關（瞄準、開火）

Toil 的擴充方法：失敗條件、進度條、action

Toil 本身是純資料，靠一票 fluent 擴充方法（多在 Verse.AI.ToilFailConditions、Verse.AI.ToilEffects）串接行為與守衛：

擴充方法作用
.FailOnDespawnedOrNull(ind) 若 target 消失/銷毀/為 null，Job 立即失敗（最常用的守衛）
.FailOnDestroyedOrNull(ind) target 被摧毀或 null 則失敗（不管是否 despawn）
.FailOnForbidden(ind) target 被玩家標記禁止則失敗
.FailOn(() => bool) 自訂 lambda，每 tick 檢查，回 true 則失敗
.FailOnBurningImmobile(ind) target 著火且不可移動則失敗
.WithProgressBarToilDelay(ind) 依 Toil 的 defaultDuration 在 target 上顯示進度條
.PlaySustainerOrSound(...) 執行期間播放持續音效
FailOn 是註冊不是當下判斷FailOnXXX 把條件登記到 Toil 的 endConditions 清單，每 tick 都會檢查，並非只在 Toil 開始時判一次。所以要把守衛掛在「實際工作的那個 Toil」上，而不是只掛在 GotoThing 上——否則走到一半目標消失才會被抓到，工作 Toil 期間目標被搶走則無人理會。慣用做法是把 FailOn 串在工廠回傳值上，整條鏈共享。

defaultCompleteMode：Toil 何時算完成

每個自製 Toil 一定要指定 defaultCompleteMode，否則永遠不會結束、Pawn 卡死：

ToilCompleteMode何時完成
Instant當 tick 執行完 initAction 立即完成（瞬間動作）
Delay等待 defaultDuration 個 tick 後完成（配進度條）
Never永不自動完成，必須靠 FailOn / 程式碼 / EndJobWith 結束（如戰鬥瞄準）
PatherArrival抵達路徑終點時完成（GotoThing 內部用）

完整範例：走到某物 → 工作 N tick → 產生效果

下面這個 JobDriver 模擬「走到自訂機器 → 工作 120 tick（顯示進度條）→ 補充燃料」。注意 TargetIndex 的用法（TargetA = 機器）：

using System.Collections.Generic;
using RimWorld;
using Verse;
using Verse.AI;

namespace MyMod
{
    public class JobDriver_RefuelCustomMachine : JobDriver
    {
        // 方便讀取的快捷屬性
        private Thing Machine => job.GetTarget(TargetIndex.A).Thing;

        // 工作所需 tick；60 tick = 1 秒
        private const int RefuelDuration = 120;

        public override bool TryMakePreToilReservations(bool errorOnFailed)
        {
            // 預約機器；失敗（已被別人預約）就放棄此 Job
            return pawn.Reserve(Machine, job, 1, -1, null, errorOnFailed);
        }

        protected override IEnumerable<Toil> MakeNewToils()
        {
            // 整條鏈共用的守衛：機器消失就整個 Job 失敗
            this.FailOnDespawnedOrNull(TargetIndex.A);
            this.FailOnBurningImmobile(TargetIndex.A);

            // 1) 走到機器的互動格
            yield return Toils_Goto.GotoThing(TargetIndex.A, PathEndMode.InteractionCell)
                .FailOnDespawnedNullOrForbidden(TargetIndex.A);

            // 2) 工作 Toil：等待 N tick，顯示進度條，期間播音效
            Toil work = Toils_General.Wait(RefuelDuration, TargetIndex.A);
            work.WithProgressBarToilDelay(TargetIndex.A);
            work.FailOnCannotTouch(TargetIndex.A, PathEndMode.InteractionCell);
            work.PlaySustainerOrSound(SoundDefOf.Interact_Tend);
            work.activeSkill = () => SkillDefOf.Crafting;  // 練手藝技能
            yield return work;

            // 3) 工作完成的瞬間動作：產生實際效果
            yield return new Toil
            {
                initAction = () =>
                {
                    // 這裡才真正改變世界狀態
                    CompRefuelable comp = Machine.TryGetComp<CompRefuelable>();
                    comp?.Refuel(25f);
                    Machine.Map.designationManager
                        .DesignationOn(Machine, MyMod_DesignationDefOf.RefuelDesignation)
                        ?.Delete();
                },
                defaultCompleteMode = ToilCompleteMode.Instant
            };
        }
    }
}
initAction vs tickActioninitAction 在 Toil 開始的那一個 tick 跑一次（適合「結算瞬間效果」）；tickAction 每 tick 跑一次（適合「持續累積」，如逐漸補燃料、逐漸建造）。若需要漸進效果，用 tickAction 配 defaultCompleteMode = Delay，並在 tickAction 內呼叫 readyForNextToil = true 提前結束，或用 EndOnDespawnedOrNull 收尾。
§4 WorkGiver：把工作插進玩家的工作面板

當你要的是「殖民者平常自動會做、且玩家可以在工作面板開關與排優先序」的勞動，就要寫 WorkGiver。它由兩部分組成：WorkGiverDef（XML 資料）+ WorkGiver_Scanner 的 C# 子類（掃描邏輯）。

WorkGiver_Scanner 的三個關鍵方法

方法職責
PotentialWorkThingsGlobal(pawn) 回傳「可能需要這份工作」的候選 Thing 集合。回傳越精準（用 designation / ListerThings 過濾）效能越好。回 null 則退回掃全地圖（慢）
HasJobOnThing(pawn, thing, forced) 快速布林檢查：這個 pawn 對這個 thing 現在有沒有事可做。便宜的篩選，先擋掉不合格者
JobOnThing(pawn, thing, forced) 真正產生 Job。通常先做完整可達/可預約檢查，再 JobMaker.MakeJob 回傳

WorkGiver 範例：為帶有指定標記的機器加燃料

using RimWorld;
using Verse;
using Verse.AI;
using System.Collections.Generic;

namespace MyMod
{
    public class WorkGiver_RefuelCustomMachine : WorkGiver_Scanner
    {
        // 用 designation 縮小候選範圍，避免掃全地圖
        public override IEnumerable<Thing> PotentialWorkThingsGlobal(Pawn pawn)
        {
            foreach (Designation d in pawn.Map.designationManager
                .SpawnedDesignationsOfDef(MyMod_DesignationDefOf.RefuelDesignation))
            {
                yield return d.target.Thing;
            }
        }

        // 走 designation 路徑時告訴引擎用哪種 designation
        public override PathEndMode PathEndMode => PathEndMode.InteractionCell;

        public override bool HasJobOnThing(Pawn pawn, Thing t, bool forced = false)
        {
            // 便宜檢查：有沒有 comp、需不需要燃料
            CompRefuelable comp = t.TryGetComp<CompRefuelable>();
            if (comp == null || comp.IsFull) return false;

            // 是否被禁止 / 能否預約且抵達
            if (t.IsForbidden(pawn)) return false;
            if (!pawn.CanReserveAndReach(t, PathEndMode.InteractionCell,
                    pawn.NormalMaxDanger(), 1, -1, null, forced))
                return false;

            return true;
        }

        public override Job JobOnThing(Pawn pawn, Thing t, bool forced = false)
        {
            // HasJobOnThing 已通過，這裡直接組 Job
            return JobMaker.MakeJob(MyMod_JobDefOf.MyMod_RefuelCustomMachine, t);
        }
    }
}
HasJobOnThing 與 JobOnThing 必須一致引擎先用 HasJobOnThing 篩、再用 JobOnThing 取。如果 HasJobOnThing 回 true 但 JobOnThing 回 null，小人會「想做卻拿不到工作」，導致每 tick 反覆嘗試、出現遲滯或 spam log。兩者的判定條件要嚴格對齊。

WorkGiverDef + WorkTypeDef（XML）

WorkGiverDef 把你的 C# scanner 接到一個 WorkTypeDef（工作大類，如 Hauling、Crafting）。把自訂工作掛到既有 WorkType 最省事：

<!-- Defs/WorkGivers/WorkGivers_MyMod.xml -->
<Defs>
  <WorkGiverDef>
    <defName>MyMod_RefuelCustomMachine</defName>
    <label>refuel custom machine</label>
    <giverClass>MyMod.WorkGiver_RefuelCustomMachine</giverClass>
    <!-- 掛到香草「基礎勞動」工作類別 -->
    <workType>BasicWorker</workType>
    <!-- 在同一 workType 內的相對優先序，數字大者先 -->
    <priorityInType>50</priorityInType>
    <verb>refuel</verb>
    <gerund>refueling</gerund>
    <!-- 命令小人去做時，若不可達顯示的提示 -->
    <requiredCapacities>
      <li>Manipulation</li>
    </requiredCapacities>
  </WorkGiverDef>
</Defs>

如果要做全新的工作大類（在工作面板多一欄），則需新增 WorkTypeDef，並指定 workGiversByPriority 把 WorkGiverDef 串進去。但 workGiversByPriority 多由引擎依各 WorkGiverDef 的 workType + priorityInType 自動組裝，一般不必手填：

<!-- 只有需要全新欄位時才寫 WorkTypeDef -->
<WorkTypeDef>
  <defName>MyMod_MachineTending</defName>
  <labelShort>tend</labelShort>
  <pawnLabel>technician</pawnLabel>
  <gerundLabel>tending</gerundLabel>
  <description>Maintain custom machines.</description>
  <verb>tend</verb>
  <!-- 工作面板由左到右的排序權重 -->
  <naturalPriority>410</naturalPriority>
  <workTags>
    <li>ManualSkilled</li>
  </workTags>
  <relevantSkills>
    <li>Crafting</li>
  </relevantSkills>
</WorkTypeDef>
CanReserveAndReach 的意義pawn.CanReserveAndReach(target, peMode, danger, ...) 一次檢查三件事：(1) 路徑可達、(2) 預約沒被別人佔、(3) 沿途危險度可接受。它是 WorkGiver 的標準守門員，幾乎所有香草 WorkGiver 都在 JobOnThing 或 HasJobOnThing 呼叫它。
§5 ThinkTree 與 ThinkNode：決策樹本身

前面說的 WorkGiver / JobGiver 都掛在 Verse.ThinkTreeDef 編成的決策樹上。香草最重要的兩棵是 Humanlike（人類主樹）與 Humanlike_PostMentalState。樹由一層層 ThinkNode 組成，從根往下走，第一個能吐出 Job 的節點獲勝。

常見 ThinkNode 類型

ThinkNode 子類行為
ThinkNode_Priority 依序嘗試每個子節點，第一個成功者勝出（最常見的選擇器）
ThinkNode_PrioritySorter 動態依優先值排序子節點再嘗試（如各種逃跑/治療的緊急度）
ThinkNode_Subtree 引用另一棵 ThinkTreeDef（模組化複用，如 subtreesToInsert 的目標）
ThinkNode_ConditionalXXX 守衛節點：只有條件成立才往下走（如 ThinkNode_ConditionalColonist）
JobGiver_Work 掃描所有啟用 WorkGiver、依優先序產生勞動 Job 的葉節點
JobGiver_GetRest / JobGiver_GetFood 需求驅動的葉節點（睡覺、吃飯）
JobGiver_Wander* 沒事做時到處晃的葉節點（樹的最底層 fallback）

ThinkTreeDef 結構（節錄香草 Humanlike）

<!-- 香草樹的概念結構（簡化） -->
<ThinkTreeDef>
  <defName>Humanlike</defName>
  <thinkRoot Class="ThinkNode_Priority">
    <subNodes>
      <!-- 1) 最高優先：被俘/受控/精神狀態 -->
      <li Class="ThinkNode_ConditionalCanDoConstantThinkTreeJobNow"> ... </li>
      <!-- 2) Duty（被 Lord 控制時，見 §6） -->
      <li Class="ThinkNode_ConditionalHasLord">
        <subNodes>
          <li Class="ThinkNode_DutyConstant" />
        </subNodes>
      </li>
      <!-- 3) 緊急需求：逃離火、找掩護 -->
      <li Class="JobGiver_ExitMapBest" />
      <!-- 4) 玩家勞動工作（這裡掃 WorkGiver） -->
      <li Class="ThinkNode_ConditionalColonist">
        <subNodes>
          <li Class="JobGiver_Work" />
        </subNodes>
      </li>
      <!-- 5) 滿足需求：吃、睡、娛樂 -->
      <!-- 6) 最底層 fallback：到處晃 -->
      <li Class="JobGiver_WanderColony" />
    </subNodes>
  </thinkRoot>
</ThinkTreeDef>

用 insertTag 插入自訂 ThinkNode（相容做法）

絕對不要整棵複製 vanilla 樹再改——相容性會死。RimWorld 內建的相容插入機制是：另外宣告一棵獨立的 ThinkTreeDef，在它身上設 insertTag 與 insertPriority（這兩個欄位定義在 Verse.ThinkTreeDef 上）。香草主樹裡的 Verse.AI.ThinkNode_SubtreesByTag 節點會在執行期把所有 insertTag 相符的子樹依 insertPriority 拉進來：

<!-- Defs/ThinkTrees/MyInsertedSubtree.xml -->
<ThinkTreeDef>
  <defName>MyMod_SpecialChoreSubtree</defName>
  <!-- 對準香草樹中某個 ThinkNode_SubtreesByTag 的 insertTag -->
  <insertTag>Humanlike_PreMain</insertTag>
  <!-- 數字越大越早被嘗試（越高優先） -->
  <insertPriority>50</insertPriority>
  <thinkRoot Class="ThinkNode_ConditionalColonist">
    <subNodes>
      <li Class="MyMod.JobGiver_DoSpecialChore" />
    </subNodes>
  </thinkRoot>
</ThinkTreeDef>
沒有 PatchOperationInsertThinkNode 這種東西香草只有通用的 PatchOperationAdd / PatchOperationInsert 等（皆在 Verse），並沒有 think-node 專用的 PatchOperation。要在指定 xpath 位置硬插節點可用 PatchOperationAdd，但首選仍是上面的 insertTag 子樹機制，相容性最好。insertTag 必須對準香草樹中既有的 ThinkNode_SubtreesByTag 標籤。

自訂 JobGiver（非玩家可見的念頭）

ThinkNode_JobGiver 只要覆寫一個 TryGiveJob。回傳 null 代表「我沒事可給」，樹就往下個節點走：

using Verse;
using Verse.AI;

namespace MyMod
{
    public class JobGiver_DoSpecialChore : ThinkNode_JobGiver
    {
        protected override Job TryGiveJob(Pawn pawn)
        {
            // 只有殖民者、心情夠好、地圖上有目標才給
            if (!pawn.IsColonist) return null;

            Thing target = GenClosest.ClosestThingReachable(
                pawn.Position, pawn.Map,
                ThingRequest.ForDef(MyMod_ThingDefOf.SpecialChoreTarget),
                PathEndMode.Touch,
                TraverseParms.For(pawn),
                30f,
                t => !t.IsForbidden(pawn) && pawn.CanReserve(t));

            if (target == null) return null;

            return JobMaker.MakeJob(MyMod_JobDefOf.DoSpecialChore, target);
        }

        // 可覆寫 GetPriority 讓父節點是 PrioritySorter 時動態排序
        public override float GetPriority(Pawn pawn) => 5f;
    }
}
ThinkResult樹節點實際回傳的是 Verse.AI.ThinkResult，封裝了 Job + 來源 ThinkNode + JobTag + 是否 fromQueue。多數時候你不會直接碰它（JobGiver 幫你包好），但除錯思維樹時知道它的存在很有用——它正是 "View thinkTree" 顯示鏈條的依據。
§6 Duty 與 Mental State：另兩個 Job 來源

除了「自主思維樹」這條主線，還有兩個重要的 Job 來源覆蓋在它之上：被 Lord 控制時的 Duty，以及精神崩潰時的 MentalState。

Duty（群體 AI 的命令）

當 Pawn 隸屬某個 Verse.AI.Group.Lord（襲擊隊、商隊、典禮參與者），Lord 會給它一個 PawnDuty。思維樹頂部的 ThinkNode_ConditionalHasLord 會優先走 Duty 對應的子樹，覆蓋日常勞動。這是第 30 章群體 AI 的核心，這裡先建立銜接。

MentalState（精神狀態）

當 Pawn 進入 Verse.AI.MentalState（暴走、放火、麻木閒晃），pawn.mindState.mentalStateHandler 會切換到專屬的思維樹 Humanlike_MentalState，覆蓋一切正常行為。與第 20 章「需求心情」互補：心情崩到閾值以下才觸發。

DutyDef 與 Duty 思維樹

Verse.AI.DutyDef 把一個職責名稱接到一棵 duty 思維樹（thinkNode）。Lord 的 LordToil 在每個 Pawn 身上設 pawn.mindState.duty = new PawnDuty(dutyDef, focusTarget)，思維樹便依此產生對應 Job：

<!-- 一個簡化的 DutyDef -->
<DutyDef>
  <defName>MyMod_GuardSpot</defName>
  <hook>HighPriority</hook>
  <thinkNode Class="ThinkNode_Priority">
    <subNodes>
      <!-- 受攻擊時還擊 -->
      <li Class="JobGiver_AISapper" />
      <!-- 否則站在 duty.focus 指定的點上 -->
      <li Class="JobGiver_Standby">
        <radius>4</radius>
      </li>
    </subNodes>
  </thinkNode>
</DutyDef>

在 C# 端的 LordToil 設定 duty（細節留待第 30 章）：

public override void UpdateAllDuties()
{
    foreach (Pawn p in lord.ownedPawns)
    {
        PawnDuty duty = new PawnDuty(MyMod_DutyDefOf.MyMod_GuardSpot);
        duty.focus = new LocalTargetInfo(guardCell);
        p.mindState.duty = duty;
    }
}

MentalStateDef 簡述

Verse.MentalStateDef 定義一種精神狀態：用哪個 stateClass（MentalState 子類）、持續時間、是否攻擊、結束條件等。觸發通常由 MentalBreakDef（心情崩潰類型，見第 20 章）呼叫 pawn.mindState.mentalStateHandler.TryStartMentalState(def)：

<!-- 自訂精神狀態：到處砸東西 -->
<MentalStateDef>
  <defName>MyMod_SmashThings</defName>
  <stateClass>MentalState_Tantrum</stateClass>
  <category>Misc</category>
  <recoveryMtbDays>0.5</recoveryMtbDays>
  <minTicksBeforeRecovery>1250</minTicksBeforeRecovery>
  <baseInspectLine>Mental state: smashing things</baseInspectLine>
</MentalStateDef>
三條來源的優先級從思維樹頂往下看：Constant tree（被俘/出口） > MentalState > Duty（有 Lord 時） > 日常勞動 JobGiver_Work > 需求 > 閒晃 fallback。你要插入的自訂行為該放在哪一層，取決於它的「不可抗拒程度」。
§7 預約與路徑：Reservation 與 Reach

RimWorld 用預約系統避免多個小人搶同一個目標或同一格。Verse.AI.ReservationManager（map.reservationManager）管理所有 target 預約。每個 Job 在開始前要「預約」它的目標，否則別人會插隊，或同一個目標被兩人同時施工。

預約相關 API

API用途
pawn.Reserve(target, job, maxPawns, stackCount, layer, errorOnFailed) 在 JobDriver.TryMakePreToilReservations 內預約。成功回 true
map.reservationManager.CanReserve(pawn, target, ...) 檢查能否預約（不真的佔用），WorkGiver 篩選用
pawn.CanReserve(target, maxPawns, stackCount, layer, ignoreOtherReservations) Pawn 層的便捷封裝
pawn.CanReserveAndReach(target, peMode, danger, ...) 同時檢查預約 + 路徑可達 + 危險度（WorkGiver 標準守門）
map.reservationManager.ReleaseAllForTarget(thing) 手動釋放某 target 的所有預約（特殊情境）

路徑可達：CanReach 與 PathEndMode

「能不能走到」由 pawn.CanReach(target, peMode, danger, ...) 判定。PathEndMode 決定「走到哪裡才算到」，這個選錯會讓小人站錯位置或永遠到不了：

PathEndMode抵達定義
OnCell必須踩到 target 那一格本身
Touch站在 target 相鄰（含斜角）任一格即可
ClosestTouch走到最近的可觸碰格（Touch 的最佳化版）
InteractionCell站在建築物指定的互動格（工作台、床等有固定操作位）
ClosestTouchUseShootingCheck近身或可射擊位置（戰鬥用）
"could not reserve" 紅字成因這是 modding 最常見的崩錯之一，通常是：(1) WorkGiver 沒檢查 CanReserve 就回 Job，導致兩個小人同時被指派；(2) JobDriver.TryMakePreToilReservations 回 true 但實際沒呼叫 pawn.Reserve，Toil 中途又去 Reserve 撞車；(3) maxPawns / stackCount 與工作實際需求不符（如要搬 30 個但只預約了 1）。對策：WorkGiver 端用 CanReserveAndReach(forced) 守門，JobDriver 端在 PreToil 確實預約同一批 target。
forced 參數很重要當玩家右鍵「強制執行」時，引擎傳 forced = true。許多檢查（如工作優先序為 0、距離過遠）在 forced 時要放寬。把 forced 一路傳進 CanReserveAndReach(..., forced)，否則玩家強制命令會莫名失效。
§8 除錯：看穿小人的腦袋

AI 行為最難除錯，因為「小人不做我要的事」可能出在鏈條任何一環。RimWorld 的 Dev mode 提供了直接的思維樹檢視工具。先在設定開啟 Development mode。

Dev mode 工具

View thinkTree

Dev 工具列 → 選中一個 Pawn → 開啟 thinkTree 檢視。它會即時印出當前思維樹被遍歷的路徑，告訴你是哪個 ThinkNode 產出了當前 Job，以及為什麼其他節點被跳過。這是定位「我的 JobGiver 為何沒被選中」的首要工具。

Pawn debug / reportString

選中 Pawn 後右下角會顯示 JobDef.reportString 解析後的當前工作描述。配合 Dev 的 "Pawn debug info" 可看 curJob、curDriver、目前在第幾個 Toil。

用 Log 印出當前 Job 狀態

// 臨時除錯：印出小人現在在做什麼、哪個 driver、第幾個 Toil
Job cur = pawn.CurJob;
Log.Message($"[MyMod] {pawn.LabelShort} curJob=" +
    $"{cur?.def.defName ?? \"(none)\"} " +
    $"targetA={cur?.targetA.ToStringSafe()} " +
    $"driver={pawn.jobs.curDriver?.GetType().Name}");

// 想知道為何 WorkGiver 沒給工作？手動跑一次它的判定
var wg = new WorkGiver_RefuelCustomMachine();
Log.Message($"HasJobOnThing = {wg.HasJobOnThing(pawn, target, forced: false)}");
Log.Message($"JobOnThing = {wg.JobOnThing(pawn, target)?.def.defName ?? \"null\"}");

最常見的 AI 卡死陷阱

症狀根因修法
小人站著不動、Toil 永遠不結束 自製 Toil 忘了設 defaultCompleteMode 明確設 Instant / Delay（配 defaultDuration）/ 用 FailOn 收尾
每 tick 反覆嘗試同工作、log spam HasJobOnThing 回 true 但 JobOnThing 回 null（判定不一致） 對齊兩者的條件，尤其 CanReserveAndReach
"could not reserve X" WorkGiver 沒守門 / PreToil 未真正 Reserve WorkGiver 用 CanReserveAndReach，PreToil 確實 pawn.Reserve
JobGiver 從不被選中 插入點優先序太低、或被上層 Conditional 擋掉 用 View thinkTree 看遍歷路徑；調整 insertPriority / 移除過嚴守衛
走到目標旁卻不工作 PathEndMode 選錯（如該用 InteractionCell 卻用 Touch） GotoThing 與 CanReach 都改成正確的 PathEndMode
工作做到一半莫名中斷 FailOn 條件過嚴 / target 被其他 designation 改動 檢查 FailOnForbidden、FailOnDespawnedOrNull 是否誤觸發
EndJobWith 與 ReadyForNextToil若要在 tickAction 內手動結束 Toil，設 readyForNextToil = true（進下一個 Toil）；若要結束整個 Job，呼叫 EndJobWith(JobCondition.Succeeded) 或 JobCondition.Incompletable。混淆兩者會導致「跳過剩餘 Toil」或「Job 不結束」。

除錯流程速查

  1. 小人不做工作？先開 View thinkTree，確認你的節點有沒有被遍歷到。
  2. 節點被遍歷但回 null？在 TryGiveJob / JobOnThing 加 Log，逐條印出篩選條件結果。
  3. 給了 Job 卻立刻中止？檢查 JobDriver 的 TryMakePreToilReservations 是否回 false，以及 FailOn 守衛。
  4. 開始做卻卡死？檢查每個自製 Toil 的 defaultCompleteMode。
  5. "could not reserve"？對齊 WorkGiver 守門與 PreToil 預約的 target / count。
