健康與醫療系統

Hediff 是 RimWorld 一切「身體狀態」的統一抽象——傷口、疾病、義體、藥效、成癮全是它。掌握 Hediff/HediffComp/手術，就掌握了角色身體的可編程介面。

Chapter 28 · Health & Medical
§1 健康模型總覽：從 Pawn 到 Capacity

RimWorld 把角色身體建模成一條清晰的物件鏈：每個 Verse.Pawn 持有一個 Verse.Pawn_HealthTracker（欄位 pawn.health），其中最核心的子物件是 Verse.HediffSet（pawn.health.hediffSet）。HediffSet 就是「這具身體目前發生的所有事情」的集合——它內部維護一個 List<Hediff>，而幾乎所有你能想到的身體狀態，從一道刀傷、一場瘟疫、一條義肢到一次酒精微醺，全部都是這個列表裡的一個 Verse.Hediff 物件。

理解這條鏈是本章一切操作的前提。你不會直接「設定角色的移動速度」，而是「在身體上掛一個會影響 Moving capacity 的 Hediff」。整個系統是宣告式的：你描述狀態，引擎推算後果。

Pawn
 └─ health : Pawn_HealthTracker          // 健康總管：管理 hediff、capacity、生死
     ├─ hediffSet : HediffSet          // 所有狀態的集合（真相層）
     │   ├─ hediffs : List<Hediff>     // 傷口 / 疾病 / 義體 / 藥效 …
     │   └─ pawn.RaceProps.body : BodyDef // 身體部位樹（HumanlikeBody 等）
     ├─ capacities : PawnCapacitiesHandler // 由 hediff 推算出的能力百分比快取
     ├─ summaryHealth : SummaryHealthHandler // 整體健康 0~1 的摘要
     └─ immunity : ImmunityHandler     // 疾病免疫進度（見 §5）

BodyDef 與 BodyPartRecord 樹

每個種族在 RaceProperties.body 指向一個 Verse.BodyDef（人類是 Human）。BodyDef 在載入時會被展開成一棵 Verse.BodyPartRecord 樹——這是實體化的部位實例，與只是定義的 BodyPartDef 不同。樹的根是 Torso（軀幹），往下掛 Neck → Head、LeftShoulder → LeftArm → LeftHand → 各手指 等。每個 Hediff 透過 hediff.Part 欄位綁到樹上的某個節點；Part == null 代表「全身性」狀態（如瘟疫、酒醉）。

BodyPartDef（定義）

純資料：部位的名稱、hitPoints 比例、frostbiteVulnerability、可裝載的 tags（如 ManipulationLimbCore）。同一個 def 可在樹上出現多次（左右手）。dnSpy 定位：Verse.BodyPartDef。

BodyPartRecord（實例）

樹上的實際節點，有 parent / parts（子節點）、coverageAbs（被命中機率權重）、height（Top/Middle/Bottom，影響命中分佈）。Hediff 綁定的是它，不是 def。

Capacity 是怎麼算出來的

RimWorld.PawnCapacityDef（如 Consciousness、Moving、Manipulation、Sight、BloodPumping）代表一種「身體能力」，數值是 0~1+ 的百分比。它不是儲存值，而是即時推算的：每個 capacity 有一個 RimWorld.PawnCapacityWorker，遍歷身體部位的健康狀況與所有 hediff 的 capMods，乘算出最終百分比。

  • 部位健康貢獻：例如 Moving 由雙腿的剩餘 efficiency 決定，砍掉一條腿就掉一半。
  • Hediff capMods：stage 上宣告的 PawnCapacityModifier 會直接加減或設下限（見 §2）。
  • 層級依賴：Consciousness 是「總開關」——它低落時幾乎所有其他 capacity 都會被連動壓低，因為清醒度是大腦運作的基礎。
查值的正確入口讀某個能力百分比請用 pawn.health.capacities.GetLevel(PawnCapacityDefOf.Moving)，而非自己加總 hediff。PawnCapacitiesHandler 內部有快取與 dirty 機制，掛/移除 hediff 時會自動失效重算，你只要負責改變 hediff。 
// 讀取角色目前的移動能力（0 = 完全不能動，1 = 正常，>1 = 義肢超人）
float moving = pawn.health.capacities.GetLevel(PawnCapacityDefOf.Moving);

// 整體健康摘要（血條那條的依據）
float summary = pawn.health.summaryHealth.SummaryHealthPercent; // 0~1

// 判斷是否還活著、有沒有倒下
if (pawn.health.capacities.GetLevel(PawnCapacityDefOf.Consciousness) < 0.3f)
    // 意識過低，pawn 會 downed
§2 HediffDef 完整 XML

Verse.HediffDef 是 hediff 的資料藍圖。它最重要的設計概念是 severity（嚴重度）與 stages（階段）的對應：hediff 的 severity 是一個連續浮點數，而 stages 是一串依 minSeverity 排序的離散區間；引擎依當前 severity 落在哪個區間，套用該 HediffStage 的所有效果。這讓「同一個病隨時間惡化」變成純資料描述。

HediffDef 頂層欄位速查

欄位型別說明
hediffClassType對應的 C# 類別，預設 Verse.HediffWithComps（支援 comps）；傷口用 Hediff_Injury、成癮用 Hediff_Addiction
isBadbooltrue 代表負面狀態（UI 紅字、會被治療優先處理）；增益藥效設 false
tendablebool能否被「治療（Tend）」；感染、出血等設 true
initialSeverityfloat首次加掛時的 severity，預設 0.5
maxSeverityfloatseverity 上限（comp 加 severity 不會超過此值）
lethalSeverityfloat達到此 severity 即觸發死亡（-1 表示永不致死）
scenarioCanAddbool能否在劇本編輯器中加入起始 pawn
defaultLabelColorColorhealth 面板中此 hediff 的文字顏色
compsList<HediffCompProperties>掛載的行為元件（見 §3），需 hediffClass 繼承自 HediffWithComps

HediffStage 欄位速查

欄位型別效果
minSeverityfloat進入此 stage 的 severity 門檻（升冪排列，引擎取符合的最後一個）
labelstring此階段顯示名稱（如「輕微」「嚴重」），UI 顯示為 主名稱 (label)
painOffset / painFactorfloat對全身 Pain 的加值 / 乘值（Pain 過高會 downed）
capModsList<PawnCapacityModifier>修改某 capacity：offset 加減、setMax 設上限、postFactor 乘算

becomeVisiblebool此階段才在 health 面板顯示（潛伏期疾病設 false）
lifeThreateningbool標記為危及生命，醫療 AI 會緊急處理
deathMtbDaysfloat進入此階段後的「平均致死天數」（機率性死亡，比 lethalSeverity 柔和）
vomitMtbDaysfloat嘔吐的平均間隔天數
capMods 三種寫法的差別offset 是加減（-0.5 扣半）；setMax 把該 capacity 上限壓到某值（0 等於直接讓該能力歸零，常用於癱瘓）；postFactor 在所有 offset 算完後乘上去。需要「砍腿後無論其他多好都只能爬」用 setMax；需要「拖慢一點」用 offset。

severityPerDay 為負且無 stage 門檻時的歸零行為HediffComp_SeverityPerDay 把 severity 降到 0 後，hediff 並不會自動消失，除非你另外掛 HediffCompProperties_Disappears，或 hediffClass 本身在 severity≤0 時自我移除。增益藥效最穩的做法是搭配 Disappears 用真實時間倒數，或在 §3 的自訂 comp 裡於 severity≤0 時呼叫 pawn.health.RemoveHediff(parent)。
§3 HediffComp 體系

HediffComp 是把行為掛到 hediff 上的元件系統，和 ThingComp 的設計如出一轍：每個 comp 由一對類別組成——Verse.HediffCompProperties（XML 資料）+ Verse.HediffComp（C# 邏輯）。XML 中 <comps> 每個 <li Class="..."> 指向一個 properties 類，引擎在建立 hediff 時依此實例化對應的 comp，並把 properties 指派到 comp.props。前提：hediffClass 必須繼承 HediffWithComps，否則 comps 完全被忽略。

常用內建 Comp

Comp（properties）作用關鍵欄位
HediffComp_SeverityPerDay每天線性增減 severityseverityPerDay
HediffComp_Immunizable疾病模型：severity 與 immunity 賽跑severityPerDaySick、immunityPerDaySick
HediffComp_TendDuration治療有效時長與治療後 severity 變化baseTendDurationHours、severityPerDayTended
HediffComp_Disappears真實時間倒數後自我移除disappearsAfterTicks
HediffComp_GetsPermanent傷口痊癒後可能轉為永久傷疤permanentLabel、becomePermanentChanceFactor

HediffComp 生命週期掛勾

覆寫這些虛擬方法就能介入 hediff 的一生。最常用的是 CompPostTick（每 tick，注意效能）與 CompExposeData（存讀檔）。

  • CompPostMake()：comp 建立後初始化（一次）。
  • CompPostTick(ref float severityAdjustment)：每 tick 呼叫；不要直接改 parent.Severity，而是把這 tick 想增減的量累加到 severityAdjustment，引擎統一結算（避免多 comp 互踩，見 §8）。
  • CompTended(float quality, float maxQuality, int batchPosition)：被治療時呼叫，quality 是 0~1 治療品質。
  • CompPostInjuryHeal(float amount)：關聯傷口痊癒一定量時。
  • CompPostMerged() / CompShouldRemove：合併同類 hediff、是否該移除。
  • CompExposeData()：用 Scribe_Values.Look 等存讀自訂欄位。
  • CompDebugString()：在 Dev mode 的 hediff inspect 顯示除錯字串。

實作：自訂 HediffComp（依環境溫度動態增減 severity）

下面這個 comp 模擬「鏽蝕感染在寒冷環境惡化較慢、炎熱環境惡化較快」，並在被治療時記錄品質、severity 歸零自我移除，所有自訂狀態都正確存檔。

using Verse;
using RimWorld;

// 1) Properties：對應 XML <li Class="MyMod.HediffCompProperties_TempReactive">
public class HediffCompProperties_TempReactive : HediffCompProperties
{
    public float baseSeverityPerDay = 0.1f;
    public float hotMultiplier      = 2f;   // 30°C 以上加速
    public float coldMultiplier     = 0.5f; // 0°C 以下減速

    public HediffCompProperties_TempReactive()
    {
        compClass = typeof(HediffComp_TempReactive);
    }
}

// 2) Comp：實際每 tick 邏輯
public class HediffComp_TempReactive : HediffComp
{
    private float lastTendQuality = -1f; // 需要存檔的自訂狀態

    // 強型別存取 props，省去到處轉型
    public HediffCompProperties_TempReactive Props
        => (HediffCompProperties_TempReactive)props;

    public override void CompPostTick(ref float severityAdjustment)
    {
        // 每天才需要算一次，這裡用每 200 tick 折算成「每天份額」
        if (!Pawn.IsHashIntervalTick(200)) return;

        float temp = Pawn.AmbientTemperature;
        float mult = 1f;
        if (temp > 30f) mult = Props.hotMultiplier;
        else if (temp < 0f) mult = Props.coldMultiplier;

        // 200 tick = 200/60000 天；把每日量折算到此 tick 區間
        float perTick = Props.baseSeverityPerDay * mult * (200f / 60000f);

        // 不直接改 Severity，累加到 out 參數讓引擎結算
        severityAdjustment += perTick;

        // severity 歸零 → 自我移除（增益/可痊癒類常見）
        if (parent.Severity + perTick <= 0f)
            Pawn.health.RemoveHediff(parent);
    }

    public override void CompTended(float quality, float maxQuality, int batchPosition = 0)
    {
        lastTendQuality = quality;
        // 高品質治療立刻壓低嚴重度
        parent.Severity -= quality * 0.1f;
    }

    public override void CompExposeData()
    {
        Scribe_Values.Look(ref lastTendQuality, "lastTendQuality", -1f);
    }

    public override string CompDebugString()
        => $"tempReactive: lastTendQuality={lastTendQuality:F2}";
}

對應 XML 掛載：

<comps>
  <li Class="MyMod.HediffCompProperties_TempReactive">
    <baseSeverityPerDay>0.1</baseSeverityPerDay>
    <hotMultiplier>2</hotMultiplier>
    <coldMultiplier>0.5</coldMultiplier>
  </li>
</comps>
CompPostTick 是熱路徑每個 hediff 的每個 comp 每 tick 都會被呼叫一次。殖民地有上百個 pawn、每人多個 hediff 時，這是巨大的呼叫量。務必用 Pawn.IsHashIntervalTick(N) 把昂貴計算節流到每 N tick，並用該區間折算每日量；切勿在 PostTick 裡做 LINQ、字串拼接或 def 查找。
§4 加掛 / 移除 Hediff 的 C# API

絕大多數 mod 不會手刻 new Hediff()，而是走工廠：Verse.HediffMaker.MakeHediff(HediffDef, Pawn, BodyPartRecord) 建立物件，再交給 pawn.health.AddHediff 掛上。AddHediff 會處理合併（同 def 同部位的可疊加 hediff）、觸發 PostAdd、刷新 capacity 快取等，是唯一正確入口。

加掛：全身 vs 指定部位

// (a) 加一個全身性 hediff（Part 留 null）
Hediff boost = HediffMaker.MakeHediff(MyDefOf.MyMod_NaniteBoost, pawn);
boost.Severity = 1f;
pawn.health.AddHediff(boost);

// (b) 對特定 BodyPartRecord 下藥——先從身體樹找出該部位
BodyPartRecord leftLeg = pawn.RaceProps.body
    .GetPartsWithDef(BodyPartDefOf.Leg)
    .FirstOrDefault();

if (leftLeg != null)
{
    Hediff infection = HediffMaker.MakeHediff(
        MyDefOf.MyMod_RustInfection, pawn, leftLeg);
    pawn.health.AddHediff(infection);
}

// (c) 1.5 的 AddHediff 也接受 (HediffDef, BodyPartRecord) 重載，
//     內部自動 MakeHediff，適合不需先設 severity 的情況
pawn.health.AddHediff(MyDefOf.MyMod_RustInfection, leftLeg);

查詢：是否已有某 hediff

// 取第一個指定 def 的 hediff（沒有則回 null）
Hediff existing = pawn.health.hediffSet
    .GetFirstHediffOfDef(MyDefOf.MyMod_NaniteBoost);

// 只想知道有沒有
bool hasIt = pawn.health.hediffSet
    .HasHediff(MyDefOf.MyMod_RustInfection);

// 取得泛型化的特定 Hediff 子類（如自訂 Hediff_MyDisease）
var disease = pawn.health.hediffSet
    .GetFirstHediffOfDef(MyDefOf.MyDisease) as Hediff_MyDisease;

// 「加一層或疊加」的慣用法：有就加 severity，沒有就建立
if (existing != null)
    existing.Severity += 0.5f;
else
    pawn.health.AddHediff(MyDefOf.MyMod_NaniteBoost);

移除

// 直接移除一個 hediff 物件
Hediff h = pawn.health.hediffSet.GetFirstHediffOfDef(MyDefOf.MyMod_NaniteBoost);
if (h != null)
    pawn.health.RemoveHediff(h);

// 較柔的做法：把 severity 降到 0 讓內建機制自然清理（若有 Disappears/自移除）

傷害如何變成 injury hediff

你很少自己掛 Hediff_Injury——它由戰鬥管線產生。流程是 Thing.TakeDamage(DamageInfo) → DamageDef.workerClass 對應的 Verse.DamageWorker（如 DamageWorker_AddInjury）→ 選定命中的 BodyPartRecord → 建立 Hediff_Injury 並 AddHediff。詳見 第 15 章 戰鬥傷害。若你只是想「直接讓某部位受傷」而不走整條戰鬥管線，可手動建構 DamageInfo 再呼叫 pawn.TakeDamage：

var dinfo = new DamageInfo(
    DamageDefOf.Cut,            // 傷害類型
    amount: 12f,
    armorPenetration: 0.2f,
    angle: -1f,
    instigator: attacker,
    hitPart: leftLeg);            // 指定命中部位（null 則隨機）
pawn.TakeDamage(dinfo);
§5 免疫與疾病進程

RimWorld 疾病的精髓是一場賽跑：疾病的 severity 一路爬向致死，而 pawn 的 immunity（免疫度）也一路爬向 1.0；誰先到終點誰贏。severity 先達 lethalSeverity（或 stage 的 deathMtbDays 命中）→ 死亡；immunity 先達 1.0 → severity 開始衰退、痊癒。這兩條曲線分別由 HediffComp_Immunizable 與 Verse.ImmunityHandler 驅動。

severity 線（病情）

由 HediffComp_Immunizable 的 severityPerDaySick 推升。當 immunity ≥ severity 時引擎改用 severityPerDayImmune（通常為負）讓病情衰退。

immunity 線（抵抗力）

由 ImmunityHandler 維護，每個疾病一筆 ImmunityRecord。生病期間用 immunityPerDaySick 累積；治療品質越高，累積越快。

tend quality 如何介入

治療（Tend）不直接「降低 severity」，而是同時：(1) 由 HediffComp_TendDuration 在一段時間內提供 severity 修正；(2) 提升免疫累積效率。治療品質 quality 受醫者 MedicalTendQuality stat、所用藥物的 MedicalQualityMax、床位的醫療加成共同決定。對致命疾病，「先用好藥好醫者把免疫拉到領先」往往比硬扛更關鍵。

// 程式化查詢某疾病的免疫進度（0~1）
float imm = pawn.health.immunity.GetImmunity(MyDefOf.MyDisease);

// 比較免疫與病情，判斷這場仗誰領先
Hediff dis = pawn.health.hediffSet.GetFirstHediffOfDef(MyDefOf.MyDisease);
if (dis != null)
{
    bool winning = imm >= dis.Severity;
    // winning == true → 免疫領先，病情即將反轉
}

拆解：瘟疫（Plague）

內建 HediffDef Plague 是理解此機制的最佳範本。其 HediffComp_Immunizable 設定大致為：severityPerDaySick 正值（病情自然惡化）、immunityPerDaySick 正值（生病就累積免疫）、severityPerDayImmune 負值（免疫超車後病情消退）。stages 分「未察覺 / 輕微 / 嚴重 / 極度（lifeThreatening + deathMtbDays）」。玩家體感的「能不能扛過瘟疫」本質上就是「免疫曲線能否在 severity 撞線前超車」——而藥物與醫者品質就是調這條曲線斜率的旋鈕。

欄位意義調高的後果
severityPerDaySick生病期間病情爬升速率更兇、更難扛
immunityPerDaySick生病期間免疫累積速率更容易自癒
severityPerDayImmune免疫領先後病情衰退速率痊癒更快
immunityPerDayNotSick潛伏期免疫累積（通常 0 或負）影響「染病前是否已有抗體」
免疫上限與獲得方式免疫最終由 ImmunityHandler 統一管理，且能透過「曾患病痊癒」保留部分抗體。若做新疾病，務必讓 immunity 與 severity 的斜率相對可控——immunity 太快疾病形同無害，太慢則必死，平衡點要靠實測（用 Dev mode 直接設 severity 觀察）。
§6 手術即配方（Surgery = RecipeDef）

RimWorld 沒有獨立的「手術系統」——手術就是一條掛在身體部位上的 RimWorld.RecipeDef。它和工作台配方共用同一套 RecipeDef 結構，差別只在：workerClass 是 Recipe_Surgery 的子類、targetsBodyParts 為 true、並透過 recipeUsers 或 ThingDef.recipes 綁到「會接受此手術的 pawn 種類」。手術成功與否、失敗後果，都由 RecipeDef 欄位與 worker 決定。

手術相關 RecipeDef 欄位

欄位說明
workerClassRecipe_Surgery 子類，如 Recipe_InstallArtificialBodyPart、Recipe_RemoveHediff、Recipe_AdministerIngestible
recipeUsers哪些 ThingDef（種族）能執行此手術，如 Human
appliedOnFixedBodyParts此手術作用於哪些 BodyPartDef（如 Leg）
addsHediff手術成功後在目標部位掛上的 hediff（義肢的 hediff）
surgerySuccessChanceFactor成功率乘數（1 = 標準，>1 較易成功）
deathOnFailedSurgeryChance手術失敗時直接死亡的機率
ingredients / fixedIngredientFilter所需材料（義體本體 + 藥物）
skillRequirements執行者最低技能（通常 Medicine）
anesthetize是否需要麻醉（多數侵入性手術為 true）

範例：安裝自訂義肢的完整 XML

三件套：(1) 義肢 hediff（裝上後給予 capMods）、(2) 義體 ThingDef（可被生產 / 交易的物品）、(3) 手術 RecipeDef（把物品變成身上的 hediff）。

<!-- (1) 義肢效果 hediff：裝在 Leg 上，移動力 +0.25 -->
<HediffDef>
  <defName>MyMod_HydraulicLeg</defName>
  <label>液壓義腿</label>
  <hediffClass>Hediff_Implant</hediffClass>
  <countsAsAddedPartOrImplant>true</countsAsAddedPartOrImplant>
  <isBad>false</isBad>
  <spawnThingOnRemoved>MyMod_HydraulicLegThing</spawnThingOnRemoved>
  <addedPartProps>
    <solid>true</solid>
    <partEfficiency>1.25</partEfficiency>
  </addedPartProps>
  <stages>
    <li>
      <capMods>
        <li>
          <capacity>Moving</capacity>
          <offset>0.25</offset>
        </li>
      </capMods>
    </li>
  </stages>
</HediffDef>

<!-- (2) 義體物品本身（可生產 / 安裝 / 移除後掉回） -->
<ThingDef ParentName="BodyPartArtificialBase">
  <defName>MyMod_HydraulicLegThing</defName>
  <label>液壓義腿</label>
  <techHediffsTags>
    <li>Advanced</li>
  </techHediffsTags>
  <statBases>
    <MarketValue>1200</MarketValue>
    <Mass>5</Mass>
  </statBases>
</ThingDef>

<!-- (3) 安裝手術 RecipeDef -->
<RecipeDef>
  <defName>MyMod_InstallHydraulicLeg</defName>
  <label>安裝液壓義腿</label>
  <workerClass>Recipe_InstallArtificialBodyPart</workerClass>
  <targetsBodyParts>true</targetsBodyParts>
  <anesthetize>true</anesthetize>
  <surgerySuccessChanceFactor>1.0</surgerySuccessChanceFactor>
  <deathOnFailedSurgeryChance>0.02</deathOnFailedSurgeryChance>
  <addsHediff>MyMod_HydraulicLeg</addsHediff>
  <skillRequirements>
    <Medicine>5</Medicine>
  </skillRequirements>
  <workAmount>3000</workAmount>
  <ingredients>
    <li>
      <filter>
        <thingDefs>
          <li>MyMod_HydraulicLegThing</li>
        </thingDefs>
      </filter>
      <count>1</count>
    </li>
    <li>
      <filter>
        <categories>
          <li>Medicine</li>
        </categories>
      </filter>
      <count>2</count>
    </li>
  </ingredients>
  <appliedOnFixedBodyParts>
    <li>Leg</li>
  </appliedOnFixedBodyParts>
  <recipeUsers>
    <li>Human</li>
  </recipeUsers>
</RecipeDef>
Recipe_InstallArtificialBodyPart 自動做了什麼它會在安裝前先移除目標部位上原有的傷病與舊義體（包含被它取代的整個子樹），再掛上 addsHediff。所以裝義腿會自動清掉那條腿上的所有傷口——你不必手動處理。若要做「不取代而是附加」的植入物（如義眼增益），請改用 Hediff_Implant + 對應 worker，並注意 countsAsAddedPartOrImplant 的語意。
recipeUsers vs ThingDef.recipes把手術掛到種族有兩種路徑：在 RecipeDef 寫 recipeUsers（推薦，集中管理），或反過來在種族 ThingDef 的 <recipes> 列出 recipe defName。兩者擇一即可，不要兩邊都寫否則重複。要替既有種族（如香草 Human）追加手術，用 recipeUsers 或 XML PatchOperation 較乾淨。
§7 義體與身體部位置換

義體系統建立在 §1 的身體樹之上。當一個部位「缺失」（被打掉、被手術移除），引擎在 hediffSet 為它掛一個 Hediff_MissingPart；裝上義體則是掛一個 countsAsAddedPartOrImplant=true 的 hediff（如 Hediff_AddedPart / Hediff_Implant），引擎據此把該部位視為「已被人工部位佔據」，並用 addedPartProps.partEfficiency 取代原生效率。

查詢身體部位狀態的 API

// 所有「還在、沒缺失」的部位
IEnumerable<BodyPartRecord> intact =
    pawn.health.hediffSet.GetNotMissingParts();

// 只要某種 tag（如可裝義眼的部位）
var sightParts = pawn.health.hediffSet
    .GetNotMissingParts(tag: BodyPartTagDefOf.SightSource);

// 某部位是否已被人工部位佔據
bool hasArtificial = pawn.health.hediffSet
    .HasDirectlyAddedPartFor(leftLeg);

// 某部位當前的綜合效率（含義體加成 / 傷病扣減）
float eff = pawn.health.hediffSet
    .GetPartHealth(leftLeg);

capMods 正向加成的「超人義肢」

讓義肢比原生更強有兩條互補管道：

  1. partEfficiency > 1（在 hediff 的 addedPartProps）：直接提升該部位本身的效率，連帶推升依賴此部位的 capacity。例如 partEfficiency 1.25 的義腿讓 Moving 提升。
  2. stage capMods offset：對特定 capacity 額外加值，可疊在 partEfficiency 之上做「精準加成」。

bodyPartTags 與採用者

BodyPartTagDef（如 ManipulationLimbCore、SightSource、BreathingSource、MovingLimbCore）是 capacity 計算「哪些部位算數」的索引。當你做全新部位或全新種族時，給部位掛上正確的 tag，capacity worker 才知道要把它納入運算。例如某 capacity 的 worker 會 GetNotMissingParts(tag: ...) 找出貢獻部位再加總效率。

spawnThingOnRemoved 的雙向性義體 hediff 上的 spawnThingOnRemoved 讓「手術移除義體時掉回物品」成立，使義體可回收再利用。而 ThingDef 的 techHediffsTags 則控制此物品能透過哪些手術安裝（標籤需與 recipe 對應）。兩者搭配才構成完整的「安裝 ⇄ 移除」循環。
§8 陷阱與除錯

同 tick 多 comp 競寫 severity

多個 comp 在 CompPostTick 直接寫 parent.Severity 會互相覆蓋、結果不確定。正確做法是全部寫進 ref severityAdjustment，由引擎在所有 comp 跑完後一次性套用，順序無關、可累加。

spawn 前 AddHediff 的順序問題

在 pawn 還沒 Spawned（如 PawnGenerator 中途）時掛 hediff，某些依賴 map / position 的 comp（如讀 AmbientTemperature）會拿到無效值或丟例外。把這類加掛延後到 pawn.SpawnSetup 之後，或在 comp 內先檢查 Pawn.Spawned。

capMods vs statOffsets 用錯層級

capMods 改的是 PawnCapacityDef（Moving/Manipulation…，會層層連動）；statOffsets 改的是 StatDef（WorkSpeedGlobal/MoveSpeed…，最終派生值）。想影響「能不能動」用 capMods；想直接調某個 stat 數字用 statOffsets。混用會得到非預期的疊乘。

用 Dev mode「Add Hediff」測

開發者模式 → 工具列「Add Hediff」可對任意 pawn 任意部位掛任意 hediff，並能即時拉 severity 觀察 stage 切換與 capacity 變化。這是驗證 §2 stage 門檻與 §5 免疫曲線最快的手段，免重啟。

capMods 三寫法的數學差異（速記）

寫法結算位置典型用途
offset加總階段輕度增減（拖慢、加快）
postFactoroffset 之後乘算整體縮放（疲勞讓所有動作打折）
setMax封頂硬上限（癱瘓 setMax=0，無論其他加成都歸零）

除錯檢查清單

  1. hediff 沒效果？先確認 hediffClass 是 HediffWithComps（或其子類），否則 <comps> 被無視。
  2. stage 沒切換？檢查 minSeverity 是否升冪排列；引擎取「符合條件的最後一個」stage。
  3. capMods 無感？確認當前 severity 真的落在含該 capMods 的 stage（用 Dev mode 拉 severity 驗證）。
  4. 義肢裝不上？檢查 recipe 的 appliedOnFixedBodyParts 與目標部位 BodyPartDef 是否一致，以及 recipeUsers 是否含該種族。
  5. 存檔後自訂狀態消失？comp 一定要覆寫 CompExposeData 並用 Scribe_* 存欄位。
  6. PostTick 造成卡頓？用 IsHashIntervalTick 節流，並把 def 查找移到 CompPostMake 快取。
不要在 CompPostTick 直接呼叫 RemoveHediff 而不收尾在遍歷 hediff 的 tick 迴圈中移除自己可能擾動列表。若要自移除，確認透過 Pawn.health.RemoveHediff(parent)（它內部安全處理），且該 tick 不再對 parent 做後續存取。更穩的替代是設 parent.Severity = 0 並依賴 HediffComp_Disappears 或 hediffClass 的自然清理。
務實心法能用 XML（HediffDef + stages + 內建 comp）表達的就別寫 C#；需要動態邏輯時優先寫 HediffComp 而非 Harmony patch——comp 與引擎生命週期整合、存檔自動、相容性最佳。Harmony 留給「改既有 hediff/手術行為」這類無法靠新增 def 達成的情境。
