1 · MapGeneratorDef 與 GenStep 流水線

RimWorld 的地圖生成是一條有序的流水線（pipeline）。 一張 Map 被建立後，MapGenerator.GenerateMap 會依序執行一連串 GenStep， 每個 GenStep 對地圖做一件事——鋪地形、放石塊、挖洞穴、撒礦脈、塞動植物、刷出敵人…… 最終疊加成一張完整地圖。哪些 GenStep、以什麼順序執行，由 MapGeneratorDef 決定。

MapGeneratorDef

定義一種「地圖類型」要跑哪些 GenStep（genSteps 是一串 GenStepDef 的引用）。

GenStepDef

包裝單一 GenStep 的 Def，攜帶 order（排序鍵）與 genStep（實際邏輯類別）。

GenStep

真正幹活的 C# 類別。覆寫 Generate(Map, GenStepParams) 改寫地圖內容。

核心呼叫流程

// Verse.MapGenerator.GenerateMap(...) 的概念骨架
public static Map GenerateMap(IntVec3 mapSize, MapParent parent,
    MapGeneratorDef mapGenerator, ...)
{
    Map map = new Map();
    // ... 初始化 map.info、components、grids ...

    // 取出所有 GenStepDef，依 order 由小到大排序（穩定排序）
    IEnumerable<GenStepWithParams> steps =
        mapGenerator.genSteps.Select(d => new GenStepWithParams(d, default))
                             .OrderBy(g => g.def.order);

    foreach (var gswp in steps)
    {
        // 每個 GenStep 都吃到同一張 map，依序往上疊加修改
        gswp.def.genStep.Generate(map, gswp.parms);
    }
    return map;
}
關鍵心智模型 GenStep 之間是共享同一張可變 Map 的副作用鏈。後面的 GenStep 看得到前面的成果 （例如「撒植物」需要先有「地形」與「肥沃度」）。因此 order 不是裝飾，而是正確性的一部分。

標準地圖（Base）的 MapGeneratorDef

遊戲核心定義在 Core/Defs/MapGeneratorDefs/MapGenerators.xml，玩家殖民地圖用 Base：

<MapGeneratorDef>
  <defName>Base</defName>
  <genSteps>
    <li>ElevationFertility</li>
    <li>Terrain</li>
    <li>Caves</li>
    <li>RockChunks</li>
    <li>ScatterRuinsSimple</li>
    <li>Plants</li>
    <li>AnimalsWildlife</li>
    <!-- ... 還有更多 ... -->
  </genSteps>
</MapGeneratorDef>
注意 <genSteps> 內的 <li> 是 GenStepDef 的 defName 引用， 不是 C# 類別名。實際執行順序仍以各 GenStepDef 的 order 為準，而非 XML 中的列出順序（見第 8 節）。
2 · 內建 GenStep 巡覽

理解內建 GenStep 是模仿與插隊的前提。下面是 Base 流水線上幾個關鍵的執行階段（依典型 order）：

order 100GenStep_ElevationFertility
產生高度/肥沃度噪聲圖
order 200GenStep_Terrain
依高度/肥沃度鋪地形
order 230GenStep_Caves
用噪聲挖洞穴空間
order 240GenStep_RockChunks
散落岩石碎塊
order 400GenStep_ScatterThings
通用物件散佈
order 500+GenStep_Plants / Animals
動植物刷出
GenStep 類別職責主要寫入的資料
GenStep_ElevationFertility 用 Perlin 噪聲產生「高度」與「肥沃度」兩張 float grid，作為後續地形/洞穴/礦脈的依據。 MapGenFloatGrid（Elevation / Fertility）
GenStep_Terrain 遍歷每格，依肥沃度、高度、海拔與河流/海岸資料決定 TerrainDef。高度超過岩石閾值的格子鋪上岩牆。 map.terrainGrid
GenStep_Caves 在岩石區域內用第二層噪聲挖出「洞穴」通道（自然形成可探索的地下空間）。 洞穴 grid（影響後續 RockChunks / 礦脈的可放置區）
GenStep_RockChunks 在開闊地面散落可採集的岩石碎塊（chunk filth/thing），增加地表變化。 Thing（Chunk）via GenSpawn
GenStep_ScatterThings 通用散佈基礎類別：依 thingDef、count、clusterSize、allowFoggedPositions 等參數隨機撒物件。礦脈、廢墟、鋼鐵塊都用它。 Thing via GenSpawn
查源碼的捷徑 用 dnSpy 反編譯 Assembly-CSharp.dll，搜尋 namespace Verse 或 RimWorld 下以 GenStep_ 開頭的類別。每個都很短，是最好的「官方範例」。對照 Core/Defs/GenStepDefs/ 的 XML 看參數綁定。
3 · 自訂 GenStep

自訂地圖生成的核心動作只有兩步：(a) 寫一個繼承 GenStep 的類別、覆寫 Generate； (b) 寫一個 GenStepDef 把它接進某條 MapGeneratorDef 流水線。

GenStep 基底簽章

public abstract class GenStep
{
    // 給隨機數種子用：同一張地圖的不同 GenStep 要回傳不同的穩定 seed，
    // 以確保「同一個世界 tile 重生地圖」時結果可重現。
    public abstract int SeedPart { get; }

    // 真正幹活的地方。map 是目標地圖，parms 攜帶生成上下文。
    public abstract void Generate(Map map, GenStepParams parms);
}

範例：撒出一片「特殊礦石」礦脈群

目標：在地圖開闊地面上隨機撒出數簇自訂礦物 MineralRich，避開已有建築與洞穴。

using System.Linq;
using RimWorld;
using Verse;

namespace MyMod
{
    public class GenStep_ScatterMineralRich : GenStep
    {
        // XML 可注入的參數（public 欄位 → Def 自動反序列化綁定）
        public ThingDef mineralDef;
        public IntRange clusterCount = new IntRange(2, 4);
        public IntRange clusterSize  = new IntRange(8, 16);

        // SeedPart 取一個唯一常數，避免和其他 GenStep 撞種子
        public override int SeedPart => 9482176;

        public override void Generate(Map map, GenStepParams parms)
        {
            if (mineralDef == null)
            {
                Log.Error("[MyMod] GenStep_ScatterMineralRich: mineralDef 未設定");
                return;
            }

            int clusters = clusterCount.RandomInRange;
            for (int c = 0; c < clusters; c++)
            {
                // 找一個合法的起點：在岩石牆相鄰處、未被霧遮、可站立
                if (!CellFinder.TryFindRandomCell(map, IsGoodSeedCell, out IntVec3 seed))
                    continue;

                GrowCluster(map, seed, clusterSize.RandomInRange);
            }
        }

        // 起點條件：必須位於岩石（mineable）格、且地圖內部
        private bool IsGoodSeedCell(IntVec3 cell, Map map)
        {
            if (!cell.InBounds(map)) return false;
            Building edifice = cell.GetEdifice(map);
            return edifice != null && edifice.def.building.isNaturalRock;
        }

        // 從種子格用洪水填充式（BFS）長出一簇礦脈
        private void GrowCluster(Map map, IntVec3 root, int size)
        {
            GenSpawn.Spawn(mineralDef, root, map); // 放下第一塊

            int placed = 1;
            map.floodFiller.FloodFill(
                root,
                // 可擴散條件：仍是天然岩石、尚未放滿
                passCheck: cell =>
                {
                    Building e = cell.GetEdifice(map);
                    return e != null && e.def.building.isNaturalRock;
                },
                processor: cell =>
                {
                    if (placed >= size) return true; // 回傳 true 終止填充
                    if (cell != root)
                    {
                        cell.GetEdifice(map)?.Destroy();        // 移除原岩牆
                        GenSpawn.Spawn(mineralDef, cell, map); // 換成礦脈
                        placed++;
                    }
                    return false;
                });
        }
    }
}

對應的 GenStepDef（XML）

<Defs>
  <GenStepDef>
    <defName>ScatterMineralRich</defName>
    <!-- order 決定它在流水線中的位置；見第 8 節 -->
    <order>410</order>
    <genStep Class="MyMod.GenStep_ScatterMineralRich">
      <!-- 這些 li/欄位對應 C# 類別的 public 欄位 -->
      <mineralDef>MineralRich</mineralDef>
      <clusterCount>2~4</clusterCount>
      <clusterSize>8~16</clusterSize>
    </genStep>
  </GenStepDef>
</Defs>
把 GenStep 接進流水線的兩種方式
  1. 新建 MapGeneratorDef：複製 Base 並在 <genSteps> 加入你的 defName——適合做全新地圖類型。
  2. Harmony / XML PatchOperation 注入既有流水線：用 PatchOperationAdd 把 <li>ScatterMineralRich</li> 加進現有 Base 的 <genSteps>——適合「在所有地圖都生效」。

用 XML PatchOperation 注入 Base 流水線

<Patch>
  <Operation Class="PatchOperationAdd">
    <xpath>/Defs/MapGeneratorDef[defName="Base"]/genSteps</xpath>
    <value>
      <li>ScatterMineralRich</li>
    </value>
  </Operation>
</Patch>
4 · TerrainDef 與地形

地形（terrain）是地圖的「地板層」——每格都有一個 TerrainDef， 決定能否種植、移動成本、能蓋什麼建築。GenStep_Terrain 在生成期就是把高度/肥沃度噪聲映射成這些 TerrainDef。

關鍵欄位

欄位型別意義
fertilityfloat肥沃度乘數。0 = 不可種植；草地約 1.0；肥沃土壤 > 1。影響植物生長與作物產量。
pathCostint移動成本加成（tick）。沙、沼澤較高；鋪過的地板較低。0 表無額外成本。
affordancesList<TerrainAffordanceDef>此地形「能承載什麼」。例如 Light / Medium / Heavy / GrowSoil / Bridgeable。建築的 terrainAffordanceNeeded 須被此清單滿足才能蓋。
generatedFilthThingDef踩過產生的污物類型（如泥土）。
tagsList<string>分類標記，供其他系統查詢（如 "Water"、"Road"）。
passabilityTraversability是否可通行（Standable / Impassable，如深水）。

自訂 TerrainDef（XML）

<TerrainDef>
  <defName>AshSoil</defName>
  <label>ash soil</label>
  <texturePath>Terrain/Surfaces/AshSoil</texturePath>
  <edgeType>FadeRough</edgeType>
  <renderPrecedence>210</renderPrecedence>
  <pathCost>2</pathCost>
  <fertility>0.6</fertility>
  <affordances>
    <li>Light</li>
    <li>Medium</li>
    <li>Heavy</li>
    <li>GrowSoil</li>
  </affordances>
  <tags>
    <li>Soil</li>
  </tags>
</TerrainDef>

在 GenStep 中設定地形

// 把某格地形換成自訂的 AshSoil
TerrainDef ash = DefDatabase<TerrainDef>.GetNamed("AshSoil");
map.terrainGrid.SetTerrain(cell, ash);

// 讀取目前地形（檢查肥沃度是否足以種植）
TerrainDef cur = map.terrainGrid.TerrainAt(cell);
if (cur.fertility > 0f) { /* 可種植 */ }
affordances 是最常見的「蓋不了東西」根因 若玩家在你的自訂地形上無法放置某建築，多半是該建築的 terrainAffordanceNeeded（如 Heavy）不在地形的 affordances 清單裡。 水體預設不含 Heavy，需鋪橋（Bridge，提供臨時 affordance）才能蓋重型建築。
5 · 程式化放置物件

在 GenStep 內放東西，圍繞三組工具：GenSpawn（怎麼放）、CellFinder（放哪裡）、ScatterThingsUtility 與相關散佈基底（批量放）。

GenSpawn

把單一 Thing/Blueprint 實例化到指定 cell。處理朝向、堆疊、覆蓋既有物件等細節。

CellFinder

各種「找一個合法格子」的演算法：隨機、依條件、靠近邊緣、可達性檢查等。

GenSpawn 常用多載

// 最常用：放一個 ThingDef 在 cell（自動 new + Spawn）
Thing t = GenSpawn.Spawn(ThingDefOf.Steel, cell, map);

// 放已建立的 Thing 實例（可先設定 stackCount 等）
Thing steel = ThingMaker.MakeThing(ThingDefOf.Steel);
steel.stackCount = 75;
GenSpawn.Spawn(steel, cell, map, Rot4.North);

// 帶 WipeMode：生成期常用 Vanish/FullRefund 避免和既有物件衝突
GenSpawn.Spawn(thing, cell, map, Rot4.North, WipeMode.VanishOrMoveAside);

CellFinder：找合法格子

// 全圖隨機（無條件）
IntVec3 r = CellFinder.RandomCell(map);

// 帶條件 + 嘗試次數上限（找不到回傳 false）
if (CellFinder.TryFindRandomCell(
        map,
        validator: c => c.Standable(map) && !c.Fogged(map),
        out IntVec3 cell))
{
    GenSpawn.Spawn(ThingDefOf.Steel, cell, map);
}

// 在某中心點附近找（半徑內、可達）
if (CellFinder.TryFindRandomCellNear(center, map, radius: 8,
        validator: c => c.Standable(map), out IntVec3 near))
{
    // ...
}

// 靠近地圖邊緣（用於商隊/敵人入場點）
IntVec3 edge = CellFinder.RandomEdgeCell(map);

批量散佈：GenStep_Scatterer 家族

與其自己寫迴圈，更常見是繼承 GenStep_Scatterer（GenStep_ScatterThings 的父類別）， 只覆寫「能不能放這裡」與「放什麼」。父類別已處理 count、clusterSize、最小間距、嘗試上限等樣板邏輯。 

public class GenStep_ScatterMyDebris : GenStep_Scatterer
{
    public ThingDef debrisDef;

    public override int SeedPart => 771143;

    // 父類別呼叫：這個 cell 能不能放？
    protected override bool CanScatterAt(IntVec3 c, Map map)
    {
        if (!base.CanScatterAt(c, map)) return false;
        if (!c.Standable(map)) return false;
        if (c.GetEdifice(map) != null) return false; // 別蓋到建築/岩牆
        return true;
    }

    // 父類別在合法 cell 上呼叫：實際放東西
    protected override void ScatterAt(IntVec3 c, Map map, GenStepParams parms, int count = 1)
    {
        GenSpawn.Spawn(debrisDef, c, map);
    }
}
<!-- 對應 GenStepDef：count / 群集大小由父類別欄位接收 -->
<GenStepDef>
  <defName>ScatterMyDebris</defName>
  <order>420</order>
  <genStep Class="MyMod.GenStep_ScatterMyDebris">
    <debrisDef>ChunkSlagSteel</debrisDef>
    <count>30~50</count>
    <clusterSize>3~6</clusterSize>
  </genStep>
</GenStepDef>
ScatterThingsUtility 若只是想在某些 cell 周邊「智慧地」放下一批 thing（自動處理堆疊上限、相鄰格分配）， 可直接呼叫 ScatterThingsUtility.ScatterThingsAround(...)，省去自己分配堆疊的麻煩—— GenStep_ScatterThings 內部就是用它把超過單格 stackLimit 的數量攤到鄰格。
6 · ElevationNoise / 洞穴 / 礦脈（噪聲生成原理）

RimWorld 的自然地貌幾乎都建立在連貫噪聲（coherent noise）之上—— Perlin/value 噪聲產生平滑、隨機卻有空間相關性的 float 值，作為「這裡是高地還是低地 / 肥沃還是貧瘠 / 是否該挖洞」的連續決策依據。

三張關鍵 grid 的生成邏輯

產物噪聲輸入映射規則（概念）
Elevation（高度） 低頻 Perlin（大尺度起伏） 值 > 岩石閾值 → 鋪天然岩牆（mineable）；其餘為開闊地面。高度也影響洞穴可挖區。
Fertility（肥沃度） 另一組 Perlin（與高度不同 seed/freq） 值高 → 富土/草地；值低 → 沙/裸岩。GenStep_Terrain 與植物 GenStep 都讀它。
Caves（洞穴） 在「岩石區」內再跑一層 Perlin / 嶺脊噪聲 噪聲落在特定區間 → 把該岩石格「挖空」成可通行洞穴通道。

礦脈的生成思路

原版礦脈（如鋼鐵/鈽礦）走的是 GenStep_PreciousLump / scatterer 路線： 在天然岩石格中選種子點，再以群集（lump）方式向相鄰岩石格擴張，把岩牆替換成礦物牆。 第 3 節的 GenStep_ScatterMineralRich 就是這個模式的簡化重製： 選岩石種子 → flood fill 擴張 → 替換 edifice。

為什麼用噪聲而不是純隨機？ 純 Rand 撒點會得到「鹽粒雜訊」般的均勻散佈，看起來不自然。 連貫噪聲讓相鄰格的值彼此相關，於是高地連成山、肥沃區連成草原、洞穴連成通道——這正是「地形感」的來源。
7 · RimWorld 的 ModuleBase / Perlin 噪聲使用

RimWorld 內嵌了 LibNoise 的移植版（命名空間 Verse.Noise）。 所有噪聲產生器都繼承抽象基底 ModuleBase，核心方法是 double GetValue(double x, double y, double z)。 你用「模組積木」組合出想要的噪聲場，再對每格取值。

常用噪聲模組

類別作用
Perlin標準 Perlin 噪聲。建構參數：frequency、lacunarity、persistence、octaves、seed、quality。
RidgedMultifractal嶺脊噪聲，產生山脊/洞穴通道狀的線性特徵。
ScaleBias對輸入模組的輸出做 value * scale + bias 線性變換（調幅度/平移）。
Multiply / Add把兩個噪聲模組的輸出相乘/相加（疊合特徵）。
Abs / Clamp取絕對值 / 夾限到區間。

範例：建立一張自訂高度噪聲並寫進 MapGenFloatGrid

using Verse;
using Verse.Noise;

public override void Generate(Map map, GenStepParams parms)
{
    // 1) 建立基礎 Perlin。frequency 越大 → 特徵越細碎
    ModuleBase noise = new Perlin(
        frequency:   0.015,
        lacunarity:  2.0,
        persistence: 0.5,
        octaves:     6,
        seed:        Rand.Range(0, int.MaxValue),
        quality:     QualityMode.High);

    // 2) 線性調整輸出範圍（Perlin 約落在 [-1, 1]，拉到 [0, 1]）
    noise = new ScaleBias(0.5, 0.5, noise);

    // 3) RimWorld 提供的工具：依世界 tile 種子讓噪聲在地圖內可重現
    NoiseDebugUI.StoreNoiseRender(noise, "MyElevation"); // 選用：Debug 視覺化

    // 4) 取得（或建立）一張 float grid，逐格寫入噪聲值
    MapGenFloatGrid elevation = MapGenerator.FloatGridNamed("Elevation", map);
    foreach (IntVec3 c in map.AllCells)
    {
        double v = noise.GetValue(c.x, c.y, c.z); // c.y 通常為 0（2D 地圖）
        elevation[c] = Mathf.Clamp01((float)v);
    }
}
MapGenFloatGrid 是 GenStep 間的「黑板」 MapGenerator.FloatGridNamed("Elevation"/"Fertility"/"Caves", map) 取得的是 跨 GenStep 共享、僅在生成期存在的暫存格網。你的 GenStep 可以讀前面寫的高度圖、 也可以覆寫它影響後面的 GenStep——這是和原版地形系統互動的正規管道。
NoiseDebugUI 視覺化 開發期用 NoiseDebugUI.StoreNoiseRender(module, "name") 把噪聲場存起來， 在遊戲內 Debug 工具（God mode → View settings / 地圖生成除錯）即可疊圖檢視， 快速調 frequency/octaves 而不必反覆重生地圖。
8 · GenStep order（GenStepDef order 欄位）

GenStepDef.order（float）是全域排序鍵。 MapGenerator 把該 MapGeneratorDef 引用到的所有 GenStepDef 蒐集起來， 依 order 由小到大穩定排序後依序執行。 <genSteps> 在 XML 裡的排列順序不直接決定執行順序——order 才是。

關鍵 order 區段（記住相對關係即可）

order 區間階段原則
~100噪聲基底（Elevation / Fertility）必須最先，後面全靠它
~200地形（Terrain）、河流、海岸讀噪聲 → 鋪地板與岩牆
~230–260洞穴、岩石碎塊、礦脈須在岩牆已存在之後
~400–450通用散佈（廢墟、特殊物件）地形/岩石已定型，可安全放物
~500+植物、動物、敵人、霧幾乎最後，依賴前面所有結果

插隊的決策

  • 要踩在岩石上（礦脈/洞穴內裝飾）：order 設在 Terrain(~200)、Caves(~230) 之後，例如 240~260。
  • 要在開闊地面撒東西：order 跟 ScatterThings 相近，400~450。
  • 要影響植物分佈：改 Fertility grid 的話 order 必須 < 植物 GenStep（~500），通常擺在地形之後、植物之前。
  • 要最後收尾（清理/校驗）：給很大的 order（如 9000），確保跑在所有原版 GenStep 之後。
order 設錯 = 看似沒生效 最典型的錯誤：自訂礦脈 order 設成 50（早於 Terrain），執行時岩牆還沒鋪好， GetEdifice 一律回 null，整個 GenStep 靜默地什麼都沒放。 症狀是「程式碼沒報錯、地圖上卻找不到東西」。
9 · 結構生成（SymbolResolver / KCSG 概念簡述）

撒散物件可以靠 scatterer，但要生成有結構的東西（古代遺跡、機械船骸、敵方據點、含房間/牆/門/陷阱的建築群） 就需要更高階的「藍圖→實體」系統。RimWorld 原版用 BaseGen + SymbolResolver，社群常用框架是 KCSG。

BaseGen / SymbolResolver（原版）

把「生成一座據點」拆成階層化的 符號（symbol）："settlement" → "edificeWalls" + "interior_threats" ... 每個符號對應一個 SymbolResolver 子類別，在指定矩形區（ResolveParams.rect）內解析出實際內容，並可遞迴 push 更細的符號。

KCSG（Custom Structure Generation）

社群函式庫（隨 Vanilla Expanded 普及）。用 XML 定義「結構藍圖」（StructureLayoutDef：以符號矩陣描述每格的 terrain/thing/wall），再由一個 SymbolResolver 把藍圖印到地圖上。讓設計師不寫 C# 就能畫出複雜建築。

SymbolResolver 概念骨架

using RimWorld.BaseGen;
using Verse;

public class SymbolResolver_MyBunker : SymbolResolver
{
    // 回傳此符號能否在給定區域解析（大小/地形檢查）
    public override bool CanResolve(ResolveParams rp)
    {
        return base.CanResolve(rp) && rp.rect.Width >= 7 && rp.rect.Height >= 7;
    }

    // 真正生成：在 rp.rect 內鋪牆、放門、push 子符號
    public override void Resolve(ResolveParams rp)
    {
        Map map = BaseGen.globalSettings.map;

        // 外牆：交給原版的 "edificeWalls" 符號
        ResolveParams walls = rp;
        walls.wallStuff = ThingDefOf.Steel;
        BaseGen.symbolStack.Push("edificeWalls", walls);

        // 在中央放一個自訂物件
        IntVec3 center = rp.rect.CenterCell;
        GenSpawn.Spawn(ThingDef.Named("MyReactor"), center, map);

        // 也可 push 更細的內部符號，遞迴解析
        // BaseGen.symbolStack.Push("interior_threats", rp);
    }
}

註冊符號 → 在 GenStep 觸發 BaseGen

// (a) 用 RuleDef 註冊符號（XML）：
// <RuleDef><defName>MyBunkerRule</defName> ... </RuleDef>
// 或在程式碼以 SymbolResolver 對應符號名。

// (b) 在自訂 GenStep 中啟動一輪 BaseGen：
public override void Generate(Map map, GenStepParams parms)
{
    CellRect rect = CellRect.CenteredOn(map.Center, 12, 12);

    ResolveParams rp = default;
    rp.rect = rect;
    BaseGen.globalSettings.map = map;
    BaseGen.symbolStack.Push("myBunker", rp); // 你的根符號
    BaseGen.Resolve(); // 開跑：依序彈出符號堆疊解析
}
該用哪個？
  • 程序化、規則導向（每次略有不同的房間布局）→ 自寫 SymbolResolver。
  • 固定/手繪藍圖（精確擺位的場景、預製建物）→ KCSG 的 StructureLayoutDef（XML 矩陣），開發效率高得多。
10 · 常見坑

坑 1 GenStep 執行順序錯置

最高頻的問題。GenStep 是副作用鏈，order 決定它看得到哪些前置成果。 在 Terrain 之前找岩石、在 Plants 之後改肥沃度，都會「程式正確但結果空白」。 

// 自檢：在 Generate 開頭印出當前已存在的前置產物
var elev = MapGenerator.FloatGridNamed("Elevation", map);
Log.Message($"[MyMod] order 執行中，Elevation 中心值={elev[map.Center]}");
// 若值為 0/未初始化 → 你跑得太早，調大 order

坑 2 未覆蓋全圖 / 邊界遺漏

遍歷地圖時務必用 map.AllCells 或正確的 CellRect，並對每格做 InBounds(map) 檢查。 手算索引、忘了邊界，會在地圖邊緣留下未處理的「破洞」，或拋 IndexOutOfRange。 

// ✅ 安全遍歷
foreach (IntVec3 c in map.AllCells) { /* ... */ }

// ✅ 鄰格存取前先檢查
foreach (IntVec3 adj in GenAdj.CardinalDirections)
{
    IntVec3 n = c + adj;
    if (!n.InBounds(map)) continue;
    // ...
}

坑 3 效能：每格做昂貴運算

地圖動輒 250×250 = 62,500 格，大地圖更多。 在逐格迴圈裡 DefDatabase.GetNamed、建立 LINQ、new 物件、查反射，會讓地圖生成卡頓數秒甚至更久。

  • 把 Def 查找、噪聲模組、FieldInfo 等提到迴圈外快取。
  • 避免在熱迴圈用 LINQ（Where/Select/ToList），改用 for + 條件。
  • 需要鄰域查詢時用 map.floodFiller / GenRadial 既有工具，別自己嵌套迴圈掃全圖。
// ❌ 每格都查 Def
foreach (IntVec3 c in map.AllCells)
    map.terrainGrid.SetTerrain(c, DefDatabase<TerrainDef>.GetNamed("AshSoil"));

// ✅ 查一次，重複用
TerrainDef ash = DefDatabase<TerrainDef>.GetNamed("AshSoil");
foreach (IntVec3 c in map.AllCells)
    map.terrainGrid.SetTerrain(c, ash);

坑 4 生成期用 WipeMode 不當導致物件互相清除

生成期物件密集，GenSpawn.Spawn 預設的 WipeMode 可能把你剛放的東西「移開或消滅」以讓位給新物件， 或反之。放岩牆/礦脈這類占據 edifice 槽的東西時，先 GetEdifice(map)?.Destroy() 清出空位再放，行為最可預期。

坑 5 SeedPart 重複 → 多個 GenStep 隨機結果相關

SeedPart 是隨機種子的一部分，用來保證「同一世界 tile 重生地圖」可重現。 若兩個 GenStep 回傳相同 SeedPart，它們的隨機序列會耦合，可能出現位置重疊或詭異關聯。 給每個自訂 GenStep 一個獨一無二的常數。

坑 6 public 欄位才能被 Def 注入

GenStep 上要從 XML 接收的參數必須是 public 欄位（或具 setter 的屬性）， 且 XML 標籤名須大小寫完全一致。打錯名稱或寫成 private，欄位會保持預設值，又是一個「沒報錯卻不生效」。

地圖生成除錯 Checklist
  1. 用 Dev Mode → Generate map 反覆重生，快速迭代。
  2. 在 Generate 開頭/結尾各印一行 Log，確認 GenStep 真的被呼叫、order 正確。
  3. 用 NoiseDebugUI 疊圖檢視噪聲場，調 frequency/octaves。
  4. 放完物件後印出實際 spawn 數量，對照預期。
  5. 大地圖（300+）下測效能，留意生成耗時。
