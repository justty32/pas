思想體系（Ideology DLC）

Ideology 把「殖民地信什麼」變成可編程的系統：Meme 組出 Ideo，Precept 規定善惡與心情，Ritual 把信仰變成可執行的典禮。

Chapter 32 · Ideology
§1 系統總覽與 DLC 守門

Ideology 在資料層的核心問題只有一句：「這個殖民地相信什麼，以及相信得有多堅定？」 把它拆成 C# 工程師熟悉的語彙，整個 DLC 是一條 「模板 → 組裝 → 執行期 instance」 的管線。和 Biotech（第 21 章）的 GeneDef→XenotypeDef→Gene 三層結構幾乎是同構的，理解一個就能類比另一個。

MemeDef

信念基石（XML Def）。對應 RimWorld.MemeDef : Def。像 trait 集合：規定一個 Ideo 能/必須帶哪些 precept、用哪套敘事文法。

PreceptDef

具體戒律的「模板」（RimWorld.PreceptDef : Def）。掛在某個 IssueDef 下，與同 issue 的其它 precept 互斥。

Ideo

執行期物件（RimWorld.Ideo），由玩家/生成器把多個 meme + 一串 Precept instance 組裝而成。存活在 Find.IdeoManager。

pawn.Ideo

每個 pawn 透過 Pawn_IdeoTracker（pawn.ideo）持有一個 Ideo 與一個 certainty（0–1 的信仰堅定度）。

注意 Def 是模板、Ideo/Precept 是 instance 的分野：你在 XML 寫的是 MemeDef / PreceptDef，但玩家存檔裡裝的是動態生成的 Ideo 物件、其中每個戒律是一個 RimWorld.Precept（或子類，如 Precept_Role / Precept_Ritual）。同一個 PreceptDef 可被多個 Ideo 實體化，數值與綁定的角色都記在 instance 上。

DLC 守門：絕對不能省的兩件事

Ideology 是付費 DLC。玩家若沒裝，你引用的 MemeDef / IssueDef 等型別根本不會被載入， 硬引用會在 def 載入階段直接拋例外、整包 mod 掛掉。守門分兩個層級：

① XML 層：MayRequire

對任何「只在裝了 Ideology 時才有意義」的節點掛 MayRequire="Ludeon.RimWorld.Ideology"。沒裝 DLC 時該節點被靜默略過，整個 def 仍能載入。

② C# 層：ModLister 檢查

所有觸碰 Find.IdeoManager / pawn.Ideo 的程式碼路徑，先過 ModLister.IdeologyInstalled 短路返回，否則 NRE。

// 任何 Harmony patch / GameComponent 進入 Ideology 邏輯前的標準守門
using Verse;
using RimWorld;

public static class IdeoGate
{
    public static bool TryGetIdeo(Pawn pawn, out Ideo ideo)
    {
        ideo = null;
        // 1) 沒裝 DLC → 直接退出，連 pawn.Ideo 都別碰
        if (!ModLister.IdeologyInstalled) return false;
        // 2) 動物 / 機械 等沒有信仰追蹤器的 pawn，ideo 會是 null
        ideo = pawn?.Ideo;
        return ideo != null;
    }
}
最常見的崩潰來源 不要用 ModsConfig.IdeologyActive 以外的方式硬判斷，也不要假設 pawn.Ideo 必不為 null。即使裝了 DLC，動物、機械人、以及 IdeoManager 尚未生成 Ideo 的早期載入階段，pawn.Ideo 都可能是 null。
反編譯定位（dnSpy / Assembly-CSharp.dll）角色
RimWorld.MemeDef信念基石模板
RimWorld.PreceptDef / RimWorld.IssueDef戒律模板與其分類軸
RimWorld.Ideo / RimWorld.IdeoManager執行期信仰物件與其管理器
RimWorld.Precept / Precept_Role / Precept_Ritual執行期戒律 instance
RimWorld.Pawn_IdeoTracker掛在 pawn 上的 pawn.ideo
RimWorld.IdeoUtility常用靜態輔助方法
§2 MemeDef：信念基石

MemeDef 決定一個 Ideo 的「人格骨架」。組裝 Ideo 時規則是： 必選恰好一個 Structure meme（結構基石），再加數個 Normal meme。 Structure 例如 Supremacist（至上主義）、Collectivist（集體主義）， 它們界定整個信仰的世界觀；Normal meme 例如 Cannibal（食人）、Nudism（裸體主義） 則是疊加的傾向。

欄位型別用途
categoryMemeCategoryStructure 或 Normal；Structure 每個 Ideo 只能有一個
impactint對 Ideo 多樣性的「重量」，影響隨機生成器的選用機率
requireOneList<List<PreceptDef>>選了此 meme，每個內層清單至少要採用一個 precept
preceptsToAddList<PreceptDef>採用此 meme 時自動加入的 precept
exclusionTagsList<string>同 tag 的 meme 互斥（避免「食人」與「動物連結·素食」共存）
generalRulesRulePackDef敘事文法：產生信仰名稱、教義段落的詞彙規則
renderOrderPriorityintUI 排序權重

範例：自訂一個「工匠崇拜」Structure meme

我們做一個強調「親手鍛造才神聖」的結構基石：它強制信仰必須在「製造」這個 issue 上採用某些 precept， 並自動加入一個崇拜工作台的 precept。 

<!-- Defs/MemeDefs/Memes_Artisan.xml -->
<Defs>

  <MemeDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_Artisanal</defName>
    <label>工匠主義</label>
    <description>真正的價值出自雙手。機器量產是對技藝之神的褻瀆。</description>
    <iconPath>UI/Memes/MyMod_Artisanal</iconPath>

    <!-- Structure = 世界觀骨架，一個 Ideo 只能挑一個 -->
    <category>Structure</category>
    <impact>3</impact>

    <!-- 採用此 meme 時自動帶上的 precept -->
    <preceptsToAdd>
      <li>MyMod_Smithy_Veneration</li>
    </preceptsToAdd>

    <!-- 在「成衣 / 製造」issue 上，至少要從這組挑一個 precept -->
    <requireOne>
      <li>
        <li>MyMod_Production_Handmade_Required</li>
        <li>MyMod_Production_Handmade_Preferred</li>
      </li>
    </requireOne>

    <!-- 與「工業化崇拜」這類 meme 互斥 -->
    <exclusionTags>
      <li>ProductionPhilosophy</li>
    </exclusionTags>

    <!-- 敘事文法：信仰自動命名與教義文案會用到 -->
    <generalRules>
      <rulesStrings>
        <li>memeConcept->技藝</li>
        <li>memeConcept->鍛造</li>
        <li>ideoName->[memeConcept]之道</li>
      </rulesStrings>
    </generalRules>
  </MemeDef>

</Defs>
requireOne 是「約束求解」，不是「自動加入」 preceptsToAdd 會無條件塞進去；requireOne 只是告訴生成器「這幾組各挑一個」。 若玩家在自訂 Ideo 介面手動移除了 requireOne 涵蓋的 precept，遊戲會跳警告但仍允許—— 所以別把 requireOne 當成硬性保證，真正要「一定存在」的請用 preceptsToAdd。
impact 不是強度 impact 只影響隨機 Ideo 生成器挑到此 meme 的傾向與「衝擊預算」消耗，不影響任何遊戲內數值。 想做平衡請改 precept 的心情值，而非調 impact。
§3 PreceptDef 與 Precept：戒律

Precept 是 Ideology 的真正肉身——它把抽象信念換算成具體的善惡判定與心情。 每個 PreceptDef 都掛在一個 IssueDef（議題軸）下， 同一個 issue 下的 precept 互斥，一個 Ideo 在每個 issue 上最多採用一個立場。

IssueDef

議題軸（RimWorld.IssueDef）。例如 Drugs、Cannibalism、Apparel_Mood、WeaponPreference。

PreceptDef

該軸上的一個立場。Cannibalism 軸上有 Cannibalism_Acceptable、Cannibalism_Horrible 等互斥選項。

ThoughtDef 銜接

信徒做了符合/違反戒律的事 → 觸發 comps 內定義的心情（接第 20 章 needs/mood/thoughts）。

PreceptDef 重要欄位速查

欄位用途
issue所屬議題軸（IssueDef）；同 issue 互斥
impact戒律「分量」（None/Low/Medium/High），影響生成偏好與 UI
associatedMemes傾向哪些 meme 採用此 precept（生成器加權）
requiredMemes硬性：沒這些 meme 不能採用此 precept
conflictingMemes有這些 meme 時不可採用
comps戒律行為的核心：PreceptComp_* 清單，掛心情/自我表達等效果
preceptInstanceDef把抽象 precept 綁到某個具體目標（如「崇拜某 ThingDef」）
ritualPatternBase若此 precept 是 ritual，指向 RitualPatternDef（見 §5）
preceptClass執行期 instance 類別，預設 RimWorld.Precept，role/ritual 用子類

PreceptComp：把戒律接到心情系統

戒律的效果幾乎都靠 comps 表達，最常見的是 PreceptComp_SituationalThought（情境心情，例如「身處有某設施的房間」）與 PreceptComp_UnaccompaniedThought / 事件驅動的 thought。 這些 comp 內引用的 ThoughtDef 必須是 Ideology 專屬的 Thought_Situational / Thought_Memory 子型別並設定 requiredTraits / nullifyingTraits 或 requiredMemes， 以確保只對信徒生效（見 §8 陷阱）。

範例 A：崇拜某個 ThingDef（preceptInstanceDef）

我們要做「崇拜深度鑽井」——信徒在能看見深度鑽井的房間裡會獲得正面心情。 這牽涉一個 IssueDef（崇拜對象軸）+ 一個 PreceptDef，以及一個情境 ThoughtDef。 

<!-- Defs/PreceptDefs/Precepts_Smithy.xml -->
<Defs>

  <!-- 議題軸：對「神聖工作台」的態度 -->
  <IssueDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_SacredWorkbench</defName>
    <label>神聖工作台</label>
    <iconPath>UI/Issues/MyMod_SacredWorkbench</iconPath>
  </IssueDef>

  <!-- 立場：崇拜鐵砧 -->
  <PreceptDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_Smithy_Veneration</defName>
    <issue>MyMod_SacredWorkbench</issue>
    <label>崇拜鍛造台</label>
    <description>鍛造台是技藝之神顯化之處，信徒見之心安。</description>
    <impact>High</impact>
    <associatedMemes>
      <li>MyMod_Artisanal</li>
    </associatedMemes>

    <!-- 情境心情：身處有鍛造台的房間 → +心情 -->
    <comps>
      <li Class="PreceptComp_SituationalThought">
        <thought>MyMod_NearSacredSmithy</thought>
      </li>
    </comps>
  </PreceptDef>

</Defs>
<!-- Defs/ThoughtDefs/Thoughts_Smithy.xml -->
<Defs>
  <ThoughtDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_NearSacredSmithy</defName>
    <!-- 情境思想，需在房間內偵測到指定建築 -->
    <thoughtClass>Thought_Situational</thoughtClass>
    <workerClass>ThoughtWorker_PreceptNearBuilding</workerClass>
    <buildingDef>FueledSmithy</buildingDef>
    <stages>
      <li>
        <label>身處神聖之鍛造</label>
        <baseMoodEffect>4</baseMoodEffect>
      </li>
    </stages>
  </ThoughtDef>
</Defs>
ThoughtWorker 是真實 API ThoughtWorker_PreceptNearBuilding（namespace RimWorld）就是原版「崇拜建築」戒律的 worker， 它讀 thought.buildingDef 並掃描房間。若你的「崇拜對象」不是建築（例如崇拜某動物或地形）， 就需要自己寫 ThoughtWorker_Precept 子類；別硬塞不相容的 worker。

範例 B：禁用某類武器（WeaponPreference 風格）

原版有「武器偏好」這條軸，常見作法是覆寫/新增 precept，搭配 PreceptComp_Apparel 之外的「裝備不悅」心情。下面示範一個「禁用遠程武器」的立場， 讓信徒攜帶遠程武器時掉心情： 

<!-- Defs/PreceptDefs/Precepts_WeaponPurity.xml -->
<Defs>
  <PreceptDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_Weapons_MeleeOnly</defName>
    <issue>WeaponPreference</issue>  <!-- 原版 IssueDef -->
    <label>遠程武器：可憎</label>
    <description>懦夫才從遠處殺人。真正的信徒近身搏鬥。</description>
    <impact>Medium</impact>
    <comps>
      <!-- 攜帶遠程武器 → 心情懲罰；此 worker 由 thought 內判斷武器類別 -->
      <li Class="PreceptComp_SituationalThought">
        <thought>MyMod_CarryingRangedWeapon</thought>
      </li>
    </comps>
  </PreceptDef>
</Defs>
// 自訂 ThoughtWorker：偵測信徒手持遠程武器
// 放在 namespace MyMod，繼承 RimWorld.ThoughtWorker_Precept
using RimWorld;
using Verse;

public class ThoughtWorker_PreceptCarryingRanged : ThoughtWorker_Precept
{
    protected override ThoughtState ShouldHaveThought(Pawn p)
    {
        if (!ModLister.IdeologyInstalled) return false;
        ThingWithComps weapon = p.equipment?.Primary;
        if (weapon == null) return false;
        // def.IsRangedWeapon 是 Verse.ThingDef 上的便利屬性
        return weapon.def.IsRangedWeapon
            ? ThoughtState.ActiveAtStage(0)
            : ThoughtState.Inactive;
    }
}
ThoughtWorker_Precept 自帶信徒過濾 繼承 RimWorld.ThoughtWorker_Precept 的好處是：基底已會檢查「該 pawn 的 Ideo 是否真的採用了此 precept」， 你只要管「行為條件」。若你直接繼承 Verse.ThoughtWorker，就必須自己加 p.Ideo.HasPrecept(...) 判斷，否則會對所有 pawn 生效（典型 §8 陷阱）。
§4 Role：信仰角色

角色是一種特殊的 PreceptDef——preceptClass 設為 RimWorld.Precept_Role（其下又分 Precept_RoleSingle 領袖型、 Precept_RoleMulti 多人型）。一個 Ideo 可以有領袖（Leader）、道德導師（Moral Guide） 等指派給特定 pawn 的職位，被指派者會獲得 服裝要求、能力、以及一組 role-only 的心情/權能。

Role 的關鍵欄位

欄位用途
preceptClassPrecept_RoleSingle（唯一，如領袖）或 Precept_RoleMulti（可多人，如專家）
roleEffectsList<RoleEffect>：賦予能力、stat 加成、特殊權能（見下）
roleApparelRequirements擔任此角色時的服裝硬性需求（接第 17 章服裝層）
roleRequiredWorkTags / roleDisabledWorkTags指派資格 / 上任後的工作限制
leaderRole布林，標記此角色為「信仰領袖」（影響某些典禮資格）
grantedAbilities (via roleEffect)透過 RoleEffect_AbilityAvailable 賦予 AbilityDef（接第 24 章）

範例：定義「大鐵匠」領袖角色並賦予能力

<!-- Defs/PreceptDefs/Precepts_RoleSmith.xml -->
<Defs>
  <PreceptDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_Role_HighSmith</defName>
    <issue>Role_Leader</issue>          <!-- 領袖議題軸 -->
    <label>大鐵匠</label>
    <preceptClass>RimWorld.Precept_RoleSingle</preceptClass>
    <leaderRole>true</leaderRole>
    <impact>High</impact>
    <associatedMemes>
      <li>MyMod_Artisanal</li>
    </associatedMemes>

    <!-- 服裝硬性需求：必須戴特定頭飾才能維持角色 -->
    <roleApparelRequirements>
      <li>
        <requirement Class="ApparelRequirement">
          <requiredDefs>
            <li>Apparel_SmithCrown</li>
          </requiredDefs>
        </requirement>
      </li>
    </roleApparelRequirements>

    <!-- 角色效果：賦予一個 AbilityDef + 提供 stat 加成 -->
    <roleEffects>
      <li Class="RoleEffect_AbilityAvailable">
        <abilityDef>MyMod_BlessTheForge</abilityDef>
      </li>
      <li Class="RoleEffect_PawnStatFactor">
        <statDef>GeneralLaborSpeed</statDef>
        <modifier>1.15</modifier>
      </li>
    </roleEffects>
  </PreceptDef>
</Defs>

C# 端：指派與查詢角色

// 取得某 Ideo 的領袖角色，並把 pawn 指派上去
using System.Linq;
using RimWorld;
using Verse;

public static void AssignAsLeader(Pawn pawn)
{
    if (!ModLister.IdeologyInstalled || pawn?.Ideo == null) return;
    Ideo ideo = pawn.Ideo;

    // RolesListForReading 列出此 Ideo 全部角色 precept
    Precept_Role leaderRole = ideo.RolesListForReading
        .FirstOrDefault(r => r.def.leaderRole);
    if (leaderRole == null) return;

    // Assign 會自動把原任者卸任（單人角色）
    leaderRole.Assign(pawn, addThoughts: true);
    Messages.Message($"{pawn.LabelShort} 成為 {leaderRole.LabelCap}",
        MessageTypeDefOf.PositiveEvent);
}

// 查詢某 pawn 目前擔任的角色（沒擔任回 null）
public static Precept_Role CurrentRoleOf(Pawn pawn)
{
    if (!ModLister.IdeologyInstalled || pawn?.Ideo == null) return null;
    // IdeoUtility / Ideo.GetRole 都能用，後者最直接
    return pawn.Ideo.GetRole(pawn);
}
失格（disqualification）會自動發生 若 pawn 死亡、改信、或脫掉 roleApparelRequirements 要求的服裝超過容忍時間，遊戲會自動把角色卸任， 你不必手動監控。但若你要在「卸任瞬間」做事（例如收回自製道具），可 Harmony patch Precept_Role.Unassign 或監聽 pawn 的相關狀態。
§5 Ritual：典禮

典禮是 Ideology 最複雜也最有展示性的系統。它把「信仰」變成一場由多個 pawn 參與、 有階段、有成果品質的群體活動，執行期由 RimWorld.LordJob_Ritual 驅動——這正是第 30 章 Lord/GroupAI 的延伸應用。

典禮的五個 Def 拼圖

Def 型別角色
PreceptDef（preceptClass=Precept_Ritual）典禮在信仰中的「存在」，玩家在 UI 看到的條目
RitualPatternDef把 precept、behavior、outcome 綁在一起的「藍圖」
RitualBehaviorDef典禮怎麼進行：階段（RitualStage）、角色（RitualRole）、worker
RitualOutcomeEffectDef典禮結束時的成果：品質（quality）→ 結果（outcome）對應表
DutyDef / 既有RitualStage 內各 RitualRole 的 duty（站位、跳舞、講道…）

資料流：從點擊到落幕

  1. 玩家在 Ideo 面板或祭壇 gizmo 觸發典禮 → 建立一個 LordJob_Ritual（Lord = 群體 AI 大腦）。
  2. Lord 依 RitualBehaviorDef 的 stages 推進；每個 RitualStage 給各 RitualRole 指派 duty。
  3. 參與者按 duty 行動（移動到 cell、跳舞、觀禮），由 RitualBehaviorWorker（C#）監控進度。
  4. 結束時 RitualOutcomeEffectDef 依累積的 quality 抽出一個 outcome（成功/普通/失敗），套用心情/獎勵。

範例：一個簡單的「鍛造慶典」聚會典禮

下面是把骨架拼齊的最小可行範例。先是 behavior（怎麼進行）：

<!-- Defs/RitualBehaviorDefs/Behavior_ForgeFeast.xml -->
<Defs>
  <RitualBehaviorDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_ForgeFeastBehavior</defName>
    <!-- worker 可用原版聚會型 worker，或自寫子類 -->
    <workerClass>RitualBehaviorWorker_Default</workerClass>
    <durationTicks>5000</durationTicks>

    <!-- 典禮角色：召集人（領袖）+ 一般參與者 -->
    <roles>
      <li Class="RitualRoleTag">
        <label>主祭</label>
        <id>leader</id>
        <tag>Moralist</tag>
        <required>true</required>
      </li>
      <li Class="RitualRoleColonist">
        <label>觀禮者</label>
        <id>participant</id>
        <maxCount>10</maxCount>
      </li>
    </roles>

    <!-- 階段：主祭講道 → 眾人慶祝 -->
    <stages>
      <li>
        <defaultDuty>Spectate</defaultDuty>
        <roleBehaviors>
          <li>
            <roleId>leader</roleId>
            <dutyDef>DeliverSermon</dutyDef>
          </li>
        </roleBehaviors>
      </li>
    </stages>
  </RitualBehaviorDef>
</Defs>

再是 outcome（品質如何換算結果）：

<!-- Defs/RitualOutcomeEffectDefs/Outcome_ForgeFeast.xml -->
<Defs>
  <RitualOutcomeEffectDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_ForgeFeastOutcome</defName>
    <workerClass>RitualOutcomeEffectWorker_FromQuality</workerClass>
    <comps>
      <!-- quality 來源：參與人數、主祭社交、是否在祭壇旁 -->
      <li Class="RitualOutcomeComp_ParticipantCount">
        <label>參與人數</label>
        <curve>
          <points>
            <li>(1, 0.1)</li>
            <li>(5, 0.3)</li>
            <li>(10, 0.5)</li>
          </points>
        </curve>
      </li>
    </comps>
    <!-- quality → outcome 區間：0~0.4 失望，0.4~0.8 普通，0.8+ 鼓舞 -->
    <outcomeChances>
      <li>
        <chance>0.5</chance>
        <positivityIndex>2</positivityIndex>
        <memory>MyMod_ForgeFeast_Inspired</memory>
        <label>鼓舞人心</label>
      </li>
    </outcomeChances>
  </RitualOutcomeEffectDef>
</Defs>

最後用 RitualPatternDef + PreceptDef 把三者綁起來：

<!-- Defs/RitualPatternDefs/Pattern_ForgeFeast.xml -->
<Defs>
  <RitualPatternDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_ForgeFeastPattern</defName>
    <ritualBehavior>MyMod_ForgeFeastBehavior</ritualBehavior>
    <ritualObligationTargetFilter Class="RitualObligationTargetFilter_ThingDef">
      <thingDefs>
        <li>Ideo_Altar</li>  <!-- 在祭壇旁舉行 -->
      </thingDefs>
    </ritualObligationTargetFilter>
    <ritualOutcomeEffect>MyMod_ForgeFeastOutcome</ritualOutcomeEffect>
  </RitualPatternDef>

  <PreceptDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_Ritual_ForgeFeast</defName>
    <issue>Ritual</issue>
    <label>鍛造慶典</label>
    <preceptClass>RimWorld.Precept_Ritual</preceptClass>
    <ritualPatternBase>MyMod_ForgeFeastPattern</ritualPatternBase>
    <associatedMemes>
      <li>MyMod_Artisanal</li>
    </associatedMemes>
  </PreceptDef>
</Defs>
能用原版 worker 就別自寫 RitualBehaviorWorker_Default 與 RitualOutcomeEffectWorker_FromQuality 已能滿足絕大多數「聚會 + 講道 + 品質換心情」的需求。只有要插入自訂階段邏輯 （例如典禮中段必須擊殺某目標）時才繼承 RimWorld.RitualBehaviorWorker 覆寫 Tick / ExpectedDuration。
典禮就是 Lord，除錯方法相通 典禮卡住、pawn 不就位，通常是 duty 指派或 LordToil 的問題。打開 Dev mode 的 Draw Duties 與 View settings → Lord，能直接看到每個 pawn 當前 duty。 這與第 30 章除錯 Lord 的手法完全一致。
§6 Style 與表現

Ideology 引入了「同一個物件、不同文化外觀」的概念。一張椅子，崇尚石化文明的信仰看到的是 雕花石椅，工業信仰看到的是金屬折疊椅——靠的是 StyleCategoryDef 與 ThingStyleDef（接第 8 章 graphics）。

StyleCategoryDef

一個「風格家族」（如 Spikecore 尖刺風、Animalist 動物風）。 內含 thingDefStyles 把 ThingDef 對映到 ThingStyleDef，並可帶 fixedIdeoColor / iconsForRandomization。

ThingStyleDef

單一物件的替代圖形（graphicData）。同物不同外觀，本質是換貼圖路徑與 drawSize/color，渲染管線仍走 Verse.Graphic。

<!-- Defs/StyleCategoryDefs/Style_ForgeCult.xml -->
<Defs>
  <!-- 單物件風格：把鋼椅換成「鐵砧風格」貼圖 -->
  <ThingStyleDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_Chair_Forge</defName>
    <graphicData>
      <texPath>Things/Building/Furniture/MyMod_ForgeChair</texPath>
      <graphicClass>Graphic_Single</graphicClass>
    </graphicData>
  </ThingStyleDef>

  <!-- 風格家族：把多個 ThingDef 綁進此 style -->
  <StyleCategoryDef MayRequire="Ludeon.RimWorld.Ideology">
    <defName>MyMod_ForgeStyle</defName>
    <label>鍛造風</label>
    <fixedIdeoColor>(0.55, 0.30, 0.10)</fixedIdeoColor>
    <thingDefStyles>
      <li>
        <thingDef>DiningChair</thingDef>
        <styleDef>MyMod_Chair_Forge</styleDef>
      </li>
    </thingDefStyles>
  </StyleCategoryDef>
</Defs>

生成 Ideo 時，meme 可透過 styleItemTags / 關聯把這些 StyleCategoryDef 拉進來。 執行期，styleDominance（風格支配度）決定殖民地建築被自動套用此風格的傾向； C# 中可由 ideo.style（RimWorld.IdeoStyle... 相關）查詢， 但多數情況下你不需要手動操作——擺好風格家具讓信徒自己提升 dominance 即可。

ideoColor 與貼圖染色 fixedIdeoColor 會被套到信仰旗幟、某些 UI 與可染色物件上。若你的 ThingStyleDef 貼圖本身已上好色，請確認對應 ThingDef 的 colorGenerator / mask 設定不會把它再染一次， 否則會出現顏色疊加錯亂（詳見第 8 章貼圖 mask 章節）。
§7 C# 互動 API

實務上你 80% 的程式碼只會用到下面這幾個入口。重點是「先判斷 Ideo，再做事」： 心情、能力、互動，都要先確認對象的信仰確實採用了對應 precept。

API用途
pawn.Ideo取得 pawn 信仰（可能 null）；底層是 Pawn_IdeoTracker
ideo.HasPrecept(PreceptDef)此 Ideo 是否採用某戒律
ideo.GetPrecept(PreceptDef)取得對應的 Precept instance
ideo.GetRole(Pawn)該 pawn 在此 Ideo 中擔任的角色（無則 null）
ideo.RolesListForReading此 Ideo 全部角色 precept（唯讀）
ideo.memes此 Ideo 的 meme 清單（List<MemeDef>）
ideo.PreceptsListForReading全部 precept instance（唯讀）
ideo.certaintyForReading / tracker.Certaintypawn 的信仰堅定度（0–1）
Find.IdeoManager.IdeosListForReading遊戲中所有 Ideo
IdeoUtility.*常用靜態輔助（如 IdeoUtility.IdeoChangeToWeights 等）

範例：只對「採用了某 precept 的信徒」加心情

using RimWorld;
using Verse;

public static class ForgeRites
{
    // 在 def 內以靜態欄位快取，避免每次 DefDatabase 查找
    private static PreceptDef SmithyVeneration =>
        DefDatabase<PreceptDef>.GetNamedSilentFail("MyMod_Smithy_Veneration");

    // 某事件後，給「真正崇拜鍛造台」的信徒一段記憶心情
    public static void GrantForgeBlessing(Pawn pawn)
    {
        if (!ModLister.IdeologyInstalled) return;
        Ideo ideo = pawn?.Ideo;
        if (ideo == null) return;

        // 關鍵：先確認此信仰採用了該 precept，否則跨信仰套用會出錯
        if (!ideo.HasPrecept(SmithyVeneration)) return;

        Thought_Memory mem = (Thought_Memory)ThoughtMaker.MakeThought(
            DefDatabase<ThoughtDef>.GetNamed("MyMod_ForgeBlessed"));
        pawn.needs?.mood?.thoughts?.memories?.TryGainMemory(mem);
    }
}

用 DefModExtension 掛額外資料

如果你想在 MemeDef / PreceptDef 上掛自己的設定（原版欄位沒有）， 標準作法是寫一個 DefModExtension 子類，避免去 patch 原版 Def 類別。 

// C# 端：定義擴充資料
using Verse;

public class ForgeMemeExtension : DefModExtension
{
    public float forgeSpeedBonus = 0.1f;
    public bool banAutomation = false;
}

// 讀取：
var ext = memeDef.GetModExtension<ForgeMemeExtension>();
if (ext != null && ext.banAutomation) { /* … */ }
<!-- XML 端：掛到 MemeDef 上 -->
<MemeDef MayRequire="Ludeon.RimWorld.Ideology">
  <defName>MyMod_Artisanal</defName>
  <modExtensions>
    <li Class="MyMod.ForgeMemeExtension">
      <forgeSpeedBonus>0.2</forgeSpeedBonus>
      <banAutomation>true</banAutomation>
    </li>
  </modExtensions>
</MemeDef>
查找 precept 的兩種風格 事件回呼裡用 DefDatabase<PreceptDef>.GetNamedSilentFail（找不到回 null，不崩）； Ideo 上的查詢一律走 ideo.HasPrecept / ideo.GetPrecept。 不要把 PreceptDef 和執行期 Precept instance 搞混——前者是模板，後者才帶綁定資料。
§8 常見陷阱與最佳實務

① 未裝 DLC 直接崩潰

忘了 MayRequire="Ludeon.RimWorld.Ideology"，或 C# 沒過 ModLister.IdeologyInstalled → def 載入或執行期 NRE。這是頭號殺手。

② precept issue 互斥載不進

同一個 IssueDef 下若你忘了它本就互斥，或把該 precept 設成多個 issue， 生成器會拒絕或玩家無法同時選用，呈現「戒律憑空消失」。

③ ritual outcome 曲線設錯

quality curve 的端點沒覆蓋 0~1 全域，或 outcomeChances 機率沒對齊， 導致典禮永遠抽到失敗、或品質爆表卻無正回饋。

④ role 服裝與第 17 章衝突

roleApparelRequirements 與其他服裝層（apparel layer）或別的角色需求重疊， pawn 無法同時滿足 → 反覆穿脫、卸任、心情崩。

⑤ 跨 ideo 心情套錯對象

給心情前沒判斷對象是否真採用該 precept，結果對「不信此教」的 pawn 也加減心情， 甚至對動物加心情（動物 Ideo 為 null）。

⑥ Def 名稱衝突 / 載入順序

覆寫原版 IssueDef / PreceptDef 時 defName 撞名，或 PatchOperation 在 Ideology 未啟用時報錯 （PatchOperation 也要包 MayRequire 條件）。

逐項拆解

① MayRequire 的散布要徹底

不只是頂層 <MemeDef> 要掛——若你在原版 Def 的清單裡插入一個指向 Ideology 型別的 <li>（例如往某 FactionDef 加一個只有 Ideology 才存在的內容）， 該 <li> 本身也要 MayRequire，否則沒裝 DLC 時整個父 def 連帶報錯。 

<!-- 正確：list 內個別 li 也掛 MayRequire -->
<someList>
  <li>VanillaThing</li>
  <li MayRequire="Ludeon.RimWorld.Ideology">MyMod_Ideology_Only</li>
</someList>
PatchOperation 也會在沒裝 DLC 時報錯 若你用 PatchOperation 去改 Ideology 的 Def，沒裝 DLC 時那個 XPath 找不到目標節點， PatchOperationReplace 等會在紅字 log 報「found 0 nodes」。把 patch 包進 <Operation Class="PatchOperationFindMod">（mods 設 Ludeon.RimWorld.Ideology） 的 match 內，僅在 DLC 啟用時套用。

⑤ 跨信仰心情：永遠先 HasPrecept

這是最隱蔽的邏輯錯誤，因為它不會崩潰、只會「行為錯」。標準防呆樣板：

// 反例（錯）：對所有 pawn 都加心情
pawn.needs.mood.thoughts.memories.TryGainMemory(myThought);

// 正例（對）：三道門 — DLC、有 Ideo、且真採用該 precept
if (ModLister.IdeologyInstalled
    && pawn.Ideo != null
    && pawn.Ideo.HasPrecept(myPreceptDef))
{
    pawn.needs?.mood?.thoughts?.memories?.TryGainMemory(myThought);
}
最佳實務總結
  • 能用 ThoughtWorker_Precept / 原版 ritual worker 就別自寫，它們自帶信徒過濾。
  • 所有 Ideology 相關 XML（含 list 內 li 與 PatchOperation）一律考慮 MayRequire。
  • C# 進 Ideology 路徑的第一行永遠是 ModLister.IdeologyInstalled。
  • 分清 Def（模板）與 instance（Ideo/Precept/Precept_Role）——查詢用 HasPrecept，別拿 Def 當 instance。
  • 自訂額外欄位用 DefModExtension，不要 Harmony patch 原版 Def 類別結構。
  • 典禮就是 Lord（第 30 章），卡關用 Dev mode 的 Draw Duties 除錯。
