皇室與榮譽（Royalty DLC）

Royalty 引入帝國這個特殊派系、可晉升的爵位、隨爵位解鎖的特權與要求，以及超能（psycast）的榮譽來源。本章拆解 RoyalTitleDef 與許可體系。

Chapter 33 · Royalty & Honor
§1 DLC 守門與帝國派系（Empire）

Royalty 的所有內容都掛在 Ludeon.RimWorld.Royalty 這個 DLC package 底下。 你的 mod 若想「在有 Royalty 時加料、沒有時也能正常載入」，必須做兩層守門： XML 層用 MayRequire（防止缺 Def 時整份 Def 報紅、被丟棄）， C# 層用 ModLister.RoyaltyInstalled（防止存取不存在的型別 / 欄位導致 NullReferenceException 或硬崩潰）。

Empire 派系

一個特殊的 FactionDef（defName Empire），玩家無法宣戰殲滅，固定友好/中立，且是唯一能授爵的派系。串接第 31 章。

Royal Favor（榮譽）

一種「對帝國的貨幣」，存在每個 pawn 身上（pawn.royalty.GetFavor(faction)），用來換爵位、換許可點數，可由任務 / 交付物資 / 禮物賺取。

RoyalTitleDef

階梯式爵位定義，從 Freeholder 到 High Stellarch。爵位帶來特權，也帶來房間 / 服裝 / 期望的義務。

XML 守門：MayRequire

當你的 PawnKind、Quest、Scenario 等 Def 內引用了 Royalty 專屬的 Def（如某個 RoyalTitleDef）， 在引用節點加上 MayRequire="Ludeon.RimWorld.Royalty"。若 DLC 不存在，Verse.DirectXmlLoader / DirectXmlCrossRefLoader 會靜默略過該節點，而不是整份 Def 解析失敗。 

<?xml version="1.0" encoding="utf-8"?>
<Defs>
  <PawnKindDef>
    <defName>MyMod_Noble</defName>
    <label>流亡貴族</label>
    <race>Human</race>
    <!-- 只有裝了 Royalty 才會帶這個爵位欄位；缺 DLC 時整段被略過，pawn 仍正常生成 -->
    <titleRequired MayRequire="Ludeon.RimWorld.Royalty">Knight</titleRequired>
  </PawnKindDef>
</Defs>
MayRequire 的兩種寫法 單一 DLC 用 MayRequire="Ludeon.RimWorld.Royalty"；需要「同時裝多個」用 MayRequire="A,B"（逗號＝AND）； 「任一裝即可」用 MayRequireAnyOf="A,B"。packageId 大小寫不敏感，但建議照官方原樣寫。

C# 守門：ModLister.RoyaltyInstalled

程式碼路徑（patch、comp、ITab、gizmo）一律先檢查 Verse.ModLister.RoyaltyInstalled。 這是一個布林快取，內部讀 ModsConfig.IsActive(...)，呼叫成本極低，可放在熱路徑。 

using RimWorld;
using Verse;

public static class RoyaltyGate
{
    // 在任何碰 royalty 型別/欄位的程式前先擋一次
    public static bool CanUseRoyalty(Pawn pawn)
    {
        if (!ModLister.RoyaltyInstalled) return false;
        // pawn.royalty 在缺 DLC 時可能為 null（Pawn_RoyaltyTracker 不一定被建立）
        return pawn?.royalty != null;
    }
}
絕對不要把 Royalty 型別寫進「總是執行」的靜態建構式或 StaticConstructorOnStartup 只要 JIT 觸碰到那個型別的中繼資料，缺 DLC 時就會 TypeLoadException。把 royalty 邏輯隔離到獨立類別、由守門方法呼叫， 讓 CLR 在守門失敗時根本不去載入那個型別（見第 7 章的「型別隔離」模式）。
§2 RoyalTitleDef：階梯式爵位

RimWorld.RoyalTitleDef 描述「一個爵位」。同一個帝國的所有爵位透過 seniority（資歷數值）排成一條鏈， RoyalTitlePermitDef 與晉升流程都以 seniority 比大小。原版帝國的爵位鏈大致如下：

defNameseniorityfavorCost（累積）典型解鎖
Freeholder10020無特殊要求，入門爵位
Yeoman20080第一級 psylink、簡單臥室要求
Acolyte300200psylink、臥室升級
Knight400400禁止部分粗活（disabledWorkTags）、獨立臥室
Praetor500700更高服裝/房間要求、援軍許可
Baron6001200需要 throne room（王座室）、貴族服裝
Count … High Stellarch700+更高套房、宴會廳、隨從、最高 psylink
favorCost 是「該級的價格」，不是累積總和 RoyalTitleDef.favorCost 指的是「從前一級升到本級需要的 favor」。上表「累積」欄是我整理的近似總成本， 實際以遊戲內 RoyalTitleUtility 計算為準。自訂爵位時，favorCost 要和它在 seniority 鏈上的位置匹配，否則晉升曲線會斷裂。

爵位的雙面性：特權 vs 義務

RoyalTitleDef 同時掛著「給你的好處」與「對你的要求」。這是設計爵位 mod 時最該先想清楚的張力：

特權（解鎖 / 加成）

  • permits / grantedAbilities：許可與超能
  • 晉升 psylink 等級（透過 psylinkLevel 配合 reward）
  • 可穿戴帝國高級服裝而不被當作不合身分
  • 呼叫帝國援軍 / 砲擊（見 §4）

義務（要求 / 懲罰）

  • minExpectation：拉高期望底線，房間 / 環境不達標 → 持續扣心情
  • requiredApparel / apparelRequirements：必須穿指定服裝
  • bedroomRequirements / throneRoomRequirements：房間角色與印象品質門檻
  • disabledWorkTags：高爵位拒絕做粗活（清潔、搬運等）

核心欄位速查

欄位型別意義
seniorityint爵位在鏈上的排序鍵，越大越高
favorCostint從前一級升到本級所需 royal favor
minExpectationExpectationDef期望底線，影響「期望」需求 → 心情
requiredApparelList<ApparelRequirement>強制服裝（依身體部位 / bodyType）
maxPsylinkLevelint此爵位允許的最高 psylink 等級（原版隨爵位提升）
bedroomRequirementsList<RoomRequirement>對 Bedroom 角色房間的要求
throneRoomRequirementsList<RoomRequirement>對 ThroneRoom 角色房間的要求
disabledWorkTagsWorkTags（flags）此爵位以上拒絕的工作標籤
permitsList<RoyalTitlePermitDef>解鎖的戰術許可（見 §4）
grantedAbilitiesList<AbilityDef>直接授予的能力 / 超能
changeHeirQuestPointsint更換繼承人任務的威脅點數（Awardable 爵位必填）
tagsList<string>授爵獎勵分流用標籤（如 Combatant、Sustaining）

自訂一個新爵位

下例在原版帝國的 Knight 與 Praetor 之間插入一個自訂爵位 MyMod_Vanguard（先鋒）。 重點：seniority 取 450（落在 Knight 400 與 Praetor 500 之間），favorCost 取兩者間的值，並掛上一個許可。 

<?xml version="1.0" encoding="utf-8"?>
<Defs>
  <RoyalTitleDef MayRequire="Ludeon.RimWorld.Royalty">
    <defName>MyMod_Vanguard</defName>
    <label>先鋒</label>
    <seniority>450</seniority>
    <favorCost>120</favorCost>   <!-- Knight→Vanguard 的價格 -->
    <changeHeirQuestPoints>800</changeHeirQuestPoints>  <!-- 換繼承人任務威脅點數；Awardable 爵位必填 -->

    <!-- 期望底線：拉到 High（房間/環境須更體面）-->
    <minExpectation>High</minExpectation>

    <!-- 拒絕粗活：清潔、搬運、植栽 -->
    <disabledWorkTags>Cleaning, Hauling, PlantWork</disabledWorkTags>

    <!-- 強制服裝：軀幹須穿帝國制服系列其中之一 -->
    <requiredApparel>
      <li>
        <bodyPartGroupsMatchAny>
          <li>Torso</li>
        </bodyPartGroupsMatchAny>
        <requiredTags>
          <li>RoyalTier3</li>  <!-- 對應 apparel 的 tradeTags/tags -->
        </requiredTags>
      </li>
    </requiredApparel>

    <!-- 臥室要求：印象品質 + 必備家具 -->
    <bedroomRequirements>
      <li Class="RoomRequirement_Impressiveness">
        <impressiveness>60</impressiveness>
      </li>
      <li Class="RoomRequirement_ThingCount">
        <thingDef>RoyalBed</thingDef>
        <count>1</count>
      </li>
    </bedroomRequirements>

    <!-- 解鎖一個自訂許可（見 §4）-->
    <permits>
      <li>MyMod_Permit_RallyCall</li>
    </permits>

    <!-- 直接授予的超能（可選）-->
    <grantedAbilities>
      <li>Skip</li>
    </grantedAbilities>
  </RoyalTitleDef>
</Defs>
怎麼把新爵位接進帝國的鏈？ 原版的爵位鏈不是寫死在 RoyalTitleDef，而是由 FactionDef.royalTitleTags + RoyalTitlePermitDef 的 faction 過濾共同決定哪些 RoyalTitleDef 屬於 Empire。 最穩做法：用 XML PatchOperation（第 2 章）把你的新爵位需要的標籤補進去，或直接 patch FactionDef 的 RoyalImpliedDef 生成設定， 避免「定義了爵位卻無法在帝國授爵 UI 看到」。
§3 取得與管理爵位（Pawn_RoyaltyTracker）

執行期的爵位狀態存在 RimWorld.Pawn_RoyaltyTracker，由 pawn.royalty 取得。 它是「per-pawn、per-faction」的——同一個 pawn 理論上能同時對不同派系持有爵位（原版只有帝國一個來源）。

常用 API

成員說明
GetCurrentTitle(Faction)取得對某派系的當前 RoyalTitleDef（無則 null）
HasAnyTitleIn(Faction)是否對該派系持有任何爵位
AllTitlesForReading所有 RoyalTitle 物件（含 faction、def、receivedFavor）唯讀清單
SetTitle(Faction, RoyalTitleDef, bool grantRewards, ...)直接設定爵位；grantRewards 決定是否一併給超能 / 許可
GetFavor(Faction) / SetFavor(...)讀 / 寫對該派系的 royal favor
GainFavor(Faction, int)增加 favor，並可能觸發「可晉升」提示
TryUpdateTitle(Faction, bool sendLetter)依目前 favor 自動把爵位升到該升的等級
GetPermitPoints(Faction)該派系下的許可點數（見 §4）
GetTitleAwardedWhenUpdating(Faction, int)給定 favor 數時，這條鏈會升到哪一級爵位

範例：賦予爵位並結算後果

using RimWorld;
using Verse;

public static class TitleOps
{
    // 直接把某 pawn 提升到指定帝國爵位（例如劇情獎勵）
    public static void GrantTitle(Pawn pawn, RoyalTitleDef title)
    {
        if (!ModLister.RoyaltyInstalled || pawn?.royalty == null) return;

        Faction empire = Faction.OfEmpire;          // 內建快捷取得帝國派系
        if (empire == null) return;

        // grantRewards: true → 一併授予 grantedAbilities / permits 與 psylink
        pawn.royalty.SetTitle(empire, title, grantRewards: true,
                              rewardsOnlyForNewestTitle: false, sendLetter: true);

        // 設定/補足對應的 favor，避免 UI 顯示成「欠費晉升」
        int needed = title.favorCost;
        if (pawn.royalty.GetFavor(empire) < needed)
            pawn.royalty.SetFavor(empire, needed);

        Messages.Message($"{pawn.LabelShort} 已晉升為 {title.GetLabelCapFor(pawn)}",
                         pawn, MessageTypeDefOf.PositiveEvent);
    }

    // 用 favor 推動「能升就升」，交給遊戲決定升到哪一級
    public static void PayFavorAndPromote(Pawn pawn, int amount)
    {
        if (!ModLister.RoyaltyInstalled || pawn?.royalty == null) return;
        Faction empire = Faction.OfEmpire;
        pawn.royalty.GainFavor(empire, amount);
        pawn.royalty.TryUpdateTitle(empire, sendLetter: true);
    }
}

爵位 → 期望 → 心情的因果鏈

minExpectation 把該 pawn 的「期望」需求底線往上抬（RimWorld.ExpectationDef，由 ExpectationsUtility.CurrentExpectationFor(pawn) 取兩者較大值決定）。 期望越高，房間 / 環境 / 食物 / 娛樂達不到時的負面想法越強，這條鏈最終落在第 20 章的 thought 系統。

  • 房間不達標：產生 ThoughtDef RoyalTitleTooLowForBedroom / ...ForThroneroom 系列負面想法。
  • 未穿指定服裝：產生「未穿著符合身分的服裝」想法（RoyalNoApparel 類）。
  • 期望落差：期望需求本身轉成 Need_Mood 上的持續扣分。
授爵不是「純加分」 你的 mod 若大方亂發高爵位，玩家會收到一個又貴又愛抱怨的 pawn：要蓋王座室、要穿貴族裝、不肯掃地。 設計獎勵時務必把「義務成本」算進去——這正是 Royalty 的核心玩法張力。
§4 RoyalTitlePermitDef：戰術許可

「許可（Permit）」是爵位附帶的主動戰術能力：呼叫帝國砲擊、空投物資、召喚援軍小隊等。 與超能不同，許可不耗 psyfocus，而是消耗「許可點數（permit points）」並走自己的冷卻。 定義型別是 RimWorld.RoyalTitlePermitDef，實際行為由 RimWorld.RoyalTitlePermitWorker 子類執行。

RoyalTitlePermitDef 核心欄位

欄位型別說明
workerClassType執行行為的 RoyalTitlePermitWorker 子類
permitPointCostint解鎖此許可需要花的許可點數
minTitleRoyalTitleDef需要的最低爵位
cooldownDaysfloat使用後的冷卻（天）
factionFactionDef提供此援助的派系（通常 Empire）
royalAidRoyalAidDef-like援助內容（援軍 pawnKinds、數量、favor 花費）
prerequisiteRoyalTitlePermitDef前置許可（形成許可樹）
permit points 從哪來？ 每升一級爵位會給固定的許可點數（透過 Pawn_RoyaltyTracker.GetPermitPoints(faction) 累計）， 玩家在「許可」分頁花點數解鎖樹狀許可。點數有限，所以許可之間是「取捨」而非「全拿」。

內建許可 Worker 一覽（常見可繼承 / 參考）

Worker效果

RoyalTitlePermitWorker_OrbitalStrike呼叫軌道砲擊（轟炸 / 燃燒指定區域）

對應的 Worker。重點是覆寫 GetRoyalAidOptions（產生援助選單項、進入點選目標模式）與目標選定後的 OrderForceTarget 執行。注意 faction 不在基底——基底只存 caller / map / free，所以本類要自己留一個 faction 欄位並在啟動 targeting 時存下來（原版 RoyalTitlePermitWorker_OrbitalStrike 就是這個寫法）。

using System;
using System.Collections.Generic;
using RimWorld;
using Verse;
using UnityEngine;

namespace MyMod
{
    public class RoyalTitlePermitWorker_RallyCall : RoyalTitlePermitWorker_Targeted
    {
        private const float Radius = 9.9f;

        // 本類自己持有 faction：基底只存 caller / map / free（皆 protected），faction 不在基底
        private Faction faction;

        // 援助選單入口：玩家在爵位援助選單選此項 → 進入點選目標模式
        public override IEnumerable<FloatMenuOption> GetRoyalAidOptions(Map map, Pawn pawn, Faction faction)
        {
            if (AidDisabled_NewTemp(map, pawn, faction, out string reason, temperatureMatters: false))
            {
                yield return new FloatMenuOption(def.LabelCap + ": " + reason, null);
                yield break;
            }
            string desc = def.LabelCap + ": ";
            Action action = null;
            if (FillAidOption(pawn, faction, ref desc, out bool free))
                action = () => BeginTargeting(pawn, faction, map, free);
            yield return new FloatMenuOption(desc, action, faction.def.FactionIcon, faction.Color);
        }

        // 存好基底欄位 + 自家 faction，啟動 Targeter（玩家點地後回呼 OrderForceTarget）
        private void BeginTargeting(Pawn caller, Faction faction, Map map, bool free)
        {
            targetingParameters = new TargetingParameters { canTargetLocations = true };
            base.caller = caller;
            base.map    = map;
            base.free   = free;
            this.faction = faction;
            Find.Targeter.BeginTargeting(this);
        }

        // 玩家點地後回呼，target 是落點（caller / map 此時已由 BeginTargeting 設好）
        public override void OrderForceTarget(LocalTargetInfo target)
        {
            foreach (IntVec3 cell in GenRadial.RadialCellsAround(target.Cell, Radius, useCenter: true))
            {
                if (!cell.InBounds(map)) continue;
                foreach (Thing t in cell.GetThingList(map))
                {
                    if (t is Pawn p && !p.Dead &&
                        p.Faction == caller.Faction && p.RaceProps.Humanlike)
                    {
                        Hediff h = HediffMaker.MakeHediff(
                            DefDatabase<HediffDef>.GetNamed("MyMod_RallyBuff"), p);
                        p.health.AddHediff(h);
                    }
                }
            }

            FleckMaker.Static(target.Cell, map, FleckDefOf.PsycastSkipFlashEntry, 2f);

            // 啟動冷卻 + 扣 favor：faction 正是 BeginTargeting 存下的那個
            caller.royalty.GetPermit(def, faction)?.Notify_Used();
            if (!free)
                caller.royalty.TryRemoveFavor(faction, def.royalAid.favorCost);
        }

        // 額外命中檢查（基底已提供 ValidateTarget，可在此擋掉非法落點）
        public override bool ValidateTarget(LocalTargetInfo target, bool showMessages = true)
            => base.ValidateTarget(target, showMessages);
    }
}
找對基底類的捷徑 在 dnSpy 裡搜 RoyalTitlePermitWorker，看哪個原版許可（如 OrbitalStrike）的 Worker 行為最接近你要的，直接繼承它並覆寫關鍵方法， 比從抽象基底硬刻 gizmo / targeting 省一大半工。所有 Worker 都在 RimWorld 命名空間。
§5 Psycast 與榮譽（psylink 的來源）

Royalty 的超能（psycast）建立在第 24 章的 Ability 框架上， 本章只聚焦「能力從哪來、爵位如何提升 psylink」。psylink 是 pawn 的「超能等級」， 由一個特殊 hediff Hediff_Psylink（Verse）承載，等級越高能用的超能越強。

psylink 的取得途徑

爵位授予

晉升到帶 reward 的爵位時，授爵流程可給一級 psylink。上限受 RoyalTitleDef.MaxAllowedPsylinkLevel(FactionDef) 約束。

儀式 / 樹靈

透過 psytrainer（藥物式 ThingDef）或在古樹（anima tree）旁的冥想儀式（Ideology 串接）提升 psylink。

PsicastingTradition

儀式的傳統 / 流派（如 PsicastingTradition 相關設定）影響冥想加速與超能風格，多由 RitualDef 配置。

從 C# 提升 psylink

using RimWorld;
using Verse;

public static void BumpPsylink(Pawn pawn, int levels = 1)
{
    if (!ModLister.RoyaltyInstalled || pawn?.health == null) return;

    // 找現有 psylink hediff；沒有就建一個
    Hediff_Psylink link = pawn.GetMainPsylinkSource();
    if (link == null)
    {
        // source = null → 視為「非爵位來源」的 psylink 增益
        pawn.ChangePsylinkLevel(levels, sendLetter: true);
    }
    else
    {
        // ChangePsylinkLevel 內部會夾在 MaxAllowedPsylinkLevel 之內
        pawn.ChangePsylinkLevel(levels, sendLetter: true);
    }
}
ChangePsylinkLevel 只夾全域上限，不檢查爵位上限 原始碼裡 Pawn.ChangePsylinkLevel → Hediff_Psylink.ChangeLevel 只把增量夾在 PsychicAmplifier.maxSeverity（全域 psylink 上限）之內， 並不會主動檢查 RoyalTitleDef.MaxAllowedPsylinkLevel。爵位上限是由授爵流程 / psytrainer 等「來源端」把關，而非這個方法。 所以直接呼叫它能讓 pawn 的 psylink 超過其爵位允許值（之後可能因「psylink 高於爵位」產生負面想法）。設計上仍建議先升爵位、再升 psylink。

neural heat（精神熵）與 psyfocus 速覽

超能機制兩個資源，由 RimWorld.Pawn_PsychicEntropyTracker（pawn.psychicEntropy）管理：

  • psyfocus（精神專注）：施放超能的「彈藥」，靠冥想 / 古樹回充；空了就放不出大招。
  • neural heat（精神熵 / 神經熱）：施放累積的「熱量」，超過上限 → 進入 psychic breakdown（短暫失能 + 心情）。
成員（pawn.psychicEntropy.*）用途
CurrentPsyfocus當前 psyfocus（0–1）
EntropyValue / MaxEntropy當前 / 上限神經熱
TryAddEntropy(float, ...)施放後加熱；回傳是否成功（會否超載）
OffsetPsyfocusDirectly(float)直接增減 psyfocus（補給 / 消耗）
本章只談來源，機制看第 24 章 自訂超能的 AbilityDef 結構、CompAbilityEffect 效果、目標選擇與 psyfocus 消耗綁定，都在第 24 章。 本章負責回答「這個超能要靠爵位 / psylink 等級幾才解鎖、榮譽怎麼換」。
§6 帝國互動玩法

帝國不是擺設派系，而是一條完整的「榮譽經濟」。你的 mod 要嵌進去，得理解 favor 怎麼進、爵位怎麼出問題、放棄爵位的代價。

榮譽任務（串第 22 章）

帝國透過 QuestScriptDef 派發任務（護送貴族、提供住宿、完成戰役），完成後獎勵 royal favor。 自訂任務獎勵 favor 時，用 QuestPart_GiveRoyalFavor（RimWorld）或在 quest script 內呼叫對應的 reward node。 

// QuestScriptDef 內，給某 pawn 帝國榮譽的 QuestPart（概念示意，實際多用 .quest 腳本節點）
var part = new QuestPart_GiveRoyalFavor();
part.faction        = Faction.OfEmpire;
part.giveToAccepter = true;          // 給接任務的 pawn（true 時不必設 giveTo）
part.amount         = 12;
part.inSignal       = "Quest.Completed";  // 收到此 signal 時發 favor
quest.AddPart(part);
用 .quest XML 腳本，而非硬寫 QuestPart RimWorld 推薦用 QuestScriptDef 的宣告式腳本（第 22 章）描述獎勵流程，可讀性與相容性都更好。 上面的 C# 只是讓你看清 favor 是透過哪個 QuestPart 實際發出去的。

貴族來訪與身分衝突

情境機制後果
貴族來訪 / 借宿帝國發起的 quest / lord job招待得好 → goodwill / favor；怠慢 → 關係下滑
高爵位 pawn 受傷致殘義體 / 缺肢與爵位形象帝國一般容忍，但部分劇情會對「不體面」反應
精神崩潰 / 暴走MentalState 觸發（第 20 章）高期望讓崩潰更易發生（房間不達標 → 期望落差）
頭骨 / 移除腦下垂體器官摘除影響 psylink 來源失去 psycaster 身分 / 能力，爵位可能不再匹配

放棄爵位的後果

玩家可以主動讓 pawn 放棄帝國爵位（UI 的「renounce title」）。這在程式上走 Pawn_RoyaltyTracker 的退爵流程， 會對帝國造成顯著的 goodwill 衝擊（派系關係下滑，可能轉敵對），並讓該 pawn 失去爵位帶來的許可 / psylink 上限。

放棄 / 被剝奪的代價

  • 帝國 goodwill 大幅下降（可能宣戰）
  • 失去該爵位的 permits 與 psylink 上限
  • 其他持爵 pawn 可能連帶受波及（劇情）
  • 產生負面想法（喪失身分 / 屈辱）

設計建議

  • 把「放棄爵位」做成有意義的抉擇而非無痛操作
  • mod 若強制移除爵位，記得同步處理 goodwill 與心情
  • 提供回頭路（重新賺 favor 復爵）增加深度

生成「帶爵位」的 pawn

Verse.PawnGenerationRequest 有專門欄位讓你直接生出有爵位的 pawn，不用生完再手動 SetTitle。 這在做帝國貴族、護送對象、援軍隊長時很有用。 

using RimWorld;
using Verse;

Faction empire = Faction.OfEmpire;
RoyalTitleDef knight = DefDatabase<RoyalTitleDef>.GetNamed("Knight");

var req = new PawnGenerationRequest(
    kind: PawnKindDefOf.Empire_Fighter_Trooper,   // 帝國兵種（Royalty 提供）
    faction: empire,
    context: PawnGenerationContext.NonPlayer,
    forceGenerateNewPawn: true)
{
    // ↓ 這些欄位只在裝了 Royalty 時有意義
    FixedTitle = knight,                        // 強制給定爵位（ctor 參數 fixedTitle）
    MinChanceToRedressWorldPawn = 1f,

    // 想連 psylink 一起：用對應欄位（版本間欄位名可能不同，去 dnSpy 對照）
};

Pawn noble = PawnGenerator.GeneratePawn(req);
// 若 FixedTitle 那條因版本不可用，退而求其次：生完再 SetTitle
if (noble.royalty?.GetCurrentTitle(empire) == null)
    noble.royalty?.SetTitle(empire, knight, grantRewards: true);
PawnGenerationRequest 欄位在版本間會變 Royalty 相關欄位（如 FixedTitle、psylink 強制值）名稱 / 是否存在隨版本變動。 務必用 dnSpy 對著你目標版本的 Verse.PawnGenerationRequest 結構確認，別照抄舊教學的欄位名（見第 11 章）。

ThroneDef 與房間角色（RoomRoleDef）

王座室的判定核心是房間角色：RimWorld.RoomRoleDef ThroneRoom 由 RoomRoleWorker_ThroneRoom 判定， 條件是房間內有一個被指派的王座（ThingDef 帶 Building_Throne / 對應 comp，定義可視為 ThroneDef 角色）。 Bedroom 角色同理由 RoomRoleWorker_Bedroom 判定。 

// 判斷某房間是不是合格的王座室、印象品質夠不夠
Room room = pawn.GetRoom();
if (room != null)
{
    RoomRoleDef role = room.Role;                  // e.g. RoomRoleDefOf.ThroneRoom
    bool isThrone = role == RoomRoleDefOf.ThroneRoom;
    float impress = room.GetStat(RoomStatDefOf.Impressiveness);

    RoyalTitleDef t = pawn.royalty?.GetCurrentTitle(Faction.OfEmpire);
    if (t != null && t.throneRoomRequirements != null)
    {
        foreach (RoomRequirement r in t.throneRoomRequirements)
        {
            bool met = r.Met(room, pawn);
            Log.Message($"王座室要求 {r.LabelCap(room)} = {met}");
        }
    }
}
§8 陷阱與最佳實務
陷阱 1：未裝 DLC 直接崩潰 最常見的災難。只要 C# 在沒守門的情況下觸碰 RoyalTitleDef、Pawn_RoyaltyTracker、Hediff_Psylink 等型別， 缺 DLC 時就 TypeLoadException / NRE。鐵則：所有 royalty 程式碼先過 ModLister.RoyaltyInstalled，並把這些型別關進獨立類別（§1）。
陷阱 2：服裝要求三方打架（爵位 × 第 17 章裝備 × 第 32 章思想） 爵位 requiredApparel 要求穿 A，Ideology（第 32 章）的 precept 可能要求穿 B 或裸體， 你自訂的 ApparelRequirement（第 17 章）又要求 C——三者衝突時 pawn 怎麼穿都會掉心情。 設計爵位服裝時，盡量用寬鬆的 tag 匹配（requiredTags 給多個可選項），別綁死單一 ThingDef。
陷阱 3：roomRequirements 太苛 → 永久負心情 把 impressiveness / space 門檻設得超高，玩家前期根本蓋不出來，結果是「升爵反而長期扣心情」。 用 RoomRequirement 的各子類（Impressiveness / NoBeauty 之類）時，數值要對照原版同級爵位，並考慮玩家所處階段。
陷阱 4：favorCost 失衡破壞晉升曲線 新爵位的 favorCost 與 seniority 不匹配，會讓晉升「跳級太快」或「卡死賺不夠」。 插入自訂爵位前，先把整條鏈的 (seniority, favorCost) 列成表（§2），確認你的新點落在曲線上而非離群。
陷阱 5：改 RoyalTitleDef 影響既有存檔 存檔裡的 RoyalTitle 物件保存的是 RoyalTitleDef 的 defName 參照 + 已收 favor 數，不是欄位快照。 所以你改 favorCost / minExpectation 後，讀檔時會立即套用新值——可能讓原本「達標」的房間突然變不達標、或 pawn 期望暴漲。 若你移除或改名了某個被存檔引用的 RoyalTitleDef，讀檔會出現 cross-ref 解析失敗（爵位變 null），務必保留舊 defName 或提供遷移 patch。
最佳實務小結
  • 守門兩層：XML MayRequire + C# ModLister.RoyaltyInstalled，型別隔離。
  • 新爵位先畫 (seniority, favorCost, minExpectation) 對照表，落在原版曲線上。
  • 服裝要求用寬鬆 tag，避開與 Ideology / 自訂 apparel 的死結。
  • 房間要求對照原版同級，別讓升爵變成永久心情懲罰。
  • 不要無痛亂發高爵位：義務成本是玩法核心，不是 bug。
  • 改 / 刪既有 RoyalTitleDef 前想清楚存檔相容性，必要時保留舊 defName。
