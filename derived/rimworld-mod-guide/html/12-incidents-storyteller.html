事件與敘事者

Incidents 與 Storyteller——RimWorld 敘事引擎的核心。從 IncidentDef 到威脅點數計算，打造你自己的襲擊、訪客與天災。

Chapter 12 · Incidents & Storyteller
§0 敘事引擎資料流總覽

RimWorld 沒有固定劇本，所有事件都由 Storyteller（敘事者）在執行期動態決定。整條鏈路是：Storyteller 內含一組 StorytellerComp，每個 Comp 依自己的節律（時間週期、隨機骰、派系互動）產出候選的 FiringIncident，經由 IncidentWorker 的可觸發性檢查（CanFireNow）後執行，最終透過 Letter 通知玩家。

三個核心抽象的職責劃分 IncidentDef = 「事件是什麼」（資料）；IncidentWorker = 「事件怎麼觸發與執行」（邏輯）；StorytellerComp = 「何時、以什麼頻率挑選事件」（節奏）。三者透過 IncidentParms 傳遞執行期參數（威脅點數、目標、派系）。 
// 敘事引擎一次 tick 的概念流程（Verse / RimWorld 命名空間）
Storyteller.StorytellerTick()
  └─ foreach (StorytellerComp comp in storytellerComps)
       └─ comp.MakeIntervalIncidents(target)   // 產出 FiringIncident 候選
            └─ 每個 FiringIncident { def, parms, source }
                 └─ def.Worker.CanFireNow(parms)        // 可觸發性閘門
                      └─ def.Worker.TryExecute(parms)     // 真正執行
                           └─ TryExecuteWorker(parms)        // 你要覆寫的方法

IncidentDef

XML 定義的事件元資料：分類、目標型別、基礎機率、冷卻、指向的 Worker class。

IncidentWorker

C# 邏輯：判斷現在能否觸發、計算實際效果、生成 Pawn / GameCondition / Letter。

StorytellerComp

節奏控制器：OnOffCycle、RandomMain、FactionInteraction 等決定事件頻率與時機。

§1 IncidentDef 結構解析

IncidentDef 是事件的資料骨架，定義於 XML。對資深工程師而言，最重要的是理解每個欄位如何被敘事引擎讀取——它們不是裝飾，而是 Storyteller 過濾與選擇候選事件的判據。

核心欄位一覽

欄位型別作用
category IncidentCategoryDef 事件分類（如 ThreatBig、ThreatSmall、Misc、FactionArrival）。Storyteller 各 Comp 只挑選自己負責分類的事件。
workerClass Type 處理此事件的 IncidentWorker 子類別全名。實際邏輯都在這裡。
targetTags List<IncidentTargetTagDef> 事件作用的目標型別標籤（地圖 / 商隊 / 世界）。與目標的 acceptedTags 必須相交，否則永遠不會觸發。
baseChance float 基礎權重（非百分比）。在同分類候選池中做加權隨機抽選時使用，數值愈大愈常被選中。
minRefireDays float 同事件兩次觸發之間的最短間隔（遊戲日）。冷卻期間 CanFireNow 自動回傳 false。
earliestDay / minPopulation int 最早可觸發日 / 最低殖民者人數門檻。常用於避免開局就出現高難度事件。
pointsScaleable bool true 時威脅點數會隨財富與時間縮放（襲擊類必開）。
letterDef / letterLabel / letterText LetterDef / string 事件觸發後預設通知信的樣式與文字（也可在 Worker 內手動發信覆蓋）。
allowedBiomes / disallowedBiomes List<BiomeDef> 生態域白/黑名單（如「火山冬天」不在沙漠觸發）。

典型 IncidentDef XML

<!-- Defs/IncidentDefs/Incidents_MyMod.xml -->
<Defs>
  <IncidentDef>
    <defName>MyMod_GiftCaravan</defName>
    <label>友好商隊來訪</label>
    <!-- category 決定哪個 StorytellerComp 會挑到它 -->
    <category>FactionArrival</category>
    <!-- workerClass 指向你的 C# 邏輯 -->
    <workerClass>MyMod.IncidentWorker_GiftCaravan</workerClass>
    <!-- targetTags：作用於「地圖」型目標 -->
    <targetTags>
      <li>Map_PlayerHome</li>
    </targetTags>
    <baseChance>2.0</baseChance>
    <minRefireDays>15</minRefireDays>
    <earliestDay>5</earliestDay>
    <minPopulation>2</minPopulation>
    <letterDef>PositiveEvent</letterDef>
    <letterLabel>友好商隊</letterLabel>
    <letterText>一支友好的商隊帶著禮物抵達了你的殖民地。</letterText>
  </IncidentDef>
</Defs>

常見 IncidentCategoryDef

威脅類

ThreatBig — 大型威脅（襲擊、機械蟲、墜艦）
ThreatSmall — 小型威脅（瘋狂動物、毒鼠）
由 OnOffCycle / Threat 類 Comp 驅動

非威脅類

Misc — 雜項（天氣、資源、商隊）
FactionArrival — 派系到訪（訪客、貿易商）
AllyAssistance — 盟友求援
DiseaseHuman / DiseaseAnimal — 疾病

§2 繼承 IncidentWorker：CanFireNowSub 與 TryExecuteWorker

IncidentWorker 是事件的行為核心。你幾乎只會覆寫兩個 protected 方法：CanFireNowSub（額外的可觸發性條件）與 TryExecuteWorker（實際執行）。注意基底的 CanFireNow 與 TryExecute 是 public 入口，內部已處理 minRefireDays、targetTags、人口門檻等通用檢查，不要直接覆寫它們，否則會繞過內建守衛。

方法職責對照

方法可見性你該做的事
CanFireNowSub(parms) protected virtual 回傳是否符合「本事件專屬」的觸發條件。預設回傳 true。例：襲擊需要有敵對派系、天氣事件需要當前無相同 GameCondition。
TryExecuteWorker(parms) protected abstract 真正執行事件並回傳 bool。回傳 false 代表「執行失敗，視為未觸發」（敘事引擎會另選事件，不消耗冷卻）。
CanFireNow(parms) public（基底實作） 勿覆寫。基底會檢查 def 條件後再呼叫你的 CanFireNowSub。 
using RimWorld;
using Verse;

namespace MyMod
{
    public class IncidentWorker_GiftCaravan : IncidentWorker
    {
        // ① 額外觸發條件——基底通用檢查通過後才會呼叫
        protected override bool CanFireNowSub(IncidentParms parms)
        {
            Map map = (Map)parms.target;

            // 需要存在一個可作為「禮物來源」的友好派系
            return Find.FactionManager.AllFactionsVisible
                .Any(f => !f.IsPlayer
                       && !f.def.hidden
                       && f.PlayerRelationKind == FactionRelationKind.Ally);
        }

        // ② 實際執行；回傳 false = 視為未觸發
        protected override bool TryExecuteWorker(IncidentParms parms)
        {
            Map map = (Map)parms.target;

            // 取得來源派系（parms 沒帶就自己挑一個盟友）
            Faction faction = parms.faction
                ?? Find.FactionManager.RandomAlliedFaction();
            if (faction == null) return false;

            // 找一個邊緣可生成點
            if (!CellFinder.TryFindRandomEdgeCellWith(
                    c => map.reachability.CanReachColony(c),
                    map, CellFinder.EdgeRoadChance_Neutral,
                    out IntVec3 spawnCell))
            {
                return false;
            }

            // ... 生成 Pawn / 放置禮物（見 §4 完整範例）...

            // 發送通知信
            SendStandardLetter(parms, new TargetInfo(spawnCell, map));
            return true;
        }
    }
}
SendStandardLetter 自動讀 def 的信件欄位 基底提供的 SendStandardLetter(parms, lookTargets) 會自動使用 def.letterDef、def.letterLabel、def.letterText 發信，並把點擊信件時的鏡頭定位到 lookTargets。若需要動態文字，改用 §8 的 Find.LetterStack.ReceiveLetter 手動發信。
§3 IncidentParms：執行期參數載體

IncidentParms 是事件「這一次」觸發的執行期上下文。Storyteller 生成候選事件時建立它，並沿著 CanFireNow → TryExecute → TryExecuteWorker 一路傳遞。最關鍵的三個欄位是 points、target、faction。

欄位型別說明
points float 威脅點數（threat points）。決定襲擊規模、敵人數量與裝備強度。由 Storyteller 依財富與難度計算（見 §7）。
target IIncidentTarget 事件作用對象。可能是 Map、Caravan 或 World。需依 def.targetTags 與目標相容。
faction Faction 觸發此事件的派系（襲擊者 / 訪客來源）。可能為 null，Worker 內常需自行挑選。
raidArrivalMode PawnsArrivalModeDef 襲擊到達方式（步行入侵 / 空投 / 爬行而出）。null 時 RaidWorker 會依點數隨機挑選。
raidStrategy RaidStrategyDef 襲擊策略（立即攻擊 / 圍城 / 偷襲）。
pawnGroupMakerSeed int? 生成敵人群組的隨機種子，便於重現特定組合。

手動建立 IncidentParms（除錯 / 觸發器）

當你想在自訂邏輯（如某個建築完工、研究解鎖）中主動觸發事件，可手動組裝 parms：

// 用 StorytellerUtility 取得「依目標縮放後」的預設 parms
IncidentParms parms = StorytellerUtility.DefaultParmsNow(
    IncidentCategoryDefOf.ThreatBig,   // 分類（決定 points 縮放曲線）
    map);                                // IIncidentTarget

// 覆寫想要的參數
parms.points = Mathf.Max(parms.points, 500f);   // 至少 500 點
parms.faction = Find.FactionManager.RandomEnemyFaction();
parms.raidArrivalMode = PawnsArrivalModeDefOf.EdgeWalkIn;

// 取得 IncidentDef 並觸發
IncidentDef incident = IncidentDefOf.RaidEnemy;
if (incident.Worker.CanFireNow(parms))
{
    incident.Worker.TryExecute(parms);
}
DefaultParmsNow 的分類必須對應 傳給 DefaultParmsNow 的 IncidentCategoryDef 決定了 points 的縮放方式。給襲擊用 ThreatBig、給非威脅事件用 Misc。用錯分類會得到不合理的點數（例如禮物事件拿到襲擊等級的點數，雖然點數對禮物無意義，但若你的 Worker 依 points 縮放規模就會出問題）。
§4 完整範例：三種事件型別

4.1 自訂襲擊事件（繼承 IncidentWorker_Raid）

襲擊類最好繼承 IncidentWorker_RaidEnemy 而非從零自製——可重用原版的群組生成、到達模式、策略選擇邏輯。下例覆寫 ResolveRaidPoints 與 TryGenerateRaidInfo 來注入自訂行為。

using RimWorld;
using Verse;
using UnityEngine;

namespace MyMod
{
    // 「狂怒襲擊」：點數翻倍、強制步行入侵、全員 berserk 傾向
    public class IncidentWorker_FrenzyRaid : IncidentWorker_RaidEnemy
    {
        protected override bool CanFireNowSub(IncidentParms parms)
        {
            // 必須先滿足原版襲擊的所有條件（有敵對派系等）
            if (!base.CanFireNowSub(parms)) return false;

            Map map = (Map)parms.target;
            // 額外條件：殖民地財富需達門檻才有資格觸發狂怒襲擊
            return map.wealthWatcher.WealthTotal > 50000f;
        }

        protected override void ResolveRaidPoints(IncidentParms parms)
        {
            if (parms.points <= 0f)
                parms.points = StorytellerUtility.DefaultThreatPointsNow(parms.target);
            parms.points *= 2f;   // 狂怒：點數翻倍
        }

        protected override bool TryExecuteWorker(IncidentParms parms)
        {
            // 強制步行入侵 + 立即攻擊策略
            parms.raidArrivalMode = PawnsArrivalModeDefOf.EdgeWalkIn;
            parms.raidStrategy = RaidStrategyDefOf.ImmediateAttack;

            // 交給基底生成敵人群組並安排到達
            bool result = base.TryExecuteWorker(parms);

            if (result)
            {
                // 額外：讓所有襲擊者進入狂暴心智狀態
                foreach (Pawn p in ((Map)parms.target)
                         .mapPawns.SpawnedPawnsInFaction(parms.faction))
                {
                    p.mindState.mentalStateHandler
                        .TryStartMentalState(MentalStateDefOf.Berserk);
                }
            }
            return result;
        }
    }
}

4.2 自訂訪客事件（生成友好 Pawn 群組）

public class IncidentWorker_GiftCaravan : IncidentWorker
{
    protected override bool CanFireNowSub(IncidentParms parms)
    {
        return Find.FactionManager.RandomAlliedFaction(
            allowHidden: false, allowDefeated: false,
            allowNonHumanlike: false) != null;
    }

    protected override bool TryExecuteWorker(IncidentParms parms)
    {
        Map map = (Map)parms.target;
        Faction faction = parms.faction
            ?? Find.FactionManager.RandomAlliedFaction();
        if (faction == null) return false;

        // 找入場點
        if (!RCellFinder.TryFindRandomPawnEntryCell(
                out IntVec3 entry, map, CellFinder.EdgeRoadChance_Friendly))
        {
            return false;
        }

        // 用 PawnGroupMaker 生成一群訪客
        var groupParms = new PawnGroupMakerParms
        {
            groupKind = PawnGroupKindDefOf.Peaceful,
            tile = map.Tile,
            faction = faction,
            points = Mathf.Max(parms.points, faction.def.MinPointsToGeneratePawnGroup(
                PawnGroupKindDefOf.Peaceful))
        };

        List<Pawn> visitors =
            PawnGroupMakerUtility.GeneratePawns(groupParms).ToList();
        if (visitors.Count == 0) return false;

        foreach (Pawn p in visitors)
        {
            IntVec3 loc = CellFinder.RandomClosewalkCellNear(entry, map, 5);
            GenSpawn.Spawn(p, loc, map);
        }

        // 指派「友好訪問」LordJob，讓 AI 表現得像訪客
        LordMaker.MakeNewLord(faction,
            new LordJob_VisitColony(faction, entry), map, visitors);

        SendStandardLetter(parms, visitors[0]);
        return true;
    }
}

4.3 自訂天氣事件（觸發 GameCondition）

public class IncidentWorker_AshStorm : IncidentWorker
{
    protected override bool CanFireNowSub(IncidentParms parms)
    {
        Map map = (Map)parms.target;
        // 已經有同類 GameCondition 時不重複觸發
        return !map.gameConditionManager
            .ConditionIsActive(MyDefOf.MyMod_AshStorm);
    }

    protected override bool TryExecuteWorker(IncidentParms parms)
    {
        Map map = (Map)parms.target;

        // 持續時間：3~7 天（以 tick 計）
        int duration = Rand.Range(60000, 140000);

        GameCondition cond = GameConditionMaker.MakeCondition(
            MyDefOf.MyMod_AshStorm, duration);
        map.gameConditionManager.RegisterCondition(cond);

        SendStandardLetter(parms, map.Parent);  // 鏡頭定位到地圖
        return true;
    }
}

對應的 GameConditionDef XML（決定視覺與環境效果）：

<!-- Defs/GameConditionDefs/GC_AshStorm.xml -->
<GameConditionDef>
  <defName>MyMod_AshStorm</defName>
  <conditionClass>GameCondition_NoSunlight</conditionClass>
  <label>灰燼風暴</label>
  <description>火山灰遮蔽天空，作物生長與太陽能發電停擺。</description>
  <endMessage>灰燼風暴已散去。</endMessage>

[DefOf] 快取你的 Def 引用 上例的 MyDefOf.MyMod_AshStorm 來自一個標註 [DefOf] 的靜態類別，欄位名與 defName 相同，遊戲啟動時自動填入。避免每次用 DefDatabase<GameConditionDef>.GetNamed("...") 字串查詢。
§5 Storyteller 架構

Storyteller（如 Cassandra、Phoebe、Randy）本身只是個薄外殼——真正的節奏邏輯全在它所掛載的一組 StorytellerComp 裡。一個 StorytellerDef 透過 <comps> 列表組合多個 Comp，每個 Comp 用對應的 StorytellerCompProperties 攜帶參數。

StorytellerDef XML 結構

<!-- 仿原版 Cassandra 的精簡結構 -->
<StorytellerDef>
  <defName>MyMod_CustomTeller</defName>
  <label>自訂敘事者</label>
  <listOrder>100</listOrder>
  <comps>
    <!-- 大型威脅：開/關循環，營造「平靜—危機」節奏 -->
    <li Class="StorytellerCompProperties_OnOffCycle">
      <category>ThreatBig</category>
      <onDays>1.5</onDays>
      <offDays>5</offDays>
      <minSpacingDays>0.4</minSpacingDays>
      <numIncidentsRange>0.6~1.5</numIncidentsRange>
      <acceptFractionByDaysPassedCurve>
        <points>
          <li>(0, 0.05)</li>
          <li>(8, 1)</li>
        </points>
      </acceptFractionByDaysPassedCurve>
    </li>
    <!-- 雜項事件：純隨機抽選 -->
    <li Class="StorytellerCompProperties_RandomMain">
      <mtbDays>4</mtbDays>
      <randomChance>1</randomChance>
      <categoryWeights>
        <Misc>1</Misc>
        <ThreatSmall>0.5</ThreatSmall>
      </categoryWeights>
    </li>
    <!-- 派系互動：訪客、貿易商、求援 -->
    <li Class="StorytellerCompProperties_FactionInteraction">
      <incident>VisitorGroup</incident>
      <baseIncidentsPerYear>12</baseIncidentsPerYear>
      <minSpacingDays>2.5</minSpacingDays>
      <fullAlliesOnly>false</fullAlliesOnly>
    </li>
  </comps>
</StorytellerDef>
Comp 與 Properties 的配對慣例 每個 StorytellerComp_X 都有一個對應的 StorytellerCompProperties_X。XML 的 <li Class="...Properties_X"> 會在載入時實例化 Properties，並由它的 CompClass 屬性反向建立對應的 Comp。寫自訂 Comp 時兩者都要實作。
§6 OnOffCycle / RandomMain / FactionInteraction

OnOffCycle

在「開啟期」密集產生事件、「關閉期」沉寂。這就是 Cassandra 著名的「危機—喘息」節奏來源，最常用於 ThreatBig。

RandomMain

以 MTB（平均間隔天數）為基礎做隨機抽骰，依 categoryWeights 加權選分類。Randy 大量依賴它，節奏較不可預測。

FactionInteraction

專責派系到訪類事件（訪客、貿易商、盟友求援），以「每年次數」為節律，並做派系關係篩選。

關鍵參數對照

Comp關鍵參數作用
OnOffCycle onDays / offDays 開啟期 / 關閉期的長度（遊戲日），構成循環週期。
numIncidentsRange 單一開啟期內預期觸發的事件數量範圍。
acceptFractionByDaysPassedCurve 隨遊戲天數調整接受率的曲線（前期溫和、後期全開）。
RandomMain mtbDays 事件平均觸發間隔（Mean Time Between）。
categoryWeights 各 IncidentCategory 的抽選權重。
FactionInteraction baseIncidentsPerYear 每遊戲年的基礎觸發次數。
minSpacingDays 同類事件最小間隔，避免訪客連續刷屏。

自訂 StorytellerComp 範例

若要實作全新節律（例如「每滿月觸發一次威脅」），繼承 StorytellerComp 覆寫 MakeIntervalIncidents：

// Properties：攜帶 XML 參數
public class StorytellerCompProperties_FullMoon : StorytellerCompProperties
{
    public IncidentDef incident;
    public float pointsFactor = 1f;

    public StorytellerCompProperties_FullMoon()
    {
        // 把 Properties 綁定到對應的 Comp 類別
        compClass = typeof(StorytellerComp_FullMoon);
    }
}

// Comp：實際節律邏輯
public class StorytellerComp_FullMoon : StorytellerComp
{
    private StorytellerCompProperties_FullMoon Props
        => (StorytellerCompProperties_FullMoon)props;

    public override IEnumerable<FiringIncident> MakeIntervalIncidents(
        IIncidentTarget target)
    {
        // 每天被呼叫一次；只在「滿月日」產出事件
        int dayOfMonth = GenLocalDate.DayOfTwelfth(target);
        if (dayOfMonth != 6) yield break;   // 非滿月不觸發

        IncidentParms parms = GenerateParms(
            Props.incident.category, target);
        parms.points *= Props.pointsFactor;

        if (Props.incident.Worker.CanFireNow(parms))
        {
            yield return new FiringIncident(
                Props.incident, this, parms);
        }
    }
}
§7 威脅點數 (Threat Points / Wealth) 計算

威脅點數是 RimWorld 難度曲線的核心。它由殖民地財富、殖民者數量、遊戲天數、難度設定共同決定，並透過 IncidentParms.points 影響襲擊規模。理解計算來源能讓你的自訂事件「合理地」隨進度變難。

點數來源 API

API回傳用途
StorytellerUtility.DefaultThreatPointsNow(target) float 當下的標準威脅點數（已套用財富、人數、天數、難度與 Storyteller 自身的調整）。自訂威脅事件首選。
StorytellerUtility.DefaultParmsNow(category, target) IncidentParms 連同 points 一起組好的 parms（內部就是呼叫上面那個算點數）。
map.wealthWatcher.WealthTotal float 地圖總財富（建築 + 物品 + Pawn）。是點數的主要輸入。
map.wealthWatcher.WealthItems / WealthBuildings float 分項財富，用於診斷「為何襲擊變強」。

點數計算的概念公式

// 概念示意（非原始碼逐行，說明影響因子）
float basePoints =
      財富因子(wealth)              // 財富愈高點數愈高（非線性，有遞減）
    + 殖民者因子(colonistCount)     // 每名殖民者貢獻固定點數
    + 已馴服動物因子;

basePoints *= 天數曲線(daysPassed); // 開局數天有大幅折扣，逐步爬升到 1.0
basePoints *= 難度設定.threatScale;  // 玩家選的難度（和平~極限）
basePoints *= storyteller 隨機波動;  // ±隨機，避免每次都一樣

// 最終 clamp 在合理範圍
points = Mathf.Clamp(basePoints, 35f, 10000f);
財富爆炸 = 襲擊爆炸 新手常見抱怨「我才剛起步就被打爆」，根因通常是堆了大量高價值物品/建築拉高 WealthTotal。設計自訂威脅事件時若用 DefaultThreatPointsNow，就自動繼承這套機制；若自己硬寫點數，務必也納入財富縮放，否則後期會過於簡單。

把點數轉成敵人群組

// 點數 → 一群敵兵的標準流程
float points = StorytellerUtility.DefaultThreatPointsNow(map);

var groupParms = new PawnGroupMakerParms
{
    groupKind = PawnGroupKindDefOf.Combat,
    tile      = map.Tile,
    faction   = enemyFaction,
    points    = points       // PawnGroupMaker 依點數決定兵力與裝備
};

List<Pawn> raiders =
    PawnGroupMakerUtility.GeneratePawns(groupParms).ToList();
除錯點數：開發者面板的 Incident Tester Dev Mode 下的 Debug Actions → Incidents 可手動觸發任意 IncidentDef 並指定點數，是調校威脅曲線最快的方式。搭配 Find.Storyteller 的內部狀態檢視，能看到各 Comp 的下次觸發倒數。
§8 LetterDef 與通知信件

事件對玩家的可見性幾乎完全靠 Letter（左下角的通知信）。LetterDef 定義信件的視覺樣式（顏色、音效、是否暫停遊戲）；發信則透過 Find.LetterStack.ReceiveLetter。

常用內建 LetterDef

LetterDef顏色 / 行為適用
PositiveEvent藍色，溫和禮物、好事、研究完成
NeutralEvent灰色中性通知（商隊路過）
NegativeEvent紅色壞事但非急迫（疾病、財損）
ThreatSmall橙色 + 提示音小型威脅
ThreatBig紅色 + 警報音 + 可暫停大型襲擊

手動發送信件（動態內容）

using RimWorld;
using Verse;

// 組裝動態文字（含 named arguments，支援在地化 grammar）
TaggedString label = "友好商隊";
TaggedString text = "來自 " + faction.Name
    + " 的商隊帶來了 " + giftCount + " 件禮物。";

Find.LetterStack.ReceiveLetter(
    label,
    text,
    LetterDefOf.PositiveEvent,
    new LookTargets(visitors[0]),  // 點信件時鏡頭定位
    faction);                          // 相關派系（信件可顯示派系資訊）

// 威脅類：可加入「選擇威脅目標」的快捷按鈕
var letter = (ChoiceLetter)LetterMaker.MakeLetter(
    "襲擊！", "一支敵對部隊正在逼近。",
    LetterDefOf.ThreatBig,
    new LookTargets(raiders[0]),
    enemyFaction);
Find.LetterStack.ReceiveLetter(letter);

§9 觸發 GameCondition（天氣、日蝕等）

GameCondition 是「持續一段時間、影響整張地圖或世界」的狀態（日蝕、極光、火山冬天、毒霧）。它與一次性 Incident 的差別在於「有持續期 + 每 tick 施加效果 + 結束時自動清理」。

GameCondition 生命週期

  1. GameConditionMaker.MakeCondition(def, durationTicks) 建立實例。
  2. map.gameConditionManager.RegisterCondition(cond) 註冊（地圖層級）；世界層級用 Find.World.gameConditionManager。
  3. 每 tick 引擎呼叫 cond.GameConditionTick()，子類別在此施加效果（如 GameCondition_NoSunlight 關閉日照）。
  4. 到期或被 cond.End() 移除時自動清理並顯示 endMessage。

內建 conditionClass 速查

conditionClass效果
GameCondition_NoSunlight日蝕：太陽能停擺、亮度下降（日蝕、灰燼風暴的基底）
GameCondition_PsychicEmanation靈能放射：影響特定性別/全體心情
GameCondition_ColdSnap / HeatWave寒流 / 熱浪：氣溫偏移
GameCondition_ToxicFallout毒霧：戶外 Pawn 與動植物中毒
GameCondition_VolcanicWinter火山冬天：長期降溫 + 減產
GameCondition_Aurora極光：夜間視覺 + 微心情加成

自訂 GameCondition 子類別（每 tick 效果）

public class GameCondition_AcidRain : GameCondition
{
    private const int DamageInterval = 300;  // 每 5 秒一次

    public override void GameConditionTick()
    {
        base.GameConditionTick();

        if (Find.TickManager.TicksGame % DamageInterval != 0)
            return;

        foreach (Map map in this.AffectedMaps)
        {
            foreach (Pawn p in map.mapPawns.AllPawnsSpawned)
            {
                // 只傷害露天（無屋頂）的 Pawn
                if (!p.Position.Roofed(map))
                {
                    p.TakeDamage(new DamageInfo(
                        DamageDefOf.Deterioration, 1f));
                }
            }
        }
    }

    // 影響天氣/亮度時可覆寫 SkyTarget / SkyGazeChanceFactor 等
}
AffectedMaps 自動處理地圖/世界範疇 GameCondition.AffectedMaps 會依 condition 註冊在哪個 manager（單一地圖 vs 世界）回傳正確的地圖集合，不需自己判斷。世界層級的 condition（如全域火山冬天）會套用到所有當前地圖。
§10 常見坑：事件無法觸發的系統化除錯

「我的事件就是不出現」是 incident 開發最常見的問題。由於敘事引擎有多層過濾閘門，逐層排查最有效。下表按出現頻率排序。

症狀 / 原因診斷修正
targetTags 與目標不相交
最常見。def 寫 Map_PlayerHome 但你以為作用於商隊。		 檢查目標的 IIncidentTarget.IncidentTargetTags() 是否包含 def.targetTags 之一。 確認 targetTags 與目標型別匹配。地圖事件用 Map_PlayerHome，世界事件用 World。
沒有任何 StorytellerComp 負責此 category 事件的 category 是否出現在當前 Storyteller 的某個 Comp 設定中？ 用既有 category（如 ThreatBig/Misc），或 PatchOperation 把你的 category 加進 Storyteller comps。
CanFireNowSub 永遠回傳 false 在方法首行加 Log.Message 確認是否被呼叫、條件結果。 檢查條件邏輯（如要求盟友派系但存檔沒有盟友）。
earliestDay / minPopulation 門檻未達 對照當前遊戲天數與殖民者數。 除錯時暫時調低門檻或用 Dev Mode 強制觸發。
minRefireDays 冷卻中 剛觸發過？冷卻內 CanFireNow 必為 false。 等待冷卻或除錯時設為 0。
baseChance 過低被其他事件淹沒 同分類池內其他 def 權重遠高於你的。 提高 baseChance，或用獨立 FactionInteraction/自訂 Comp 隔離。
workerClass 寫錯（型別找不到） 啟動時 Log 會報 Could not resolve cross-reference 或型別載入失敗。 用完整命名空間 + 類別名，確認 DLL 已正確編譯載入。
TryExecuteWorker 回傳 false
邏輯執行但因找不到生成點等而失敗。		 在各 early-return 處加 Log，定位是哪個前置條件失敗。 處理邊界情況（無可用 cell、無合適派系）並記錄。

系統化除錯流程

// 在 CanFireNowSub / TryExecuteWorker 灑診斷日誌
protected override bool CanFireNowSub(IncidentParms parms)
{
    bool hasAlly = Find.FactionManager.RandomAlliedFaction() != null;

    // 只在 Dev Mode 輸出，避免污染正式日誌
    if (Prefs.DevMode)
    {
        Log.Message($"[MyMod] GiftCaravan CanFireNowSub: "
            + $"hasAlly={hasAlly}, target={parms.target}, "
            + $"points={parms.points}");
    }
    return hasAlly;
}
  1. 開啟 Dev Mode，用 Debug Actions → Incidents → Do incident (map) 直接觸發你的 def——能觸發代表 Worker 本身沒問題，問題在 Storyteller 挑選層。
  2. 若手動觸發也失敗，看 Log 是否印出 Worker 的診斷訊息；沒印出代表 CanFireNow 在基底層就被擋下（targetTags / 冷卻 / 門檻）。
  3. 用 Debug Inspector → View Settings → Storyteller 觀察各 Comp 的下次觸發倒數與當前點數。
  4. 確認 def 已被載入：DefDatabase<IncidentDef>.GetNamedSilentFail("MyDef") != null。
targetTags 不符 = 永遠靜默失敗 這是最隱蔽的坑：def 完全正確載入、Worker 沒報錯，但事件就是不觸發，且不會有任何錯誤訊息。因為 Storyteller 在挑選階段就因 targetTags 不相交而跳過它。務必第一個檢查：map.IncidentTargetTags()（或 caravan/world 對應方法）是否包含 def.targetTags 中至少一個。
本章重點回顧

IncidentDef

category 決定誰挑它，targetTags 必須與目標相交，workerClass 指向邏輯

IncidentWorker

覆寫 CanFireNowSub + TryExecuteWorker，勿動 public 入口

Storyteller

OnOffCycle 危機節奏、RandomMain 隨機、FactionInteraction 派系

除錯

先查 targetTags，再查 category 是否有 Comp 負責，Dev Mode 強制觸發
