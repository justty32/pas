心智模型：三層分工

RimWorld 的建築是「資料（Def）+ 行為（Building class）+ 可插拔功能（Comp）」的組合。 多數建築甚至不需要自訂 class——一個 ThingDef 掛上幾個 ThingComp 就能跑。 理解這三層各自的職責，是寫出乾淨 mod 的關鍵。

ThingDef（資料）

XML 定義。尺寸、外觀、材質需求、掛哪些 Comp、放在哪個建造選單。能用 XML 就別寫 C#。

Building（行為）

Verse.Building 子類別。需要每 tick 邏輯、自訂 Gizmo、繪製覆寫、檢視面板文字時才繼承。

ThingComp（功能）

橫切功能模組：電力、燃料、開關、散熱。可組合、可重用，不綁定特定 class。

命名空間速記 Building、ThingComp、ThingWithComps、Map 在 Verse； CompPowerTrader、CompPower、PowerNet、Building_Battery 在 RimWorld。
1 · Building ThingDef 結構

建築和物品共用同一個 ThingDef，差別在 <category>Building</category> 以及一個 <building> 子區塊。下面是一個帶電力消耗的自訂機器的完整骨架。 

<?xml version="1.0" encoding="utf-8"?>
<Defs>
  <ThingDef ParentName="BuildingBase">
    <defName>MyMachine</defName>
    <label>共振機</label>
    <description>一台會嗡嗡作響的神祕機器。</description>

    <!-- 關鍵：歸類為建築 -->
    <category>Building</category>

    <!-- 指向自訂 Building 子類別（純資料建築可省略，預設 Verse.Building） -->
    <thingClass>MyMod.Building_MyMachine</thingClass>

    <!-- 外觀 -->
    <graphicData>
      <texPath>Things/Building/MyMachine</texPath>
      <graphicClass>Graphic_Single</graphicClass>
      <!-- drawSize 可大於 size 以突出格線 -->
      <drawSize>(2, 2)</drawSize>
    </graphicData>

    <!-- 佔地大小（寬, 高），以格為單位 -->
    <size>(2, 2)</size>

    <!-- 可否被穿越：Impassable / PassThroughOnly / Standable -->
    <passability>Impassable</passability>

    <!-- 是否可旋轉（家具/機台常為 true，方正機器設 false 較單純） -->
    <rotatable>false</rotatable>

    <!-- 阻擋光照與視線 -->
    <blockLight>true</blockLight>
    <fillPercent>0.5</fillPercent>

    <!-- 建造分類選單（見第 8 節） -->
    <designationCategory>Production</designationCategory>

    <!-- 統計：血量、易燃度、製作工時 -->
    <statBases>
      <MaxHitPoints>180</MaxHitPoints>
      <WorkToBuild>2400</WorkToBuild>
      <Flammability>0.5</Flammability>
    </statBases>

    <!-- 建材需求 -->
    <costList>
      <Steel>120</Steel>
      <ComponentIndustrial>4</ComponentIndustrial>
    </costList>

    <!-- building 子區塊：建築專屬旗標 -->
    <building>
      <isEdifice>true</isEdifice>      <!-- 是否為「主結構物」，一格一個 -->
      <ai_chillDestination>false</ai_chillDestination>
      <destroySound>BuildingDestroyed_Metal_Small</destroySound>
    </building>

    <!-- 掛載功能 Comp（電力消耗 + 開關 + 散熱）-->
    <comps>
      <li Class="CompProperties_Power">
        <compClass>CompPowerTrader</compClass>
        <basePowerConsumption>200</basePowerConsumption>
        <shortCircuitInRain>false</shortCircuitInRain>
      </li>
      <li Class="CompProperties_Flickable" />
    </comps>
  </ThingDef>
</Defs>

thingClassType對應的 C# class；省略則用父類別繼承的（通常 Building）

2 · 繼承 Building 自訂建築 class

當你需要 tick 邏輯、自訂按鈕、特殊繪製或檢視面板文字時，繼承 Verse.Building。 下面展示最常覆寫的幾個方法，並標出各自的呼叫時機與注意事項。 

using System.Collections.Generic;
using System.Text;
using RimWorld;
using UnityEngine;
using Verse;

namespace MyMod
{
    public class Building_MyMachine : Building
    {
        // 快取 Comp 參照，避免每次 GetComp 反覆查表
        private CompPowerTrader powerComp;
        private int resonanceLevel;   // 需存檔的自訂狀態

        // ── 生成（放置完成 / 讀檔重建時呼叫）──────────────
        public override void SpawnSetup(Map map, bool respawningAfterLoad)
        {
            // 一定要先呼叫 base，否則 Comp 不會初始化、PowerNet 不會掛上
            base.SpawnSetup(map, respawningAfterLoad);

            // base.SpawnSetup 之後才能安全取得 Comp
            powerComp = this.TryGetComp<CompPowerTrader>();

            if (!respawningAfterLoad)
            {
                // 只在「首次放置」執行的初始化（讀檔時不要重做）
                resonanceLevel = 0;
            }
        }

        // ── 存檔 / 讀檔（自訂欄位序列化）──────────────────
        public override void ExposeData()
        {
            base.ExposeData();
            Scribe_Values.Look(ref resonanceLevel, "resonanceLevel", 0);
        }

        // ── 低頻 tick（每 250 ticks ≈ 4 秒一次，省效能）────
        public override void TickRare()
        {
            base.TickRare();

            // powerComp 可能為 null（極少數情況），務必判空
            if (powerComp == null || !powerComp.PowerOn)
                return;   // 沒電就什麼都不做

            // 有電才累積共振（每次 TickRare 加一）
            resonanceLevel = Mathf.Min(resonanceLevel + 1, 100);
        }

        // ── 繪製（疊加效果在主貼圖之上）──────────────────
        public override void Draw()
        {
            base.Draw();   // 先畫本體

            if (powerComp != null && powerComp.PowerOn && resonanceLevel > 50)
            {
                // 在建築上方畫一個發光特效（範例用 OverlayDrawer 思路）
                this.Map.overlayDrawer.DrawOverlay(this, OverlayTypes.PowerOn);
            }
        }

        // ── 右鍵 / 選取時的按鈕（Gizmo）──────────────────
        public override IEnumerable<Gizmo> GetGizmos()
        {
            // 保留 base 的 gizmo（複製/拆除/Flick 開關等）
            foreach (Gizmo g in base.GetGizmos())
                yield return g;

            // 追加自訂按鈕
            yield return new Command_Action
            {
                defaultLabel  = "重置共振",
                defaultDesc   = "將共振等級歸零。",
                icon          = ContentFinder<Texture2D>.Get("UI/Commands/ResetResonance"),
                action        = () => resonanceLevel = 0
            };
        }

        // ── 檢視面板（左下角資訊）的附加文字──────────────
        public override string GetInspectString()
        {
            var sb = new StringBuilder();
            // base 會輸出血量、電力狀態、Flick 狀態等標準行
            sb.Append(base.GetInspectString());

            if (sb.Length > 0) sb.AppendLine();
            sb.Append($"共振等級：{resonanceLevel} / 100");

            // 慣例：結尾不要留多餘換行，用 TrimEndNewlines
            return sb.ToString().TrimEndNewlines();
        }
    }
}

tick 三種頻率

  • Tick() — 每 tick（60/秒），高耗能，需在 ThingDef 設 tickerType=Normal
  • TickRare() — 每 250 ticks，預設 tickerType=Rare
  • TickLong() — 每 2000 ticks，超低頻背景邏輯

呼叫順序要點

  • 所有覆寫先呼叫對應的 base.XXX()
  • 取 Comp 必須在 base.SpawnSetup 之後
  • GetGizmos 先 yield base 的，再追加自己的
  • 自訂欄位一律寫進 ExposeData
tickerType 必須對得上 只覆寫 TickRare() 卻在 ThingDef 設 <tickerType>Normal</tickerType>， 你的 TickRare 永遠不會被呼叫（反之亦然）。預設 BuildingBase 多為 Rare， 要每 tick 邏輯記得改成 Normal。
3 · CompPowerTrader / CompPowerPlant / CompPowerBattery 與 PowerNet

電力系統的核心是 PowerNet——由相連的電纜（conduit）構成的一張網。 每個耗電/發電/儲電建築透過一個 CompPower 子類別接入這張網。

CompPowerTrader

「交易者」：可發電（正值）也可耗電（負值）。一般機器用它，PowerOn 表示目前是否通電。

CompPowerPlant

發電廠基底。子類別 CompPowerPlantWind / Solar / Steam 各自算發電量。

CompPowerBattery

儲電。StoredEnergy / AmountCanAccept，PowerNet 自動充放電。

PowerNet 的角色

PowerNet 每 tick 結算：把所有 CompPowerTrader.PowerOutput 為正的加總成供給， 負的加總成需求；不足時從電池放電，仍不足則整網斷電（所有 PowerOn 變 false）。 你幾乎不直接操作 PowerNet，但讀取它的狀態很常見： 

// 從任一接電 Comp 取得所屬 PowerNet
CompPowerTrader trader = this.TryGetComp<CompPowerTrader>();
PowerNet net = trader?.PowerNet;   // ⚠ 未接上電纜時為 null！

if (net != null)
{
    float gain   = net.CurrentEnergyGainRate();   // 本 tick 淨增益（含電池）
    float stored = net.CurrentStoredEnergy();    // 全網電池總存量 Wd

    // 遍歷網內所有發電/耗電者
    foreach (CompPowerTrader c in net.powerComps)
    {
        // c.PowerOutput > 0 → 發電；< 0 → 耗電
    }
}

三者比較

Comp典型用途關鍵屬性PowerOutput 正負
CompPowerTrader機器、燈、門PowerOn、PowerOutput耗電為負
CompPowerPlant各式發電廠PowerOutput（由子類計算）發電為正
CompPowerBattery電池StoredEnergy、AmountCanAccept不直接交易，由網調度
電纜其實也是建築 電纜 PowerConduit 的 ThingDef 在 building 區塊設了 <isPowerConduit>true</isPowerConduit>。建築若 isPowerConduit 為 true， 自身就是網的一部分（如核心反應爐的整合電纜）。
4 · CompProperties_Power 設定

CompProperties_Power 是電力 Comp 的 XML 設定容器。 用 <compClass> 指定要實例化哪個 CompPower 子類別。

耗電機器

<li Class="CompProperties_Power">
  <compClass>CompPowerTrader</compClass>
  <!-- 正值 = 耗電（單位 W）。引擎內部存成負的 PowerOutput -->
  <basePowerConsumption>200</basePowerConsumption>
  <!-- 下雨時是否可能短路（戶外電器設 true）-->
  <shortCircuitInRain>false</shortCircuitInRain>
  <!-- 待機耗電（部分建築閒置時較省電）-->
  <idlePowerDraw>-1</idlePowerDraw>
</li>

固定輸出發電機（如地熱）

<li Class="CompProperties_Power">
  <compClass>CompPowerPlant</compClass>
  <!-- 負值 = 發電。-3600 表示穩定供給 3600W -->
  <basePowerConsumption>-3600</basePowerConsumption>
</li>

電池

<li Class="CompProperties_Battery">
  <storedEnergyMax>1000</storedEnergyMax>     <!-- 容量 Wd -->
  <efficiency>0.5</efficiency>             <!-- 充電效率（核心電池 50%）-->
</li>
欄位型別說明
compClassTypeCompPowerTrader / CompPowerPlant / 自訂子類
basePowerConsumptionfloat正=耗電、負=發電（W）。引擎以負值代表淨輸出
shortCircuitInRainbool戶外暴露電器下雨短路的事件來源
idlePowerDrawfloat待機耗電；-1 表示不啟用待機機制
transmitsPowerbool此 Comp 本身是否傳導電力（電纜為 true）
正負號是新手最常踩的坑 basePowerConsumption 在 XML 裡「正數代表耗電」是直覺寫法； 但 C# 端 CompPowerTrader.PowerOutput 耗電是負值、發電是正值。 兩邊符號相反，改 C# 時務必記住這個轉換。
5 · 從 C# 控制耗電與發電

動態調整功耗是常見需求：機器運轉時全功率、待機時省電；或發電量隨某個條件變動。 關鍵是 CompPowerTrader.PowerOutput（屬性，內部欄位 powerOutputInt）與 PowerOn。 

public class Building_VariableMachine : Building
{
    private CompPowerTrader power;
    private bool running;

    public override void SpawnSetup(Map map, bool respawningAfterLoad)
    {
        base.SpawnSetup(map, respawningAfterLoad);
        power = this.TryGetComp<CompPowerTrader>();
    }

    public override void Tick()   // 需 tickerType=Normal
    {
        base.Tick();
        if (power == null) return;

        // PowerOn 是唯讀的（由 PowerNet 結算決定），不能直接設定。
        // 沒電時直接停擺。
        if (!power.PowerOn)
        {
            running = false;
            return;
        }

        // ✅ 可寫：動態改變功耗。耗電為「負值」！
        // PowerOutput = -消耗瓦數。
        float baseDraw = -this.def.GetCompProperties<CompProperties_Power>().basePowerConsumption;
        power.PowerOutput = running
            ? baseDraw            // 全功率（已是負值）
            : baseDraw * 0.1f;    // 待機只用 10%
    }
}

三種寫法對照

目標API備註
讀目前是否通電power.PowerOn唯讀；PowerNet 結算結果
動態改耗電/發電power.PowerOutput = value耗電負值、發電正值
取設定的基礎值props.PowerConsumption / basePowerConsumption從 CompProperties_Power 讀
不要直接寫 powerOutputInt powerOutputInt 是 backing field，存取修飾子為 protected。 請走公開屬性 PowerOutput（它做了必要的內部更新）。 若你真的需要繞過（例如 Harmony patch 別人的建築），用 AccessTools.FieldRefAccess， 細節見 Harmony 章節。
發電量隨條件變動：覆寫 CompPowerPlant 風力/太陽能發電廠透過繼承 CompPowerPlant 並覆寫 protected override float DesiredPowerOutput 來回傳即時發電量。 自訂發電機照此模式即可，無需每 tick 手動賦值。
6 · CompFlickable / CompRefuelable / CompHeatPusher

CompFlickable 開關

加上它，建築就會有「開/關」按鈕（Flick），由小人執行撥動 Job。 關閉時 CompPowerTrader 會自動停止耗電。從 C# 讀 SwitchIsOn。 

<li Class="CompProperties_Flickable" />
CompFlickable flick = this.TryGetComp<CompFlickable>();
if (flick != null && flick.SwitchIsOn)
{
    // 開關開啟。注意：剛撥動到實際生效中間有 Job 延遲
}

CompRefuelable 燃料

以燃料驅動（如化學燃料發電機、火爐）。小人會自動補充燃料。 從 C# 用 Fuel / HasFuel 判斷，用 ConsumeFuel 手動消耗。 

<li Class="CompProperties_Refuelable">
  <fuelConsumptionRate>4.5</fuelConsumptionRate>   <!-- 每天消耗 -->
  <fuelCapacity>75</fuelCapacity>
  <fuelFilter>
    <thingDefs><li>Chemfuel</li></thingDefs>
  </fuelFilter>
  <consumeFuelOnlyWhenUsed>false</consumeFuelOnlyWhenUsed>
</li>
CompRefuelable fuel = this.TryGetComp<CompRefuelable>();
if (fuel != null && fuel.HasFuel)
{
    fuel.ConsumeFuel(1f);   // 自訂事件消耗一單位
}

CompHeatPusher 散熱

每 tick 向周圍格子注入（或抽走）熱量。發電機、火爐都靠它升溫；冷氣則用負值。 通常搭配 CompProperties_HeatPusherPowered，只有通電時才推熱。 

<!-- 純散熱（永遠推熱） -->
<li Class="CompProperties_HeatPusher">
  <compClass>CompHeatPusher</compClass>
  <heatPerSecond>3.5</heatPerSecond>          <!-- 正=放熱 負=吸熱 -->
  <heatPushMaxTemperature>28</heatPushMaxTemperature> <!-- 超過此溫不再放熱 -->
</li>

<!-- 只在通電時放熱 -->
<li Class="CompProperties_HeatPusherPowered">
  <compClass>CompHeatPusherPowered</compClass>
  <heatPerSecond>3.5</heatPerSecond>
</li>
Comp 之間會互相感知 CompHeatPusherPowered 內部會 TryGetComp<CompPowerTrader>() 檢查 PowerOn； CompFlickable 關閉時 CompPowerTrader 自動停耗。 這種 Comp 互查是 RimWorld 模組化的精髓——你的自訂 Comp 也可以照樣查詢同物件上的其他 Comp。
7 · PlaceWorker — 放置限制驗證

PlaceWorker 在玩家拖放建築藍圖時即時驗證位置是否合法，並可繪製輔助提示（範圍圈、相鄰高亮）。 它不影響已放置的建築，純粹是「能不能蓋在這」的規則。 

using Verse;
using RimWorld;

public class PlaceWorker_NotNearOther : PlaceWorker
{
    // 回傳是否允許放置；不允許時回傳的字串會顯示為紅字理由
    public override AcceptanceReport AllowsPlacing(
        BuildableDef def,
        IntVec3      center,
        Rot4         rot,
        Map          map,
        Thing        thingToIgnore = null,
        Thing        thing = null)
    {
        // 範例：方圓 5 格內不能有另一座同類建築
        int radius = 5;
        foreach (IntVec3 cell in GenRadial.RadialCellsAround(center, radius, useCenter: true))
        {
            if (!cell.InBounds(map)) continue;

            foreach (Thing t in cell.GetThingList(map))
            {
                if (t.def == def && t != thingToIgnore)
                    return new AcceptanceReport("附近已有一座共振機。");
            }
        }
        return AcceptanceReport.WasAccepted;
    }

    // 選取藍圖 / 拖放時繪製輔助圖示（選用）
    public override void DrawGhost(
        ThingDef def, IntVec3 center, Rot4 rot,
        Color ghostCol, Thing thing = null)
    {
        // 畫出 5 格半徑的範圍圈
        GenDraw.DrawRadiusRing(center, 5);
    }
}

掛載到 ThingDef

<placeWorkers>
  <li>MyMod.PlaceWorker_NotNearOther</li>
  <!-- 也可疊加核心提供的，例如顯示需要研究、需要屋頂等 -->
  <li>PlaceWorker_ShowFacilitiesConnections</li>
</placeWorkers>
常用核心 PlaceWorker PlaceWorker_Heater（顯示影響的房間）、PlaceWorker_OnSteamGeyser（限定間歇泉上）、 PlaceWorker_NeverAdjacentTrap、PlaceWorker_WindTurbine（風機間距）。 要做相似行為時先找有沒有現成的，能省很多力。
8 · designationCategory 與建造選單分類

<designationCategory> 決定建築出現在底部「建造（Architect）」選單的哪個分頁。 值必須對應一個既有或自訂的 DesignationCategoryDef。

核心內建分類（部分）

defName分頁

Production生產（工作台、機器）
Furniture家具
Power電力
Security防禦
Temperature溫度（冷暖氣）

自訂一個新分頁

<Defs>
  <DesignationCategoryDef>
    <defName>MyModCategory</defName>
    <label>共振科技</label>
    <order>250</order>   <!-- 數字越大越靠右；參考核心值排位 -->
  </DesignationCategoryDef>
</Defs>
<!-- 然後在 ThingDef 引用 -->
<designationCategory>MyModCategory</designationCategory>
把建築塞進「別人的」分頁 若只想把一兩個建築加進既有分頁，直接設 designationCategory 即可，不要 PatchOperation。 但若要往核心分頁加 specialDesignatorClasses 之類，才需要 XML PatchOperation。 自訂分頁太多會讓玩家選單雜亂——能借用既有分頁就借用。
9 · 連線建築（CompPipe 概念與 graphic linking）

「連線建築」指相鄰時會自動拼接外觀或共享資源的建築，分兩個獨立面向： 外觀拼接（graphic linking）與資源網路（如電力、或 mod 自訂的管線網）。

外觀拼接：graphic linking

牆、電纜、管線那種「會自動和鄰居接起來」的視覺，靠 Graphic_Linked 系列實現。 引擎依相鄰同類建築選擇對應的接合貼圖（直線、轉角、T 字、十字）。 

<graphicData>
  <texPath>Things/Building/Linked/MyPipe_Atlas</texPath>
  <!-- Graphic_LinkedTransmitter：和電纜一樣的接線視覺 -->
  <graphicClass>Graphic_LinkedTransmitter</graphicClass>
  <linkType>Transmitter</linkType>
  <linkFlags>
    <li>Custom1</li>   <!-- 只和帶相同 flag 的鄰居拼接 -->
  </linkFlags>
</graphicData>
<building>
  <linkFlags>
    <li>Custom1</li>
  </linkFlags>
</building>

資源網路：CompPipe 概念

RimWorld 核心沒有通用「管線」Comp——電力的 PowerNet 是寫死的特例。 DLC（如 Biotech 的氣體管）與大型 mod（PRF、VFE）各自實作了自己的管網系統， 社群慣稱這類為 CompPipe / CompResource。核心思路與 PowerNet 同構：

節點 Comp

每個接網建築掛一個 Comp，持有對所屬 Net 的參照，並在 PostSpawnSetup / PostDeSpawn 時通知網重建。

Net Manager

一個 MapComponent 維護所有 Net，用 flood-fill 從相鄰節點重建連通分量，每 tick 結算供需。

// 自訂管網節點 Comp 的骨架（概念示意）
public class CompPipe : ThingComp
{
    public PipeNet net;   // 由 MapComponent 指派；未連網時為 null

    public override void PostSpawnSetup(bool respawningAfterLoad)
    {
        base.PostSpawnSetup(respawningAfterLoad);
        // 通知地圖的管網管理器：有新節點，請重建受影響的網
        parent.Map.GetComponent<PipeNetManager>()?.RegisterNode(this);
    }

    public override void PostDeSpawn(Map map)
    {
        map.GetComponent<PipeNetManager>()?.DeregisterNode(this);
        base.PostDeSpawn(map);
    }
}
要做管網？先看 PipeSystem 函式庫 社群開源的 PipeSystem（Vanilla Expanded 系列依賴）已把這套抽象做成可重用框架。 除非要深度客製，否則直接依賴它比自己重寫一遍 PowerNet 划算得多。
10 · 常見坑

坑 1 PowerNet 為 null

建築剛放下、或從未接到電纜時，CompPowerTrader.PowerNet 是 null。 在 SpawnSetup 當下、或孤立建築上存取 PowerNet.xxx() 會直接 NRE。 

// ❌ 危險
float g = powerComp.PowerNet.CurrentEnergyGainRate();

// ✅ 一律判空
PowerNet net = powerComp?.PowerNet;
if (net != null)
    float g = net.CurrentEnergyGainRate();

坑 2 忽略 respawningAfterLoad

SpawnSetup 在「首次放置」與「讀檔重建」兩種情境都會呼叫。 把一次性初始化（給予初始資源、播放放置特效、註冊到管理器計數）寫在沒判斷 flag 的分支， 會在每次讀檔時重複執行，造成存檔污染或數值翻倍。 

public override void SpawnSetup(Map map, bool respawningAfterLoad)
{
    base.SpawnSetup(map, respawningAfterLoad);

    if (!respawningAfterLoad)
    {
        // ✅ 只有真正首次放置才做：發特效、給初始狀態
        FleckMaker.Static(this.Position, map, FleckDefOf.PsycastAreaEffect);
    }

    // 兩種情境都要做的（取 Comp、註冊 tick 監聽）放外面
    powerComp = this.TryGetComp<CompPowerTrader>();
}

坑 3 忘記呼叫 base.SpawnSetup

漏掉 base.SpawnSetup(map, respawningAfterLoad)，Comp 不會 PostSpawnSetup、 電力不會接網、繪製器不會註冊。症狀通常是「建築看不見」或「TryGetComp 回 null」。 所有覆寫的生命週期方法都要記得鏈式呼叫 base。

坑 4 在 ExposeData 漏存 Comp 自身狀態的誤解

Comp 的狀態由 Comp 自己的 PostExposeData 處理，不需要你在 Building 的 ExposeData 裡手動存。你只需存 Building class 上自己宣告的欄位。 重複存反而會出現「節點重複」之類的讀檔警告。

坑 5 耗電/發電正負號搞反

XML basePowerConsumption：正=耗電。 C# PowerOutput：負=耗電。兩邊相反。 若設定後機器反而在「發電」，十之八九是把 C# 端的符號寫成正的了。

坑 6 Draw 裡做重運算

Draw() 每幀呼叫（非每 tick），且和畫面更新率綁定。 在裡面做尋路、查 PowerNet 遍歷、配置記憶體都會明顯掉幀。 把計算放 TickRare 算好快取，Draw 只負責畫。

上線前 Checklist
  1. 所有覆寫方法都呼叫了對應的 base.XXX()
  2. 取 Comp 在 base.SpawnSetup 之後，且結果判空
  3. 存取 PowerNet 前判 null
  4. 一次性初始化包在 if (!respawningAfterLoad)
  5. tickerType 與你實際覆寫的 tick 方法相符
  6. 正負號：XML 正=耗電 / C# 負=耗電
  7. 自訂欄位寫進 ExposeData，且未重複存 Comp 狀態
