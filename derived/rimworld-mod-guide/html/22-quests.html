心智模型：兩個世界

RimWorld 的任務系統有兩層需要清楚分離。生成期 (generation-time) 由 QuestScriptDef 的 QuestNode 樹負責——它讀取一個叫 Slate 的變數袋，產出一個具體的 Quest 實例， 實例由一群 QuestPart 組成。執行期 (runtime) 則完全由 QuestPart 接收 signal（字串訊號）來推進——QuestNode 此時早已退場。

QuestNode（生成期）

XML 腳本的節點，決定「這次任務要長什麼樣」。執行 RunInt(Slate) 往 Quest 塞入 QuestPart。產出後即丟棄。

QuestPart（執行期）

序列化進存檔的執行單元。透過 Notify_QuestSignalReceived 對 signal 反應，是任務真正的狀態機。

Slate（橋樑）

生成期的變數容器（Dictionary<string, object> 的包裝）。QuestNode 之間靠它傳值，生成結束後不存在。

關鍵分界 Slate 與 QuestNode 是 暫時的，不進存檔；Quest 與 QuestPart 是 持久的，會被 IExposable 序列化。任何要在任務存活期間使用的資料，必須在生成期從 Slate「烘焙」進 QuestPart 的欄位。
1 · QuestScriptDef 與 quest 腳本系統

QuestScriptDef（繼承自 Def）是一個任務的「模板」。它的核心是一棵 <root> 節點樹， 外加 QuestGen 相關的篩選與權重欄位。RimWorld 的 QuestManager 與 Storyteller 會週期性地挑選符合條件的 QuestScriptDef 來生成任務。

最小可用的 QuestScriptDef XML

<?xml version="1.0" encoding="utf-8"?>
<Defs>
  <QuestScriptDef>
    <defName>MyMod_RescueColonist</defName>

    <!-- 是否能被 storyteller 自然觸發（false = 只能由 incident/C# 觸發） -->
    <rootSelectionWeight>1</rootSelectionWeight>
    <rootMinPoints>400</rootMinPoints>

    <!-- 任務金額（points）→ 獎勵價值的曲線 -->
    <rootMinProgressScore>0</rootMinProgressScore>

    <!-- root：整個任務的節點樹入口 -->
    <root Class="QuestNode_Sequence">
      <nodes>

        <!-- 1. 取得一個目標據點 -->
        <li Class="QuestNode_GetSiteTile">
          <storeAs>siteTile</storeAs>
        </li>

        <!-- 2. 設定任務說明文字 -->
        <li Class="QuestNode_SetQuestNameAndDescription">
          <name>救援殖民者</name>
          <description>有人受困於 [siteTile_factionName]…</description>
        </li>

      </nodes>
    </root>
  </QuestScriptDef>
</Defs>
學習捷徑：抄官方腳本 RimWorld 本體把所有任務腳本放在 Data/Core/Defs/QuestScriptDefs/ 與 Data/Royalty|Ideology|Biotech/Defs/QuestScriptDefs/。 OpportunitySite_ItemStash.xml 與 Hospitality_* 是極佳的範本。 用 dnSpy 反編譯 RimWorld assembly 對照各 QuestNode_* 的實作。
2 · QuestNode — 腳本節點樹

QuestNode 是抽象基底，每個節點代表生成期的一個動作。核心契約是兩個方法： TestRunInt(Slate)（試跑，回傳 bool 判斷此腳本在當前條件下能否生成）與 RunInt(Slate)（實跑，真正對 Quest 塞入 QuestPart 或對 Slate 寫值）。 

// RimWorld 內部的 QuestNode 抽象（簡化示意）
public abstract class QuestNode
{
    // 試跑：判斷能否生成（不可有副作用，但可寫 Slate 的試算值）
    protected abstract bool TestRunInt(Slate slate);

    // 實跑：真正執行生成邏輯
    protected abstract void RunInt();

    // 對外入口（內部會設定 QuestGen.slate 後呼叫上面兩個）
    public bool TestRun(Slate slate) { /* ... */ }
    public void Run() { /* ... */ }
}

三類常見節點

QuestNode_Sequence

容器節點。依序執行 <nodes> 內的子節點。任一子節點 TestRun 失敗，整個序列失敗。

QuestNode_Root_*

root 通常是 Sequence，但官方常用 QuestNode_Root_* 這類專屬根節點封裝一整套生成邏輯（如 QuestNode_Root_ItemStash）。

條件節點

QuestNode_Chance、QuestNode_RandomNode、QuestNode_If、QuestNode_GetFaction 等，控制分支與篩選。

條件分支範例：QuestNode_If 與 QuestNode_Chance

<root Class="QuestNode_Sequence">
  <nodes>

    <!-- 取一個敵對派系，存到 Slate 的 enemyFaction -->
    <li Class="QuestNode_GetFaction">
      <storeAs>enemyFaction</storeAs>
      <allowEnemy>true</allowEnemy>
      <allowAlly>false</allowAlly>
    </li>

    <!-- 50% 機率加碼一個額外的威脅 -->
    <li Class="QuestNode_Chance">
      <chance>0.5</chance>
      <node Class="QuestNode_Sequence">
        <nodes>
          <li Class="QuestNode_Raid">
            <faction>$enemyFaction</faction>
            <points>$points</points>
          </li>
        </nodes>
      </node>
    </li>

  </nodes>
</root>
$變數 vs [變數] 在 XML 中，$slateName 是把 Slate 變數整顆物件注入欄位（SlateRef 解析，見第 4 節）； [slateName] 是描述文字裡的字串內插（會呼叫 ToString / grammar resolver）。兩者用途不同，別混用。
3 · QuestPart — 任務的執行單元

QuestPart 是任務在執行期的最小單位，會被序列化進存檔。一個 Quest 就是 List<QuestPart>。生成期由 QuestNode 把 QuestPart Add 到 Quest 上； 執行期所有 QuestPart 都會收到任務廣播的每一個 signal。

QuestPart 基底重點成員

成員說明
quest反向引用所屬的 Quest
InSignal*觸發此 part 行動的 signal（如 inSignal）
Notify_QuestSignalReceived(Signal)核心：收到 signal 時被呼叫，是行為入口
QuestPartStateChanged / Enable / Disablepart 的啟用狀態管理
ExposeData()IExposable：序列化自己的欄位
QuestObjectsToTexts / questLookTargetsUI 顯示用的相關物件與描述

三個必懂的內建 QuestPart

QuestPart_Choice

玩家在任務接受時的「選項」。每個 choice 帶一組獎勵 (Reward)，玩家選一個。內含 QuestPart_Choice.Choice 子物件。

QuestPart_GiveRewards

收到指定 signal 時實際發放獎勵。底層委派給 RewardsGenerator 產出的 Reward 清單。

QuestPart_*Signal

QuestPart_PassOutInterval、QuestPart_Filter、QuestPart_Delay 等——signal 的轉接、過濾與計時器。

QuestPart 的生命週期（執行期）

// 概念示意：Quest 廣播 signal 給所有 part
public class Quest : IExposable
{
    public List<QuestPart> parts = new List<QuestPart>();

    public void Notify_SignalReceived(Signal signal)
    {
        for (int i = 0; i < parts.Count; i++)
        {
            if (parts[i].State == QuestPartState.Enabled)
                parts[i].Notify_QuestSignalReceived(signal);
        }
    }
}
4 · Slate 與 SlateRef — 變數傳遞

Slate 是生成期的鍵值容器（內部是 Dictionary<string, object>）。 QuestNode 之間透過 Slate 共享資料：上游節點 Set("siteTile", tile)，下游節點 Get<int>("siteTile")。 呼叫方在生成前通常會預先塞入 points、map 等共同變數。

Slate 常用 API

// 寫入
slate.Set("enemyFaction", faction);
slate.Set("points", 1500f);

// 讀取（強型別；找不到回 default 或丟例外，看多載）
Faction f      = slate.Get<Faction>("enemyFaction");
float   points  = slate.Get<float>("points", 0f);   // 第二參數為預設值

// 存在性檢查（TestRun 中極常用）
if (!slate.Exists("map")) return false;

// TryGet 模式
if (slate.TryGet<Map>("map", out var map)) { /* ... */ }

SlateRef<T> — 讓 QuestNode 欄位接受 XML 變數或字面值

SlateRef<T> 是 QuestNode 欄位的型別包裝。它讓同一個欄位能接受兩種寫法： 直接寫字面值（<chance>0.5</chance>），或寫 $變數 引用 Slate （<faction>$enemyFaction</faction>）。在 C# 中用 .GetValue(slate) 解析。 

using RimWorld.QuestGen;

public class QuestNode_MyThreat : QuestNode
{
    // XML: <faction>$enemyFaction</faction> 或 <faction>Pirate</faction>
    public SlateRef<Faction> faction;

    // XML: <points>$points</points>
    public SlateRef<float> points;

    // 把生成結果寫回哪個 Slate key
    public SlateRef<string> storeAs;

    protected override bool TestRunInt(Slate slate)
    {
        // 解析 SlateRef：傳入 slate 取得實際值
        return faction.GetValue(slate) != null
            && points.GetValue(slate) > 0f;
    }

    protected override void RunInt()
    {
        Slate slate = QuestGen.slate;
        Faction f = faction.GetValue(slate);
        float pts = points.GetValue(slate);

        // ...產生 QuestPart 並 Add 到 QuestGen.quest...
    }
}
TestRunInt 不能有真副作用 TestRunInt 可能被呼叫多次（storyteller 評估多個候選腳本），只能讀 Slate 或寫「試算用」的值。 真正建立 QuestPart、消耗資源、產生地圖物件，一律放 RunInt。
5 · 從 C# 生成 quest

不想透過 storyteller 隨機觸發，而要由你的 mod 程式碼主動發任務時， 最直接的入口是 QuestUtility.GenerateQuestAndMakeAvailable。它一步到位：跑生成流程、把任務加入 Find.QuestManager、並（依參數）發出可接受通知。 

using RimWorld;
using RimWorld.QuestGen;
using Verse;

public static class MyQuestSpawner
{
    public static Quest FireRescueQuest(float points)
    {
        // 1. 取得腳本 def
        QuestScriptDef script = DefDatabase<QuestScriptDef>
            .GetNamed("MyMod_RescueColonist");

        // 2. 準備 Slate：放入生成期需要的共同變數
        Slate slate = new Slate();
        slate.Set("points", points);
        slate.Set("map", Find.CurrentMap);

        // 3. 生成並掛上（一步完成）
        Quest quest = QuestUtility
            .GenerateQuestAndMakeAvailable(script, slate);

        // 4. （選用）跳出可接受通知信
        if (!quest.hidden)
        {
            QuestUtility.SendLetterQuestAvailable(quest);
        }

        return quest;
    }
}

更底層：手動跑 QuestGen.Generate

若你需要在掛上 QuestManager 之前先檢查生成結果，可以直接呼叫 QuestGen.Generate， 再自己決定要不要 Find.QuestManager.Add(quest)。 

QuestScriptDef script = /* ... */;
Slate slate = new Slate();
slate.Set("points", 1200f);

// 只生成，不掛上 QuestManager
Quest quest = QuestGen.Generate(script, slate);

// 自行檢視 / 決定
if (quest.parts.Count > 0)
{
    Find.QuestManager.Add(quest);
}
必須先驗證 CanRun 在生成前用 script.CanRun(slate, map)（或自己跑一次 TestRun）確認腳本在當前 Slate 條件下可生成。 直接對不滿足前提的腳本呼叫 Generate，可能產出殘缺任務（缺 QuestPart）或在中途丟例外。
6 · QuestGen 與生成流程

QuestGen 是生成期的靜態上下文。當一個任務正在生成時， QuestGen.quest（正在組裝的 Quest）與 QuestGen.slate（當前 Slate） 都是執行緒區域的「環境變數」，所有 QuestNode 的 RunInt 都靠它們存取共享狀態。

生成流程（時序）

  1. 呼叫端準備 Slate（塞入 points / map 等），呼叫 QuestGen.Generate(script, slate)。
  2. QuestGen 建立空白 Quest，設定 QuestGen.quest 與 QuestGen.slate。
  3. 對 script.root 先 TestRun（試跑全樹），任一節點失敗則整體生成失敗。
  4. 試跑通過後 Run（實跑），各 QuestNode 在 RunInt 中往 QuestGen.quest.AddPart(...) 加入 QuestPart。
  5. 解析所有描述文字的 [變數] 內插、計算 reward、產生 initiateSignal 等。
  6. 回傳組裝完成的 Quest；清空 QuestGen 的環境變數。

在 QuestNode 中加入 QuestPart

protected override void RunInt()
{
    Quest quest = QuestGen.quest;
    Slate slate = QuestGen.slate;

    // 產生唯一 signal 名稱（避免和其他 part 撞名）
    string completeSignal = QuestGen.GenerateNewSignal("RescueComplete");

    // 建立自訂 QuestPart 並掛上
    var part = new QuestPart_TrackRescue
    {
        inSignalEnable = slate.Get<string>("inSignal"),
        outSignalComplete = completeSignal,
        targetPawn = slate.Get<Pawn>("victim")
    };
    quest.AddPart(part);

    // 把產出的 signal 寫回 Slate，供後續節點接續
    slate.Set("rescueCompleteSignal", completeSignal);
}
永遠用 QuestGen.GenerateNewSignal 手寫 signal 字串常數會在同一任務有多個同類 part 時撞名。 QuestGen.GenerateNewSignal("Prefix") 會回傳帶唯一序號的字串（如 "Quest42.Prefix"），保證不衝突。
7 · 獎勵系統

獎勵的核心是 RewardsGenerator：給它一個 RewardsGeneratorParams（含 rewardValue、giverFaction 等），它回傳一組 Reward 物件。 每個 Reward 能產出對應的 QuestPart（透過 GenerateQuestParts）， 把「發放動作」掛進任務。

Reward 類型一覽

類型內容底層 QuestPart
Reward_Items一批物品（武器、材料、藝術品…）QuestPart_DropPods / GiveItems
Reward_Pawn加入殖民地的 pawn（俘虜、難民、動物）QuestPart_Give...（依情境）
Reward_Goodwill派系好感度變化QuestPart_FactionGoodwillChange
Reward_RoyalFavor帝國榮譽（Royalty）QuestPart_GiveRoyalFavor
Reward_VomitInfo純資訊獎勵（地圖情報等）對應 info part

在 C# 中產生獎勵並掛上任務

using RimWorld;
using RimWorld.QuestGen;

protected override void RunInt()
{
    Quest quest = QuestGen.quest;
    Slate slate = QuestGen.slate;

    // 1. 設定獎勵參數
    var parms = new RewardsGeneratorParams
    {
        rewardValue   = slate.Get<float>("points") * 0.7f,
        giverFaction  = slate.Get<Faction>("enemyFaction"),
        chosenPawnSignal = null
    };

    // 2. 產生 reward 清單
    List<Reward> rewards =
        new RewardsGenerator().Generate(parms).ToList();

    // 3. 用 QuestPart_Choice 包成「玩家選一個」
    var choicePart = new QuestPart_Choice
    {
        inSignalChoiceUsed = slate.Get<string>("inSignal")
    };

    var choice = new QuestPart_Choice.Choice();
    foreach (Reward r in rewards)
    {
        choice.rewards.Add(r);
        // 每個 reward 把自己的發放 QuestPart 掛到 quest
        choice.questParts.AddRange(
            r.GenerateQuestParts(0, parms,
                /* customLetterLabel */ null,
                /* customLetterText  */ null,
                /* chosenPawnSignal  */ null,
                /* useChosenPawnsForRef */ null));
    }
    choicePart.choices.Add(choice);

    quest.AddPart(choicePart);
}

XML 端的等價寫法（更常用）

多數情況不必手寫 C#，官方提供現成 QuestNode 包好整套獎勵流程。

<li Class="QuestNode_GiveRewards">
  <parms>
    <rewardValue>$rewardValue</rewardValue>
  </parms>
  <inSignal>$questCompleteSignal</inSignal>
</li>
8 · 自訂 QuestPart

要表達內建 QuestPart 無法涵蓋的任務行為，就繼承 QuestPart，覆寫 Notify_QuestSignalReceived，並務必正確實作 ExposeData（否則讀檔崩潰）。 下面是一個完整、可直接套用的範例：追蹤目標 pawn 是否被救回殖民地。 

using System.Collections.Generic;
using RimWorld;
using RimWorld.QuestGen;
using Verse;

public class QuestPart_TrackRescue : QuestPart
{
    // ── 由 QuestNode 在生成期填入的欄位 ──
    public string inSignalEnable;     // 收到此 signal 才啟用追蹤
    public string inSignalRescued;    // pawn 被救回時觸發
    public string inSignalVictimDied;  // pawn 死亡時觸發
    public string outSignalComplete;   // 任務成功時對外廣播
    public string outSignalFailed;     // 任務失敗時對外廣播
    public Pawn   targetPawn;

    private bool active;             // 是否已啟用追蹤（會存檔）

    // 讓這些 signal 在 part 啟用狀態下也能被接收
    public override IEnumerable<GlobalTargetInfo> QuestLookTargets
    {
        get
        {
            if (targetPawn != null)
                yield return targetPawn;
        }
    }

    public override void Notify_QuestSignalReceived(Signal signal)
    {
        base.Notify_QuestSignalReceived(signal);

        // 1. 啟用追蹤
        if (signal.tag == inSignalEnable)
        {
            active = true;
            return;
        }

        if (!active) return;

        // 2. 成功：對外廣播 complete signal（其他 part 接力發獎）
        if (signal.tag == inSignalRescued)
        {
            Find.SignalManager.SendSignal(
                new Signal(outSignalComplete, signal.args));
            active = false;
        }
        // 3. 失敗
        else if (signal.tag == inSignalVictimDied)
        {
            Find.SignalManager.SendSignal(
                new Signal(outSignalFailed, signal.args));
            active = false;
        }
    }

    // ── 序列化：每個欄位都要 Look，否則讀檔後變 null ──
    public override void ExposeData()
    {
        base.ExposeData();
        Scribe_Values.Look(ref inSignalEnable,    "inSignalEnable");
        Scribe_Values.Look(ref inSignalRescued,   "inSignalRescued");
        Scribe_Values.Look(ref inSignalVictimDied,"inSignalVictimDied");
        Scribe_Values.Look(ref outSignalComplete, "outSignalComplete");
        Scribe_Values.Look(ref outSignalFailed,   "outSignalFailed");
        Scribe_Values.Look(ref active,            "active", false);

        // 引用遊戲物件：用 Scribe_References（存的是 loadID）
        Scribe_References.Look(ref targetPawn, "targetPawn");
    }
}
ExposeData 是讀檔崩潰的頭號來源 漏掉任一欄位的 Scribe_*.Look，讀檔後該欄位為 null， Notify_QuestSignalReceived 比對 signal tag 時就 silent fail，任務永遠卡住。 引用 Thing / Pawn / Faction 一律用 Scribe_References（存 loadID）， 不要用 Scribe_Deep（會把整個物件複製進存檔）。
9 · Signal 驅動的任務狀態機

任務執行期沒有中央「腳本直譯器」。整個任務就是一群 QuestPart，靠 Signal（一個 tag 字串 + SignalArgs 參數）互相串接成的事件驅動狀態機。 一個 part 的 out-signal 是另一個 part 的 in-signal，如此鏈式推進。

Signal 流向（典型救援任務）

// 用文字描述 signal 鏈（→ 表示 out-signal 觸發 in-signal）

quest.Initiate
   └─→ QuestPart_Choice           // 玩家接受任務、選獎勵
          └─→ inSignalEnable
                 └─→ QuestPart_TrackRescue (active=true)
                        ├─ pawn.Rescued  ─→ outSignalComplete
                        │                       └─→ QuestPart_GiveRewards
                        │                       └─→ QuestPart_End (Success)
                        └─ pawn.Died     ─→ outSignalFailed
                                                └─→ QuestPart_End (Failed)

發出與接收 signal

// 發出 signal（任何地方）
Find.SignalManager.SendSignal(new Signal("MyMod.CustomEvent"));

// 帶參數的 signal（傳遞上下文給下游 part）
var args = new SignalArgs(
    new NamedArgument(rescuedPawn, "SUBJECT"));
Find.SignalManager.SendSignal(new Signal(outSignalComplete, args));

// Quest 也提供便利方法（會自動加 quest 前綴）
quest.End(QuestEndOutcome.Success);

內建的 signal 轉接 part

QuestPart作用
QuestPart_Delay收到 in-signal 後延遲 N tick，再發 out-signal（任務倒數計時）
QuestPart_Filter條件過濾：滿足才轉發 signal（用來做 if 分支）
QuestPart_PassAll / PassAny多個 in-signal 全到 / 任一到，才發 out-signal（AND / OR 邏輯閘）
QuestPart_SignalPassWithFaction轉發 signal 並夾帶 faction 參數
QuestPartActivable有 enable/disable 狀態的 part 基底（你的自訂 part 常繼承這個而非裸 QuestPart）

坑 1 Quest 根本不出現

storyteller 沒選中你的腳本，幾乎都是篩選條件沒過。逐項排查：

  • rootSelectionWeight 為 0 或缺失 → 永遠不會被自然挑中（這類腳本只能由 C# / incident 觸發）。
  • rootMinPoints 高於當前 storyteller 給的 points → 跳過。
  • root 樹中任一節點 TestRun 回 false（如找不到合適 faction / site）→ 整個腳本作廢。
  • 用 dev mode 的 「Execute quest...」 或 「Make quest」 debug action 強制生成，能立刻分辨是「篩選沒過」還是「生成出錯」。
// 在 C# 中診斷：自己跑一次 TestRun 看哪個節點擋下
bool ok = script.root.TestRun(slate);
Log.Message($"[MyMod] {script.defName} TestRun = {ok}, "
    + $"slate has map={slate.Exists("map")}, points={slate.Exists("points")}");

坑 2 Signal 名稱對不上 → 任務卡死

in-signal 與 out-signal 是純字串比對，差一個字、大小寫不符，或忘了加 quest 前綴，signal 就永遠送不到目標 part， 任務停在某狀態不動，也不報錯。

  • 一律用 QuestGen.GenerateNewSignal("Name") 產生 signal，把結果存進 Slate 傳給下游 part，不要兩端各自硬寫字串常數。
  • 內建事件 signal（如 pawn 被俘、死亡）名稱要從官方 QuestPart 反編譯確認，例如 QuestPart_RequirementsToAcceptThingCount 等。
  • Debug：在 Notify_QuestSignalReceived 開頭 Log.Message($"recv {signal.tag}")，對照你期望的 tag。

坑 3 Slate 變數缺失 → 生成期 NRE

下游節點 slate.Get<T>("foo") 取一個上游沒設定的 key，依多載不同會回 default（null）或丟例外， 接著就是一連串 NullReference。

  • 節點開頭先 if (!slate.Exists("foo")) return false;（在 TestRunInt 中守門）。
  • 呼叫端生成前，務必塞齊腳本要求的共同變數（最常漏 map 與 points）。
  • 用有預設值的多載 slate.Get<float>("points", 0f) 降低硬崩潰風險，但別用它掩蓋真正的設定遺漏。

坑 4 描述文字 [變數] 顯示成原始字串

任務描述裡寫了 [siteTile_factionName] 但畫面照原樣顯示中括號，代表該 Slate 變數在生成期沒被設定， 或變數名拼錯。文字內插在生成期一次性解析——生成後才設定的值不會反映到描述上。

坑 5 讀檔後自訂 QuestPart 行為消失

通常是 ExposeData 漏 Look 某欄位（見第 8 節）。signal 字串欄位變 null 後， 比對永遠不成立，part 形同失效。存檔讀回後任務「卡住不前進」是這個坑的典型症狀。
