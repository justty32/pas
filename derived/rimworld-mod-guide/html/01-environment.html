環境設定

IDE 配置、Assembly 參考、反編譯工具、開發迴圈——一次到位，之後不再碰設定。

Visual Studio 2022 / JetBrains Rider · .NET Framework 4.7.2
1 · Visual Studio / Rider 專案設定

為什麼是 .NET Framework 4.7.2？

RimWorld 跑在 Unity 的 Mono 執行期（非 CoreCLR）。 Unity 2022 LTS 內建的 Mono 對應 .NET Framework 4.7.2 的 API surface， 因此你的 Class Library 目標框架必須是 net472。 .NET 5 / 6 / 8 編譯出來的 assembly 格式（PE 特徵、runtime directive）Mono 無法載入， 會在遊戲啟動時拋 BadImageFormatException。

常見誤區 .NET Standard 2.0 Class Library 看似相容，但它的 TFM 是 netstandard2.0， NuGet restore 會拉 .NET Standard 版本的傳遞依賴，最終 DLL 仍可能引入 CoreCLR 限定 API。 直接用 net472，避免隱性問題。

建立專案（Visual Studio 2022）

  1. 新增專案 → 類別庫 (.NET Framework)，框架選 .NET Framework 4.7.2。
  2. 建議命名為 YourName.YourMod（與 packageId 一致，方便追蹤）。
  3. 刪除自動生成的 Class1.cs。
  4. 在 專案屬性 → 建置 → 輸出路徑 填入 mod 資料夾的絕對路徑：
    C:\...\RimWorld\Mods\YourMod\Assemblies\（末尾加反斜線）。
  5. 將 平台目標 改為 x64（RimWorldWin64 是 64-bit 執行檔）。
輸出路徑直接指向 Assemblies/ 設定後，每次 Build 都會把 DLL 直接放到 mod 目錄，免去手動複製。 啟動 RimWorld 即可測試最新版。

加入 Assembly References

所有參考 DLL 都位於 RimWorld/RimWorldWin64_Data/Managed/（Steam 預設路徑 C:\Program Files (x86)\Steam\steamapps\common\RimWorld\RimWorldWin64_Data\Managed\）。 每一個 Reference 都必須將 Copy Local 設為 false， 否則建置輸出會把遊戲 DLL 複製到你的 Assemblies/ 目錄，造成版本衝突。

DLL 檔名必要性用途
Assembly-CSharp.dll 必要 所有 RimWorld 遊戲類別（ThingDef、Pawn、Map…）
0Harmony.dll 必要 Harmony 2.x 修補框架，遊戲已內建，不要打包
UnityEngine.CoreModule.dll 必要 Vector2/3、Color、GameObject 等 Unity 核心型別
UnityEngine.IMGUIModule.dll UI 開發 GUI、GUIStyle、GUILayout
UnityEngine.TextRenderingModule.dll UI 開發 Font、TextMesh
UnityEngine.AudioModule.dll 音效 AudioClip、AudioSource

.csproj 範例片段

以下是手工編輯後的典型 .csproj（SDK-style 之前的舊格式，net472 仍用舊格式）：

<Project ToolsVersion="15.0" xmlns="http://schemas.microsoft.com/developer/msbuild/2003">
  <PropertyGroup>
    <Configuration Condition="'$(Configuration)'==''">Debug</Configuration>
    <Platform Condition="'$(Platform)'==''">x64</Platform>
    <TargetFrameworkVersion>v4.7.2</TargetFrameworkVersion>
    <AssemblyName>YourName.YourMod</AssemblyName>
    <OutputType>Library</OutputType>
  </PropertyGroup>

  <PropertyGroup Condition="'$(Configuration)'=='Debug'">
    <!-- 輸出直接進 mod 目錄，省去手動複製 -->
    <OutputPath>..\..\Mods\YourMod\Assemblies\</OutputPath>
    <DebugType>portable</DebugType>
    <DebugSymbols>true</DebugSymbols>
  </PropertyGroup>

  <PropertyGroup Condition="'$(Configuration)'=='Release'">
    <OutputPath>..\..\Mods\YourMod\Assemblies\</OutputPath>
    <Optimize>true</Optimize>
  </PropertyGroup>

  <ItemGroup>
    <!-- RimWorld 主要 API ─────────────────────────────────────────── -->
    <Reference Include="Assembly-CSharp">
      <HintPath>$(RIMWORLD_PATH)\RimWorldWin64_Data\Managed\Assembly-CSharp.dll</HintPath>
      <Private>false</Private>    <!-- Copy Local = false -->
    </Reference>

    <!-- Harmony (遊戲已內建，絕對不能 Copy Local) ──────────────── -->
    <Reference Include="0Harmony">
      <HintPath>$(RIMWORLD_PATH)\RimWorldWin64_Data\Managed\0Harmony.dll</HintPath>
      <Private>false</Private>
    </Reference>

    <!-- Unity 核心 ──────────────────────────────────────────────── -->
    <Reference Include="UnityEngine.CoreModule">
      <HintPath>$(RIMWORLD_PATH)\RimWorldWin64_Data\Managed\UnityEngine.CoreModule.dll</HintPath>
      <Private>false</Private>
    </Reference>

    <Reference Include="UnityEngine.IMGUIModule">
      <HintPath>$(RIMWORLD_PATH)\RimWorldWin64_Data\Managed\UnityEngine.IMGUIModule.dll</HintPath>
      <Private>false</Private>
    </Reference>
  </ItemGroup>

  <ItemGroup>
    <Compile Include="**\*.cs" />
  </ItemGroup>

  <Import Project="$(MSBuildToolsPath)\Microsoft.CSharp.targets" />
</Project>
使用環境變數 RIMWORLD_PATH 在系統環境變數加一條 RIMWORLD_PATH=C:\...\RimWorld， HintPath 就可以跨機器共用（CI / 多台電腦）。 Visual Studio 重啟後即可讀取。

Rider 使用者的注意事項

Rider 可直接開啟 .csproj，所有設定與 Visual Studio 相同。 例外：Rider 的「Output Path」對話框會把路徑寫成絕對路徑； 建議直接手動編輯 .csproj XML 使用相對路徑或環境變數， 避免把本機絕對路徑 commit 進版本庫。

2 · 反編譯工具

RimWorld 的核心邏輯封裝在 Assembly-CSharp.dll（閉源）， 但 IL（Intermediate Language）可完整反編譯回接近原始的 C#。 搞清楚某個系統怎麼跑，幾乎都靠反編譯，沒有捷徑。

dnSpy 推薦

開源的 .NET 除錯器兼反編譯器。除了閱讀反編譯結果， 更支援 Attach to Process（附加到執行中的 RimWorld）， 在反編譯出來的假 source 上設斷點，實際命中時暫停遊戲進程並顯示區域變數。 是 modder 的首選利器。

下載：github.com/dnSpy/dnSpy（已停止主線維護，但功能完整）
或使用社群 fork dnSpyEx（持續更新）

ILSpy 輕量替代

官方持續維護，有獨立 GUI 程式與 Visual Studio Extension 兩種形式。 不支援 attach debug，但反編譯品質與搜尋速度俱佳。 ILSpy VS Extension 可讓你在 VS 內對著型別名稱按 F12 直接跳到反編譯結果，工作流程最順暢。

下載：github.com/icsharpcode/ILSpy

Assembly-CSharp.dll 的位置

# Windows Steam
C:\Program Files (x86)\Steam\steamapps\common\RimWorld\RimWorldWin64_Data\Managed\Assembly-CSharp.dll

# macOS Steam
~/Library/Application Support/Steam/steamapps/common/RimWorld/RimWorld.app/Contents/Resources/Data/Managed/Assembly-CSharp.dll

# Linux Steam
~/.steam/steam/steamapps/common/RimWorld/RimWorld_Data/Managed/Assembly-CSharp.dll

dnSpy 使用技巧

  1. 開啟 DLL：File → Open → 選 Assembly-CSharp.dll。 一次可以把整個 Managed 目錄拖進去，dnSpy 會自動掃描。
  2. 搜尋類別：Ctrl+Shift+K → 輸入類別名稱（如 JobGiver_Work）， 支援 camelCase 縮寫（JGW 即可找到）。
  3. 追蹤呼叫鏈：右鍵方法名稱 → Analyze， 左側 Analyzer 視窗會展示「誰呼叫了這個方法」（Used By） 與「這個方法呼叫了誰」（Uses）。 從 Pawn.Tick() 追到底層業務邏輯只需幾次點擊。
  4. Attach to Process debug：啟動 RimWorld， dnSpy → Debug → Attach to Process → 選 RimWorld64.exe。 在反編譯出的方法體內設斷點（F9），當遊戲執行到該 IL 時會暫停。 注意：Debug build 的中斷點命中率較高；Release 優化可能讓某些斷點永遠不命中。
  5. 搜尋字串 / 欄位值：Edit → Search Assemblies， Kind 選 String 或 Field，可快速定位硬編碼的標籤或 defName。
搜尋策略建議 先從 XML Def 的 <thingClass> 或 <workerClass> 欄位找到進入點類別， 再用 Analyzer 展開呼叫樹。不要從 Main() 追，Unity 的生命週期是 MonoBehaviour 驅動的，入口在 Root.Start()（Verse 命名空間）。
3 · 開發迴圈（Development Loop）

XML / Def 更改

改完存檔後，在遊戲內 Dev mode 介面按 Reload Defs， 無需重啟。適用於調整數值、新增 Def 等純資料變更。

C# DLL 更改

必須完全重啟遊戲。Mono 不支援 assembly unload / hot reload， 改了 DLL 進程仍持有舊版本的記憶體映射。

開啟 Dev Mode

有兩種方式，效果相同：

方法 A：啟動參數

在 Steam 右鍵 → 內容 → 啟動選項，填入：

-dev

每次啟動都進入 Dev mode，適合開發中的機器。

方法 B：Preferences.xml

編輯 RimWorld 設定檔，把 <devMode> 改為 True：

# 檔案位置（Windows）
%USERPROFILE%\AppData\LocalLow\Ludeon Studios\
  RimWorld by Ludeon Studios\Config\Preferences.xml

<devMode>True</devMode>

Dev Mode 在畫面上提供什麼

進入遊戲後，右上角會出現一排工具列，關鍵按鈕包括：

按鈕功能Modding 用途
Reload Defs 重新載入所有 XML Def 調整數值後立即看效果，免重啟
Log 打開遊戲內 Log 視窗 看 Log.Error / Log.Warning 輸出
Spawn Pawn 生成任意 Pawn 快速測試自訂 Hediff、ThingComp 行為
Inspect String 滑鼠懸停顯示物件偵錯資訊 確認 Def 是否正確綁定
God Mode 殖民者無敵，建設瞬間完成 跳過遊戲前置，快速到達測試場景

Log 檔案位置

# Windows（Unity Player.log）
%USERPROFILE%\AppData\LocalLow\Ludeon Studios\RimWorld by Ludeon Studios\Player.log

# macOS
~/Library/Logs/Unity/Player.log

# Linux
~/.config/unity3d/Ludeon Studios/RimWorld by Ludeon Studios/Player.log
Log 在開發中的重要性 RimWorld 的 Mod loading 錯誤（型別找不到、Def parse 失敗、Harmony patch 例外） 全都輸出到 Player.log。遊戲閃退時這是第一個要看的地方。 在 C# 裡用 Log.Error("msg")、Log.Warning("msg") 寫出的訊息也會進這個檔，並帶上呼叫堆疊。

推薦輔助 Mod

HugsLib

提供更好用的 Log 視窗（支援篩選、複製、上傳分享）、 Mod setting 框架、更完善的 Harmony bootstrapping。 許多 mod 以它為基底。Steam Workshop 有現成安裝。

RimWorld Dev Tools

擴充 Dev mode 工具列，增加 Pawn property editor、 Hediff 注入介面、Research 一鍵解鎖等。 開發期間大幅縮短從「build」到「看到效果」的時間。

典型開發迴圈（C# Mod）

  1. 在 IDE 修改 C# 邏輯，Build（Ctrl+Shift+B），DLL 自動輸出到 Assemblies/。
  2. 切換到 RimWorld，關閉遊戲（Alt+F4），重新啟動。
  3. 進入存檔或新遊戲，在 Dev Log 確認沒有 Load Error。
  4. 觸發測試場景，觀察行為；若有 Bug，看 Log 或用 dnSpy Attach debug。
  5. 重複。
4 · About.xml — 進階欄位

總覽頁已展示最小範例。這裡補充 modDependencies 與 loadAfter / loadBefore 的正確寫法——這些欄位在 多 mod 互動時至關重要。 

<?xml version="1.0" encoding="utf-8"?>
<ModMetaData>
  <name>My Advanced Mod</name>
  <author>YourName</author>

  <!-- packageId 全域唯一，格式：Author.ModName，只用 ASCII 英數與句點 -->
  <packageId>YourName.MyAdvancedMod</packageId>

  <supportedVersions>
    <li>1.6</li>
  </supportedVersions>

  <!-- 宣告硬性依賴：缺少時啟動器警告，並拒絕啟動遊戲 -->
  <modDependencies>
    <li>
      <packageId>brrainz.harmony</packageId>
      <displayName>Harmony</displayName>
      <steamWorkshopUrl>https://steamcommunity.com/sharedfiles/filedetails/?id=2009463077</steamWorkshopUrl>
      <downloadUrl>https://github.com/pardeike/HarmonyRimWorld/releases</downloadUrl>
    </li>
    <li>
      <packageId>UnlimitedHugs.HugsLib</packageId>
      <displayName>HugsLib</displayName>
      <steamWorkshopUrl>https://steamcommunity.com/sharedfiles/filedetails/?id=818773962</steamWorkshopUrl>
    </li>
  </modDependencies>

  <!-- 宣告載入順序：Def 繼承與 Harmony patch 執行順序依賴此設定 -->
  <loadAfter>
    <li>brrainz.harmony</li>
    <li>UnlimitedHugs.HugsLib</li>
  </loadAfter>

  <!-- 宣告必須在某 mod 之前載入（較少用） -->
  <loadBefore>
    <li>SomeOther.CompatMod</li>
  </loadBefore>

  <description>An example mod with proper dependency declarations.</description>
</ModMetaData>
modDependencies vs loadAfter 的差異 modDependencies 是強制需求：依賴的 Mod 若未啟用，RimWorld 啟動器會警告， 甚至拒絕進入遊戲。loadAfter 只影響載入順序，不強制對方存在。 兩者通常需要同時宣告：先用 modDependencies 強制依賴， 再用 loadAfter 確保載入序正確。
5 · 常見坑

Copy Local = true 高頻

預設值是 true，建置時會把 Assembly-CSharp.dll、 0Harmony.dll、UnityEngine.*.dll 都複製到輸出目錄。 遊戲啟動時若在 Assemblies/ 發現這些 DLL，會載入你複製的舊版， 蓋掉執行時的正確版本，導致型別衝突、TypeLoadException、或詭異的執行錯誤。 所有 Reference 的 Copy Local 都必須是 false。

packageId 不唯一 中頻

RimWorld Mod Manager 以 packageId 識別 Mod，重複的 ID 會導致 Mod 載入衝突、存檔損壞、依賴解析失敗。 格式規範：AuthorName.ModName，只用 ASCII 英數字母與句點， 大小寫不敏感。永遠不要使用 example.mod 或 Author.NewMod 這類通用名稱。

.NET Standard / Core 不相容 中頻

Visual Studio 新專案精靈預設可能產生 netstandard2.0 或 net8.0 的 TFM。這些格式 Mono 無法載入，遊戲 Log 會出現 BadImageFormatException 或靜默忽略（DLL 存在但類別不注冊）。 確認 .csproj 內有 <TargetFrameworkVersion>v4.7.2</TargetFrameworkVersion> 而非 <TargetFramework>net472</TargetFramework> （後者是 SDK-style 格式，部分版本 Mono 仍可接受，但前者更保險）。

Harmony 重複載入 中頻

若把 0Harmony.dll 打包進自己的 Assemblies/， 且其他 Mod 也打包了不同版本的 Harmony，CLR 會載入多個版本的 Harmony 到同一 AppDomain， Patch 追蹤表錯亂，導致覆蓋彼此的 Patch 或無限遞迴。 永遠只 Reference，絕不打包 0Harmony.dll。

HintPath 指向絕對路徑 低頻但惱人

把 C:\Program Files (x86)\Steam\... 寫死在 .csproj 後 commit 進版本庫，其他協作者或 CI 機器立刻爆紅。 改用 環境變數（$(RIMWORLD_PATH)） 或相對於 solution 的 libs/ 符號連結目錄。

Debug symbols 缺失 低頻

DebugType 預設是 full（Windows PDB 格式）， Mono 讀不懂，dnSpy Attach debug 時看不到區域變數名稱。 改為 <DebugType>portable</DebugType>， Mono 可讀取，dnSpy 也相容。

存檔相容性警告 修改已存在 Def 的 defName、或刪除 ThingComp、 Hediff 類別後，舊存檔嘗試讀取對應資料時會觸發 NullReferenceException，嚴重時整個存檔損壞。 開發期間用獨立測試存檔，正式發布前絕不重命名已上線的 defName 或類別名稱。
快速檢查清單

每次新建 Mod 專案時，過一遍以下確認事項：

項目正確設定狀態
目標框架 .NET Framework 4.7.2 必查
平台目標 x64（或 Any CPU） 必查
所有 Reference 的 Copy Local false 必查
輸出路徑 指向 Mods/YourMod/Assemblies/ 推薦
DebugType portable 推薦
HintPath 使用環境變數或相對路徑 推薦
About.xml — packageId 全域唯一，符合 Author.ModName 格式 必查
