XML Def 系統

RimWorld 的資料驅動核心——從反序列化原理到 PatchOperation 跨 mod 修補

Chapter 02
§1　Def 是什麼

RimWorld 的幾乎所有遊戲資料——物品、生物、配方、科技、特質——都以 Def 為單位定義。 每一份 XML 節點對應 C# 端一個繼承自 Verse.Def 的類別；遊戲啟動時透過反射將 XML 反序列化成記憶體中的物件圖。

反序列化流程

DeepProfiler.Start() → DirectXmlLoader.LoadModXmlFiles() → XmlInheritance.Resolve() → DefGenerator.GenerateImpliedDefs() → DefDatabase<T>.ResolveAllReferences()

欄位對應規則

XML 子元素名稱直接對應 C# 的 public 欄位（field）或帶有 [LoadAlias] 的屬性名稱。名稱區分大小寫，拼錯只會在 debug log 裡警告，不會拋例外。

Def 的 defName

<defName> 是全域唯一鍵，跨 mod 衝突時後載入的 mod 會覆蓋先前的定義。 命名慣例：ModPrefix_DefName（如 MyCorp_Sword_Plasma）。

反射細節 DirectXmlToObject.ObjectFromXml<T>() 在內部對每個 XML 子節點呼叫 fieldInfo.SetValue()。集合欄位（List<T>）由 DirectXmlToObject.ListFromXml() 逐一 Add；若元素有 Class attribute，則以 Type.GetType() 解析後再反序列化——這就是多型列表的原理。
§2　常用 Def 類型速查表
Def 類型 命名空間 XML 根節點 代表什麼
ThingDef Verse <ThingDef> 物品、建築、植物、屍體（category 區分）
PawnKindDef RimWorld <PawnKindDef> 生物種類（殖民者、野生動物、突襲者族群）
RecipeDef RimWorld <RecipeDef> 工作台配方（製作、外科手術）
ResearchProjectDef RimWorld <ResearchProjectDef> 科技樹節點
TraitDef RimWorld <TraitDef> 殖民者特質（勤勞、暴力狂等）
HediffDef RimWorld <HediffDef> 傷勢、疾病、手術後狀態
JobDef RimWorld <JobDef> AI 任務定義（搭配 JobDriver C# 類別）
WorkTypeDef RimWorld <WorkTypeDef> 殖民者工作欄位（採礦、種植等）
ThoughtDef RimWorld <ThoughtDef> 心情思想（睡好覺了、吃到人肉等）
StatDef RimWorld <StatDef> 能力值定義（移動速度、近戰傷害等）
快速查定義欄位 用 dnSpy / ILSpy 打開 Assembly-CSharp.dll，搜尋對應類別即可看到所有 public 欄位。 也可參考 RimWorld Wiki ThingDef 參考。
§3　ThingDef 詳細範例——自訂武器

以下定義一把「電漿劍」，繼承自原版近戰武器父 Def，帶完整欄位說明。

<!-- Defs/Weapons/MyCorp_Sword_Plasma.xml -->
<Defs>

  <!-- ① 抽象父 Def（本檔自定義，也可繼承原版） -->
  <ThingDef Abstract="True" Name="MyCorp_BasePlasmaBlade">
    <thingClass>ThingWithComps</thingClass>        <!-- C# 實體類別 -->
    <category>Item</category>                        <!-- ThingCategory enum -->
    <drawerType>MapMeshAndRealTime</drawerType>
    <techLevel>Ultra</techLevel>
    <alwaysHaulable>true</alwaysHaulable>
    <rotatable>false</rotatable>
    <equipmentType>Primary</equipmentType>              <!-- 主武器槽 -->
    <weaponTags>
      <li>MeleeWeapon</li>
    </weaponTags>
  </ThingDef>

  <!-- ② 具體 Def，ParentName 繼承上方父 Def -->
  <ThingDef ParentName="MyCorp_BasePlasmaBlade">
    <defName>MyCorp_Sword_Plasma</defName>
    <label>電漿劍</label>
    <description>以電漿束包覆刀刃，切割效率遠超傳統金屬。</description>

    <!-- 圖形設定 -->
    <graphicData>
      <texPath>Things/Item/Equipment/MyCorp/PlasmaSword</texPath>
      <graphicClass>Graphic_Single</graphicClass>
      <drawSize>1.1</drawSize>
    </graphicData>

    <!-- 基礎能力值——對應 StatDef.defName -->
    <statBases>
      <WorkToMake>40000</WorkToMake>
      <Mass>1.0</Mass>
      <MarketValue>2500</MarketValue>
    </statBases>

    <!-- 製作材料 -->
    <costList>
      <ComponentSpacer>4</ComponentSpacer>
      <Plasteel>60</Plasteel>
    </costList>

    <!-- 近戰攻擊列表（多型 List<Tool>） -->
    <tools>
      <li>
        <label>劍身</label>
        <capacities>
          <li>Cut</li>
        </capacities>
        <power>18</power>              <!-- 基礎傷害 -->
        <cooldownTime>1.6</cooldownTime>   <!-- 攻擊間隔（秒） -->
        <armorPenetration>0.3</armorPenetration>
      </li>
      <li>
        <label>刀柄</label>
        <capacities>
          <li>Blunt</li>
        </capacities>
        <power>8</power>
        <cooldownTime>2.0</cooldownTime>
      </li>
    </tools>

    <!-- CompProperties：掛載組件（多型 List<CompProperties>） -->
    <comps>
      <li Class="CompProperties_Forbiddable"/>
    </comps>

    <!-- 科技前置（需對應 ResearchProjectDef.defName） -->
    <recipeMaker>
      <researchPrerequisite>MyCorp_PlasmaForging</researchPrerequisite>
      <skillRequirements>
        <Crafting>8</Crafting>
      </skillRequirements>
    </recipeMaker>
  </ThingDef>

</Defs>

thingClass

指定遊戲實例化時使用的 C# 類別。預設繼承父 Def；可改成自訂類別（需繼承 ThingWithComps 等）。

equipmentType

實務上只用 Primary（主武器槽）。EquipmentType enum 僅有 None 與 Primary，並無 Secondary；盾帶（shield belt）屬於 Apparel 而非 equipment。

weaponTags

用於 PawnKindDef 的 weaponMoney 篩選及敵人武器生成邏輯；自訂 tag 可配合 C# 查詢。

§4　Def 繼承系統

RimWorld 以自己的 XML 繼承機制模擬物件繼承，在 XmlInheritance.Resolve() 階段展開，早於反序列化。

Abstract 父 Def

<ThingDef Abstract="True" Name="BaseWeapon">
  <category>Item</category>
  <alwaysHaulable>true</alwaysHaulable>
</ThingDef>
Abstract="True" 不會進入 DefDatabase 設定 Abstract="True" 的 Def 只作為繼承模板，不會被 DefDatabase<T>.GetNamed() 找到，也不會在遊戲中生成任何實例。

ParentName 繼承與覆蓋

<ThingDef ParentName="BaseWeapon">
  <defName>MyMod_Sword_Iron</defName>
  <!-- 未出現的欄位直接繼承父值 -->
  <!-- 重複定義的欄位：子 Def 覆蓋父 Def -->
  <label>鐵劍</label>
</ThingDef>

多型列表元素（Class attribute）

當 C# 欄位型別是基底類別（如 List<CompProperties>），XML 的 <li> 需指定具體型別：

<comps>
  <!-- Class 必須是能從 Assembly 解析的完整型別名或命名空間縮寫 -->
  <li Class="CompProperties_Power">
    <compClass>CompPowerTrader</compClass>
    <basePowerConsumption>200</basePowerConsumption>
  </li>
  <li Class="CompProperties_Flickable"/>
</comps>

繼承規則摘要

情況行為
子 Def 未定義某欄位 直接沿用父 Def 的值
子 Def 重新定義純量欄位（string、int、float） 子值覆蓋父值
子 Def 重新定義集合欄位（List、dict） 子集合完整取代父集合（非合併）
多層繼承 支援，Name 可跨檔案引用；解析順序由載入順序決定
跨 mod 繼承 可以，但須確保父 mod 在 loadAfter 之前
§5　PatchOperation（XML Patching）

PatchOperation 讓你不修改原始 Def 檔案就能變更任何 Def 的內容，是 mod 相容性的關鍵機制。 Patch 檔案放在 mod 的 Patches/ 資料夾，在所有 mod 的 XML 載入合併後、XmlInheritance.Resolve()（繼承展開）前套用。

常用操作類型

PatchOperationAdd

在 XPath 指向的節點內部插入新子節點（預設加在末尾，可用 order 指定 Prepend）。

PatchOperationRemove

刪除 XPath 指向的節點（含其所有子節點）。

PatchOperationReplace

用 <value> 內的 XML 完整取代 XPath 指向的節點。

PatchOperationAddOrReplace

目標存在則取代，不存在則新增——最安全的更新方式。

XPath 語法速查

XPath 片段意義
Defs/ThingDef[defName="Steel_Sword"] 選取 defName 為 Steel_Sword 的 ThingDef 節點
.../statBases 選取 statBases 子節點
.../tools/li[label="blade"] 選取 tools 下 label 為 blade 的 li
.../comps/li[@Class="CompProperties_Power"] 選取有特定 Class attribute 的 li
.../weaponTags/li[.="MeleeWeapon"] 選取文字內容為 MeleeWeapon 的 li

完整範例：在鋼鐵劍 statBases 加入自訂屬性

<!-- Patches/MyMod_SteelSword_Patch.xml -->
<Patches>

  <!-- 單一操作 -->
  <Operation Class="PatchOperationAdd">
    <xpath>Defs/ThingDef[defName="Steel_Sword"]/statBases</xpath>
    <value>
      <!-- 新增自訂 StatDef 的值 -->
      <MyMod_PlasmaAffinity>0.4</MyMod_PlasmaAffinity>
    </value>
  </Operation>

</Patches>

PatchOperationSequence 批次操作

<Patches>
  <Operation Class="PatchOperationSequence">
    <success>Always</success>   <!-- Always / Normal / Never -->
    <operations>

      <!-- 步驟 1：移除舊市場價 -->
      <li Class="PatchOperationRemove">
        <xpath>Defs/ThingDef[defName="Steel_Sword"]/statBases/MarketValue</xpath>
      </li>

      <!-- 步驟 2：新增修改後的市場價 -->
      <li Class="PatchOperationAdd">
        <xpath>Defs/ThingDef[defName="Steel_Sword"]/statBases</xpath>
        <value>
          <MarketValue>350</MarketValue>
        </value>
      </li>

      <!-- 步驟 3：替換標籤文字 -->
      <li Class="PatchOperationReplace">
        <xpath>Defs/ThingDef[defName="Steel_Sword"]/label</xpath>
        <value>
          <label>精鋼劍</label>
        </value>
      </li>

    </operations>
  </Operation>
</Patches>

MayRequire 條件修補

當 Patch 依賴其他 mod（或 DLC）時，用 MayRequire/MayRequireAnyOf 包裝，缺少依賴時整個 Operation 會靜默跳過。

<Patches>
  <!-- 僅在 Royalty DLC 存在時套用 -->
  <Operation Class="PatchOperationSequence"
              MayRequire="Ludeon.RimWorld.Royalty">
    <success>Always</success>
    <operations>
      <li Class="PatchOperationAdd">
        <xpath>Defs/ThingDef[defName="MyCorp_Sword_Plasma"]/weaponTags</xpath>
        <value>
          <li>Neurotrainer</li>    <!-- Royalty DLC 特有 tag -->
        </value>
      </li>
    </operations>
  </Operation>

  <!-- 依賴任一 mod 即套用（用 PackageId 逗號分隔） -->
  <Operation Class="PatchOperationAdd"
              MayRequireAnyOf="OtherMod.PackageId,AnotherMod.PackageId">
    <xpath>Defs/ThingDef[defName="MyCorp_Sword_Plasma"]/comps</xpath>
    <value>
      <li Class="OtherMod.CompProperties_CustomEffect"/>
    </value>
  </Operation>
</Patches>
PatchOperationFindMod 已棄用 舊版教學常見的 PatchOperationFindMod 在 1.1 後已不建議使用，應改用 MayRequire attribute，後者語意更清晰且由引擎直接支援。
§6　DefOf 與 C# 互動

[DefOf] 自動注入

在 C# 端建立 [DefOf] 靜態類別，引擎會在 DefDatabase 就緒後自動將同名欄位填入對應 Def 實例， 免去手動 GetNamed() 呼叫，也讓編譯期有型別檢查。 

using Verse;

// 加上 [DefOf] attribute；命名空間內的 static class 即可
[DefOf]
public static class MyCorpDefOf
{
    // 欄位名稱必須與 defName 完全一致
    public static ThingDef            MyCorp_Sword_Plasma;
    public static ResearchProjectDef  MyCorp_PlasmaForging;

    // 若依賴 DLC，加 [MayRequireRoyalty] / [MayRequire("PackageId")]
    [MayRequireRoyalty]
    public static ThingDef MyCorp_PsyBlade;
}
注入時機 [DefOf] 欄位在 StaticConstructorOnStartup 之前就已填入，可以安全地在任何遊戲初始化後的上下文使用。 但不能在 static 建構函式（型別初始化器）裡直接讀取，因為載入順序不保證。

DefDatabase<T> 動態查詢

using Verse;

// 依 defName 查詢（不存在時回傳 null，可選 errorOnFail: true 拋例外）
ThingDef swordDef = DefDatabase<ThingDef>.GetNamed("MyCorp_Sword_Plasma");

// 不確定是否存在時用 GetNamedSilentFail
ThingDef maybeDef = DefDatabase<ThingDef>.GetNamedSilentFail("MyCorp_Sword_Plasma");
if (maybeDef != null) { /* ... */ }

// 遍歷所有同類 Def
foreach (ThingDef def in DefDatabase<ThingDef>.AllDefs)
{
    if (def.IsWeapon && def.weaponTags?.Contains("MeleeWeapon") == true)
        Log.Message(def.defName);
}

ThingMaker.MakeThing() 從 Def 生成實例

using Verse;

// 生成物品 Thing（不帶材質）
Thing sword = ThingMaker.MakeThing(MyCorpDefOf.MyCorp_Sword_Plasma);

// 帶材質（stuff）生成——適用有 stuffCategories 的 ThingDef
Thing plastiSword = ThingMaker.MakeThing(
    MyCorpDefOf.MyCorp_Sword_Plasma,
    ThingDefOf.Plasteel       // stuff ThingDef
);

// 設定初始品質（需有 CompQuality）
CompQuality comp = plastiSword.TryGetComp<CompQuality>();
comp?.SetQuality(QualityCategory.Masterwork, ArtGenerationContext.Colony);

// 生成 Pawn（透過 PawnGenerator）
Pawn raider = PawnGenerator.GeneratePawn(
    new PawnGenerationRequest(
        PawnKindDefOf.Mercenary_Fighter,
        faction: Faction.OfPlayer
    )
);

生成後放置到地圖

// GenSpawn.Spawn 將 Thing 放到指定位置
GenSpawn.Spawn(sword, new IntVec3(10, 0, 10), Find.CurrentMap);

// 或用 DropPodUtility 從天而降（突襲 / 補給艙）
DropPodUtility.DropThingsNear(
    Find.CurrentMap.Center,
    Find.CurrentMap,
    new List<Thing> { sword }
);
Def vs Instance 的關鍵概念 ThingDef（及所有 Def）是不可變的資料模板，整個遊戲只有一份。 Thing（及所有遊戲實體）是執行期實例，持有對 Def 的參考（thing.def）。 修改 Def 欄位會影響所有現有及未來的實例，因此在執行期改動 Def 是極度危險的反模式。
