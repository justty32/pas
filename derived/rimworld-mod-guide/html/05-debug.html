§1 Dev Mode 開啟與核心功能

Dev Mode 是 RimWorld 官方內建的除錯環境，開啟後右上角出現一排工具圖示，並在畫面四角顯示額外資訊。 對 mod 開發者而言，這是唯一可信賴的即時回饋管道。

開啟方式

啟動參數（推薦）

在 Steam 遊戲屬性 → 啟動選項加入 -dev，或在捷徑目標末加入參數：

# Steam 啟動選項欄位
-dev

每次啟動都自動進入 Dev Mode，不需手動設定。

Prefs.xml 持久化

直接編輯 Preferences 設定檔：

# 路徑（Windows）
%APPDATA%\..\LocalLow\Ludeon Studios\
  RimWorld by Ludeon Studios\Config\Prefs.xml

<DevMode>True</DevMode>
<LogVerbose>True</LogVerbose>
💡 建議同時開啟 LogVerbose LogVerbose = True 會讓 RimWorld 輸出更細緻的框架日誌，包含 Def 載入進度、Harmony patch 套用結果等，是追蹤初始化期問題的利器。

Debug Action Menu（右上角工具列）

進入 Dev Mode 後，畫面右上角出現四個小圖示，由左至右：

⚙ Dev Actions

主選單，包含生成 Pawn / Item / Event、刷新地圖等數百個操作。

🔍 Inspector

點擊地圖上的任何物件即可查看其所有欄位值，相當於 Unity Inspector 的 readonly 版本。

📋 Log

開啟內建 Log 視窗，顯示 Message / Warning / Error，可篩選與清除。

🐛 Package Mode

顯示所有已載入 mod 的 packageId，用於確認 mod 清單與載入順序。

常用快捷操作

操作路徑 / 快捷用途
生成 Pawn Dev Actions → Spawn pawn… 快速在游標位置生成敵友 Pawn，測試 AI 行為
生成 Item Dev Actions → Spawn thing… 輸入 defName，立刻取得任何物品
觸發 Incident Dev Actions → Execute incident… 強制觸發突襲、疾病、飼育等任何事件
強制 Tick Dev Actions → Do tick（或按 T） 手動推進一個遊戲 tick，測試定時邏輯
季節跳躍 Dev Actions → Set season → … 瞬間切換春夏秋冬，測試溫度 / 作物邏輯
地圖填充 Dev Actions → Fill with… 整張地圖鋪滿指定地形或物品，壓力測試用
God Mode Dev Actions → Toggle god mode 無限資源、無需材料建造，快速搭建測試場景

訊息等級識別

ERROR 紅色

必須修。通常伴隨 stack trace，遊戲可能繼續但行為已損壞。是 mod 除錯的首要目標。

WARN 黃色

潛在問題。常見如 Def 欄位缺失、Patch 未命中等，不一定會 crash 但需要確認。

LOG 白色

一般資訊。Log.Message() 輸出至此。用於追蹤執行流程，不代表錯誤。

閱讀 Stack Trace 的要點

// 典型錯誤輸出結構
Exception in RimWorld.StatWorker.GetValueAbstract: NullReferenceException: ...
  at Verse.ThingDef.get_IsStuff ()     <-- 最接近根因的框架位置
  at RimWorld.StatWorker.GetValueAbstract (...)
  at YourMod.MyPatch.Prefix (...)      <-- 你的 Patch 呼叫點
  at (wrapper dynamic-method) ...Harmony_Patch...

讀 trace 的方向：從下往上 找第一個屬於你的 mod namespace 的 frame，那通常是你的程式碼觸發問題的位置。

Harmony 相關錯誤特徵

錯誤訊息特徵意義排查方向
HarmonyException: method not found 目標方法不存在或簽名不符 用 dnSpy 確認方法名稱與參數型別
InvalidProgramException in IL Transpiler 產生的 IL 非法 用 HarmonyLib.PatchClassProcessor 的 debug 模式輸出 IL
Ambiguous match found 方法多載衝突，Harmony 無法決定目標 在 [HarmonyPatch] Attribute 中明確指定參數型別
No annotations found Patch class 沒有正確的 Attribute 確認 [HarmonyPatch(typeof(X), "MethodName")] 放在 class 上
Patch 套用後無反應 目標組件可能在 patch 前就已 JIT 確認 harmony.PatchAll() 在 [StaticConstructorOnStartup] 或 Mod 建構子中呼叫

使用 HugsLib Log Window

HugsLib 提供一個遠比內建 Log 更好用的 in-game Log viewer，按 Ctrl+F12 呼出：

  • 可搜尋關鍵字篩選訊息
  • 一鍵複製全部 Log 到剪貼簿（回報 bug 時非常有用）
  • 顯示 mod 載入清單與版本資訊
  • 持久保留跨 map 的訊息（內建視窗每次開關地圖會清空）
HugsLib 相依性說明 若你的 mod 已相依 HugsLib，可繼承 ModBase 並用 Logger.Message() 系列 API，自動加上 mod 前綴避免和其他 mod 的 log 混淆。若不想引入相依，用原生 Log.Message() 加上明確的 mod 名稱前綴即可。
§3 C# Log API 與自訂 Dev 選單

基本 Log 方法

using Verse;

// 一般訊息 → Player.log 白色
Log.Message("[YourMod] 目前 pawn 數量: " + pawns.Count);

// 警告 → 黃色
Log.Warning("[YourMod] graphicData 為 null，使用預設值");

// 錯誤 → 紅色 + stack trace
Log.Error("[YourMod] 無法找到 ThingDef: " + defName);

// 防洗版：相同 key 只輸出一次（整個遊戲 session）
Log.ErrorOnce("[YourMod] 組態錯誤，請檢查 modSettings", 0xDEAD);

// 條件式 verbose（只在 DevMode 下輸出，避免正式版洗版）
if (Prefs.DevMode)
    Log.Message("[YourMod] Transpiler IL offset = " + offset);
⚠ 正式發布前清除或保護 Log 呼叫 高頻率執行的方法（Tick()、Draw()）如果有 Log.Message()，每秒可能輸出數百行，嚴重拖慢效能並讓其他玩家的 log 難以閱讀。發布前用 Prefs.DevMode 包覆，或改用 Log.ErrorOnce()。

物件狀態快照輔助

// 輸出 Pawn 的完整標籤樹（Scribe_Deep 序列化結果）
var sb = new System.Text.StringBuilder();
Scribe.saver.InitSaving(ScribeMetaHeaderUtility.ScribeHeaderMode.Map);
// 或用反射逐欄位輸出：
foreach (var fi in pawn.GetType().GetFields(
    System.Reflection.BindingFlags.Public |
    System.Reflection.BindingFlags.NonPublic |
    System.Reflection.BindingFlags.Instance))
{
    sb.AppendLine($"{fi.Name} = {fi.GetValue(pawn)}");
}
Log.Message(sb.ToString());

自訂 Dev Action 選單項目（DebugActionsModMenu）

RimWorld 1.5 將 [DebugAction] 等 Dev 工具搬到 LudeonTK 命名空間，讓 mod 可以在 Dev Actions 選單中加入自訂按鈕，無需繼承任何基底類別：

using Verse;
using LudeonTK;   // 1.5：DebugAction/DebugActionType 已搬到此命名空間

[DebugAction(
    "YourMod",            // 分類名稱（會在選單中作為子選單標題）
    "強制觸發自訂事件",    // 按鈕顯示文字
    actionType = DebugActionType.Action,
    allowedGameStates = AllowedGameStates.PlayingOnMap
)]
private static void DebugForceMyEvent()
{
    IncidentDef def = DefDatabase<IncidentDef>.GetNamed("MyMod_MyIncident");
    IncidentParms parms = StorytellerUtility.DefaultParmsNow(
        def.category, Find.CurrentMap);
    def.Worker.TryExecute(parms);
    Log.Message("[YourMod] 事件已觸發");
}

// 帶選項的 Dev Action（使用 FloatMenuOption 讓使用者選擇）
[DebugAction("YourMod", "給玩家 Pawn 加 Hediff",
    actionType = DebugActionType.ToolMapForPawns)]
private static void DebugAddHediff(Pawn pawn)
{
    HediffDef def = DefDatabase<HediffDef>.GetNamed("MyMod_MyHediff");
    pawn.health.AddHediff(def);
    Log.Message($"[YourMod] 已對 {pawn.LabelShort} 加入 {def.defName}");
}
💡 DebugActionType 種類 Action（直接執行）、ToolMapForPawns（點擊地圖上的 Pawn）、ToolMapForThings（點擊 Thing）、ToolMap（點擊地圖任意格）是最常用的四種。選擇適當的 type 能讓開發選單更直觀。
§4 dnSpy Attach to Process 即時除錯

dnSpy 是 .NET IL 的反編譯器兼除錯器，可附加至正在執行的 Unity Mono 進程設置中斷點， 是深度追蹤 RimWorld 框架行為最強力的工具。

前置需求 下載 dnSpy（建議 x64 版）。 RimWorld 使用 Unity Mono 執行時，dnSpy 支援直接在反編譯的 IL 上設置中斷點，不需要原始 PDB。

附加步驟

  1. 啟動 RimWorld（含 Dev Mode），確認遊戲已進入主選單或地圖（Mono 執行時完全初始化後才能附加）。
  2. 開啟 dnSpy，點選選單 Debug → Attach to Process…（或 Ctrl+Alt+P）。
  3. 在清單中找到 RimWorld（進程名稱通常是 RimWorldWin64.exe 或 RimWorldLinux）， 確認 Runtime 欄顯示 Unity（Mono），點選 Attach。
  4. 載入你的 DLL：File → Open → 選取 YourMod.dll（或 Assembly-CSharp.dll）， dnSpy 會反編譯並顯示原始碼。
  5. 設置中斷點：在反編譯的方法行號旁點擊即可設置（紅點），或按 F9。
  6. 切回 RimWorld 觸發路徑，例如建造物件、移動 Pawn 等；dnSpy 會在中斷點暫停， 此時可查看 locals、call stack、即時執行運算式（Immediate Window Ctrl+G）。

Unity Mono 的限制

部分中斷點無效

已被 JIT 編譯為原生碼的路徑有時無法中斷。解決方法：在方法進入前加入 Thread.Sleep(1)（讓 JIT 重新計算），或在 Debug → Options 啟用 Break on Module Load。

Harmony Patch 的 IL 追蹤

Harmony 動態生成的方法 wrapper 有時在 dnSpy 中顯示為 <dynamic method>， 無法直接設中斷點。此時改用 Log.Message 二分法，或在你的 Prefix/Postfix 方法自身設中斷點。

Edit-and-Continue 不支援

Unity Mono 不支援 Edit-and-Continue。修改 C# 後必須：停止附加 → 重新編譯 → 重啟遊戲 → 再次附加。

自備 PDB 加速定位

在 csproj 中設定 <DebugType>full</DebugType> 並將產出的 .pdb 與 .dll 放在同目錄，dnSpy 會優先讀取 PDB，中斷點命中率顯著提升。

快速除錯替代方案：條件式 Log 二分法

// 當 dnSpy 中斷點不可靠時，用靜態旗標縮小問題範圍
public static bool DebugVerbose = false;

public override void Tick()
{
    base.Tick();
    if (DebugVerbose && Prefs.DevMode)
        Log.Message($"[MyComp] Tick #{Find.TickManager.TicksGame}, owner={parent?.Label}");

    // 懷疑問題出在這裡 → 用 DebugAction 動態開啟 DebugVerbose
}

§6 Steam Workshop 發布流程

發布前準備

About/About.xml 完整性

<ModMetaData>
  <name>Your Mod Name</name>
  <author>YourSteamName</author>
  <packageId>Author.ModName</packageId>
  <supportedVersions>
    <li>1.6</li>
  </supportedVersions>
  <description>…</description>
</ModMetaData>

packageId 是全域唯一識別碼，格式 Author.ModName，一旦發布後不可更改（更改等同於新 mod）。

About/Preview.png

Steam Workshop 封面圖規格：

  • 建議解析度：320 × 240 px（4:3 比例）
  • 格式：PNG 或 JPG
  • 上限：1 MB
  • Workshop 頁面顯示時會裁切為 268 × 268，建議把主視覺放在中央安全區

使用 RimWorld 內建上傳介面

  1. 在本地測試通過：將 mod 資料夾直接放在 RimWorld/Mods/YourModName/， 在遊戲 Mod Manager 啟用並完整測試。
  2. 進入 Dev Mode，開啟 Mod Manager（主選單 → Mods），找到你的 mod， 點選右下角的 … 選單 → Upload to Steam Workshop。
  3. 設定 Visibility（第一次上傳建議選 Friends Only 先做最終測試）， 確認後點 Upload。
  4. 上傳完成後，遊戲會在 mod 資料夾產生 About/PublishedFileId.txt， 內含 Workshop Item ID。保留此檔案，後續更新時用同一介面上傳會覆蓋同一條目。
  5. 在 Steam Workshop 頁面補充說明文字、標籤、截圖，改為 Public 可見度。
⚠ PublishedFileId.txt 遺失問題 若此檔案遺失或意外刪除，下次上傳時會建立新的 Workshop 條目而非更新舊的。 解決方法：手動建立 About/PublishedFileId.txt，內容只需一行 Workshop Item ID（純數字）。 Item ID 可在 Workshop 頁面 URL 中找到：steamcommunity.com/sharedfiles/filedetails/?id=1234567890。

版本號管理與 Changelog

RimWorld 本身不強制語意化版本，但建議建立一致的版本策略：

<ModMetaData>
  <!-- About.xml 中沒有官方的 version 欄位，但可自訂 -->
  <version>1.3.0</version>  <!-- 部分框架（HugsLib）會讀取此欄位 -->
</ModMetaData>

Changelog 通常附在 Workshop 描述的末尾，或放在 About/ChangeNotes.txt（部分工具會讀取）。 建議格式：

## v1.3.0 (2025-06-01)
- 新增：XXX 功能
- 修正：YYY 造成的 NRE
- 相容：支援 RimWorld 1.6

本地 Mods/ 資料夾測試

不透過 Steam Workshop 也能測試，直接將 mod 資料夾放入 RimWorld 的 Mods/ 目錄：

# 路徑（Windows，Steam 版）
C:\Program Files (x86)\Steam\steamapps\common\RimWorld\Mods\YourModName\

# 或使用符號連結避免複製（PowerShell 管理員模式）

§7 Mod 相容性最佳實踐

MayRequire 條件 Def Patch

當你的 mod 需要與另一個 mod 整合時，使用 PatchOperations 而非硬依賴：

<!-- Patches/patch_integration_somemod.xml -->
<Patch>
  <Operation Class="PatchOperationFindMod">
    <mods><li>Author.SomeMod</li></mods>
    <match Class="PatchOperationAdd">
      <xpath>Defs/ThingDef[defName="SomeMod_Widget"]/comps</xpath>
      <value>
        <li Class="YourMod.CompMyFeature"/>
      </value>
    </match>
    <!-- nomatch 可省略，表示找不到 mod 時靜默忽略 -->
  </Operation>
</Patch>

或在單一 Def 的 XML 中用 MayRequire：

<li MayRequire="Author.SomeMod">
  <filter><thingDefs><li>SomeMod_Resource</li></thingDefs></filter>
</li>

Harmony Patch 加 try/catch 保護

線上環境中，一個 Patch 崩潰可能導致連鎖失敗。在可能出錯的 Patch 中加入防護：

[HarmonyPatch(typeof(JobDriver_Ingest), "MakeNewToils")]
public static class Patch_JobDriver_Ingest_MakeNewToils
{
    static void Postfix(JobDriver_Ingest __instance, ref IEnumerable<Toil> __result)
    {
        try
        {
            __result = AddCustomToils(__instance, __result);
        }
        catch (Exception ex)
        {
            Log.ErrorOnce(
                "[YourMod] Patch_JobDriver_Ingest_MakeNewToils 失敗，已停用本 Patch: "
                + ex.ToString(), 0xABC1);
            // 不重拋例外，讓原方法的結果照常使用
        }
    }
}

動態偵測其他 Mod

using RimWorld;
using Verse;

// 在 Mod 建構子或 [StaticConstructorOnStartup] 中一次性偵測
public static bool HasCombatExtended;
public static bool HasVanillaExpanded;

static YourModMain()
{
    HasCombatExtended  = ModLister.HasActiveModWithName("Combat Extended");
    HasVanillaExpanded = ModLister.GetActiveModWithIdentifier(
        "vanillaexpanded.vfe.core") != null;

    if (HasCombatExtended)
        Log.Message("[YourMod] 偵測到 Combat Extended，啟用相容模式");
}

// 在邏輯中根據旗標走不同路徑
public static float GetDamage(Thing weapon)
{
    if (HasCombatExtended)
        return GetDamageCE(weapon);   // 呼叫 CE 的 API
    return weapon.GetStatValue(StatDefOf.RangedWeapon_DamageMultiplier);
}
⚠ 用 packageId 而非顯示名稱偵測 HasActiveModWithName() 比對的是顯示名稱，可能因玩家更改 About.xml 或翻譯而不符。 建議改用 GetActiveModWithIdentifier("author.modname")，比對的是不可變的 packageId，更可靠。

避免 Static Field 殘留狀態

RimWorld 的「返回主選單」不會重啟進程，所有 static field 的值會保留到下一局。 這在 Harmony Patch 中特別危險：

// ❌ 危險：靜態快取在跨局時不清空
public static class BadPatch
{
    private static Dictionary<Pawn, MyData> cache = new();

    static void Postfix(Pawn __instance) { /* 使用 cache */ }
}

// ✅ 正確：在 GameComponent 中管理，隨地圖生命週期清空
public class MyGameComp : GameComponent
{
    private Dictionary<Pawn, MyData> cache = new();

    public MyGameComp(Game game) { } // 每局開始時重新建構
}

// ✅ 或在 NewGame/LoadedGame 時重置
public class MyMod : Mod
{
    public override void WriteSettings()
    {
        base.WriteSettings();
        BadPatch.cache?.Clear(); // 設定改變時清快取
    }
}
🚨 Harmony Patch 中的靜態旗標尤其危險 若你在 Prefix 中用靜態 bool 做「once-per-execution」防止重入，而這個 bool 在例外發生後沒有被重置， 整個遊戲 session 中該 Patch 可能永遠不會再執行。務必在 finally 區塊中重置此類旗標。

相容性最佳實踐總覽

Def 層隔離

所有 defName 加 mod 前綴；跨 mod 整合用 PatchOperations 而非硬寫 defName；可選依賴用 MayRequire。

Patch 防禦性設計

高風險 Patch 包 try/catch；Prefix 返回 false 前確認副作用；避免 Transpiler 除非絕對必要（維護成本高）。

狀態生命週期

遊戲資料用 GameComponent / MapComponent 持有，確保跨局乾淨重置；不在 Patch class 持有遊戲狀態。

依賴聲明

About.xml 中用 <modDependencies> 列出硬依賴；可選相依只做執行時偵測，不設載入順序強制。

版本保護

在 supportedVersions 只列出測試過的版本；新版 RimWorld 釋出後用空白存檔完整跑過再更新版本號。

Log 禮儀

正式版不在熱路徑輸出 Log；使用 ErrorOnce 防洗版；所有輸出加 [YourMod] 前綴方便玩家過濾。
