## 標準實現規範的本質（2026-05-21）

目前在定義的標準實現規範，其作用是：

**對不確定性進行建模，同時對缺損智能進行引導的框架。**

具體來說：
- LLM 的行為具有不確定性（同樣的輸入可能產生不同輸出）
- LLM 的邏輯鍊條豐富但有缺損（能推理，但推理可能中途出錯或跳步）
- 標準實現規範提供結構，讓 LLM 在這個結構內行動，不確定性被限縮到規範許可的空間內

## 跨環境統一管理規範的願景（2026-05-21）

對於各種「可執行的東西」——程式片段、腳本、單檔程式、程式內函數——的管理，包括：

- **納管**：如何將它們收入某個系統
- **增刪查改**：如何操作它們
- **路由**：如何找到並呼叫它們

這一整套應該能夠**跨越不同的環境邊界**：

| 環境 | 例子 |
|---|---|
| Terminal / shell | 單檔程式、shell 腳本、pipe |
| 程式內部 | 函數、class method、lambda |
| 作業系統層 | process、subprocess、daemon |
| 分散式系統 | 跨機器的 API 呼叫、遠端服務 |

目標是整合成一套**貫徹始終的規範**，涵蓋呼叫、實作、管理三個面向，讓開發者不需要為不同環境各自發明一套做法。

目前 ai_core 的各元件（metadata 協定、router、forge、SFC、outside_progs 等）都是朝這個方向在走，但尚未被明確地以這個統一願景來框架。

**環境清單補充說明：**
- Container 內部是 OS 環境，對外是 API 呼叫環境（分散式）；Serverless 同理，從 caller 角度是 API 呼叫，但從 function 作者角度有特殊約束（stateless、冷啟動、無常駐 process）
- Browser 本質是程式內環境，頂多算 OS 層；WASM 是可攜式執行模型，能跨越多個環境邊界，值得留意
- **GPU kernel**（CUDA / OpenCL）：獨立記憶體空間、不同處理器、走 driver 介面——不屬於上表任何一格，是真正額外的執行環境，暫不深入
- **同步 vs 非同步呼叫模型**是橫切所有環境的面向：Terminal / in-program 預設同步，distributed / browser 預設 async / event-driven；這會影響「呼叫」規範在不同環境下的設計

## 執行狀態分類的跨環境性（2026-05-21）

可執行的東西，可以依據其**執行狀態**做分類：

| 分類 | 語意 |
|---|---|
| one-shot | 呼叫一次、執行完畢、結束 |
| multi-shot | 跨多次呼叫保有狀態，上次結果影響下次執行 |
| persistent | 持續存在，等待呼叫 |
| singleton | 資源有限，同時只能有一個（或受限數量的）實例 |

這套分類**應該跨越所有執行環境**——無論是 terminal 中的 shell 腳本、程式內部的函數、分散式系統中的服務，都可以用這四種狀態來描述其執行性質。

這是比「環境」更本質的分類軸：環境描述「在哪裡跑」，執行狀態描述「怎麼跑、跑完後發生什麼」。兩個軸正交，合起來才是完整的描述。

**補充——三個待深入設計的面向：**

**Streaming（增量輸出）**：one-shot 是執行完才輸出，streaming 是邊執行邊輸出（`tail -f`、串流 LLM 回應）。跟 persistent 不同：persistent 等待多個獨立呼叫，streaming 是單次呼叫產生連續輸出。可能值得列為獨立執行狀態。

**觸發機制（另一個正交軸）**：上表四種狀態描述「執行後的行為」，觸發方式是另一個軸：

| 觸發機制 | 說明 |
|---|---|
| 直接呼叫 | 上表隱含的預設 |
| Event-driven | 被外部事件推送觸發（message queue、webhook、filesystem watch） |
| Scheduled | 由時間觸發（cron） |

觸發機制可疊在任何執行狀態上（如「定時跑的 one-shot」、「收到 event 才啟動的 persistent」）。

**Lazy / warm（中間態）**：介於 one-shot 與 persistent 之間——第一次呼叫才啟動，之後保持活著一段時間。可作為獨立分類或 persistent 的子類，待定。

以上三點待之後針對執行狀態分類做更完整的設計。

## 牢籠與猴子：系統的整體隱喻（2026-05-21）

我預期要做的東西，是一個由**無數細線編織出的牢籠**。

**細線 = 硬編碼邏輯（符號主義的象徵）。** 之所以說是「細」線，是因為相比於 LLM 的不可知性與龐大的可能性空間，那些硬編碼邏輯就像細線一樣——單獨一條微不足道，但編織起來能成形。

**整個運作過程 = 猴子在牢籠中玩耍。** 猴子就是 LLM（及其行為）。

### 出入口：本質是一個函數

牢籠有出入口，**出入口本質上是一個函數，目標是處理猴子**。出入口同樣用細線編織出特定的形狀，只讓**特定狀態的猴子**通過：

- **入口 = 篩選**（對輸入的 filter）
- **出口 = 最後的處理**（對輸出的 post-processing）

### 牢籠 = 邊界

牢籠表示了猴子的**邊界**。雖然仍有一些毛髮細屑會通過（少量洩漏無法完全杜絕），但至少**不會讓整隻猴子通過**——即不會整體失控。

牢籠的邊界可能是**多層嵌套**的，代表多個沙盒作為安全措施。

這些邊界是用**工業編織**的：代表這方面通常死板但嚴謹可靠，無需人工處理細項——是自動化、標準化的部分。

### 空中纜線：手工設置的欄杆與導航

牢籠中會用**粗線手工設置空中纜線**，這就像是欄杆和導航，能讓猴子抓住欄杆前往出口。

- 猴子仍然可能**手滑抓不住**——這代表 LLM 的隨機出錯性。
- 所以手工設置時需要**精心設計**，兩個目標：
  1. 讓猴子能很好地找到下一根**牢固且容易抓握**的纜繩；
  2. 盡可能讓纜繩的**數量減少**——這是為了減少人類的手工心力。

### 最終目標：飛輪

精心設計關卡，讓猴子在牢籠中**經過訓練後，能自行編織纜繩**，建立另一個更好的牢籠，最終實現**飛輪**——猴子一代比一代聰明。

猴子本身的記憶，就像是 agent 的記憶或 skill。

## 滑坡軌道與球：執行形式的本質隱喻（2026-05-21）

關於 one-shot、multi-shot 這些用來描述「可執行的東西」之形式的字詞，它們的本質是什麼？可以類比為一顆球在**滑坡軌道**上，從頂端滑落到底端的過程。

- **軌道 = 函數**
- **球在頂端的狀態 = 輸入**
- **球在底端的狀態 = 輸出**

### 最基本形式

什麼都沒有——球乖乖滑到底之後，什麼都沒變。（無狀態、無副作用的純通過）

### one-shot

在滑道中央添加一些**噴漆口**，球經過之後會被上色。這就是最簡單的 one-shot 函數。

### multi-shot

在滑道之外立一個標示：

> 「此滑道可供已上色的球使用，結果是球的顏色會愈變愈深。」

使用者當然可以每次都拿沒上色的球去滑，但也可以拿已經上色的球去滑。

而真正的精髓在於，滑道又另外立了一個牌子說明：

> 「此滑道的目的就是加深球的顏色，尚未被上色的球不會受影響。」

### 記憶體與執行時間管理

這些都可以很簡單地想出相應的比喻。（待補）

### persist 函數

球在正常情況下會**一直卡在滑道中等待互動**。

### 異常狀況：錯誤與強制終止

額外要說明的兩種情況：

- **球滑出滑道外** → 代表錯誤（escape / 例外）
- **使用者在球正在滑落時，手動阻止球滑落** → 代表使用者強行終止（interrupt）

### 自有儲存狀態的變體

球每經過滑道一次，**滑道本身**就會做一些變化或計數。這種與 multi-shot 類似，但其函數狀態是**自有儲存**的（state 存在函數自己身上，而非由球攜帶）。

### 範疇延伸

現在講的都是**單一函數**的狀況。之後我們要進入**多個函數**的狀況——也就是 hub、router、indexer 等等。
