🏗 架構元件

五個核心元件的設計職責、Singleton Resource Manager Pattern，以及 --metadata 協議的完整規則。

整體架構圖
caller（人類 shell / AI agent / 其他 function）
   │
   ├── 直接 shell call ─────────────────────────────────┐
   │                                                    ▼
   │                               任意語言寫的 function
   │                               （唯一硬規則：--metadata 印合法 JSON）
   │                                       │
   │                                       ├── 單純函式：直接執行（stateless one-shot）
   │                                       └── 重資源 ──▶ wrapper CLI ──▶ singleton server
   │
   ├── 想知道有哪些 function ──▶ Function Hub (one-shot / server)
   │                              ├── scan → metadata → skill list
   │                              └── export: MCP / openai-tools / anthropic-tools
   │
   └── 想新增 function ─────────▶ ai-core-author
                                   generate → dry-run → register

LLM 呼叫路徑：
   ai-core-call ──HTTP──▶ ai-core-server (FastAPI + asyncio)
       ├─ per-model FIFO queue + worker
       ├─ rate limit (rpm / tokens / cost)
       └─ litellm router ──▶ ollama / lm-studio / gemini
五個核心元件
元件 1 · Entry Manager

LLM Entry Manager

ai-core-server + ai-core-call

統一管理多個 LLM 模型/API 的呼叫入口。LLM 是 singleton 資源（本地或遠端一次只能處理一個請求），採佇列模式集中管理 consume rate。

  • 常駐 server（FastAPI + uvicorn），HTTP 對外
  • per-model FIFO asyncio queue + worker
  • 三軸限額：rpm / tokens_per_day / cost_per_day
  • litellm 接通 ollama、lm-studio（OpenAI 相容）、gemini
  • ai-core-call wrapper：shell 不須手動 curl
pattern: Entry-managing Singleton
--metadata（工具自身）+ --entry-metadata <model>（個別模型）
元件 2 · Calling Packing

LLM Calling Packing

ai-core-call (context binding)

把 LLM 呼叫包裝成具語意的函式。基底介面是 llm_call(string) → string，再疊加 context 與 post-processing 形成新函式。

def llm_call_coding_question(string):
    input = "you are a professor of coding..." + string
    output = llm_call(input)
    return output + " -- at 20240505"
  • user-profile 橫切層（「我是繁中使用者 → 產出都用繁中」）
  • calling pack 可被組合：pipe / fanout / retry_until_valid
  • M6 里程碑：llm_call_coding_question 等典型範例集
context binding 是跨輪次的 memory 積累點
見 thinking.md memory_hints 欄位（待補）
元件 3 · Function Hub

Function Hub

ai-core-hub + ai-core-hub-server

掃描函式集，呼叫每個函式的 --metadata 並彙總成「給 LLM 的 skill 清單」。Hub 是透鏡（lens）——聚合＋轉換 metadata，不只是被動目錄列表。

  • 掃描策略：遍歷目錄，對每個可執行檔 call --metadata
  • 容錯：缺 metadata / 不合法 JSON → graceful 降級
  • context budget 收斂：過大清單自動省略低優先函式
  • export 格式：MCP / openai-tools / anthropic-tools
  • one-shot（ai-core-hub）+ server（ai-core-hub-server）雙形態
  • Hub 可被組合：meta-hub 把多個 hub 再聚合
Hub 的三種轉換：擴增（加分類/tag）、過濾（省略 description）、彙總（語意相近群組合並）
元件 4 · Small Function Center

Small Function Center

ai-core-sfc

許多函式只是一行 shell 或薄 LLM 包裝，為避免檔案爆炸，集中到一個 dispatcher。git-style subcommand CLI。

  • Layer 0：store（functions.json + index.json）
  • Layer 1a：intake（從外部收入新函式）
  • Layer 1b：router（name → dispatch）
  • Layer 2：forge persistent server（NDJSON 行協議）
  • Layer 3：動態 add/remove/persist（執行期管理）
  • Layer 4：shell-kind timeout + 標準錯誤封套
kind: python（真 in-process exec）或 shell（subprocess）
signature: fn(stdin: str, args: dict) → str
元件 5 · Author

ai-core-author（自舉）

ai-core-author

一站式產生函式骨架 + dry-run 驗證 + 進入 registry。AI 生的函式需通過驗證才能進 hub，避免「AI 生 AI 用」的內部黑箱。

① Generate

--spec 介面，透過 ai-core-call 用 LLM 產函式骨架（bash / python）

② Dry-run

跑範例驗證、失敗時把回饋送回 LLM 重試，不通過不放行

③ Register

進入 registry，自動觸發 hub 更新（auto/ 目錄同步）

author 自身遵守 --metadata，可被 hub 發現（自舉性）。
成功 register 後自動更新 auto/FUNCTIONS.md + AGENTS.md。
Singleton Resource Manager Pattern

對「一次只能服務一個請求、建立成本高」的單例資源，統一採用此 pattern。適用場景：LLM 模型、heavy GPU 計算、配額受限 API、長連線本地服務。

Pattern 四步：① 常駐 server 包住資源，HTTP / IPC 對外 → ② per-resource FIFO queue + worker → ③ 配套一支 shell wrapper CLI（ai-core-call）→ ④ wrapper 實作 --metadata（工具自身）
Entry-managing 亞型

管理多個具名資源（entries）。例：LLM Entry Manager（多個 model）。需在 metadata 中宣告 has_entries: true，支援 --entry-metadata <name>。

Simple singleton 亞型

包住單一資源，不需 entry 概念。功能上類似常駐 MCP 工具伺服器，只是變成 singleton 以保護底層資源。只有 --metadata，無 --entry-metadata。

共用樣板抽取時機（YAGNI）：等第二個 singleton manager 出現時才從 entry_manager/ 抽出 protocol/queue.py + protocol/ratelimit.py，目前目錄先預留。

--metadata 協議

跨元件契約。唯一強制要求：輸出合法 JSON。schema 不強制，所有 caller/hub/author 都必須預期函式可能沒寫 metadata 或寫得不完整。

最小合法輸出範例
{"lifecycle": "one_shot",
 "state": "stateless",
 "interruptible": true,
 "guarantee": "idempotent",
 "resources": {},
 "entries": {
   "stdin": {"mode": "text"}
 }
}

# subcommand-scoped
sfc --metadata          → lifecycle: one_shot
sfc forge --metadata    → lifecycle: persistent

# entry-managing
ai-core-call --metadata          → 工具自身
ai-core-call --entry-metadata M  → 個別 model M
設計規則
  • ▸合法 JSON 是唯一強制；schema 不強制
  • ▸容錯優先：缺欄位 / 不合法 → graceful 降級，不 crash
  • ▸輸入（stdin/flags/files）不影響 metadata 輸出
  • ▸可變維度只有 subcommand 層次（靜態命令層次，非執行期資料）
  • ▸--metadata 必須是唯一旗標，不與資料旗標混用
  • ▸hub 可做擴增 / 過濾 / 彙總（透鏡式轉換）
  • ▸待補欄位：memory_hints、nondeterministic、memoized（詳見 待決議）
Metadata 八軸（core_nature/lib_spec.md）

目前已定義的八個執行特性軸。注意：這些軸描述的是執行特性，而非語意用途（後者是 B1 懸案）。

lifecycle 執行生命週期 one_shot | persistent | stateful_session
state 函式是否有外部狀態 stateless | state_dirs | external
interruptible 是否可安全中斷並恢復 true | false
guarantee 重複執行的保證強度 none | idempotent | transactional
resources 預期資源用量（時間、記憶體） {"time_seconds": 2, "memory_mb": 64}
entries I/O 通道的方向/mode/type {"stdin": {"mode": "text"}, "stdout": ...}
state_dirs 狀態目錄宣告（.config/.cache/.state/.data） {"cache": "~/.cache/myapp"}
has_entries 是否有可用的 --entry-metadata（entry-managing singleton 用） true | false
⚠️ 待補軸（候選）：nondeterministic（LLM 馴化框架觸發根）、memoized（快取策略）。詳見 待決議 C 組。
