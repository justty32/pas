🎯 目標問題（v0 切片）

第一個目標問題：程式碼輔助助手。最小垂直切片設計，以及它會逼出哪些規範懸案。

為什麼要有「目標問題」

⚠️ 沒有目標的框架會無限長

每條軸、每個複合規範看起來都「可能有用」。DECISIONS.md 的所有懸案都找不到優先序。開發會在「規範越做越完整」的感覺中漂移，但什麼都不能用。

✅ 目標問題 = 停止鍵

一旦釘死具體目標，它變成剎車：只有這個目標用得到的部分才做。DECISIONS.md 的懸案也會自動排好序：擋路的先解，沒用到的放著。

方法論翻轉：不要「先把八軸/複合/多函數合作的標準全定完，再選問題」。要「先選目標問題，讓它告訴你哪些標準該先定」。
A/B 兩層分法（避免把兩個東西當一個做）

A 層 — 能用的最小工具（產品）

SFC + 幾個組合原語 + singleton 管理。今天就能交付、忠於初心、天然小。這是產品。

B 層 — 北極星願景（羅盤）

規範家族 + 治理原則 + 飛輪。這是羅盤，用來指引 A 怎麼長，不是 v1。v0 期間 B 層只補「擋路的」，其餘凍結。

第一個目標問題：程式碼輔助助手

場景

手上有一個程式框架。以前：智慧大模型 + AI agent（如 Claude Code）讀框架原始碼 → 生 skill → 輔助寫程式碼。效果好，但每次都燒貴模型。

ai_core 的版本：把它拆成兩層。聰明模型只跑一次性的上層；笨模型跑高頻的下層。

上層（一次性）

讀框架 → 生資產（結構化抽取）

聰明模型讀原始碼，抽出有 schema 的資產：

  • API 簽名 / 用法 snippet
  • 觸發條件
  • few-shot 範例
  • 護欄（guard）條件
  • verifier hook

本質是結構化抽取問題：schema 可驗證，retry_until_valid / guard 咬得住。「結構化抽取也不錯」這個念頭安放在這——它不是另一個目標，是同一個目標的上層。

下層（高頻）

在精準標記的位置填碼（受約束生成）

笨模型最不該做的是「重寫整個檔」（自由度爆炸 → 必錯）。策略：

  • 固定 snippet 寫進 prompt（上層生的資產）
  • 行數助手精準標記「在第 N 行插入 / 填這個洞」
  • 自由度愈小，笨模型愈不會搞砸
  • 先笨模型填，沒過驗證再升級給聰明模型

類比：Claude Code 自己的 Edit 工具要求 exact string match、不准重寫整檔——聰明模型都需要這層約束，笨模型只會更需要。

✅ v0 驗收標準

拿一個真實框架 → 用一次聰明模型呼叫，生出一小份資產（程序 + snippet 庫）→ 再讓一個便宜/笨模型 + 行數助手 + retry/guard，產出「raw 笨模型做不到、包了鷹架就做得到」的正確程式碼編輯。

同一個編碼任務：對照組（raw 笨模型直接寫）失敗，實驗組（ai_core 鷹架）成功。一條線打通，勝過十條規範。

橫切層：使用者習慣

user-profile 橫切層

使用者習慣（如「我是繁中使用者 → 產出都用繁中」）疊在整條鏈上，是 llm_call context binding 的 user-profile 級升級。這不需要一個獨立目標，就是 Calling Packing（元件 2）的延伸。

v0 切片會逼出的規範懸案

跑通 v0 的過程，會讓 DECISIONS.md 的這幾條從「憑空猜」變成「擋路、得解」：

懸案為何被 v0 逼出來出處
B1：語意/用途描述欄位 生資產一定撞到——八軸只描述執行特性，沒有「這環節做什麼、吃什麼參數」 DECISIONS B1
B3：in-process 沙箱 要跑程式碼/測試來驗證笨模型的產出，需要資源與安全隔離邊界 DECISIONS B3
C2：memoize 快取 便宜≠免費；笨模型也要錢/算力。重複輸入該快取 DECISIONS C2
C1：nondeterministic 軸（含證書） 治理原則需要它落地——不只「這裡隨機」，而是「測試組/模型/穩定度%」的證書 DECISIONS C1
A4：組合軸推導 資產被組成調用鏈時，複合 metadata 要能自動算。優先序待 v0 驗證後再定 DECISIONS A4
結論：v0 切片本身就是規範的篩子——跑通它的過程，會告訴我們上面哪幾條是真的擋路、哪幾條可以再放著。
❓ 開放問題（roadmap §8）
  • 最高優先
    固化引擎手動 vs 自動：v0 要不要把「把 LLM 處理過的模糊案例固化成新 matcher」也做進去？還是 v0 先手動固化、只證明「洞填得準」？
    回答這題決定 v0 的工程量差異：手動 vs 自動的複雜度差幾個量級。
  • 待定
    v0 框架選什麼：「一個真實框架」選哪個？用使用者自己手邊的框架最真實。
  • 待定
    笨模型選型：本地 <10B 具體選哪個（llama3、qwen2.5 等）？對照組怎麼設才公平（同一個任務直接 raw 呼叫 vs 加鷹架）？
  • 待定
    行數助手形態：獨立 CLI 函式？還是 Claude Code Edit 那種 exact-match 風格？前者更符合 ai_core 的 function-as-CLI 哲學。
  • 後議
    生資產的 schema：上層「讀框架 → 生資產」的 schema 長什麼樣？這直接連到 B1（語意用途欄位）與 C1（證書）的設計。
  • 後議
    證書放哪、誰簽發、撤照流程（roadmap §3.4）——這是治理原則能不能落地的關鍵。v0 先手動管，後續考慮自動化。
  • 後議
    A/B 開發節奏：v0 期間 B 層（規範家族）只補「擋路的」，其餘凍結？這有助於防止「規範無限擴張」的陷阱。
🧠 v0 在 Heuristic Learning 飛輪中的位置

v0 = 飛輪第一圈

用現在便宜的聰明模型當「資產工廠」，輸出上層抽取的資產（程序/snippet/few-shot）。這些資產讓笨模型在「洞」裡填得更準，等同降低了 LLM 被需要的面積。

這正是 roadmap §2 說的：趁現在聰明大模型還便宜，先用它生產出耐久的資產，讓未來的笨模型去消費。 v0 切片就是飛輪轉第一圈的具體形狀。

v0 驗證

「加了鷹架的笨模型 > raw 笨模型」——這是飛輪能轉的最小證明。

後續固化

v0 成功後，把 LLM 成功處理的案例固化成確定性 matcher，LLM 的「洞」縮小。飛輪正式開轉。
