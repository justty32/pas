A2

單一執行檔多種 lifecycle

高優先
現況

sfc 預設是 one_shot dispatch，sfc forge 子命令是 persistent server。register() 只能宣告一個 lifecycle。已用 register_subcommand("forge", lifecycle="persistent") 表達；sfc forge --metadata 回 persistent、頂層回 one_shot。

建議

採「頂層描述 dispatcher、各 subcommand 各有 scoped metadata」模型（隨 A1 一起）。

你要決定

接受 scoped-metadata 模型？還是規定「不同 lifecycle 必須拆成不同執行檔」？

A3

register() 的 import-time 副作用

高優先
現況

register() 放 module 頂層 → 一 import 就讀 argv / 攔截 / 佔旗標 → 工具無法被當 library 重用。hub 想 import indexer 時撞到。已把 indexer 的 register 移進 main() 修掉。

建議

規範明訂「register 的副作用應延遲到確定以腳本身分執行」——只在 __main__ 呼叫，或 library 提供 lazy 模式。

你要決定

是否把「register 僅在 __main__」寫成慣例 / 由 library 強制？

B3

in-process 的資源隔離

v0 若用沙箱會撞到
現況

python-kind tiny function 是真 in-process（exec），標準庫無法乾淨設記憶體/時間上限。Layer 4 的 --call-timeout 因此只對 shell-kind 生效。

選項

(a) 資源上限只承諾 shell-kind
(b) python-kind 要限制時退回 subprocess（失去 in-process 速度）
(c) 引入非標準庫沙箱（違背 least dependency）

你要決定

Layer 2「真 in-process」與 Layer 4「資源上限」如何取捨？

C 組 — 候選新 metadata 軸/欄位（牽動軸層，需審慎）
C1

nondeterministic: true 新軸

LLM 馴化框架的觸發根
現況

整套 LLM 馴化框架（docs/llm_taming_framework.md）需要一個「這函式是隨機的」的標記作為觸發根。八軸目前無此概念。memoize / retry_until_valid 等馴化組合子都需要知道「輸入函式是不是隨機的」。

背景

與 memoized 同樣處於「沒有既有軸值可隱含觸發」的處境，和「中斷恢復慣例」當初不加欄位的根本不同。

你要決定

加 nondeterministic 軸嗎？（這是 LLM 馴化慣例的入口；不加的話，馴化組合子只能靠呼叫方手動標注。）

C2

memoized: {version, ttl} 欄位

中優先
現況

lib/memoize.py 已實作，使用 sha256 key 組成（D1 已拍板）。但「函式是否啟用 memoize」這件事，在 metadata 中沒有欄位可以宣告——caller 無法從 metadata 知道「呼叫這個函式的結果是否會被快取」。

情境

memoized 沒有既有軸值可隱含（不像 interruptible 有 guarantee 可推導）→ 可能真的需要新欄位。

你要決定

加欄位，還是接受 memoized 純屬 runtime、不入 metadata？

C3

lifecycle 變體「依賴外部 server」

複合規範影響
現況

含 persistent 成員的複合函式（如 sfc + llm_entry_manager），其呼叫本身是 one_shot，但需要 server 在線才能運作。八軸的 lifecycle 沒有這個值。compose_meta.py 目前在複合 metadata 里列 requires_persistent。

背景

這是 A4 組合軸推導的延伸——若組合規則被收進規範，就需要能表達「外部依賴」的軸值。

你要決定

加這個 lifecycle 值/欄位嗎？（可以等 A4 一起決定）

C4

交互終止條件標準化

多函數交互安全
現況

多函數交互（actor↔critic、debate）需要標準化的終止語意，否則可能無限互踢。目前 lib/interact.py 用 max_rounds 強制安全閥（D7 已拍板）。

背景

docs/multi_function_interaction.md §5 提出終止條件應標準化。max_rounds 只是底線，不是語意終止。

你要決定

要不要像 interruptible 那樣把「終止條件」標準化成一個 metadata 概念？
