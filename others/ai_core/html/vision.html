🌟 願景演化

從最初「幾支小 CLI 輔助笨模型」到「LLM 退縮到兩介面的自我改進系統」的演化鏈，以及治理這個系統的核心原則。

演化鏈（一條有內在邏輯的路）
①

前提（約束）

AI 泡沫化 → 聰明可靠的 LLM 太貴 → 假設手上只有便宜的本地 <10B 小笨模型可用。

這個前提不是悲觀，是設計約束。約束產生設計。

②

初心（最小形態）

既然只能用小笨模型，就得做框架去輔助它。CLI 天生適合 LLM → 最初只是幾支小命令行程式：prompt 片段管理 + 組合 + 輸出後處理（＝最原始的 SFC）。

③

第一次擴張

推論需要時間、一次只能回一串 → singleton 資源 + 時間/記憶體管理（＝ LLM Entry Manager 的由來）。

④

質變的領悟

擴著擴著發現這形狀像符號神經網路——一張計算圖，多數節點是確定性的 shell/py 函式，少數節點是那個隨機的 LLM。

⑤

野心的引爆點

LLM 能自己生 prompt → 自我改進迴圈 → 數據飛輪。為此東西從「幾支小程式」膨脹成「整套規範 + 框架 + 概念」。

設計哲學（KISS / Lightweight / No wheel-remake / Least dependency）不是美學偏好，是前提的邏輯必然：一個前提是「窮、只能用便宜算力」的專案，更付不起龐大基礎設施的維護成本。

北極星：LLM 退縮到兩介面

終點形狀

在成熟系統裡，系統大部分是確定性程式碼。LLM 只剩兩個介面，且這兩個介面永遠不會完全閉合（總有新意圖、總有新模糊案例）：

① 意圖翻譯層

人類的模糊意圖 → 程式可理解的結構化意圖。

② 模糊前沿的暫時處理者

還沒被固化成程式碼的模糊地帶，暫由 LLM 扛著，直到被固化撤照。

單位不是「skill」，是「帶 LLM 留白的程序」

LLM 是 cache-miss handler，或一個巨大 switch 的 default: 分支。請求先過確定性前濾網（format matcher），命中走純程式碼；只有落穿前濾網的，才掉進 LLM。

笨模型（未來）

matcher 更多、洞更小（不敢信它）。自由度收縮到最窄，填碼才不會搞砸。

聰明模型（現在）

matcher 更少、洞更大（可放手）。同一套程序，不同的「填充比例」。

佐證：Claude Code 自己的 Edit 工具要求 exact string match、不准重寫整檔——聰明模型都需要這層約束，笨模型只會更需要。

⚖️ 治理原則：拒絕為預設 + 憑證准入

核心命題

舉證責任在「讓 LLM 進來」那邊。 LLM 預設零領地；要佔據某個環節，得通過兩部分閘門。失敗則 LLM 不得進入。兩個類比：最小權限（least privilege）；新藥審批（療效 + 安全性都要過）。

① 必要性（療效）

千錘百鍊證明確定性程式碼辦不到、非 LLM 不可。

② 穩定性（安全性）

在測試組下做得夠穩，過可靠度門檻。

通過後，LLM 環節取得一張「證書」：此環節經測試組 B、用模型 Z、認證穩定度 X%。LLM 不是被信任，是被認證；可被撤照（哪天有確定性解法就踢出去）。整個系統因此可稽核。
🔬 最硬的未決問題：固化引擎

手動固化

人觀察 LLM 處理的案例 → 人寫新的確定性分支。這個專案是「一個好用的工具」，使用者主導固化。v0 切片選擇這條路：先證明「洞填得準」。

自動固化（北極星方向）

聰明模型定期掃 log → 自動提案新的確定性分支。這個專案是「一個會自我改進的系統」。比手動難非常多。v0 是否含固化：待議（最高優先開放問題）。

🧭 設計哲學（前提的邏輯必然）

✂️ KISS

簡單優先。考慮要不要引入框架、抽象層、DSL 時，預設答案是「不要」，除非有明確理由。

🪶 Lightweight

前提是「窮、便宜算力」的專案，付不起龐大基礎設施的維護成本。輕量是邏輯必然。

🚫 No wheel-remake

litellm、FastAPI、uv 已存在就用。try_implement 原型只用 Python 標準庫，示範極限情境。

📦 Least dependency

每多一個相依就多一個維護負擔。相依是負債，不是資產，要謹慎引入。
