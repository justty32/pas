ai_core

讓 LLM 退縮到「意圖翻譯」與「模糊前沿」兩介面的基底
以「拒絕為預設、憑證准入」治理 LLM 領地，並用飛輪持續把模糊地帶固化成確定性程式碼

— 為 Heuristic Learning 提供基礎設施 —

KISS Lightweight No wheel-remake Least dependency 構想 + 規範 + 原型 try_implement: 140 assertions passing Python 3.11+ / CLI-first
💡 願景演化鏈
🌱

初心（最小形態）

假設未來只剩便宜的本地 <10B 小模型可用 → 需要框架輔助小笨模型。CLI 天生適合 LLM，最初概念只是幾支小命令行程式：管理 prompt 片段、組合、後處理。

SFC + LLM Entry Manager = 起點
🌟

北極星（成熟形態）

系統大部分是確定性程式碼，LLM 只剩兩個介面：

① 意圖翻譯層：人類模糊意圖 → 程式可理解意圖

② 模糊前沿：尚未被固化的模糊地帶，暫由 LLM 扛

LLM 不是被信任，是被「認證」
⚙️

飛輪（遷移機制）

現在：趁聰明模型還便宜，讓它作為「資產工廠」產出耐久資產（程序/snippet/few-shot/骨架）。

未來：笨模型作為「資產消費者」在窄任務上填碼套用。

固化 = 把 LLM 換成確定性程式碼（撤照）

🏗 架構元件（五個核心）
元件 1

LLM Entry Manager

ai-core-server + ai-core-call

統一管理多個 LLM 模型/API 的呼叫入口。LLM 是 singleton 資源，採佇列模式集中管理 consume rate（token / 金錢 / GPU）。

pattern: Entry-managing Singleton
litellm → ollama / lm-studio / gemini
元件 2

LLM Calling Packing

ai-core-call (context binding)

把 LLM 呼叫包裝成具語意的函式。基底 llm_call(str)→str，疊加 context 與 post-processing 形成新函式。

user-profile 橫切層
llm_call_coding_question() 等
元件 3

Function Hub

ai-core-hub + ai-core-hub-server

掃描函式集，呼叫 --metadata 彙整清單。Hub 是「透鏡」——聚合＋轉換 metadata，對 LLM 呈現 skill 清單。

one-shot + server 雙形態
MCP / openai-tools / anthropic-tools export
元件 4

Small Function Center

ai-core-sfc

把許多 tiny function（一行 shell / 薄 LLM 包裝）集中到一個 dispatcher，避免檔案爆炸。4 層架構：store → intake → router → forge server。

git-style subcommand CLI
Layer 0–4 已原型化
元件 5

Author

ai-core-author

一站式產生函式骨架 + dry-run + 進入 registry。AI 生的函式需通過 dry-run + 範例驗證才能進 hub，避免黑箱增殖。

generate → dry-run → register
多語言骨架（bash / python）
caller (人類 shell / AI agent / 其他 function) │ ├── 直接 shell call ─────────────────────────────┐ │ ▼ │ 任意語言寫的 function │ （唯一硬規則：--metadata 印合法 JSON） │ │ │ ├── 單純函式：直接執行 │ └── 重資源 ──▶ wrapper CLI ──▶ singleton server │ ├── 想知道有哪些 function ──▶ Function Hub (one-shot / server) │ └── 想新增 function ─────────▶ ai-core-author (generate → dry-run → register) LLM 呼叫路徑： ai-core-call ──▶ ai-core-server (FastAPI + asyncio) ├─ per-model FIFO queue + worker ├─ rate limit (rpm / tokens / cost) └─ litellm router ──▶ ollama / lm-studio / gemini
📋 --metadata 協議（跨元件契約）
最小合法 metadata（任何函式都得輸出）
{
  "lifecycle": "one_shot",
  "state": "stateless",
  "interruptible": true,
  "guarantee": "idempotent",
  "resources": {},
  "entries": { "stdin": {"mode":"text"} }
}

# subcommand-scoped 範例
ai-core-sfc --metadata          # lifecycle: one_shot
ai-core-sfc forge --metadata    # lifecycle: persistent

# entry-managing singleton
ai-core-call --metadata         # 工具自身
ai-core-call --entry-metadata gemini-flash  # 個別 model
設計規則
  • 合法 JSON 是唯一強制要求，schema 不強制
  • 容錯優先：缺欄位 / 不合法 JSON → graceful 降級，不 crash
  • 輸入（stdin/flags）不影響 metadata；可變維度只有 subcommand 層次
  • --metadata 必須是唯一旗標，不與其他旗標混用
  • 所有 caller / hub / author 都必須預期函式可能根本沒寫
  • hub 可對 metadata 做擴增 / 過濾 / 彙總（透鏡式轉換）
  • 待補欄位：memory_hints / nondeterministic / memoized（見 thinking.md）
⚖️ 治理原則（拒絕為預設 + 憑證准入）

LLM 預設零領地。要進入某個環節，需通過兩部分閘門：

① 必要性（療效）

千錘百鍊證明確定性程式碼辦不到、非 LLM 不可。舉證責任在「讓 LLM 進來」那邊。

② 穩定性（安全性）

在測試組下做得夠穩，過可靠度門檻。通過後發給「證書」：哪個模型、測試組、穩定度 %。

洞的大小 = f(模型能力)

笨模型 → matcher 更多、洞更小。聰明模型 → matcher 更少、洞更大。同一程序、不同填充比例。

撤照機制

哪天有人想出確定性解法，LLM 就被踢出去（撤照）。飛輪 = 從寬鬆遷往嚴格的力。

上層（一次性）

讀框架 → 生資產（結構化抽取）

聰明模型讀原始碼，抽出有 schema 的資產：API 簽名 / 用法 snippet / 觸發條件 / few-shot / 護欄 / verifier hook。本質是結構化抽取，schema 可驗證，retry_until_valid 咬得住。

下層（高頻）

在精準標記的位置填碼（受約束生成）

笨模型不重寫整個檔。固定 snippet 寫進 prompt + 行數助手精準標記「在第 N 行插入」。自由度愈小，笨模型愈不會搞砸。類比：Claude Code 的 Edit 工具要求 exact string match。

✅ 驗收標準：同一個編碼任務，對照組（raw 笨模型）失敗，實驗組（ai_core 鷹架）成功。
