Level 1

初始探索：技術棧與整體架構概覽

定位本專案為 Godot 4.4 起手模板（非遊戲），純 GDScript、GL Compatibility 渲染器，靈感源自 Maaack。盤點頂層目錄、8 個 plugins、16 個 autoload、場景流、29 語系 i18n、CI/CD 與命名慣例。

  • 頂層目錄結構（godot/root/ 收納所有實體內容）
  • 16 個 autoload 分三類與其載入順序約束
  • 主場景 BootSplash → MenuScene 場景流
  • CI（gdlint threshold=0）/ CD（itch.io）概覽
  • 專案類型判定：模板 D（UI/框架）為主、模板 A（遊戲）為輔
速覽

一句話總結

以 16 個 autoload 為骨幹，把第三方 plugin 全部 Wrapper 化 成 enum / Resource 驅動的型別安全介面；再用 SignalBus 觀察者模式 解耦全域通訊；UI 採 Component-Driven Design（Builder 自動注入），設定與存檔分別走 INI / JSON 雙持久化。

相對 Maaack 的差異化擴充：系統化 Wrapper 層、設定/存檔分離、反射式存檔、Builder 模擬 Traits、29 語系 i18n、HACKS 文件化、多玩法範例。

Level 3 — 三個最具特色的子系統
Level 3A

設定（INI）與存檔（JSON）雙持久化

兩套獨立持久化機制的完整剖析：設定走 INI、存檔走 JSON + 簽章，皆以「約定 + 反射自動化」壓低擴充成本。

  • ConfigurationController 模板方法生命週期
  • 命名約定自動註冊（Loader 免改）
  • SaveData 反射式序列化、§§§ 簽章健壯性
  • 自動存檔、加密、base64 匯入匯出
Level 3B

場景流、玩法抽換與 Builder 元件注入

三個「換模組不改框架」的關鍵接縫：Wrapper + Resource 驅動轉場、GameScene 鴨子型別抽換玩法、Builder 條件式元件注入。

  • 轉場呼叫鏈：enum 尋址 + 預載 Options .tres
  • GameScene 以鴨子型別弱耦合（"player" in game_content）
  • UiBuilder 自動注入 TwistMotion / ControlFocusOnHover
Level 3C

HACKS（Godot issue workaround）與 Web 整合

把引擎已知 issue 的 workaround 文件化，重點剖析 Web 剪貼簿的 JavaScriptBridge JS 注入。

  • HACKS 清單：已解 [x] / 待辦 [ ]
  • Web 剪貼簿（#81252）原生 dialog 注入流程
  • Theme → CSS 主題橋接、同步阻塞輪詢
  • font tween→scale tween、play_stream bus 等細節
