A. 純資料接點（只寫 XML）

README：「basic resources and settlements can be created in XML alone… Adding new events or policies is possible through XML, too!」

想新增Def 類別限制（何時被迫升級到 B）
新據點類型WorldSettlementDef只用內建 SettlementTypeExtension 即純 XML；要改建立/驗證/升級曲線得自訂 extension 子類
新資源類型ResourceTypeDef產量綁既有 FCStatDef 即純 XML；非線性產量/特殊過濾需 ResourceProduction/FilterExtension
新據點建築BuildingFCDef升級樹與依賴皆 XML 表達；要建築主動行為需 BuildingFCExtension
新事件FCEventDef觸發條件純 XML；有選項抉擇/特殊效果需 FCEventHandlerExtension
新敕令/政策FCPolicyDef純增益/減益/禁用動作即 XML；要每稅期主動行為需 FCPolicyBehavior 子類
biome 產出 / 事件獎勵 / 軍事工作 / stat 軸BiomeResourceDef / ResourceEventRewardDef / MilitaryJobDef / FCStatDef數值純資料；全新行為需 B

A 類風險最低：不改 DLL、不破壞存檔、不衝突 Harmony。最大限制＝只能組合既有「行為原語」；任何「主動發生的事」都需要 B。

註冊慣例：[StaticConstructorOnStartup] 內 XxxRegistry.Register(new MyImpl())。Registry 不序列化，每次開機要重註冊；Invoke 對每個參與者單獨 try/catch（壞了默默記 LogUtil.Error）。

⚠ 關鍵陷阱：Game.ClearCaches 清空全部 Registry

每次開新局/讀檔時 Game.ClearCaches 觸發 EmpireCacheUtil.InvalidateAll，清空全部 registry（HarmonyPatches/CachePatches.cs:36-50,72-86）。只在 StaticConstructorOnStartup 註冊一次 → 第一次讀檔後擴展靜默失效。

解法：用 EmpireCacheUtil.RegisterCacheInvalidator(id, RegisterAll) 重註冊 —— invalidator 字典不會被清，且回呼在 ClearAll 之後執行（CachePatches.cs:55-59）。連核心自身都這麼做（FactionFC.cs:524-527「Re-register … in case ClearCaches ran」）。

B-2. Bridge 走讀（Patch-RW）：官方橋接的 9 種常用模式
