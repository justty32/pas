# 尋路架構深化：PF 引擎與 A* 演算法 (源碼剖析)

Freeciv 的尋路系統 (Path-Finding Engine) 是其 AI 與自動化單位的靈魂。它不僅解決「如何從 A 走到 B」，更處理了複雜的敵方控制區 (ZoC)、兩棲登陸、燃料限制與風險規避。本文件深入剖析 `common/aicore/path_finding.c` 與 `pf_tools.c`。

## 1. 核心演算法：加權 Dijkstra 變體

Freeciv 的 PF 引擎本質上是 Dijkstra 演算法的擴展，透過一個精密的「複合成本模型」來達成類似 A* 的效果。

### 1.1 複合成本公式 (Combined Cost, CC)
在 `path_finding.h:60` 中定義了核心決策公式：
```c
total_CC = PF_TURN_FACTOR * total_MC + move_rate * total_EC;
```
- **`total_MC` (Move Cost)**: 時間成本。單位移動所需的總點數。
- **`total_EC` (Extra Cost)**: 偏好/風險成本。
- **`PF_TURN_FACTOR` (65536)**: 一個巨大的常數。這保證了**時間優先原則**——除非繞路能避開極大的風險，否則 AI 永遠選擇最快到達的路徑。

### 1.2 優先權隊列 (`map_index_pq`)
系統使用一個展平的地圖網格 `lattice` 與優先權隊列進行導算：
- 每次 `pf_map_iterate()` 會彈出當前 `total_CC` 最低的節點。
- 為了偏好低成本路徑，隊列儲存的是 `-total_CC`。

## 2. 動態權重與風險規避

PF 引擎的靈活性來自於一組高度抽象的校調函數 (Callbacks)：

### 2.1 移動成本 (`get_MC`)
- **基礎地形**: 從 `movement.c` 讀取（如：道路為 1/3，森林為 2）。
- **兩棲切換 (`amphibious_move`)**: 對於三棲單位，系統會動態切換陸地與海洋的成本計算邏輯，並在切換點（登船/下船）注入額外的代價。

### 2.2 敵方控制區 (Zone of Control, ZoC)
在 `pf_normal_map_iterate` 循環中，系統會執行硬性物理阻斷：
```c
if (node->zoc_number != ZOC_MINE && node1->zoc_number == ZOC_NO) {
    continue; /* ZoC 阻斷：禁止從敵方控制區直接移動到另一個敵方控制區 */
}
```
這解釋了為什麼 AI 單位能精確地繞過敵人的前線防禦工事，而不會因為尋路 bug 誤撞包圍圈。

### 2.3 風險與危險圖圖層 (`pf_danger_map`)
當單位具備「風險規避」意識時（如開拓者）：
- 系統會啟用「影子 Dijkstra」子循環。
- **死路預判**: 如果某條路徑會導致單位在回合結束時停留在敵方射程內（`to_danger && cost >= moves_left`），該路徑會被標記為 `PF_IMPOSSIBLE_MC`。

---

## 3. 特殊導航：燃料與補給 (`pf_fuel_map`)

針對空軍單位，PF 引擎引入了燃料維度：
- **續航力檢查**: 在每個節點記錄 `fuel_left`。
- **補給點吸引**: 在 `get_EC` 中，具備機場或航空母艦的方格會返回負的額外成本（吸引力），引導飛機在巡邏後能自動規劃出返回加油的路徑。

---

## 4. 尋路智慧的「層級」
Freeciv AI 根據情境使用不同的 PF 參數設定：

| 模式 | 核心校調函數 | 說明 |
| :--- | :--- | :--- |
| **快速進攻** | `normal_move` | 忽略危險，只求最短路徑，允許中途交戰。 |
| **隱密滲透** | `no_fights_or_unknown` | 排除所有非盟友單位格，禁止進入未知黑幕。 |
| **資源開發** | `overlap_move` | 允許一小步跨入非原生地形，用於處理碼頭或渡輪對接。 |
| **開拓擴張** | `adv_avoid_risks` | 將敵方射程內的方格 EC 設為無限大，強迫繞遠路。 |

---

## 5. 工程見解
- **記憶體預分配 (`lattice`)**: PF 引擎在啟動時會根據地圖大小預分配一個巨大的節點陣列。雖然耗內存，但消除了導算過程中的動態內存碎片，這對於 AI 每回合數百次尋路請求至關重要。
- **原子性操作**: `pf_map_path()` 是一個「懶加載」函數。如果目標點尚未被導算，它會自動觸發 `pf_map_iterate` 遞歸直到觸及目標，展現了極佳的調用封裝。
- **全向廣播**: 當一個單位移動後，系統不會只更新一條路徑，而是透過 `pf_map_iterate` 同時探索周邊的所有可能性，這讓 AI 具備了「中途隨機應變」的能力。
