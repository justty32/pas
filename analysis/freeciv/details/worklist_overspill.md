# 城市架構深化：工作清單與生產溢出邏輯 (源碼剖析)

在 Freeciv 的城市管理中，「生產」是帝國擴張的核心。系統不僅要處理單一物件的建造，還需要處理跨項目的產能繼承與自動化隊列。本文件剖析 `common/worklist.c` 與 `server/cityturn.c`，解構其溢出產能 (Overspill) 與工作清單 (Worklist) 機制。

## 1. 核心資料結構：生產隊列 (`struct worklist`)
定義於 `common/worklist.h`。這是一個長度有限的陣列，儲存了玩家預設的「未來計畫」。

```c
struct worklist {
  int length;
  struct universal entries[MAX_LEN_WORKLIST]; /* 最大 64 項 */
};
```
- **通用抽象 (`struct universal`)**: 由於生產隊列中可以混雜「單位」、「建築」與「奇觀」，系統使用通用物件抽象。這使得工作清單可以在不同的 ruleset 下無縫運作。
- **自動前進 (`worklist_advance`)**: 當前項目完成後，系統會自動呼叫此函數，將索引移至下一項。如果工作清單為空，城市會進入「未生產」警告狀態。

---

## 2. 產能溢出系統 (Production Overspill)
當一個城市每回合產生 50 產能，但正在建造的「戰士」只需 10 產能時，剩下的 40 產能去哪了？

### 2.1 累加與繼承
在 `server/cityturn.c` 的回合處理中，系統執行以下邏輯：
1. **產能注入**: `pcity->shield_stock += city_production_build_shields(pcity);`
2. **完工判定**: 檢查 `pcity->shield_stock >= total_cost(pcity->production)`。
3. **溢出計算**: 
   - `overspill = pcity->shield_stock - total_cost;`
   - 系統不會丟棄這部分數值，而是將其保留在 `pcity->shield_stock` 中，並立即將生產項目切換為下一項。

### 2.2 規則限制
並非所有產能都能無損轉移。Ruleset 中可以定義 **`shield_loss_on_change`**：
- **項目切換懲罰**: 如果玩家在中途手動切換生產項目（例如：從「蓋奇觀」切換到「造長槍兵」），系統會根據規則扣除 50% 或更多的已累積產能。
- **溢出上限**: 某些規則限制溢出量不能超過下一個項目的總成本，多餘的會被浪費，這防止了玩家透過低價單位「囤積」超高額產能來秒蓋奇觀。

---

## 3. 溢出產能的「變現」機制
在極少數情況下（如：建築被占領、技術進步導致無法再生產該項目），溢出的產能會觸發補償：
- **強制轉換**: 系統會自動將產能轉化為最接近的合法項目。
- **金錢補償**: 在特定 ruleset 下，無法轉換的產能會以固定比例（如：4 產能 = 1 金錢）存入國庫，雖然這在預設規則中較少見。

---

## 4. 自動化管理：全域工作清單
Freeciv 允許玩家定義「全域工作清單模板」（Global Worklists）。
- **實作邏輯**: 客戶端傳送 `PACKET_WORKLIST_APPLY`。伺服器會將模板內容深度拷貝 (`worklist_copy`) 到城市的私有結構中。
- **動態校驗**: 在載入模板時，系統會調用 `can_city_build()` 逐項校驗。如果模板中包含該城市因地形（如：內陸城市不能造船）或技術而無法建造的項目，該項會被標記為非法並跳過。

---

## 5. 工程見解
- **通用物件的好處**: 透過 `struct universal`，生產系統與具體的單位屬性完全解耦。這意味著即使是 Lua 腳本動態生成的特殊物件，只要符合 `Kind_...` 定義，就能進入生產流水線。
- **產能守恆與確定性**: Freeciv 的產能累加全部使用整數運算。即便在極高溢出的情況下（如：一回合秒蓋五個戰士），系統也能保證在網路同步中不會產生捨入誤差。
- **回饋迴圈**: 生產完成觸發 `city_refresh`，重新計算維護費與人口情感。這種鏈式反應保證了遊戲狀態的即時一致性。
