核心架構（三大支柱 + 兩大狀態）
支柱一

Signal Bus 事件匯流排

子系統不互持引用，全部透過 FieldEvents / CombatEvents 的 signal emit/connect 通訊。combat_triggered 是探索→戰鬥的唯一切換管道。

支柱二

Composition over Inheritance

Battler 用 @export PackedScene 動態掛載 BattlerAnim 與 CombatAI；Gamepiece 為「笨物件」，行為由 Controller 子節點提供。

支柱三

Resource-as-Template

BattlerStats / BattlerAction 是 .tres 原型，_ready() 時 duplicate() 成獨立實例，避免共享污染。

狀態 A

探索（field）

Field（Node2D，主場景根）。Gameboard 格子移動、Gamepiece、Cutscene、Pathfinder。常駐，戰鬥時 hide()。

狀態 B

戰鬥（combat）

Combat（CanvasLayer）。兩階段回合制：階段一全員選 action，階段二依 speed 降序執行。常駐但 hide()。

GDExtension

C++ 後端化方向

戰鬥計算 / 回合排程 / 狀態機 / 傷害公式移往 C++；Godot 只留資料接收與顯示。詳見 target/ 衍生計畫。

關鍵資料流
探索 → 戰鬥（狀態切換，全程經 Signal Bus）
CombatTrigger._execute → FieldEvents.combat_triggered → Combat.setup → CombatEvents.combat_initiated → Field.hide()
回合制戰鬥（兩階段）
next_round → 階段一：AI + 玩家逐一選 action → _play_next_action 依 speed 排序 → Battler.act → cached_action.execute → turn_finished（遞迴）
傷害結算（近戰攻擊）
base_damage + source.attack → ±10% 隨機浮動 → BattlerHit(damage, to_hit) → take_hit → stats.health -= damage
Resource 原型 → 執行期實例
.tres 原型（設計時） → @export 掛到 Battler → _ready() duplicate() → 獨立實例（互不污染）
分析重點發現
readiness / ATB 是未實裝的設計遺跡：docstring 與 readiness_saved 欄位殘留，但現行 combat.gd 為純回合制，順序僅由 speed 降序決定。
傷害公式目前未納入 defense / evasion / 元素相剋（elements.gd 的 ADVANTAGES 表未接線）——這是教學專案刻意留給學習者擴充的空白。
