Level 1
初始探索

專案定位

GDQuest 的 2D 回合制 JRPG 教學示範（非框架）。嚴守 GDQuest GDScript Style Guide，充分使用 GDScript 4 新特性。涵蓋回合制戰鬥、物品欄、角色成長、過場/對話、網格移動、多層選單。

進入點與分工

主場景 src/main.tscn → Field（Node2D）。九個 Autoload 為鬆耦合骨幹。資料層（combat/·overworld/）與邏輯層（src/）分離，設計師與程式設計師互不踩線。

Level 3 · A
回合制戰鬥系統深度剖析

兩階段回合制

階段一：敵方 AI 先選，玩家後選（_select_next_player_action 支援按「back」回溯前一位）。階段二：_play_next_action 把可行動者依 speed 降序排序逐一執行，turn_finished 遞迴推進。

Battler 生命週期

_ready() 把 stats / actions 全部 duplicate()（原型→實例）。組合驅動：battler_anim_scene / ai_scene 以 PackedScene 動態掛載並做型別檢查。

傷害 / 命中公式

modified = base_damage + source.attack；damage = modified ±10%；to_hit = hit_chance × (source.hit_chance/100)；命中判定 randf()×100 < to_hit。負傷害即治療。

BattlerStats 修飾器系統

final = round(max(0, base×(1+Σmult)+Σmod))。add_modifier/multiplier 回傳 uid 供裝備卸下時移除。

★ 重要發現：readiness/ATB 為未實裝遺跡，現行為純回合制；傷害公式尚未納入 defense/evasion/元素相剋（教學留白）。

Level 3 · B
Signal Bus 與事件系統

FieldEvents

探索態事件匯流排。combat_triggered(arena) 是探索→戰鬥的唯一切換管道；input_paused(bool) 由 Cutscene 靜態旗標自動發出，凍結/解鎖全 field 輸入。

CombatEvents

戰鬥態事件匯流排。combat_finished(victory) 一對多廣播：同時被 field.gd（show()）與 combat_trigger.gd（await 取勝負）消費，兩者互不知情。

設計原則：任何子系統都不持有彼此引用、不 get_node() 找對方——全部經 autoload 事件匯流排通訊（控制反轉）。這正是 GDExtension 後端化的天然事件接口雛形。

Level 4
Resource 驅動資料 與 場域系統

.tres 原型模式

所有戰鬥數值/戰技做成 Resource，編輯器 inspector 視覺化編輯。執行期 duplicate() 成獨立實例，避開 Godot Resource 跨節點共享的污染陷阱。

Inventory 唯一存檔

專案中只有 Inventory 持久化（user://inventory.tres，ResourceSaver/Loader）。Battler 數值、進度、地圖位置目前不存檔（教學留白）。

Gameboard 格子系統

格↔像素互轉、A* Pathfinder、GamepieceRegistry 一格一棋子。棋子移動自動觸發 Pathfinder enable/disable 佔用格，全套無需手動同步。

field ↔ combat 切換

兩狀態同時存在於場景樹，靠 show()/hide() 切換（非 change_scene），唯一觸發點為 combat_triggered 事件接力。

隊伍（party）

無獨立 party 資料結構——隊員即 BattlerRoster 下 is_player==true 的 Battler 節點；探索態操控者由 Player.gamepiece 單一持有。

Gamepiece 笨物件

繼承 Path2D，只負責佔格與沿 Curve2D 平滑移動；主動行為由 Controller 子節點提供（組合優於繼承）。
