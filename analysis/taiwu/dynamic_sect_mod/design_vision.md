# 中小門派 mod — 設計願景 (design_vision.md)

> 建立：2026-05-23。本檔是使用者口述的**設計構想留檔**，非實作規格、非可行性結論。記錄當下的願景與想法，供日後逐項展開討論與實驗。
> 接續脈絡：[`../player_faction_research/`](../player_faction_research/)（脊椎=OrganizationInfo、佔位 sect、TakeRevenge 指揮）、[`../npc_population_research/`](../npc_population_research/)（NPC 生成、九階位、武功 config）、[`../details/sect_skill_favor_ui.md`](../details/sect_skill_favor_ui.md)（門派 UI）、[`../new_sect_mod/`](../new_sect_mod/)（陳家堡＝靜態大派的踩雷經驗）。
> 與陳家堡的根本差異：陳家堡是**世界生成就存在、佔地圖聚落槽的靜態大派**；本構想是**遊戲過程中動態誕生、複用既有產業建築當據點的中小門派**。

---

## 一、核心願景

太吾世界目前**只有 15 個大門派**，缺了「中小門派」這一層。本 mod 要補上這個缺口。

- **動態生成**：中小門派不在開新世界時生成，而是**在遊戲進行過程中逐漸誕生、成長**。
- **小型 vs 中型的區別**：
  - **小型門派**：其實算不上門派，更像是**普通的師徒關係**——一個創派者帶著一兩個弟子。
  - **中型門派**：兩個條件——① 在某個地點的**產業視圖中擁有一格屬於它的地盤（據點）**；② **人數達到 10 人以上**。達成後門派「升格」，解鎖一系列功能。
- **玩家對等性（最重要的一點）**：太吾（玩家自己）也要能**成為師傅、創立門派**，享有與 NPC 師傅相同的整套機制。

## 二、框架化與 JSON 可配置（架構願景，2026-05-23 補充）

> 使用者要求：**可以的話，把這做成一個框架，讓內容能用 JSON 定義，方便其他玩家自己做 mod。**「可以的話」＝盡力而為的架構目標、非硬性需求。

- **核心理念**：本 mod 不只是「一個寫死的中小門派系統」，而是一個**資料驅動的平台**——把「門派內容」與「程式邏輯」分離，內容用 JSON 宣告式定義，玩家不必碰 C# 就能新增自己的門派類型、契機、武功變異。
- **預期可 JSON 定義的東西**（純資料部分）：
  - **門派類型/目標**：商會、鏢局、醫館、毒師、鄉勇、採藥人、工匠、家族……每種的名稱、據點建築、初始武學根源、性格傾向。
  - **創派契機**：條件 → 結果的規則（如「身份=醫師 且 仁善 且 富有 且 已悟武功 且 有眼緣弟子」→ 觸發創派為醫館）。
  - **階級表**：每個階級的名稱、福利、權利、月俸、月任務類型（採集/販賣/醫人…）。
  - **武功變異規則**：以哪個既有武功為根源、改動哪些數值/五行/階位/特效、品階升降路徑（下品→上品）。
  - **據點建設**：對應哪些既有產業建築、所需資源/財富、與城主關係的減免規則。
  - **外交/地區影響參數**：對文化/治安的影響值、與其他門派關係的初值與變化規則。
- **架構連接（已知事實）**：太吾 mod 本就走 `Config.lua` manifest（前端 GBK）＋ YAML（後端 net6 UTF-8）＋ ConfigData 注入（`AddExtraItem`/`AddOrModifyItem`）。框架做法＝**mod 啟動時讀玩家放的 JSON → 翻譯成對應的 ConfigData 注入 / 事件定義 / 行為參數**。JSON 比 ConfigData 二進位易讀易寫，適合非程式玩家。（註：太吾原生資料層是 YAML/Lua，採 JSON 是為了對外的易用性，需 mod 自己做 JSON→引擎資料的轉接層。）
- **資料 vs 邏輯的分界（關鍵設計約束）**：純資料（門派類型、階級表、數值、武功變異）天然適合 JSON。但涉及**行為判定**（契機觸發、弟子 AI、據點建設時機）的部分，不可能讓玩家寫任意程式碼——框架要預留一組**「規則原語／可組合條件」**（如賦性、屬性門檻、關係值、財富、所悟武功等可宣告的條件積木），讓玩家用組合而非編程來表達邏輯。這層原語的設計是框架成敗關鍵，須與六項基礎實驗 mod 一起演化。

## 三、門派的生命週期（成長敘事）

1. **創派者**：一個 NPC，**身具武功**。
2. **收徒**：他收下弟子。
3. **創派宣告**：某一天他說「我們來創立門派吧」，於是一個小門派誕生。
4. **初期**：可能只有一兩個弟子，**弟子跟著師傅行動**。
5. **建據點**：之後師傅可能**砸錢**，在某個地點的產業視圖中**建立一個根據地**（如鏢局、醫館之類，複用既有建築）。
6. **升格中型**：有了據點之後**招人速度變快**，人數達標 → 成為中型門派。
7. **解鎖功能**：中型門派可啟用一些功能，例如：
   - 對地區施加影響（文化、治安等）。
   - 與其他門派的外交關係。

## 四、要先做的基礎實驗 mod（驗證概念）

> 使用者列的 6 項。每項都要同時涵蓋「NPC 版」與「太吾（玩家）版」。

### 1. 收徒與傳授行為
- NPC 的**收徒行為**與**傳授（教武功）行為**。
- 太吾的**收徒**與**傳授行動**。
- 重點：這會**影響 NPC 之間的關係聯繫**（師徒關係的建立）。

### 2. 師傅對弟子的指令／要求
- 師傅可**要求弟子跟隨自己行動**。
- 師傅可**要求弟子滿足某種要求**，例如：
  - 上貢（給師傅進貢財物）。
  - 去醫好某些人。
  - **在一年內掌握某項武學**。
- 太吾成為師傅時要有相關**選項**（下達要求）。
- 太吾身為弟子時要有相關**事件**（接受/回應師傅的要求）。

### 3. 中小門派的外交、地區影響、階級制度
- **外交關係**：門派與門派之間的關係。
- **對地區的影響**：包括文化、治安等。
- **公開收徒事件**。
- **階級分類（參考《三國志》系列）**——除了階級的「名稱」之外，還要有階級的**福利與權利**：
  - **高位（如長老）**：可參與**門派會議、參與決策**，並有**月俸**。
  - **低位**：可能**每月都有任務**，包括去採集資源、販賣物品、醫治救人等。

### 4. 相關 UI
- 即「武功列表」UI（已調查＝`UI_CombatSkillTree`，見 [`../details/sect_skill_favor_ui.md`](../details/sect_skill_favor_ui.md)）。
- 觸發路徑設計：**點擊產業視圖中的門派據點 → 對話框 → 選擇「觀看門派介紹與狀況」→ 跳出此 UI**。
- 門派據點：打算**複用遊戲內的資源**（中小門派往往是鏢局、醫館之類，正好對應既有建築）。

### 5. 門派創立的「理由」與「契機」
小門派只是師徒；**要變成中型門派，必須有一個「目標」**。目標往往受**創派者身份**影響。
- **可能的門派目標／類型**（受創派者身份決定）：商會、鏢局、文化人相關、治病救人、毒師、鄉勇、採藥人、工匠、家族……等。例：醫師創立的就是醫館。
- **契機鏈（以醫師為例）**：什麼契機會讓一個醫師收徒、乃至建據點創派？
  1. 醫師**機緣巧合悟出了武功**。
  2. 機緣巧合**收下有眼緣的弟子**。
  3. 因其**仁善且富有**，並受其**賦性**等因素影響，**觸發創派契機**。
- **【重點待記】據點建設與城主關係減免**：若創派者在建立據點時，與**目標城鎮的一階 NPC（如城主身份者）關係好**，那麼所需的**資源或財富可以減免，乃至免費**。
- **家族門派**：很常見，但**往往是小型**。要成為中型，通常因為：
  - 當代師傅**想賺錢** → 成立武館、開班授課，將武學傳給家族外的人；或
  - 家族內的人太多 → 決定**開門派來管理**。

### 6. 武功的來源 — 「武功進化與變異」mod（可慢慢討論）
受限於遊戲美術資源，**武功只能從原有的那些來**，但做修改／改進。因此另需一個 mod：
- **概念**：把遊戲內的 **15 個大派當作「武學根源」**。創派祖師學了這些原本的武學後，**進行創新改進、或變化退化**：
  - 名字不變，但**效果與威力稍作修正**；或
  - **特效略有變化**；乃至
  - **整個五行都變了**。
- **實作方向**：在遊戲途中加入武功＝**把原有武功複製一份**，**遊戲美術資源不變**（除非五行或階位有變），然後**數值做變化**。
- **最期待的效果**：創派祖師習得了**下品武學**，經過一代代的改進，變成**上品武學**——雖然美術資源僅稍作變化，但那也很好。

## 六、開放問題（日後討論／實驗時釐清）

- 中型門派「升格」的判定與狀態存哪（自訂欄位？借 OrganizationInfo？過月檢查人數＋據點？）。
- 「對地區施加影響（文化/治安）」對應遊戲哪些既有數值，能否讀寫。
- 門派外交關係要不要借既有門派好感網（陳家堡發現大派好感網寫死 1..15，但有 `index>=0` 防護自動略過非大派——中小門派的關係可能要 mod 自建一張表）。
- 「公開收徒事件」「契機觸發」用官方事件編輯器（`TaiwuEventItem`）做最自然，條件鏈（賦性/仁善/富有/眼緣）對應哪些角色欄位。
- 武功進化的「下品→上品」具體是改 `CombatSkill` 的哪些欄位（品階、五行、招式數值），美術資源綁定關係（哪些欄位一改美術就會跟著變）。
- 太吾版的整套（收徒/傳授/下令/創派/升格/據點建設減免）UI 與事件入口怎麼鋪。
- **JSON 框架（§二）**：JSON schema 怎麼設計、放哪、何時讀（前後端各持一份 config → 可能前後端各讀一次）；「規則原語／可組合條件」要暴露哪些角色/門派欄位當積木；JSON→ConfigData/事件 的轉接層放前端還後端；錯誤 JSON 的容錯（後端未捕捉例外＝整個 GameData 進程崩潰，解析務必 try-catch＋防呆）。
