# Wesnoth 原始碼全解析百科全書 - 卷四：螞蟻的導航儀（路徑搜尋與空間幾何）

為什麼你在遊戲裡點一下目的地，單位就能繞過高山、避開敵軍，甚至鑽過傳送門抵達？這一切都仰賴 `src/pathfind/astarsearch.cpp` 中的 A* 演算法。

## 核心函數：`heuristic` (啟發式函數與偏置魔法)

A* 演算法的核心是「猜測」：它需要猜測當前位置到終點大約還要走多遠。猜得越準，找路越快。但在六角格地圖上，猜測會遇到一個大問題。

### 1. 六角格的「等價路徑」災難
在方形網格中，直走通常最短。但在六角格中，因為形狀的關係，往往有成千上萬條形狀不一樣（有的是閃電型，有的是圓弧型），但**步數完全一樣**的路徑。
如果不用點小手段，電腦在找路時會把這些一模一樣短的路「全部算一遍」，這會卡死你的 CPU，而且最後畫出來的路線會像蚯蚓一樣歪七扭八。

### 2. 神秘的九億分之一：破解平局
```cpp
double heuristic(const map_location& src, const map_location& dst) {
    // 1. 計算真實的投影直線距離 (歐幾里得距離的基礎)
    double xdiff = (src.x - dst.x) * 0.75;
    double ydiff = (src.y - dst.y) + ((src.x & 1) - (dst.x & 1)) * 0.5;

    // 2. 公式核心
    return distance_between(src, dst) + (xdiff*xdiff + ydiff*ydiff) / 900000000.0;
}
```
**白話解讀**：
- `distance_between`：這是六角格上的標準步數距離（例如：還要走 5 步）。
- `(xdiff*xdiff + ydiff*ydiff)`：這是標準的直線距離（畢氏定理的平方）。
- **關鍵操作**：工程師把它除以了 **900,000,000 (九億)**！
- **為什麼要這樣做？**
    1. 這個數字加起來通常只有 0.0001。它絕對不會影響「誰走得比較少步」的判定。
    2. **但是**，當電腦遇到兩條都是「5 步」的路徑時，電腦會看後面的小數點。直線距離越短的那條路，小數點就越小。
    3. 電腦就會說：「哦！雖然都是 5 步，但這條路的直線距離比較短（視覺上比較直），我選這條！」
    4. 就靠這九億分之一的微小推力，Wesnoth 完美解決了六角格 A* 演算法的效能危機，並畫出了最優美的移動路線。

## 核心函數：`a_star_search` (A* 主循環與 ZOC 阻斷)

### 3. ZOC (控制區) 如何變成一堵牆？
在 A* 演算法中，如果遇到不能走的牆壁，移動成本是無限大。但 ZOC（敵軍旁邊的格子）是可以走進去的，只是進去後「不能再動」。

這在程式碼的 `cost_calculator` (在 `pathfind.cpp` 內) 裡是如何處理的？
```cpp
double cost = n.g + calc.cost(loc, n.g);
```
**白話解讀**：
- 當演算法走到一個格子 `loc` 時，會呼叫 `calc.cost()` 來問：「這步要花多少移動力？」
- 如果這個格子有敵人的 ZOC，`calc.cost()` 會非常殘酷地回傳：**「你剩下的所有移動力！」**
- 假設你的騎兵有 8 點移動力，走了 2 步進到 ZOC，他會被扣掉剩下的 6 點，瞬間歸零。
- 對 A* 演算法來說，這代表這條路徑的「成本瞬間爆表」，它就會自動放棄這條路，轉而去搜尋有沒有可以繞過 ZOC 的路線。

### 4. 傳送門的捷徑 (Teleportation)
```cpp
if (teleports && !teleports->empty()) {
    const auto& allowed_teleports = teleports->get_adjacents(n.curr);
    locs.insert(locs.end(), allowed_teleports.begin(), allowed_teleports.end());
}
```
**白話解讀**：
一般的找路，只能看相鄰的 6 個格子。但如果地圖上有傳送陣（例如銀影精靈的技能），程式碼會作弊！它會把傳送門的出口，**直接當作你的鄰居** (`allowed_teleports`) 塞進去找路清單裡。對演算法來說，相隔十萬八千里的兩個傳送陣，在它眼裡就跟隔壁鄰居一樣近。
