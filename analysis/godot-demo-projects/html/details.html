1 · 2d/platformer — 完整 2D 平台遊戲

為何選它

repo 中最完整、最具教學價值的 2D 玩法 demo，是學「一個小遊戲怎麼組起來」的範本。主場景由 game_singleplayer.tscn / game_splitscreen.tscn 提供，掛載 game.gd。

關鍵腳本剖析

  • game.gd:1-26 頂層只管全域輸入（F 切全螢幕、Esc 切暫停），暫停用 get_tree().paused（對應 misc/pause）。
  • player/player.gd（最值得讀）：重力從專案設定讀取（:17）、終端速度上限（:37）、move_toward() 做加速度緩衝（:39-40）。
  • 跳躍手感：提前放開縮短跳躍 velocity.y *= 0.6（:33-35）、二段跳（:82-85）。
  • 動畫狀態用 get_new_animation() 輕量 if/else（:62-76），對比正式 FSM。
  • player/gun.gd：Timer 做射速冷卻、set_as_top_level(true) 解耦子彈座標（:21）。

可遷移的設計重點

  • 重力讀專案設定而非寫死，方便全域調參。
  • 跳躍手感三件套：終端速度上限、提前放開縮短、二段跳。
  • action_suffix 模式：用一個字串後綴讓同一控制腳本支援多玩家（:15,31,39,52）。
  • set_as_top_level 解耦投射物與發射者座標。

對照：物理驅動版 2d/physics_platformer；正式狀態管理 2d/finite_state_machine；3D 對應 3d/platformer。

2 · 2d/finite_state_machine — 有限狀態機設計模式

為何選它

把「狀態機」抽象成可重用的基礎類別，再用繼承實作具體玩家狀態，是組織複雜角色行為的標準模式。值得對照 2d/platformer 的「用 if/else 選動畫」輕量做法，理解何時該升級成正式 FSM。

兩層設計 + 關鍵腳本

  • state_machine/state.gd：定義所有狀態的介面契約 enter()/exit()/handle_input()/update()（:10-28），靠 signal finished(next_state)（:6-7）轉移，狀態之間不互相引用。
  • state_machine/state_machine.gd：_enter_tree() 連接子狀態 finished 訊號（:33-36）；委派 update()/_unhandled_input()；用 states_stack 支援 push/pop（:70-84），"previous" 回上一層。
  • player/player_state_machine.gd：覆寫 _change_state 把 stagger/jump/attack 用 push_front（:21-30）；覆寫 _unhandled_input 把攻擊當可打斷全域輸入（:33-43）。
  • player/player_state.gd：用 Dictionary 集中定義狀態名常數（:3-13），避免魔術字串。

可遷移的設計重點

  • 訊號驅動轉移：狀態 emit finished，由狀態機切換——狀態彼此不耦合。
  • 狀態堆疊：暫時狀態用 push/pop，永久狀態用替換；"previous" 是回上一狀態的關鍵字。
  • 基類 + 特化繼承：通用機制寫一次，遊戲特化只覆寫 _change_state / _unhandled_input。
  • 可打斷輸入上提：把能中斷任何狀態的輸入（攻擊）放在狀態機層攔截。
3 · compute/texture — Compute Shader 即時填充材質

為何選它

repo 中最完整的 compute shader 範例：用 GPU compute 即時計算水面漣漪，並把結果材質回灌到一般渲染管線。涵蓋 RenderingDevice 低階 API、push constant、ping-pong 多緩衝、以及「在 render thread 上執行」這些進階主題。比入門的 compute/heightmap 深一階。

核心流程

  • 執行緒模型（最重要）：所有 RD 操作鎖在 render thread，用 RenderingServer.call_on_render_thread(...)（water_plane.gd:39,55,129）。
  • 初始化 _initialize_compute_code（:154-195）：取主 RD、載 .glsl 建 pipeline、建 3 張 R32_SFLOAT 紋理、9 組 uniform set。
  • Ping-pong 三緩衝（:139-143）：一張寫入、一張上幀、一張上上幀；二階波方程需前兩幀狀態。每幀 next_texture = (next_texture+1)%3（:117）。
  • 每幀分派 _render_process（:198-243）：手動組 push constant 對齊到 8 個 float、dispatch group = (N-1)/8+1、compute_list_dispatch。
  • 輸入：滑鼠射線打水面 Area3D 換算紋理座標加波點（:70-92），無互動時隨機落雨（:104-110）。

可遷移的設計重點

  • compute 要接進渲染就用主 RD + call_on_render_thread，所有 RD 操作鎖在 render thread。
  • Texture2DRD 是 compute 輸出回灌材質的橋樑：改它的 texture_rd_rid 就切換顯示紋理。
  • ping-pong 多緩衝處理需要歷史幀的迭代演算法（波、流體、生命遊戲）。
  • push constant 手動對齊到 16 byte 邊界（此處湊 8 個 float = 32 byte）。
  • dispatch group = (N-1)/local_size + 1，搭配 shader 內邊界 discard。
4 · networking/websocket_chat — WebSocket 聊天室

為何選它

repo 中把 WebSocket 低階 API 講得最完整的 demo：同時示範「在 Godot 內自架 WS server」與「連到 server 的 client」，並手動處理握手、輪詢（poll）、收發訊息與廣播。相較於 websocket_minimal 與 websocket_multiplayer，本 demo 直接操作 TCPServer / WebSocketPeer，最能看懂底層運作。

Server 核心剖析

  • WebSocketServer.gd 用 class_name 註冊，對外只暴露三個訊號（:4-6）。
  • 握手狀態推進 _connect_pending（:133-169）：TCP → (TLS) → WS 的逐幀狀態機，用 PendingPeer 追蹤；逾時丟棄（:104-106）。
  • 主迴圈 poll()（:91-130）每幀由 _process 呼叫：接新 TCP、推進握手、對每 peer poll、偵測斷線、emit 收到的封包。
  • send()（:48-65）用 peer_id 正負號做路由：0=廣播、<0=廣播排除一人、>0=點對點；依型別選 send_text 或 var_to_bytes。
  • client.gd 示範「可重用網路類」與「畫面業務層」分離（:13-50）。

可遷移的設計重點

  • WebSocketPeer 是 poll 驅動：每幀 poll() 是收訊、偵測斷線的前提。
  • 握手是多階段狀態機：TCP → (TLS) → WS，用 PendingPeer 逐幀推進並設逾時。
  • peer_id 正負號當廣播語意（0=全體、負=排除一人）是簡潔的路由技巧。
  • 字串 vs 二進位用 was_string_packet() / var_to_bytes 對稱處理。
  • 把 server/client 封裝成 class_name 節點，對外只露訊號與少數方法。
關鍵心智模型：WebSocketPeer 不會主動推事件，必須每幀 poll()，否則收不到資料、也偵測不到斷線。
