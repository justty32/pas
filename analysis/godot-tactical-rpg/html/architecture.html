Level 1 — 初始探索：技術棧與整體架構
Level 1

技術棧與整體架構概覽

Godot 4.3 / mobile 渲染 / 純 GDScript 戰棋範本。頂層目錄 assets/（素材）vs data/（程式），後者再分 models/（資料+邏輯）與 modules/（場景節點）。

  • 專案基本資訊與授權（MIT）
  • Model / Module / Service 三層慣例與 service of services
  • 入口初始化流程：main.tscn → main.gd → tactics_level._ready
  • 場景節點概觀（Camera/Controls 屬 main.tscn 全域常駐）
三層架構摘要

Model / Module / Service

Module（節點，轉發）· Service（RefCounted，無狀態演算法）· Resource（.tres，持有狀態、可共享）。

project.godot 用資料夾顏色標示：addons 紅、assets 藍、models 綠、modules 黃。

Level 2 — 核心模組職責與資料流
Level 2

核心模組職責與資料流

五大模組（Arena / Tile / Pawn / Participant / Camera / Controls）的權責劃分與耦合點。頂層回合驅動器 TacticsLevel 是整局的「心臟」。

  • 八階段 stage 機（0~7）：玩家走全部、對手只走 0~4
  • 呼叫鏈：Level → Participant.act → turn_service → controls/player/combat
  • 共享 .tres 與 stage 變數即事件機制（無事件匯流排）
  • 跨層 cam.target 聚焦機制
關鍵不對稱

玩家 vs 對手 stage

玩家 0→7（移動與攻擊分兩條動作分支，輸入驅動）；對手 0→4（選棋→追擊→移動→選目標→攻擊，自動完成）。

陣營制「整隊行動」：先把己方所有 pawn 動完才換邊，無個別速度順序。

Level 3 — 四個核心子系統深度剖析
Level 3 · 場地

格子地圖、移動範圍與 Pathfinding

Blender 方塊 mesh 執行期格子化、射線探鄰居（高度差即跳躍門檻）、BFS flood-fill 移動範圍、reachable/attackable/hover 標色、pf_root 回溯重建路徑。

  • tiles_into_staticbodies 動態生成 StaticBody3D + 腳本
  • 無 (x,y) 座標系統，全靠 RayCast3D
  • BFS 全圖標記 + 事後距離過濾（一份結果服務移動與攻擊）
  • 路徑是 Array[Vector3] 世界座標串列
Level 3 · 回合與戰鬥

回合制流程、移動物理與戰鬥判定

三層嵌套回合（turn_stage / stage / can_move·can_attack）、stage 推進細節、pawn 沿路徑移動含跳躍重力、極簡傷害公式。

  • 動作選單可用性由 can_move/can_attack 動態 enable
  • 移動是逐格趨近 + move_and_slide，跳躍/落下用真實重力
  • 傷害 = curr_health += (-attack_power)，無防禦/命中/暴擊
  • 無顯式死亡移除（歸零只灰階、仍擋路）、無勝負判定
Level 3 · 敵人 AI

敵人 AI 決策鏈

stage 0~4 全自動：選棋（第一個能動）→ chase_nearest_enemy（BFS 找最近敵人鄰格，走不到就沿 pf_root 退到最近可達格）→ 等移動 → 選射程內最弱 → 攻擊。

  • 啟發式：純距離（pf_distance）+ 血量，無威脅/陣型評估
  • strategy 列舉欄位目前完全未被使用（預留 hook）
  • 擴充切入點：替換三個啟發式函式即可，不動共用層
Level 3 · 攝影機與輸入

攝影機系統與輸入處理

四層樞紐節點分離平移/yaw/pitch/zoom；五種行為 + 聚焦目標；滑鼠射線投影拾取（mask 1=tile / 2=pawn）；鍵鼠手把雙模分流（is_joystick）。

  • static var 當輕量全域狀態匯流排（代價：不利多相機）
  • 象限吸附 + fov 縮放 + 球形邊界三個「成品級」收尾
  • 聚焦目標 res.target 是相機與遊戲流程唯一耦合點
