原型系統 ✅ 完成

PrototypeManager 三步驟

① load_file() 讀 YAML 序列（raw_defs_）→ ② resolve_inheritance() DFS 拓撲排序 + component flat merge → ③ spawn(em, id) 呼叫已登錄的 ComponentLoader 建立實體。

core/prototypes/prototype_manager.h+cpp

ComponentLoader — 零反射

型別 = std::function<void(registry&, entity, const YAML::Node&)>。呼叫方用 register_loader("Spatial", lambda) 顯式登錄。未登錄的 component 在 apply_to 時靜默略過（前向相容）。

坑①：yaml-cpp iterator_value 在 MSVC 的 operator= 歧義

range-based for 回傳 YAML::detail::iterator_value，MSVC 偏好 template operator=<T> 而非 copy assignment，觸發 C2027 "use of undefined type"。

修正：改用 index 存取 root[i]

for (std::size_t i = 0; i < root.size(); ++i) { YAML::Node node = root[i]; }

坑②：YAML::Node map 複製共享 detail::node，跨原型靜默污染

複製 YAML::Node map entry 只複製 handle（引用計數），底層 detail::node 共享。子原型賦值 proto.components["Spatial"] = new_node 會呼叫 AssignNode → set_ref，修改共享節點，導致所有繼承同一父原型的子類被靜默污染。

修正：繼承 merge 時對所有 component node 呼叫 YAML::Clone()

父原型 component 複製 + 子原型自身 override，兩者都用 YAML::Clone(parent_node)，確保每個原型持有獨立的 detail::node。

save_load — 三層 API

① stream（ostream/istream）→ ② path（檔案路徑）→ ③ SaveStore（抽象後端）。FolderSaveStore 以 string slot name 對應一個 .bin 檔案。

core/serialize/save_load.h, save_store.h
坑①：entt::entity == entt::null_t 在 doctest CHECK() 造成 C2593 歧義

doctest 的 Expression_lhs<entt::entity> 有 template operator==<T>；EnTT 也有 operator==(entity, null_t)，兩者同等匹配，MSVC 報 C2593 ambiguous。

修正：先求值成 bool 再 CHECK

bool parent_null = (sp.parent == entt::null); CHECK(parent_null);

坑②：cereal 序列化 std::string / std::vector 無 include 時 static_assert

cereal 不預設知道 std containers。MetaDataComponent::proto_id（string）與 MapData::tiles（vector<Tile>）都需要顯式 include。

修正：在 save_load.h 加入兩個 include

#include <cereal/types/string.hpp> + #include <cereal/types/vector.hpp>

繼承 Node vs RefCounted

持有模擬狀態、需要 _ready 生命週期 → Node（OpenNefiaWorld）。無狀態工具 → RefCounted（OpenNefiaCore）。與 core_data_layer_design.md 慣例二/四一致。

Signal Bus 模式

核心完成動作後 emit world_changed；GDScript .connect() 訂閱並刷新渲染。C++ 不知道 GDScript 如何渲染，保持單向依賴。對應 Signal Push 模式。

坑①：MapData::get() 無 const 版本，在 const 方法中報 C2662

OpenNefiaWorld 的查詢方法標記為 const，但 MapData::get(x,y) 沒有 const 過載，MSVC 報 C2662。

修正：改用 in_bounds(x,y) + at(x,y)（at 有 const 過載）

if (!map.in_bounds(x, y)) return false; return map.at(x, y).is_walkable();

坑②：EnTT 空 struct emplace 回傳 void，EntityManager wrapper 宣告 C& 觸發 C2440

NpcAiComponent 是空 struct（無資料成員）。EnTT 對空型別使用特殊 storage，registry::emplace<T>() 回傳 void 而非 T&。EntityManager::emplace 原宣告 C& emplace(...)，MSVC 報 C2440 "cannot convert from void to C&"。

修正：回傳型別改為 decltype(auto)，自動跟隨 EnTT 行為

template<typename C, typename... Args> decltype(auto) emplace(entt::entity e, Args&&... args) { return reg_.emplace<C>(e, std::forward<Args>(args)...); }

多樓層 + BSP 地城

TILE_STAIR_DOWN（tile.h）

旗標 1 << 2。放在 BSP 地城最後一個房間中心（需至少 2 個房間）。hero 踩到 → next_floor()：++current_floor_; setup_map(); recompute_fov(); emit floor_changed()。英雄 HP 保留；NPC + 物品 + 地圖重新生成。

BSP 地城 60×40（map_gen.h+cpp）

取代原固定 20×15 測試地圖。回傳 std::vector<Room>，各房間中心依序放英雄（rooms[0]）、NPC/物品（中間）、樓梯（back）。NPC 上限 min(4 + floor, 8)。

WorldStateComponent + 存讀檔

WorldStateComponent

掛在 map_entity_ 上，隨 AllComponents 一併序列化。解決「C++ member variable（turn_count/floor）不在 ECS 中，無法隨 save/load 持久化」的問題，不需 sidecar 檔案。

core/components/world_state_component.h

save_game() / load_game()

save：同步成員變數 → WorldStateComponent → cereal::PortableBinaryOutputArchive。load：registry.clear() → snapshot_loader → 重建 map/hero entity 指標（MapData / proto_id == "hero"）→ 還原 turn/floor。

F5/F9 GDScript 整合

ProjectSettings.globalize_path("user://opennefia_save.bin") 取得真實路徑。F5/F9 在 _unhandled_input 的 if _dead: return 前攔截，game_over 後仍可操作。讀檔成功後設 _dead = false 恢復輸入。

load 後 entity 指標失效

registry.clear() 後所有 entt::entity handle 作廢。需重查：map_entity_ ← view<MapData> 第一個；hero_entity_ ← view<HeroComponent> 第一個（F6 起；舊用 proto_id == "hero"）。此處早已預警「不能依賴 exclude 模式」——F6 在 npc_ai_system 印證了這個脆弱性，見下節。

設計決策：以語意查詢重建指標，而非快取

每次 load_game() 都用語意查詢（MapData / HeroComponent）重建所有 C++ 指標。既無需快取機制，也不依賴實體 ID 的穩定性（snapshot_loader 能保留原始 entity ID，但此做法更不脆弱）。

F6 — GUI 實機觀察 + 戰鬥 bug 修復 + 英雄辨識正規化（2026-06-02）

首次在 Linux/godot-mono 圖形視窗實玩，抓到只在真機浮現的戰鬥 bug：英雄撞 NPC 能傷敵，但自身永遠不掉血。挖出雙重缺陷與一條設計通則。

主因：排除法辨識英雄選錯實體

npc_ai_system 舊用 view<SpatialComponent>(exclude<NpcAiComponent>) 取「首個非 NPC 有座標實體」當英雄，但物品也符合（有 Spatial、無 NpcAi）。EnTT 單 component view 由 storage 尾端往前迭代，真機建立順序「英雄 → NPC → 物品」使物品落尾端 → hero_ent 誤指向物品（無 Health）→ 攻擊 try_get<Health> 回 null 打空。

次因：行動機率閘門擋掉攻擊

if (pct_dist >= move_chance) continue; 擺在系統迴圈最前，把視野更新 + 鄰接攻擊一起擋掉，低 move_chance 弱怪短兵相接常還不了手。

單元測試為何沒先抓到

初版回歸測試把物品建在英雄之前，反向迭代剛好先碰到英雄、矇混過關——正是排除法脆弱性的鐵證。修正後案例 2 刻意把物品建在英雄/NPC 之後（對齊真機順序）才重現 bug。

根因消除：HeroComponent 正向標記（決策 01）

準則：辨識某類實體就給它正向 tag，用 view<Tag,...> 查；不要用 exclude 列舉「它不是什麼」。新增空 tag HeroComponent（is_empty → snapshot 空型別最佳化、零 payload），系統改 view<HeroComponent, SpatialComponent> 唯一鎖定；列入 all_components.h，gbind 建立時 emplace、load 路徑改用它找回 hero_entity_。詳見 ../../derived/opennefia-cpp/docs/decisions/01_entity_identification.md。

次因修法 + 回歸測試

重構為視野每回合必更新、警覺且鄰接必定攻擊，閘門下移只管移動。新增 tests/src/test_npc_combat.cpp 4 案例（核心戰鬥 / 真機 pool 順序 / move_chance=0 仍攻擊 / 空 tag round-trip）。ctest 40 cases / 146 assertions 全綠。

GUI 實機工具鏈

godot_test/map_view.tscn（MapView/Sprite2D/Camera2D/CanvasLayer/InfoLabel，script 以路徑引用）+ project.godot 設 run/main_scene。godot-mono 4.6.3 開窗實玩（NVIDIA OpenGL），移動/等待/重開/存讀檔/攻擊全綠。

HeroComponent — 空 tag 可序列化

std::is_empty_v == true → EnTT snapshot 只存 entity id、不呼叫 serialize()，零成本。存讀檔後英雄辨識邏輯與執行期完全一致。
