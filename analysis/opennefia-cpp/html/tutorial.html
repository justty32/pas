1. 新增一個 Component（核心層）

以「HealthComponent」為例。

① 定義 struct

建 src/core/components/health_component.h。必須有 template<class Archive> void serialize(Archive& ar) 以支援 cereal 存讀。

struct HealthComponent {
    int max_hp{ 10 };
    int cur_hp{ 10 };
    template<class Archive>
    void serialize(Archive& ar) { ar(max_hp, cur_hp); }
};

② 加入 AllComponents

src/core/serialize/all_components.h 的 type_list 加一行，save/load fold 自動覆蓋——不需要改其他序列化代碼。

using AllComponents = entt::type_list<
    MetaDataComponent,
    SpatialComponent,
    MapData,
    HealthComponent  // ← 新增
>;

③ 視需要加 YAML loader

若原型 YAML 要設定 HP，在初始化時呼叫 register_loader：

pm.register_loader("Health",
  [](auto& reg, auto e, const auto& n) {
    HealthComponent h;
    if (n["max_hp"])
      h.max_hp = n["max_hp"].as<int>();
    h.cur_hp = h.max_hp;
    reg.emplace_or_replace<HealthComponent>(e, h);
  });
重建後跑 ctest，現有 36 cases 應全部繼續通過。再寫 CHECK(em.has<HealthComponent>(e)) 的新測試。
2. 新增一個系統（自由函式）

以「每 tick 回復 1 HP 的再生系統」為例。

系統函式

void regen_system(
    entt::registry& reg,
    SystemCtx& ctx) {
  auto view = reg.view<HealthComponent>();
  for (auto e : view) {
    auto& hp = view.get<HealthComponent>(e);
    if (hp.cur_hp < hp.max_hp) ++hp.cur_hp;
  }
}

登錄到 EntityManager

em.add_system(opennefia::regen_system);
// 或 lambda：
em.add_system([](auto& reg, auto& ctx) {
  // ...
});

執行順序 = add_system 的呼叫順序。無需排序標記。

關鍵原則：系統是無狀態自由函式，所有需要的服務透過 SystemCtx 傳入——不需要繼承、不需要介面、不需要 DI 框架。

4. 橋接新資料到 Godot（gbind 層）

以「在 GDScript 取得 hero HP」為例。

C++ 端（opennefia_world_gd）

// .h 宣告
int get_hero_hp() const;

// .cpp 實作
int OpenNefiaWorld::get_hero_hp() const {
  if (!em_.registry().valid(hero_entity_))
    return 0;
  const auto* hp = em_.registry()
    .try_get<HealthComponent>(hero_entity_);
  return hp ? hp->cur_hp : 0;
}

// _bind_methods() 登錄
ClassDB::bind_method(
  D_METHOD("get_hero_hp"),
  &OpenNefiaWorld::get_hero_hp);

GDScript 端

info_label.text = "HP: %d/%d" % [
  world.get_hero_hp(),
  world.get_hero_max_hp()
]

重建：cmake --build build --target opennefia_gd，複製 .dll 到 godot_test/bin/。

新增 .cpp 後需先重跑 cmake -S . -B build ... 讓 GLOB_RECURSE 拾取新檔。

6. 新增 NPC 種類（YAML 原型，F7 流程）

以「Golem」（高 HP 低移速）為例，橫跨 5 個位置。詳細見 howto_add_npc_type.md。

    CombatStats:

④ 生成機率（pick_proto）

// setup_map() 裡的 pick_proto lambda
auto pick_proto = [&](int) {
  int r = d(rng);
  if (current_floor_ <= 2)
    return r < 8 ? "Putit" : "Golem";
  // 深層 golem 比例提高
  if (r >= 7) return "Golem";
  // ...其他分支
};

回傳**原型 id 字串**（"Golem"），pm_.spawn() 從 YAML 建實體，無需改其他 C++ 代碼。新 enum 值大於現有最大值；舊存檔未知值走 default 安全降級。

7. 新增物品類型

以「HP 上限強化藥水」（max_hp_boost）為例。詳細見 howto_add_item_type.md。

① ItemType 加值

enum class ItemType : uint8_t {
    health_potion = 0,
    max_hp_boost  = 1  // ← 新增
};

② move() 改 switch

switch (item.type) {
  case health_potion:
    hp->hp = min(hp->hp + item.value,
                 hp->max_hp); break;
  case max_hp_boost:
    hp->max_hp += item.value;
    hp->hp     += item.value; break;
  default: break;
}

③ GDScript UI

func _on_item_picked_up(
    name: String, val: int):
  match name:
    "health_potion":
      label.text = "+%d HP" % val
    "max_hp_boost":
      label.text = "最大HP +%d" % val
設計備注：item.value 欄位語義依 type 而異（回血量 vs HP 上限提升量）。若未來物品效果複雜，可考慮拆成獨立 Component（如 HealOnPickupComponent）。

9. 把 C++ 成員變數納入 ECS 存讀檔

OpenNefiaWorld 的 turn_count_ / current_floor_ 不在 ECS 中，如何讓它們隨存檔持久化？詳細見 howto_save_load_pattern.md。

解法：StateComponent 掛在 map_entity_

// 1. 定義 Component
struct WorldStateComponent {
    int turn_count{ 0 };
    int current_floor{ 1 };
    template<class Archive>
    void serialize(Archive& ar)
    { ar(turn_count, current_floor); }
};

// 2. 加入 AllComponents（自動序列化）
using AllComponents = entt::type_list<
    ..., WorldStateComponent
>;

存前同步 / 讀後重建

// 存檔：先同步 members → Component
ws->turn_count = turn_count_;
serialize::save(reg, path);

// 讀檔：clear() 後 handle 失效！
// 用「語義查詢」重建指標：
for (auto e : reg.view<MapData>())
    map_entity_ = e;
// 用 HeroComponent 正向 tag（F6+）
for (auto e : reg.view<HeroComponent>())
    hero_entity_ = e;
// 還原 members
turn_count_ = ws->turn_count;

舊作法（MetaDataComponent::proto_id == "hero" 比對）在 F6 後已廢棄；改用空 tag HeroComponent 正向辨識，抗迭代順序、可序列化。

關鍵坑：registry.clear() 讓所有舊 entt::entity handle 失效。讀檔後必須用「持有特定 Component」或「MetaData::proto_id 比對」等語義查詢重建 C++ 端的快取指標，不可保留舊值。
