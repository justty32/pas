關鍵模組詳解

server/main.py — 組裝入口

唯一啟動入口，承擔所有依賴注入：

  • game_instance dict：全局狀態容器
  • GameSessionRuntime：runtime 包裝
  • public_query_builders：查詢函數集
  • command_handlers：寫入指令集
  • ConnectionManager：WS 連線管理

classes/core/world.py — 世界容器

World dataclass 包含：

  • Map、MonthStamp（時間）
  • AvatarManager、MortalManager
  • EventManager、DeceasedManager
  • GatheringManager、RankingManager
  • Dynasty、WorldLore、SectContext
  • sect_wars、sect_relation_modifiers

classes/core/avatar/ — 角色核心

分 mixin 組合：

  • core.py：屬性、境界、靈根
  • action_mixin.py：動作執行狀態
  • inventory_mixin.py：物品/裝備管理
  • info_presenter.py：資訊格式化輸出

server/loop_runtime.py — 遊戲循環

後台異步無限迴圈：

  • 呼叫 sim.step() 推進時間
  • 收集 tick 事件與角色更新
  • 透過 WS manager.broadcast() 推送
  • 定期觸發自動存檔

sim/save/ + sim/load/ — 存讀檔

Mixin 模式，各類自行處理序列化：

  • 只存 JSON 基礎類型
  • 跨物件引用只存 ID
  • RunConfig 隨存檔保存
  • 事件庫用 SQLite 獨立文件

config/settings_service.py — 設定服務

三層設定分離：

  • 只讀：static/config.yml（世界規則）
  • 用戶：settings.json + secrets.json
  • 本局：RunConfig 快照
  • LLM API Key 不回傳前端
主要耦合點與資料流

寫入序列化（最重要）

所有外部寫入（API / 天道干預 / 存讀檔）必須在 Simulator.step() 的間隙執行，透過 command_handlers 統一序列化，禁止路由層直接操作 game_instance。

事件作為解耦介面

動作、宗門、社交的結果均以 Event 物件傳遞，由 EventManager 統一入庫（SQLite）。前端透過 WebSocket 接收事件流，REST API 可分頁查詢歷史事件。

LLM 集中管控

所有 LLM 呼叫均透過 src/utils/llm/client.py，按任務名稱自動選擇 NORMAL/FAST 模式，Semaphore 防止過度並發，錯誤統一分類回報前端。

SectContext 惰性初始化

World.sect_context 是惰性載入的宗門作用域，封裝「本局啟用宗門」邏輯。優先讀 world.existed_sects，再回退全局 sects_by_id，避免不同局的宗門互相污染。

值得注意的設計決策
決策做法原因
LLM 不用 SDK urllib 直呼，自行實作 OpenAI 相容 + Anthropic 兩套 降低依賴，凍結打包體積小，且行為完全可控
game_instance 用 dict 全局共享 Python dict，而非強類型類別 lifespan 管理方便，各服務注入靈活；代價是型別檢查較弱
存檔只存 ID 引用 跨物件引用只存 ID，加載時重建關係 避免循環引用序列化問題，存檔體積小
config.yml 只讀 世界規則不可在運行時修改，用戶設定走另一套 保持遊戲平衡可控，避免存檔跨版本兼容問題
角色扮演不進存檔 roleplay runtime 狀態不序列化 roleplay 是臨時 runtime 接管，存讀檔後顯式清空
事件 SQLite 持久化 事件用獨立 SQLite，分頁查詢 事件可累積上千筆，JSON 存檔只保留最後 1000 筆，DB 支援分頁
