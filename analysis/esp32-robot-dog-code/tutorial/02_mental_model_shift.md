# 教學 02：從資深軟體工程師視角看嵌入式韌體

作為一名資深軟體工程師，你習慣了作業系統（OS）幫你處理好一切（記憶體管理、執行緒調度、I/O 緩衝）。但在這台機器狗（ESP32）上，你就是作業系統。

## 1. 執行模型的轉變：從事件驅動到循環驅動
在 Web 或 App 開發中，你通常是在等待「事件」（如 HTTP Request 或 Click）。但在機器人控制中，核心是**控制循環 (Control Loop)**。
*   **250Hz 的承諾**：本專案的核心是一個每 4 毫秒 (4000us) 執行一次的循環。在這個循環裡，你必須完成「感知 -> 計算 -> 執行」。
*   **抖動 (Jitter) 是敵人**：如果你的計算花了 5 毫秒，或者 WiFi 中斷卡住了 CPU 1 毫秒，機器狗的腿就會「抖一下」。這在嵌入式中稱為實時性 (Real-time) 挑戰。

## 2. 記憶體：沒有虛擬記憶體，沒有 Swap
ESP32 有約 520KB 的 RAM。
*   **靜態分配優先**：在韌體開發中，資深工程師傾向於在編譯時分配好所有內存（如全域物件），避免在運行時頻繁 `new` 或 `malloc`，以防止堆積碎片 (Heap Fragmentation) 導致系統在運行數天後崩潰。
*   **Stack Overflow**：每個 FreeRTOS Task 都有固定的 Stack 大小。如果你的遞迴太深或區域變數太大，系統會直接重啟。

## 3. 硬體就是一個 API (Memory Mapped I/O)
在嵌入式中，控制硬體（如 GPIO、PWM）本質上是讀寫特定的內存地址。
*   **暫存器 (Registers)**：硬體內部的配置開關。
*   **驅動程式 (Drivers)**：就是一層薄薄的包裝，把讀寫暫存器的行為變成 `set_pwm_width()` 這樣的函數。

## 4. 本專案的 C++ 實踐
你會發現 `libs/` 下使用了 C++ 類別。這在現代嵌入式中很常見，但要注意：
*   **避免異常 (Exceptions)**：ESP32 雖然支援，但會極大增加二進位檔大小與延遲。
*   **避免 RTTI**：同樣是為了性能。
*   **精簡的類別**：類別主要用於封裝狀態（例如每條腿的當前座標）。
