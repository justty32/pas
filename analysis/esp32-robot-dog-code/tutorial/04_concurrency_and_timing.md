# 教學 04：多核心協調與任務調度 (FreeRTOS)

ESP32 是一顆雙核心處理器。對於資深開發者來說，這就是一個典型的**多執行緒併發問題**。

## 1. 任務分配：Core 0 vs Core 1
本專案利用 FreeRTOS 將任務固定在不同核心：
*   **Core 1 (運動核心)**：執行 `loop()`。它的優先級最高，邏輯必須保持簡單、無阻塞。
*   **Core 0 (服務核心)**：執行 `servicesLoop()`。處理 WebSocket、HTTP Server、IMU 讀取。

## 2. 為什麼 WiFi 會搞砸一切？
在嵌入式中，WiFi 協議棧是非常消耗資源的，且它通常運行在 Core 0。
*   **中斷風暴**：當 WiFi 收到大包數據時，Core 0 會被佔用。
*   **共享資源**：如果你的運動代碼（Core 1）需要讀取 IMU 數據（由 Core 0 更新），你就需要同步機制。
*   **阻塞風險**：如果 Core 0 在處理 WiFi 時卡住了 I2C 總線，Core 1 的運動循環也會跟著卡住，這就是為什麼機器狗會突然「癱瘓」的原因。

## 3. 定時器循環：`vTaskDelayUntil`
在 PC 上你可能會用 `sleep(4)`。但在嵌入式中，如果你想精確地每 4ms 執行一次，你需要補償計算本身消耗的時間。
*   **傳統做法**：`delay(4)`。這是不精確的，因為你的計算可能花了 1ms，結果總週期變成了 5ms。
*   **資深做法 (ESP-IDF 遷移重點)**：使用 `vTaskDelayUntil`。它會記錄上一次喚醒的時間點，自動補償計算耗時，確保兩次任務執行的開始時間點間隔永遠精確為 4ms。

## 4. 數據交換：二進位 vs JSON
作為資深工程師，你會發現本專案在 WebSocket 中使用了二進位格式 (`packagesProcess.ino`)。
*   **效能**：在 ESP32 上解析 JSON 字串是非常昂貴的（CPU 與記憶體）。
*   **延遲**：二進位數據包體積小，傳輸快，解析只需簡單的位移運算 `(b1 << 8 | b2)`。這對於 20Hz 以上的遙測數據回傳至網頁面板至關重要。
