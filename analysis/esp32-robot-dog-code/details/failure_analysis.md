# 專案失敗原因分析 (Failure Analysis)

README 提到「Project Closed」且「基本上不再運作」，經過對代碼庫與現行生態系的對比，分析出以下核心原因：

## 1. 核心庫斷裂 (Core Library Rot)
*   **ESP32_ISR_Servo 衝突**：專案依賴的高精度 PWM 庫 `ESP32_ISR_Servo` 在 ESP32 Arduino Core 2.0.x 之後版本發生了重大的 API 變更（特別是 Timer 分配邏輯）。使用新版編譯會導致舵機輸出無反應或直接 Panic。
*   **AsyncTCP 異步死鎖**：`ESPAsyncWebServer` 依賴的 `AsyncTCP` 在處理頻繁的 WebSocket (20Hz+ 遙測數據) 時，若未針對新版 LwIP 進行緩衝區優化，會導致 WiFi 堆疊崩潰。

## 2. 軟體架構瓶頸
*   **NaN 傳遞鏈**：由於 IK 計算中缺乏對 `acos` 輸入值的 `clamp` 處理，一旦運動指令稍大導致超出足端極限，生成的 `NaN` 角度會一路傳遞到 PWM 控制層，導致舵機失控「打斷」齒輪。

## 3. 硬體局限性
*   **18650 壓降問題**：2 節 18650 電池在控制 12 顆 MG90D 舵機時，若同時動作（如跳躍），瞬間電流可達 5A+，會導致 ESP32 的 3.3V 穩壓器輸入端低於臨界點而重啟。代碼中雖然有 `INA219` 電壓監測，但由於採樣率在 Core 0 較慢，無法即時觸發保護機制。

## 結論
該專案是 2020 年代的優秀嘗試，但在**數值穩定性**與**硬體電源隔離**上存在設計缺陷。對於後續學習者，應保留其「分段校準」與「多核心分工」的思路，但需更換為更穩定的 PWM 庫並引入矩陣式 IK 運算。
