# 深度模組分析：通訊與硬體控制 (Communication & HAL)

## 1. 模組定位
這部分涵蓋了如何將網路/串口指令解包，並最終轉換為驅動硬體的底層電氣訊號。涉及檔案包含 `packagesProcess.ino`, `HAL_ESP32PWM.ino`, `failsafe.ino`。

## 2. 二進位通訊協定 (`packagesProcess.ino`)
為了在 Web 介面與 ESP32 之間實現低延遲（< 20ms）的高頻控制，專案放棄了 JSON，採用自定義的二進位協定。

*   **上行解析 (`pMove`)**：
    *   接收 14 Bytes 的數據包。
    *   利用位移運算 `(byte1 << 8) | byte2` 重組 16-bit 整數。
    *   經過數學縮放（除以 10000 減 1）還原為 $[-1.0, 1.0]$ 的浮點數速度向量。
*   **下行遙測 (`pTelemetry`)**：
    *   組合電池電壓、電流與迴圈時間（`loopTime`），以位元組陣列回傳給前端。

## 3. 舵機分段線性映射 (`HAL_ESP32PWM.ino`)
這是專案最核心的硬體補償邏輯。
*   **`angleToPulse`**：
    *   將計算出的關節弧度轉換為 PWM 脈衝微秒數。
    *   不使用簡單的 `map(angle, 0, 180, 500, 2500)`，而是將 0-180 度劃分為 8 個區間（<30, <50, ... <150）。
    *   每個區間根據 EEPROM 中儲存的 `servoMainProfile` 進行獨立的局部線性插值。這種做法能完美抵消 MG90D 等廉價舵機的非線性響應曲線。

## 4. 安全機制 (`failsafe.ino`)
*   **`FS_WS_count`**：在 Core 1 的 250Hz 主循環中持續累加。
*   如果在 Core 0 的網路任務中收到 WebSocket 封包，會重置該計數器。
*   若計數器超過 `FS_WS_THR`（約 1 秒），系統判定斷線，立刻鎖定所有舵機，防止機器狗失控撞毀。
