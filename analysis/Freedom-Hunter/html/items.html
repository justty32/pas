堆疊邊界分析

add(n) 堆疊邊界行為 consumable.gd:29-36

計算公式：max_n = max_quantity - quantity，剩餘可放入容量。

add(n: int) -> int:
    var max_n := max_quantity - quantity   # 剩餘容量
    if n <= max_n:
        quantity += n
        return 0                           # 無溢出
    quantity += max_n                      # 填滿剩餘容量
    return n - max_n                       # 回傳溢出量
quantity 已滿時（max_n = 0）整批 n 個全部溢出
quantity 不超量函式保證不超過 max_quantity
無錯誤 / 警告靜默溢出，不拋例外
溢出由呼叫者處理gathering.gd 通知玩家
不自動尋找第二格inventory.add_item 只找第一個同名格
數量滿時 label 變紅set_label_color() 視覺提示
溢出處理路徑：gathering.gd 收到 overflow > 0 後觸發「無法攜帶更多」通知， 不重試部分放入。inventory.add_item 的同名格路徑回傳 overflow 給呼叫者，設計上由呼叫者自行決定處理方式。

1

物品欄（Inventory）拖放架構

Slot 內部類別 inventory.gd:161-169

1
_get_drag_data()：從物品欄移除道具，建立拖曳預覽， 設置 inventory.dragging = {'item', 'slot', 'in_flight': true}
2
_can_drop_data()：接受條件——目標格空， 或同名且 data.quantity + item.quantity ≤ max_quantity
3
_drop_data()：設 data.in_flight = false， 呼叫目標 Inventory 的 add_item()，清空 dragging

call_deferred() give_back 機制 inventory.gd:35-44

放手未命中任何格時的保護機制。 為何需要 deferred？

若直接呼叫，give_back_dragged_item() 可能在 Slot._drop_data() 之前執行， 誤判道具仍在飛行中而歸還原位，造成成功放下的道具消失。

call_deferred() 將回呼推遲到當前幀所有輸入事件處理完畢後， 確保 _drop_data 先將 in_flight = false， 再由 give_back 判斷是否歸還。

雙保護旗標：

Slot._drop_datain_flight = false + dragging = null
Inventory._input（下一幀）dragging != null && in_flight → 才歸還
跨欄拖曳時，dragging Dict 為引用型別， 兩個 Inventory 指向同一物件。in_flight 一旦被目標欄設為 false， 來源欄的 deferred 回呼也能正確判斷，不會誤觸歸還。

ItemStack 顯示元件 inventory.gd — ItemStack extends TextureRect

layout(item)：設定圖示紋理，若 max_quantity > 1 則顯示數量標籤。 數量已滿時標籤顯示紅色（set_label_color(label)），未滿顯示白色。 作為拖曳預覽節點使用。
