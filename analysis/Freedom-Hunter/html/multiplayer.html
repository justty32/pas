多人系統

ENet 架構、RPC 同步、已知安全漏洞

ENet 架構

伺服器 / 客戶端架構

  • 無 relay：所有封包直接在 peer 間傳遞，無中間轉發伺服器
  • 底層實作：ENetMultiplayerPeer
  • Authority 模型：玩家 authority = 各自的 peer_id；怪物 authority = 1（伺服器端）
RPC 同步機制
機制 RPC 模式 說明
HP 同步 call_remote authority 端在 hp_changed signal 觸發
耐力同步 call_remote 同上，authority 端驅動
死亡同步 any_peer + call_local 雙端執行確保狀態一致
復活同步 rpc —
玩家加入 register_player any_peer 廣播給所有 peer
重大 Bug
Transform 同步完全缺失
  • entity.gd:218 中 #rpc("transform", tf) 被注解掉
  • 接收端函式從未定義 → 解除注解後也會直接報錯
  • 影響：遠端玩家位置永遠凍結在出生點，無法移動同步
  • 次生影響：player.gd:183-189 的死亡方向推算永遠為零向量，擊退方向錯誤
安全漏洞

已知五項漏洞

  • ID 偽造：客戶端可覆蓋 players[1]（伺服器 authority 槽位）
  • Username 節點路徑注入：has_node 支援路徑語法，惡意 username 可遍歷場景樹
  • Transform 傳送攻擊：一旦 Transform 同步修復後，客戶端可傳送任意位置
  • 客戶端完全信任接收資料：伺服器端不驗證 RPC 資料合法性
  • 缺少 get_remote_sender_id() 比對：無法確認 RPC 是否由合法 peer 發出
大廳系統殘留問題
Godot 3 語法殘留：networking.gd:34 使用了已廢棄的 yield 語法（Godot 3 風格）， 在 Godot 4 環境下無法執行。
修正方向：改為 callback 串接或 await 語法。
